# Phase 57 Plan

Eliminate the remaining live-runtime product defects on the installed Discord surface by:

- stopping ordinary user turns from hanging until manual `/reset`
- making Brainstack graph/provider failure paths fail closed instead of half-open
- restoring truthful native scheduler / reminder behavior
- removing user-facing reset leakage
- proving the result from the `Brainstack-phase50` source-of-truth install path

## workstreams

### 1. live defect capture and fault-boundary audit

- reproduce the hanging Discord-turn defect on the installed runtime
- separate:
  - host run-loop / generation invalidation defects
  - provider/API hang behavior
  - Brainstack provider init failures
  - graph backend half-open publish/search residue
- keep the diagnosis truth-first:
  - do not assume Brainstack is the sole cause unless the runtime proves that

### 2. fail-closed graph/runtime containment

- ensure graph backend failure does not leave a half-open active lane
- if `graph.kuzu` is unhealthy or not open:
  - retrieval and publish paths must degrade cleanly
  - repeated active-request churn on `publish_entity_subgraph(...)` is not acceptable
- memory-provider init failure must not leave a misleading partially alive provider surface behind

### 3. stuck-run recovery without heuristic masking

- eliminate the class where an ordinary Discord turn sits live until manual `/reset`
- add robust, truthful abort/recovery only if the root-cause fix still leaves a bounded need for it
- do not solve the symptom only with surface wording while the run-management defect remains

### 4. native scheduler truth and delivery proof

- audit the real reminder path end-to-end
- success acknowledgements must only be emitted if a real native scheduled job exists
- near-term reminder proof must include actual delivery on the installed runtime
- if scheduler setup fails, the bot must say so clearly and not fall back to memory-only pretend scheduling

### 5. reset surface hygiene

- remove bare `Session reset.` and comparable lifecycle leakage from ordinary Discord chat
- keep reset behavior correct without exposing raw lifecycle/status strings to the user

### 6. source-of-truth install proof and final Discord proof

- install from `Brainstack-phase50` into `finafina`
- rebuild the runtime
- prove:
  - no stuck ordinary turn
  - no fake scheduler success
  - no reset leak
  - no half-open graph/provider residue in the active request path

## acceptance criteria

- `Brainstack-phase50` is the only edited source repo for the fix
- ordinary Discord turns no longer hang indefinitely waiting for manual `/reset`
- if a provider/memory path cannot complete cleanly, the runtime fails closed in a bounded, truthful way
- graph backend failure no longer leaves repeated active request-path `KuzuGraphBackend is not open` churn
- memory-provider initialization failure no longer leaves a misleading partial-active state
- reminder / cronjob success claims correspond to real native scheduled jobs and observed delivery
- bare `Session reset.` and comparable lifecycle text no longer appear in ordinary Discord chat
- the corrected runtime is installed into `finafina` from source of truth and verified there

## anti-goals

- no blame-driven shortcut that assumes Brainstack is the only cause before the runtime boundary is proven
- no fake reminder success implemented as memory-only storage
- no prompt-only or user-specific rescue logic
- no output masking that hides reset/status leakage while the underlying runtime path still emits it
- no new heuristics or benchmark-shaped tuning
