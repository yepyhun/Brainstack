# Phase 29.9 Implementation Contract

## phase type

Product verification and regression-hardening phase.

## required final architecture

- end-to-end obedience scenarios exist
- future phases can rerun them cheaply
- storage/retrieval tests remain, but no longer masquerade as full behavior proof

## minimum evidence required before calling the phase done

- at least one realistic ordinary-turn obedience suite exists
- failures are actionable and attributable
- the gate can be reused after later policy or memory changes
- verify output must distinguish:
  - compiled policy presence in system authority
  - ordinary-turn memory injection presence
  - final-answer obedience success or failure

## explicitly disallowed outcomes

- declaring success from packet internals alone
- replacing product gates with narrow synthetic checks

## cleanup requirement

- remove obsolete tests that assert the wrong proxy for behavior correctness

## compatibility requirement

- the new gate must coexist with the current retrieval/plumbing suites

## code hygiene requirement

- keep checkers bounded
- no duplicate harnesses with overlapping purpose
## recommended model level

`high`
