# Phase 29.9 Plan

## goal

Add end-to-end obedience and behavioral regression gates that check actual answers against the active compiled policy.

## scope

### in

- ordinary-prompt obedience scenarios
- regression harness design
- acceptance criteria for behavior policy

### out

- big retrieval redesign
- KG/document pipeline work

## core principles this phase must preserve

1. donor-first
2. modular, thin integration
3. zero heuristic sprawl
4. multimodal-ready
5. token discipline
6. single durable owner per truth class
7. second-brain / proactive-agent fit
8. no dead code / no half-wired paths

## architecture decision

### accepted target model

- measure actual output quality against the active behavior policy
- keep retrieval/plumbing tests, but stop pretending they are the full product gate

### rejected model

- calling the system "done" from packet tests alone
- replacing behavior gates with only benchmark wins

## workstreams

### 1. obedience scenario set

- create realistic prompts that should obey the active policy
- include prompts that never mention rules directly, to prove ordinary-turn obey-path behavior
- include reset-after-teaching scenarios so delayed activation cannot hide behind green packet tests
- include a real Hungarian long-form rule-pack fixture
- include no-dual-owner disagreement scenarios
- include principal-isolation scenarios

### 2. checker design

- define how adherence is judged without turning the system into regex theater
- allow deterministic checks for hard surface-form constraints without making runtime behavior depend on regex rewriting

### 3. regression harness

- make this rerunnable after future memory/policy changes
- add an explicit source-vs-shim-vs-final parity gate before trusting any obedience result
- separate three failure classes explicitly:
  - behavior policy missing from the ordinary-turn system authority
  - behavior policy present in prompt assembly but not visible on the final answer
  - harness/runtime seam failure unrelated to behavior-policy truth

### 4. gate adoption

- define the minimum acceptance threshold for future closes
- answer-level obedience is the primary gate, not packet-only success
- require at least one `hermes-final` runtime-path proof, not just source-repo answer checks

## validation ladder

1. scenario set review
2. checker calibration
3. rerunnable regression pass
4. future-phase adoption proof
5. Hungarian ordinary-turn obedience proof
6. verify-surface parity proof before any pass result is accepted

## anti-goals

- no fake green from synthetic packet-only tests
- no giant brittle checker pile

## recommended model level

`high`
