# Phase 55 Plan

## goal

Close the remaining product gap by making explicitly taught rule packs persist and recall with high fidelity, drive ordinary Discord behavior without warnings or leaks, and prove the result on the real running Discord product.

This phase is not a new memory-governor phase.
This phase is not a prompt-polish phase.
This phase is a kernel-fidelity and Discord-delivery proof phase.

## scope

### in

- first-class durable persistence for explicit multi-rule packs taught by the user
- exact ownership and lineage for explicit rule-pack truth:
  - native explicit truth remains host-owned
  - Brainstack mirrors or recalls it without becoming a parallel governor
- recall fidelity for explicit rule packs:
  - correct rule count
  - no omitted rules
  - no semantic inversions
- ordinary Discord-turn compliance from stored explicit truth after successful teaching
- Discord surface cleanliness:
  - no lifecycle/status leak
  - no warning spam
  - no “please resend the rules” drift after successful capture
- real Discord proof on the running Bestie product

### out

- new behavior-policy layers
- new ordinary-turn communication-governor logic
- regex farms, locale parsers, field enums, or rule-taxonomy schemas
- user-specific hacks for the current Hungarian rule pack
- benchmark-oriented success criteria
- harness-only “proof” as a substitute for Discord proof

## workstreams

### 1. Defect-family trace and owner map

- trace the full path of an explicitly taught rule pack:
  - Discord message intake
  - native explicit write
  - durable storage
  - Brainstack mirror or recall
  - prompt assembly
  - ordinary reply generation
  - reset and post-reset recall
- identify exactly where:
  - count is lost
  - rules are omitted
  - rules are inverted
  - ordinary replies ignore stored truth
  - warnings or lifecycle/status text reach Discord
- produce one owner map that separates:
  - host-owned explicit truth
  - kernel-owned mirror/recall
  - surface-owned delivery concerns

### 2. Explicit pack durability and lineage

- ensure an explicitly taught rule pack can persist as first-class durable truth
- ensure the write path remains generic:
  - not a narrow text-only schema
  - not a locale-specific parser
  - not a rule-slot taxonomy
- ensure the stored artifact preserves:
  - cardinality
  - item boundaries
  - semantic content
- ensure mirrored artifacts remain mirror-class and do not bounce back as new authority

### 3. Recall fidelity and ordinary-turn consumption

- ensure explicit rule-pack recall returns the full taught pack with no missing or inverted rules
- ensure the same stored truth also shapes ordinary replies without requiring:
  - warnings
  - re-teaching
  - extra user nudges
- ensure this happens through the owner-correct truth path, not hidden policy projection

### 4. Discord surface delivery cleanup

- remove remaining user-facing lifecycle/status leak on Discord
- ensure ordinary greetings and short replies do not drift from stored explicit truth
- ensure post-teach steady-state is quiet:
  - no “I counted 21 not 22” warnings once the user confirms the pack
  - no internal reset/rate-limit/model-switch text in ordinary chat

### 5. Final live proof

- prove on the running Discord surface:
  - explicit rule-pack teaching succeeds in one-message and multi-message forms
  - same-session recall is full and correct
  - reset-safe recall is full and correct
  - ordinary greetings and short replies obey the stored pack without warning
  - no tool/lifecycle/status leak appears
- rerun the same matrix on fresh mutable state a second time

## proof fixture vs design target

The current Hungarian 21/22-rule pack is the proof fixture for this phase.

It is not the schema target.
It is not a license to encode Hungarian-specific or punctuation-specific logic into the kernel.
The implementation must improve explicit pack fidelity in general.
