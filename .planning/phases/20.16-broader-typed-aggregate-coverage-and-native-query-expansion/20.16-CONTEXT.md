# Phase 20.16 Context

## Why This Phase Exists

- `20.15` closed at gate with a real bounded aggregate capability win.
- The project no longer needs to re-prove whether native graph-side aggregate rows can exist in bounded oracle proof.
- The post-deploy fixed `12`-item live answer-only slice showed that the next honest seam is not blind aggregate expansion.
- The later deployed reset-boundary eval raised an even more product-shaped question, but that measurement was confounded and must be cleaned before it is treated as final truth.
- The live residual split is now explicit:
  - some cases pass with zero backend support via transcript-only fallback
  - some cases fail despite nonzero backend structure
  - `empty_text` / zero-backend residual also appears in live answer-only, not only in oracle spillover
- The current native aggregate planner is still intentionally narrow around the proven total-distance road-trip sum class, but the stronger immediate question is why the existing bounded wins are not yet bridging into the default live mixed path.

## Current Truth Entering 20.16

- `20.11` through `20.14` closed the temporal structural path at gate.
- `20.15` proved that:
  - stronger Tier-2 typed extraction can write aggregate-capable graph state
  - one bounded native Kuzu sum path can surface a real `native_aggregate` row over that graph state
- The later post-deploy fixed `12`-item live answer-only slice is now the stronger deciding signal:
  - `6 / 12` pass
  - runtime sync stayed clean (`28 / 28`, `0` mismatch)
  - `4 / 6` passes happened with zero backend support
  - `4 / 6` failures happened with nonzero backend support
  - `8 / 12` cases showed at least one `empty_text` Tier-2 batch
  - `6c49646a` passed live with the correct `3,000 miles`, but without surfaced `native_aggregate`
  - `gpt4_7f6b06db` seeded nonzero backend state with `temporal_events`, but still failed answer-only live
- The stricter deployed reset-boundary eval must now be read carefully:
  - it produced `0 / 5`
  - it did surface live-relevant concerns:
    - `(empty)` seed or recall turns
    - prior-scenario content resurfacing in later scenarios
    - wrong memory preference after reset
  - but it was also confounded:
    - scenario `N+1` seed entered the prior scenario recall session before the next reset boundary
    - the harness ran on `Platform.LOCAL`, and traces showed local-tool behavior not representative of deployed Discord/Bestie use
  - review of the reset-finalize path showed this is not best explained by `flush_min_turns`; the reset route calls Brainstack session shutdown directly
- The strongest honest interpretation is therefore:
  - oracle bounded capability wins are real
  - live bridge remains incomplete
  - admission residual is live-relevant but not the whole story
  - the reset-boundary failure class is not yet cleanly measured
- A short-session oracle admission edge case still exists, but it is now secondary to the broader live residual split.

## What This Phase Is

- a live residual triage phase
- a faithful live-eval correction phase before treating reset-boundary failures as product truth
- an oracle-to-live bridge phase
- a phase to separate:
  - zero-backend / `empty_text` live failures
  - nonzero-backend but wrong-answer live failures
- a decision phase for whether broader aggregate expansion is really next
- a memory-kernel capability phase, not a benchmark-maxing detour

## What This Phase Is Not

- not another temporal micro-tuning phase
- not an oracle-only admission-rescue phase
- not a reranker / scorer bakeoff phase
- not a lexical counting hack phase
- not a full graph schema redesign
- not an excuse to relitigate the bounded `20.15` oracle win
- not a blind broader-aggregate execute just because `20.16` was already planned

## Entry Conditions

- runtime-sync verification remains mandatory before bounded proof runs
- any reset-boundary eval used by this phase must isolate scenarios cleanly rather than reusing the prior recall session
- any live eval used by this phase must avoid non-faithful `Platform.LOCAL` tool leakage or state that limitation explicitly
- the fixed temporal canary `gpt4_7f6b06db` must stay green
- the current temporal-event prompt rules remain protected and additive-only
- bounded semantic rerank and env-gated external scorer hooks remain preserved but are not this phase's center
- `session_boundary` remains a bounded benchmark-path proof tool only
- LongMemEval remains a calibration probe only

## Target End State

- the project can name the dominant live residual class without bluffing:
  - zero-backend / `empty_text`
  - nonzero-backend but wrong answer
  - missing oracle-to-live bridge for bounded aggregate win
  - or a bounded combination
- the project can say whether reset-boundary memory hygiene still fails after a faithful isolated live eval, rather than after a harness-confounded one
- the phase can show whether the next right follow-up is:
  - live bridge / selection / synthesis work
  - admission reliability work
  - or honest broader aggregate expansion after triage
- the bounded `20.15` aggregate win remains preserved while being explicitly classified as not yet live-bridged
- the short-session oracle admission edge case remains correctly classified as carry-forward context, not as the whole product diagnosis
