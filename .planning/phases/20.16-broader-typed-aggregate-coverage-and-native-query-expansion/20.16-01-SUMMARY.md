## Strongest Findings

### 1. The remaining failures are not simple admission misses

In the isolated eval temp Brainstack DB:

- `transcript_entries = 23`
- `continuity_events = 23`
- `profile_items = 0`

The failing facts were persisted into transcript/continuity shelves:

- dietary:
  - `lactose intolerance`
  - `eggs and mushrooms`
- coffee:
  - `oat flat white`
  - `extra hot`
  - `no vanilla syrup`

So the primary problem is no longer “nothing got written”.

### 2. Cross-session / cross-user retrieval contamination is now explicit

The localized working-memory packet audit showed that the failing recall packets surfaced transcript rows from **other isolated synthetic users/sessions** instead of the correct prior session for the same scenario.

- dietary recall:
  - `routing = fact`
  - `profile = 0`
  - `recent = 0`
  - `matched = 0`
  - transcript fallback pulled unrelated `gift_context` rows
- coffee recall:
  - transcript fallback pulled unrelated `dietary_constraint` rows

This means the current live recall path is not sufficiently principal-scoped for messaging-style usage.

### 3. `"order"` is still over-triggering the temporal route

The coffee recall question:

`Write my cafe order in one line the way I would actually say it.`

was routed as:

- `requested_mode = temporal`
- `applied_mode = temporal`
- reason: deterministic temporal cues: `order`

This is a generic route disambiguation bug, not a benchmark-only artifact.

### 4. Profile-style durable recall is underpowered in the reset/live path

The isolated eval DB ended with:

- `profile_items = 0`

and the current session-end ingest path still builds:

- `profile_candidates = []`

for ordinary user messages in `build_session_message_ingest_plan(...)`.

That does **not** mean the next phase should blindly restore broad handwritten profile inference, but it does explain why preference/routine-style recall is currently over-dependent on continuity/transcript competition.
