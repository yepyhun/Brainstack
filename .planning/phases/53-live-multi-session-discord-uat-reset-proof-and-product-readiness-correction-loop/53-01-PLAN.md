## live UAT ladder

### fresh mutable state semantics

Every fresh-state run in this phase must use the same mutable-state reset contract.

Must wipe before each full ladder run:
- Discord/session replay artifacts
- Brainstack mutable SQLite/session continuity state
- session-local transcript caches
- session-local graph/corpus indexes that derive from mutable session state
- Hermes gateway/session cache and runtime state

Must not wipe before each full ladder run:
- native explicit durable profile truth when the rung is specifically testing post-reset durability
- static runtime config
- auth state
- fixed deploy wiring

Interpretation rule:
- if the rung is testing fresh teaching from nothing, native explicit profile truth may be reset deliberately and that reset must be recorded in the ledger
- if the rung is testing reset durability, native explicit profile truth must survive while session-local mutable state is wiped

1. Fresh-start identity and preference capture
- wipe mutable runtime memory/state
- boot the running Discord product
- teach bounded explicit truths:
  - durable identity facts
  - one or two durable user preferences
  - one explicit rule pack or bounded operating instruction pack
- verify:
  - native explicit write lands
  - Brainstack mirrors
  - no shadow Brainstack-first profile path appears
  - explicit rule-pack teaching is stored as explicit truth, not mined later from residue

2. Same-session ordinary chat and explicit recall
- have a normal short conversation after the initial teaching
- verify ordinary chat remains natural and non-governed
- ask for explicit recall:
  - durable identity facts
  - durable preferences
  - the explicit rule pack as taught truth
- verify recall comes from the right authority, not transcript panic or ordinary-turn policy projection

3. Session reset and post-reset continuity
- force a real session reset
- continue in a fresh session
- verify:
  - durable facts survive as intended
  - ephemeral session-local clutter does not masquerade as durable truth
  - tool traces and internal mechanics do not leak to the user

4. Contradiction and supersession
- update previously taught truths with corrected values
- verify:
  - native explicit write supersedes the old value
  - Brainstack mirror converges instead of duplicating
  - recall returns the latest truth
  - no stale shadow residue resurfaces
  - no old style/profile residue silently recreates a discarded policy lane

5. Reminder and logistics flow
- create at least one realistic reminder / logistics fact flow
- verify:
  - reminder capture is correct
  - timezone handling is correct
  - follow-up recall and scheduling behavior are coherent
  - failure copy is clean and truthful if something cannot be completed

6. Long-turn naturalness and robustness
- run a longer mixed conversation:
  - casual chat
  - explicit recall
  - correction
  - reset-adjacent interaction
  - mid-run forced reset
  - fast contradiction burst
  - controlled dual-principal isolation check
- verify:
  - latency is acceptable
  - no tool trace leak
  - no blocked internal copy leak
  - no regression into host-level behavior governor patterns
  - no automatic communication-contract prose appears in ordinary turns

## rolling correction loop

1. run the full six-step live ladder
2. record every failure with concrete evidence
3. cluster failures by defect family instead of symptom wording
4. apply the smallest architecturally correct fix
5. rerun the full ladder, not just the failing step
6. a rung-only rerun may be used as a local sanity check, but it never counts as verification
7. if the defect comes from leftover governance, duplicate authority, or dead compatibility wiring, remove or sharply reduce that code instead of wrapping it in another heuristic
8. repeat until the whole matrix is green
9. wipe mutable state again and rerun the whole ladder from scratch
10. only declare readiness after two consecutive all-green fresh-state runs

The phase is not complete while any one scenario still relies on:
- lucky phrasing
- one-off retries
- manual operator interpretation
- stale state from a previous attempt
- prompt-tuning that only masks the defect
- dead or half-wired fallback paths that survive only because nobody deleted them
- schema or policy work that changes the product shape instead of proving the current one
- hidden acceptance criteria that still expect Brainstack to mine communication policy from generic free text

## correction discipline

- prefer deletion or sharp reduction over additive workaround when the failing code belongs to the old Brainstack-as-governor shape
- do not preserve behavior-governor remnants, duplicate explicit-profile paths, or fake fallback logic merely because they might have helped an older phase
- if a defect is outside phase-53 scope, record it explicitly as an external blocker instead of quietly stretching this phase into a new architecture cycle
- every fix must improve the whole ladder, not just the prompt that exposed it

## masterpiece hard gates

1. authority purity
- native explicit profile writes must remain native-first and Brainstack-mirrored only
- explicit rule-pack recall may exist, but it must be driven by explicit truth ownership, not ordinary-turn governance
- the one-way mirror contract must be visible in runtime evidence:
  - `native_write_id`
  - `source_generation`
  - `mirrored_from`
- owner-first routing must beat cue-first keyword hits when an authority owner is available
- required proof:
  - dedicated authority-precedence test
  - live evidence showing native write -> mirror only, not Brainstack-first profile ownership
  - dedicated native-write-no-bounce proof

2. deterministic truth-state
- the product is not ready while any rung is intermittent
- each ladder rung must be rerun at least `3x` on fresh mutable state
- deterministic means:
  - the same taught truths converge to the same durable authority-state and recall outcome
  - exact wording variance alone is not a failure
- any intermittent truth-state or recall result is a blocking defect, not residual risk

3. inspectability
- the shipped Brainstack plugin must remain explainable to an external auditor
- demoted governance surfaces must be:
  - removed
  - sharply reduced
  - or isolated with an explicitly bounded archival / recall-only role
- the final shipped path must not require an auditor to infer that “ordinary chat secretly depends on style cue mining”
- this phase must emit or update a dependency/responsibility map showing which module owns:
  - native explicit writes
  - Brainstack mirror ingress
  - transcript / task / operating / graph / corpus recall
  - explicit archival rule-pack recall

4. adversarial stress
- the live ladder must include non-happy-path stress, not just clean linear teaching
- minimum adversarial coverage:
  - contradiction burst
  - unexpected reset in the middle of an interaction
  - controlled dual-principal isolation check
- the goal is to prove anti-fragility, not only happy-path success

5. latency and leak hygiene
- every ladder rung must record a latency baseline
- measure both when possible:
  - end-to-end reply time
  - Brainstack-added overhead
- latency regressions are product defects, not cosmetic concerns
- internal leak budget is zero:
  - no tool traces
  - no blocker copy
  - no internal pipeline names in user chat
- required proof:
  - dedicated no-tool-trace-leak test
  - per-rung latency ledger
  - evidence that ordinary turns remain free of Brainstack-generated communication-policy sections
