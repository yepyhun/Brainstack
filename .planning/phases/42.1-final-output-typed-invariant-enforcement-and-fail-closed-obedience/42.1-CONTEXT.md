# Phase 42.1 Context

## problem statement

After the authority and routing cleanup, there is still a user-visible correctness gap in the final-output path:

- the typed invariant layer can know a rule
- the validator can detect a violation
- the runtime can still ship the violating answer anyway

The currently visible example is dash-like punctuation, but the bug class is broader than that single wording rule.

## why this phase exists

- Phase 41 surfaced a real kernel defect, not a cosmetic obedience complaint
- the product cannot claim strong communication-contract correctness while the final-output layer still tolerates known typed violations
- this must be fixed generically so typed final-output invariants become trustworthy kernel infrastructure rather than partial advisory hints

## findings this phase is intended to close

- Phase 41 Batch 6:
  - 17. Final-output typed invariant enforcement is incomplete

## architectural posture

- do not add a dash-specific hack
- do not widen regex or phrase policing
- do not push more prompt text as a substitute for runtime enforcement
- do not paraphrase the immutable principles; reference them directly

## success shape

- typed final-output invariants have explicit enforcement semantics
- known violations cannot silently pass through to the user
- trace and shipped output stay converged
- the resulting fix improves the kernel bug class, not one benchmark-shaped symptom
