# Phase 20.10 Plan

## Goal

Replace the current slow, fragile benchmark inner loop with a trustworthy fast-feedback ladder: runtime-sync verification, retrieval-quality harnessing, fixed canary benchmarking, and route-harness cleanup, while moving full `15`-question live reruns to phase-gate proof only.

## Scope

### In

- runtime-sync verification between source repo and Bestie target runtime
- retrieval-quality harness for Brainstack questions without full answer/judge loops
- fixed `5`-question canary subset for fast comparable loops
- route-harness cleanup and deterministic misroute audit support
- explicit route-harness audit coverage for `f523d9fe`
- full `15`-question live reruns reframed as phase-gate proof only

### Out

- new retrieval algorithm changes as the main deliverable
- new route-policy redesign as the main deliverable
- broad benchmark-score optimization
- re-opening route-hint `fix vs replace`
- broad deployment / installer rewrite outside what is needed for runtime-sync truth

## Planning Assumption

`20.9` already showed that valid live truth can change materially once runtime parity is corrected.

So the next highest-leverage move is not another expensive retrieval-fix loop. It is making the validation ladder fast and trustworthy enough that the next retrieval phase can iterate with short cycles and clean evidence.

## Execution Steps

1. Workstream A: add runtime-sync verification as step `0`.
   - Detect and report source-vs-Bestie drift before any harness or benchmark run starts.
   - Fail loudly rather than allowing a stale runtime benchmark artifact.
   - Keep the check specific to the Brainstack benchmark path, not a generic repo-wide checksum system.

2. Workstream B: build the retrieval-quality harness first.
   - For a given question, capture:
     - route source and applied mode
     - per-channel candidate counts
     - selected evidence rows
     - packed `<memory-context>`
   - Do this without waiting for the model answer or the judge loops.
   - Make this the default diagnostic entry point for most retrieval questions.

3. Workstream C: define and document the fixed `5`-question canary subset.
   - Keep the canary stable and named:
     - `c8c3f81d`
     - `5d3d2817`
     - `e9327a54`
     - `gpt4_7f6b06db`
     - `6c49646a`
   - Ensure the canary covers:
     - fact baseline
     - profile / identity miss
     - false-positive temporal class
     - true temporal class
     - aggregate class

4. Workstream D: clean up the route harness.
   - Make deterministic route-policy auditing fast and explicit.
   - Keep the original misroute family visible.
   - Add `f523d9fe` as an explicit route-harness audit case outside the fixed canary.
   - Do not let route harness work become a new routing-policy redesign phase.

5. Workstream E: redefine the benchmark proof ladder.
   - Full `15`-question answer-only and split runs should be explicitly treated as phase-gate proof, not inner-loop diagnostics.
   - The phase should leave behind a simple rule for when to use:
     - route harness
     - retrieval-quality harness
     - canary subset
     - full `15`

## Implementation Rules

- Do not let this phase drift into generic infra work.
- Do not let the retrieval-quality harness secretly depend on the answer/judge loop.
- Do not mutate the fixed canary opportunistically once named.
- Do not remove `f523d9fe` from explicit route-harness audit coverage just because it is outside the canary.
- Do not use full `15`-question live runs as the default debugging loop anymore.
- Do not let runtime-sync verification become advisory; it must be a hard guard.

## Output Shape Required From Phase 20.10

- a documented runtime-sync check with clear pass/fail behavior
- a retrieval-quality harness with example output
- a fixed documented `5`-question canary subset
- a route-harness audit list that explicitly includes `f523d9fe`
- a short benchmark ladder policy that states when to use:
  - route harness
  - retrieval-quality harness
  - canary subset
  - full `15`-question proof

## Exit Gate

- Runtime-sync verification exists and is mandatory before benchmark/harness execution.
- Retrieval-quality harness exists and can expose route, candidates, and packed memory context without answer/judge loops.
- The fixed `5`-question canary is documented and stable.
- Route-harness audit coverage includes `f523d9fe` in addition to the earlier deterministic misroute family.
- The benchmark ladder is explicit enough that future phases do not need to default back to full `15`-question inner-loop reruns.
