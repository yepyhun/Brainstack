# Phase 20.10: Benchmark Validation, Runtime Sync, And Fast-Feedback Harnesses - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

## Why This Phase Exists

`20.9` produced a more honest live picture, but it also exposed a workflow problem:

- source-side fixes were real
- stale Bestie runtime distorted live truth again
- patched-runtime answer-only truth improved to `5 / 15`
- the comparable full split rerun did not finish cleanly enough to be trusted as a closing proof

This means the next blocker is not only retrieval quality. The next blocker is that the benchmark validation loop itself is too slow and too operationally fragile to remain the default inner loop.

Today the project is paying too much time and uncertainty for each diagnostic question:

- full `15`-question live reruns take too long
- split proof is especially expensive and unstable
- stale runtime can silently invalidate a whole benchmark pass
- many retrieval questions do not need a full answer + judge loop to diagnose

So the next phase should not be another broad retrieval-fix phase. It should be a narrow benchmark-validation / harness-hardening phase.

## What This Phase Is

This phase is a **fast-feedback infrastructure phase for Brainstack benchmarking**.

It exists to establish a new testing ladder:

1. runtime-sync verification before any run
2. retrieval-quality harness without full answer/judge loops
3. fixed `5`-question canary subset for comparable fast loops
4. route-harness cleanup for deterministic policy audits
5. full `15`-question live reruns only as phase-gate proof

This phase should make the next retrieval-recovery work materially faster and more trustworthy.

## What This Phase Is Not

This phase is **not**:

- a new retrieval-algorithm recovery phase
- a new route-policy redesign phase
- a return to route-hint fix-vs-replace work
- a new proof-calibration phase
- a benchmark-score chase
- a broad infrastructure rewrite unrelated to the Brainstack benchmark loop

If a harness reveals a retrieval bug, that bug should be named and carried forward. But this phase should not absorb the fix itself unless it is strictly required to make the harness truthful.

## Entry Conditions

- `20.9` source execution is complete enough to establish new live truth.
- The patched Bestie runtime answer-only rerun is accepted as the current valid live truth:
  - `5 / 15`
- Runtime staleness is now a documented recurring distortion source.
- The deterministic routing path is live.
- The current full split loop is explicitly treated as too slow / unstable for default inner-loop use.
- The carry-forward benchmark-validation requirements are already agreed:
  - runtime-sync check as step `0`
  - retrieval-quality harness as first new deliverable
  - fixed `5`-question canary
  - route-harness cleanup
  - explicit extra route-harness audit for `f523d9fe`

## Target End State

- Every benchmark or harness run has an explicit runtime-sync guard before execution.
- A retrieval-quality harness exists that can show:
  - chosen candidates
  - route/mode info
  - packed memory context
  without requiring a full answer + judge loop.
- A fixed `5`-question canary subset exists and is documented for comparable quick loops:
  - `c8c3f81d`
  - `5d3d2817`
  - `e9327a54`
  - `gpt4_7f6b06db`
  - `6c49646a`
- The route harness is clean enough to audit deterministic misroutes quickly, including:
  - the original `20.8` misroute family
  - `f523d9fe` as an explicit extra live false-positive temporal case
- Full `15`-question live answer-only / split reruns are demoted from inner-loop usage to phase-gate proof usage only.
