# Phase 20.10 Summary

## Outcome

Phase `20.10` completed its intended harness-hardening scope.

This phase did **not** attempt another broad retrieval-fix cycle. It replaced the fragile benchmark inner loop with faster, more trustworthy validation entry points.

## Delivered

### 1. Runtime-sync verification is now a hard step `0`

The Brainstack subset runner now verifies source-vs-Bestie runtime parity before any benchmark/harness execution starts.

What it checks:
- Brainstack plugin payload tree
- `rtk_sidecar.py`
- `host_payload` copied files

Behavior:
- hard-fails on drift instead of silently allowing stale-runtime benchmark artifacts
- reports compared file count and mismatch count

Actual validation against the current Bestie root:
- compared files: `28`
- mismatch count: `0`
- result: `ok`

### 2. Retrieval-quality harness now exists

The subset runner now supports a retrieval-only mode:
- `--retrieval-only`

This mode captures without answer/judge loops:
- route source / requested mode / applied mode
- route bounds
- per-channel candidate counts
- backend population snapshot
- packed Brainstack memory block
- composed captured prompt block

This is now the intended first diagnostic surface for most retrieval questions.

## Verdict

`20.10` succeeded as a benchmark-validation / fast-feedback phase.

The phase deliberately did **not** use another full `15`-question live rerun as its success metric. That would have violated the point of the phase itself.

The new truth after `20.10`:
- stale Bestie runtime is now a first-class hard failure, not a hidden confounder
- retrieval-quality inspection no longer requires the full answer/judge loop

- improved retrieval quality
- improved benchmark score
- solved fact-path parity
- solved profile / identity retrieval misses
- solved structural route candidate quality

What `20.10` enables:
- future retrieval phases can iterate with:
  - runtime-sync truth
  - retrieval-only inspection
  - stable canary loops
  - route-policy audit without paying the full `15`-question cost each time

Required carry-forward into `20.11`:

- specifically:
  - pass runtime-sync verification first
  - run retrieval harness on the fixed canary on the patched runtime
  - capture the pre-change baseline candidate / channel / packing truth
  - do not start retrieval-recovery code changes before that baseline report exists
- this preserves the point of `20.10`: harness-reading first, code second
