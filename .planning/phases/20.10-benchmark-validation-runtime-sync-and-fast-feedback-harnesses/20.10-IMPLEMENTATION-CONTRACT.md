# Phase 20.10 Implementation Contract

## Purpose

Harden the Brainstack benchmark validation workflow so future retrieval-recovery phases can iterate with fast, trustworthy evidence instead of repeatedly relying on slow, drift-prone full `15`-question live reruns.

## Required Workstreams

### Workstream A. Runtime-Sync Verification

- This workstream is step `0` and must stay step `0`.
- It must verify source-vs-Bestie runtime parity before any harness or benchmark run starts.
- It must fail loudly on drift.
- It must be scoped to the actual Brainstack benchmark/runtime files, not an abstract whole-repo sync theory.

### Workstream B. Retrieval-Quality Harness

- This workstream is the first real new deliverable.
- It must expose, for a chosen question:
  - route source
  - requested/applied mode
  - channel candidate counts
  - selected evidence
  - final packed `<memory-context>`
- It must not require a model answer or judge loop.
- It must be useful as the primary diagnostic entry point for retrieval-quality questions.

### Workstream C. Fixed Canary Subset

- This workstream must define and preserve a fixed `5`-question canary subset:
  - `c8c3f81d`
  - `5d3d2817`
  - `e9327a54`
  - `gpt4_7f6b06db`
  - `6c49646a`
- The canary must remain stable for comparability.
- It must not be silently edited to chase score movement.

### Workstream D. Route-Harness Cleanup

- This workstream must make deterministic route-policy audits cheap and explicit.
- It must preserve earlier misroute visibility.
- It must explicitly include `f523d9fe` as a required live false-positive temporal audit case even though it is outside the fixed canary.
- It must not turn into a new broad routing-policy redesign phase.

### Workstream E. Benchmark Ladder Policy

- This workstream must define the new benchmark usage ladder:
  - route harness
  - retrieval-quality harness
  - fixed canary
  - full `15`-question live proof
- Full `15`-question answer-only and split runs must be demoted to phase-gate proof usage only.
- The output must make future default misuse harder.

## Protected Boundaries

### Scope Boundary

- This phase must not become the main retrieval-fix phase.
- This phase must not absorb unrelated deployment cleanup outside benchmark-runtime truth.

### Proof Boundary

- The retrieval-quality harness must not secretly depend on answer/judge execution.
- Runtime-sync verification must not be advisory-only.

### Canary Boundary

- The fixed canary must remain stable once written.
- Extra audit cases such as `f523d9fe` belong in the route harness, not as silent canary churn.

### Cost Boundary

- The point of this phase is to remove full `15`-question runs from the inner loop.
- Any implementation that still requires a full `15`-question rerun for routine diagnosis fails the purpose of the phase.

## Minimum Proof Required Before Calling Phase 20.10 Done

- A runtime-sync check exists and blocks stale-runtime runs.
- A retrieval-quality harness exists and shows route + candidate + packed-context truth without answer/judge loops.
- The fixed `5`-question canary is documented and runnable.
- Route-harness audit coverage explicitly includes `f523d9fe`.
- The benchmark ladder is written clearly enough that a future phase can choose the right validation layer without defaulting to another half-hour full rerun.
