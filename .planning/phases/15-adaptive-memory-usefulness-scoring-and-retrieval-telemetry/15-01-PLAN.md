# Phase 15 Plan

## Goal

Add bounded adaptive retrieval telemetry and usefulness-aware ranking to Brainstack without creating a new subsystem or weakening existing recall correctness.

## Scope

### In

- record telemetry for rows actually surfaced into working memory
- store telemetry inside existing row metadata
- add bounded ranking adjustments for repeated fallback / repeated matched usage
- keep identity, communication-style, and shared-work essentials safe from accidental demotion
- cover profile and graph recall paths

### Out

- no destructive deletion
- no large schema rewrite
- no answer-content parser that guesses whether a row was used
- no new runtime worker or queue

## Execution Steps

1. Add a small Brainstack-owned usefulness helper module.
   - Normalize telemetry reads/writes from `metadata_json`
   - Compute bounded adjustment values for profile and graph rows

2. Extend store helpers in `db.py`.
   - Profile rows: bump served / match / fallback telemetry by `stable_key`
   - Graph rows: return stable row ids from search and bump served telemetry by row type + id

3. Wire telemetry into the control plane.
   - After building the working-memory packet, record which rows were surfaced
   - Distinguish matched profile rows from fallback/same-session boosted rows

4. Add bounded ranking nudges.
   - Prefer repeatedly matched rows modestly
   - Gently demote repeatedly fallback-only rows
   - Never let this override core communication / identity / shared-work essentials

5. Add tests.
   - telemetry increments on surfaced rows
   - fallback-only rows can be demoted without deletion
   - core communication / identity facts remain visible
   - graph rows get retrieval telemetry

6. Reinstall into live Bestie and verify.
   - doctor pass
   - rebuild/start pass
   - inspect live runtime behavior and stored telemetry

## Exit Gate

- working-memory retrieval records bounded telemetry for surfaced profile and graph rows
- repeated fallback-only rows can be ranked lower without deleting them
- repeatedly matched rows can gain a modest priority edge
- communication contract / identity facts still remain stable and visible
- live runtime carries the Phase 15 logic through the Brainstack installer

## Recommended Effort

- `high`
