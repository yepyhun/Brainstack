# Phase 20.15 Plan

## Goal

Turn the medium-term aggregate-recovery direction into a bounded capability gain by strengthening Tier-2 typed entity extraction and adding one native Kuzu aggregate query path for a real count / sum-style question class.

## Scope

### In

- baseline reading of currently persisted graph structure for one or more named aggregate-style cases
- typed entity / state / relation extraction improvements that remain generic and donor-first
- one bounded native Kuzu aggregate query path over the improved structured data
- patched-runtime bounded proof on a representative aggregate question class
- canary regression protection after the change

### Out

- new temporal micro-tuning loops
- scorer / reranker reopening
- Jina v5 evaluation work
- benchmark-specific counting heuristics
- full graph schema redesign
- broad answer-only full-suite loops as inner-loop tools

## Execution Steps

1. Workstream A: baseline the current aggregate structure truth.
   - Start from one or more named aggregate-style cases already present in project artifacts.
   - Persist the pre-change truth for:
     - current typed / untyped entity population
     - state / relation coverage
     - whether the graph contains enough explicit structure to support native aggregate queries
   - Explicitly separate:
     - missing typed structure
     - missing query path
     - or both
   - Explicitly classify the named aggregate case type before query work begins:
     - count
     - sum
     - bounded grouped comparison
     - or another clearly named aggregate class
   - Do not assume that every canary aggregate case is a count problem.

2. Workstream B: improve Tier-2 typed entity extraction in a bounded, generic way.
   - Strengthen extraction for aggregate-relevant structure such as:
     - repeated events
     - transactions / purchases
     - trips
     - routines / breaks
   - Prefer typed output that is reusable by graph-native queries instead of prompt-only answer synthesis.
   - Keep the work generic:
     - no benchmark-specific event names
     - no hand-written one-off counters
   - The phase must name what typed structure was added or made materially more reliable.
   - Tier-2 prompt changes must be additive to the existing temporal-event extraction rules, not replacements.
   - The current temporal-event hard caps, preference for concrete real-world events, and prior-event capture rules must be preserved verbatim unless the phase produces explicit regression evidence that one of them is itself the blocker.
   - The fixed temporal canary case `gpt4_7f6b06db` must remain green after any Tier-2 prompt change.

3. Workstream C: add one bounded native Kuzu aggregate query path.
   - Implement one native graph-side query path for a concrete aggregate class such as:
     - count
     - sum
     - bounded grouped comparison
   - Keep it narrow and explicit.
   - The query path must consume structured graph data, not prompt-side string scanning.
   - If the phase discovers that typed structure is still insufficient, it must stop honestly at that boundary rather than faking an aggregate answer path.
   - If Workstream B reveals that the required schema / extraction changes are materially larger than expected, the phase may close after Workstream B with an explicit verdict that the native Kuzu aggregate query path is deferred to `20.16`.

4. Workstream D: bounded proof and regression check.
   - Re-run one targeted bounded proof on the named aggregate class.
   - Re-run the fixed retrieval-only canary afterward.
   - Do not escalate to broader answer-only proof unless the bounded artifacts show a real capability gain instead of noise.

## Implementation Rules

- Do not reopen temporal-chain work unless a new generic temporal regression appears.
- Do not solve aggregate questions by lexical cue hacks or row stuffing.
- Prefer graph-native structure and bounded graph-native queries over prompt-only counting.
- Preserve the current scorer / reranker layer as background capability, not the center of this phase.
- Keep the intervention small enough that the dominant gain can still be named.
- If typed extraction improves but native querying does not land, say so explicitly.
- Preserve the existing temporal-event extraction behavior unless explicit evidence shows a direct conflict with the new typed extraction work.

## Output Shape Required From Phase 20.15

- one explicit statement of the dominant current aggregate blocker:
  - typed-structure absence
  - query-path absence
  - or both
- before/after bounded artifacts on at least one aggregate-style case
- one canary retrieval-only regression artifact
- explicit statement whether the win came from:
  - stronger typed extraction
  - native aggregate query path
  - or both

## Exit Gate

- one bounded aggregate-capable structure improvement exists in code, or the phase states clearly why it stopped at instrumentation
- one bounded native Kuzu aggregate query path exists in code, or the phase explicitly records why typed structure is still insufficient
- targeted proof is comparable before vs after
- canary retrieval-only regression check exists
- the phase can state whether aggregate capability improved without bluffing or benchmark-specific shortcuts

- `xhigh`, because this is the next real capability seam after the temporal handoff is closed
