# Phase 20.15 Summary

## Verdict

Phase `20.15` is execution-complete at gate.

The phase did produce a real bounded aggregate capability, but the winning path was more specific than the initial first-pass live proof suggested:

- the first `20.15` live aggregate proof stayed correct on `6c49646a`, but no `native_aggregate` row surfaced
- the immediate blocker was **not** absence of all typed structure
- it was a mismatch between the **real typed graph shape** produced by Tier-2 and the **overly narrow native aggregate planner/query shape**

The closing fix therefore came from **both** sides:

- stronger typed extraction so aggregate-capable graph states are written at all
- then a bounded native Kuzu sum path that can consume the real typed entity shapes instead of only one idealized schema variant

## What 20.15 Implemented

- added a new additive Tier-2 output channel in `brainstack/tier2_extractor.py`
  - `typed_entities`
  - normalized against real turn numbers
  - compact scalar attributes such as `distance_miles`, `amount_usd`, `duration_days`, `count`
- preserved the current temporal-event rules verbatim while extending the prompt
  - temporal caps remain intact
  - concrete-event preference remains intact
  - prior-event capture rules remain intact
- reconciled `typed_entities` into the existing graph model in `brainstack/reconciler.py`
  - writes `entity_type`
  - writes `owner_subject`
  - writes normalized scalar attributes as graph states
- extended Tier-2 telemetry and benchmark reporting
  - `typed_entities` extracted counts
  - `typed_entity_samples`
- added a bounded native aggregate query path over Kuzu in:
  - `brainstack/graph_backend.py`
  - `brainstack/graph_backend_kuzu.py`
  - `brainstack/db.py`
  - `brainstack/executive_retrieval.py`
- initial native planner supported only an exact shape:
  - `entity_type = road_trip`
  - `owner_subject = User`
  - `metric_attribute = distance_miles`
- after localization on the real benchmark seed graph, broadened the planner/query to match the real bounded shape:
  - accept `mileage_history`
  - accept `*_road_trip`
  - exclude `planned_*`
  - remove the brittle exact `owner_subject = User` dependency
- added proof coverage in `tests/test_brainstack_phase20_proof.py`
  - mixed real-world aggregate shapes now sum correctly
  - `planned_road_trip` does not leak into the result

## Localization Truth

The decisive localization step was **not** another full benchmark rerun.

The key evidence came from inspecting the real `session_boundary` seed state used for the named aggregate case:

- `Yellowstone trip`
  - `entity_type = family_road_trip`
  - `distance_miles = 1200`
- `1,800-mile road trip total`
  - `entity_type = mileage_history`
  - `distance_miles = 1800`
  - `trip_count = 3`
- unrelated trip-like rows also existed:
  - `planned_road_trip`
  - other distance-bearing entities that should **not** be summed for this question

This proved that the remaining blocker was no longer “typed structure does not exist”.

It was:

- the planner/query was too narrow for the real aggregate-capable graph states
- exact `road_trip + owner_subject=User` was the wrong live assumption
