# Phase 20.15 Implementation Contract

## Objective

Follow the `20.14` closeout by turning the next recorded forward path into a bounded capability step: stronger Tier-2 typed entity extraction plus one bounded native Kuzu aggregate query path.

## Workstream A: Aggregate Structure Baseline

- Start from one or more named aggregate-style cases already referenced in current project artifacts.
- Persist:
  - current graph entity / state / relation population
  - current typed-vs-generic structure quality
  - whether the desired aggregate class is already answerable natively in Kuzu
- Explicitly classify the named aggregate case before query-path work begins:
  - count
  - sum
  - bounded grouped comparison
  - or another clearly named aggregate class
- No graph-native aggregate implementation work may begin before this baseline exists.

## Workstream B: Typed Entity Extraction Improvement

- Land one bounded, generic Tier-2 extraction improvement that materially helps aggregate-capable structure.
- Preferred classes include:
  - repeated events
  - purchases / transactions
  - trips
  - routines / breaks
- The phase must explicitly name what structure became more reliable.
- Tier-2 prompt changes must be additive to the existing temporal-event extraction rules, not replacements.
- Preserve the current temporal-event hard caps, concrete-real-world-event preference, and prior-event capture rules verbatim unless explicit evidence shows one of them is itself the blocker.
- The fixed temporal canary case `gpt4_7f6b06db` must remain green after any Tier-2 prompt change.
- This workstream must not silently drift into:
  - benchmark event hardcoding
  - lexical counting heuristics
  - answer-template hacks

## Workstream C: Native Kuzu Aggregate Query Path

- Implement one bounded graph-native aggregate query path.
- Allowed query classes include:
  - count
  - sum
  - bounded grouped comparison
- The path must use graph-native structure, not prompt-side row counting.
- If typed structure is still insufficient, the phase must stop honestly and record that as the blocker.
- If Workstream B reveals that the required schema / extraction change is materially larger than expected, the phase may close after Workstream B with an explicit `20.16` deferral for the native Kuzu aggregate query path rather than forcing both capabilities into one phase.

## Workstream D: Bounded Proof

- Re-run one targeted bounded proof on the named aggregate class.
- Re-run the fixed retrieval-only canary afterward.
- Do not escalate to broader answer-only proof unless the bounded artifacts show a real capability gain instead of noise.

## Protected Boundaries

### Anti-Benchmaxing Boundary

- No benchmark-specific event-name rules.
- No one-off counting regexes.
- No “just pack more rows and let the model count” pseudo-fix.
- No reopening of already rejected temporal or lexical micro-tuning loops.

### Architectural Boundary

- Do not reopen whether a temporal semantic reranker exists; `20.12` already answered that.
- Do not reopen whether upstream temporal-event generation exists; `20.13` already answered that under bounded proof.
- Do not reopen whether temporal chain coverage can be preserved; `20.14` already answered that under bounded proof.
- Keep `session_boundary` as a bounded benchmark-path proof tool only.

### Scope Boundary

- The phase is about aggregate capability through stronger structure plus bounded native graph querying.
- The phase must explain whether the dominant win came from:
  - typed extraction
  - native aggregate querying
  - or both
- If the blocker remains typed-structure absence, the phase must say so explicitly rather than folding it into vague “graph quality” language.
- If the phase stops after Workstream B, that must be recorded as an honest scope split, not as a hidden partial success.

## Minimum Evidence Required Before Calling Phase 20.15 Done

- one baseline aggregate-structure artifact
- one post-change bounded aggregate proof artifact
- one canary retrieval-only regression artifact
- explicit statement of the dominant aggregate blocker
- explicit statement whether graph-native aggregate capability materially improved
