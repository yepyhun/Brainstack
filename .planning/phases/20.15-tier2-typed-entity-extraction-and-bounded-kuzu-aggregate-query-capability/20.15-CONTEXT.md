# Phase 20.15 Context

## Why This Phase Exists

- `20.14` closed the bounded temporal trip-chain handoff at gate.
- The next open capability gap is no longer temporal selection compression.
- The strongest remaining non-benchmaxing forward path already recorded in the project is:
  - stronger Tier-2 typed entity extraction
  - then a bounded native Kuzu aggregate query path for count / sum-style questions
- Current aggregate handling still leans too heavily on recalled top-K evidence and answer synthesis.
- That is structurally weak for questions that are really:
  - counting
  - summation
  - set membership over repeated event classes
  - comparison over typed transactions / trips / breaks / purchases

## Current Truth Entering 20.15

- `20.12` closed scorer capability at gate.
- `20.13` closed upstream temporal-event availability / generation reliability at gate.
- `20.14` closed temporal chain coverage / selection compression at gate.
- The cleaner next seam is not more temporal micro-tuning.
- The cleaner next seam is making aggregate-capable structure exist explicitly enough that Kuzu can answer bounded aggregate classes natively.

## What This Phase Is

- a donor-first capability phase
- an aggregate-recovery phase built on stronger structured extraction
- a graph-native query phase for bounded count / sum-style questions
- a memory-kernel capability phase, not a benchmark-maxing detour

## What This Phase Is Not

- not another temporal rescue phase
- not a reranker / embedder phase
- not a lexical aggregate heuristic phase
- not a “just stuff more rows into context” phase
- not a full graph schema redesign
- not a vendor / scorer bakeoff phase

## Entry Conditions

- runtime-sync verification remains mandatory before bounded proof runs
- bounded semantic rerank and env-gated external scorer hooks remain preserved but are not this phase’s center
- Kuzu remains the active embedded graph backend target
- LongMemEval remains a calibration probe only
- anti-benchmaxing and donor-first boundaries stay active

## Target End State

- the project can name one bounded typed-entity extraction improvement that materially helps aggregate-capable structure
- one bounded native Kuzu aggregate query path exists for a concrete aggregate question class
- the phase can say with evidence whether the main gain came from:
  - stronger typed extraction
  - native aggregate querying
  - or the bounded combination
- the carry-forward is a real memory-kernel capability step, not another retrieval-only patch cycle
