# Phase 59 Plan

## goal

Close the next quality/efficiency gap by doing three things in the right order:

1. prove whether rapid context-window fill is actually being driven by Brainstack or by the wider Hermes prompt stack
2. add a real cross-shelf budget allocator so Brainstack selects the smallest useful evidence set across shelves instead of relying on fragmented per-shelf limits
3. strengthen hybrid retrieval fusion so Brainstack ranks keyword and semantic evidence more coherently before bounded packing

This phase is not a new memory capability phase. It is an attribution-first quality and token-discipline hardening phase.

### accepted point-0 diagnosis

- fast context-window fill cannot honestly be blamed on Brainstack alone
- Brainstack is still a material contributor because:
  - it injects a per-turn prefetch packet
  - it contributes a provider system-prompt block
  - its current allocation logic is fragmented rather than globally budgeted
- but the wider Hermes host prompt stack and tool schema overhead are also real contributors

This phase therefore begins with attribution, not blame.

## pre-execution improvement targets

1. missing cross-shelf allocator
   - current budgets are route-aware but mostly per-shelf
   - there is no strong global selector deciding what survives when shelves compete for the same turn budget

2. hybrid retrieval fusion is present but not yet strong enough
   - keyword and semantic signals exist
   - fusion is still not coherent enough across shelves and routes

3. token discipline is not yet measured as one whole-stack story
   - the current installed runtime can fill context quickly
   - we need proof separating:
     - host guidance
     - user profile
     - Brainstack system block
     - Brainstack prefetch block
     - tools
     - messages

## workstreams

### 1. whole-stack attribution and measurement

- measure the effective token/character contribution of each prompt layer on the installed runtime
- record at least:
  - host static system prompt
  - builtin memory block
  - builtin user-profile block
  - Brainstack system-prompt block
  - Brainstack prefetch block
  - tool schema overhead
  - message history
- prove whether the user complaint "`Hermes fills up the context window quickly`" is:
  - mostly host-driven
  - mostly Brainstack-driven
  - or mixed

### 2. cross-shelf allocator design

- define one global working-memory allocator over the eligible Brainstack shelves
- preserve route-awareness
- force shelves to compete inside one explicit turn budget instead of only inside fragmented per-shelf caps
- prefer smaller, higher-yield evidence over bigger but weaker evidence packets

### 3. hybrid retrieval fusion hardening

- audit the current keyword/semantic fusion path
- strengthen ranking without rewriting the architecture
- keep lexical and semantic retrieval complementary
- ensure the stronger fusion improves the candidate set before packing, not by adding more raw material

### 4. install-and-proof validation

- reproduce the new allocation/fusion behavior on `finafina`
- measure:
  - effective prompt size
  - evidence composition
  - retrieval quality behavior
- verify no regressions in:
  - canonical profile recall
  - continuity
  - graph truth surfacing
  - corpus recall

### 5. planning and proof normalization

- update the project state so the context-window story is explicit:
  - not "Brainstack bad"
  - not "host bad"
  - but one attributed stack with measured contributors

## acceptance criteria

- the phase begins with a measured attribution of context growth
- the project can state honestly whether Brainstack is the dominant cause, a partial cause, or not the dominant cause
- Brainstack ends the phase with a real cross-shelf allocator design and implementation, not just retuned per-shelf constants
- hybrid retrieval fusion is measurably more coherent without turning into query-specific heuristic sprawl
- token discipline improves without collapsing recall quality
- the new behavior is reproduced from source-of-truth onto the installed runtime
