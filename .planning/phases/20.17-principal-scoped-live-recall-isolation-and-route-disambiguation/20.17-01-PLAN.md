# Phase 20.17 Plan

## Goal

Eliminate cross-principal live recall contamination and generic noun-`order` temporal misrouting, then re-check the reset-boundary live baseline before any broader capability expansion resumes.

## Scope

### In

- principal-scoped live recall isolation for the current Bestie/Brainstack live path
- bounded route disambiguation for ambiguous noun-`order` queries
- one corrected reset-boundary live re-eval after those changes
- canary protection for:
  - fixed temporal path
  - bounded aggregate win

### Out

- broader aggregate trigger expansion
- new aggregate query classes
- temporal micro-tuning unrelated to the `order` ambiguity
- broad profile extraction redesign
- scorer/reranker reopening

## Execution Steps

1. Workstream A: principal-scope localization.
   - Trace why the current live recall packet can surface transcript rows from unrelated isolated users/sessions.
   - Prove whether the leak comes from:
     - continuity search scope
     - transcript search scope
     - working-memory packet assembly
     - or another bounded retrieval seam
   - Land the smallest donor-first fix that restores principal-scoped recall.

2. Workstream B: route disambiguation for noun-`order`.
   - Localize why queries like `Write my cafe order in one line ...` deterministically route to temporal mode.
   - Introduce a generic disambiguation that separates:
     - sequencing / ordering questions
     - noun uses like purchase order, cafe order, usual order, etc.
   - Keep the fixed temporal canary green.

3. Workstream C: corrected live re-eval.
   - Re-run the faithful isolated reset-boundary eval.
   - Compare directly against the `20.16` `3 / 5` baseline.
   - State whether the remaining failures, if any, now sit in:
     - profile-style durable recall weakness
     - reset-boundary bridge weakness
     - or another bounded class

4. Workstream D: honest next-phase decision.
   - If reset-boundary live behavior is now materially improved, decide whether the next phase should target:
     - bounded profile-style durable recall strengthening
     - broader aggregate expansion
     - or another named live residual
   - Explicitly state whether the deployed Bestie runtime is still operating with `semantic` / `graph` channels degraded, so graph/corpus-backed live enablement stays visible as a separate roadmap step instead of being silently assumed.
   - If not improved, state the exact residual honestly and stop there.

## Implementation Rules

- Do not widen retrieval globally if the bug is principal-scope leakage.
- Do not hardcode scenario words like `coffee`, `lactose`, or `gift`.
- Do not solve noun-`order` ambiguity with benchmark-specific keyword lists.
- Prefer narrow route-policy clarification over giant classifier rewrites.
- Preserve donor-first seams and upstream updateability.

## Output Shape Required From Phase 20.17

- one explicit statement of the source of cross-principal recall leakage
- one explicit statement of the noun-`order` route fix
- one corrected isolated live reset-eval artifact
