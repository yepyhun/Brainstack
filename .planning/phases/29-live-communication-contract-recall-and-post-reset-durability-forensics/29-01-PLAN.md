# phase 29 plan

## goal

Reproduce and explain the live communication-contract failure after reset without assuming whether the bug is in storage, retrieval, contract assembly, application, or deployed runtime parity.

## scope

### in

- live Bestie reproduction after reset
- Brainstack durable storage for identity vs style rules
- retrieval packet composition
- active communication-contract assembly
- runtime application / override behavior
- stale deployed-runtime / config / reset-path falsification

### out

- donor refresh work
- prompt-only hardening as a first move
- persona/skill/note file substitutes
- broad rewrite of unrelated memory shelves

## validation method

- one accepted live failing trace first
- one stored-state audit for the same principal
- one retrieval / contract-assembly trace for the same failing path
- one application / override check on the deployed runtime
- one final seam verdict with explicit exclusions

## decision rules

- if live behavior and stored state disagree, live behavior wins for the next decision
- if identity survives but style rules do not, treat that as a split personal-memory-axis failure until falsified
- if a seam is not falsified, do not jump over it to a later seam
- if execute reaches a real architecture fork, bring it back in simple language before locking the direction
- if one seam explains the failure completely, do not "improve" sibling modules in the same pass

## hypothesis -> falsification -> repair decision

1. ingest seam
   - hypothesis:
     - Humanizer / communication-contract facts never became durable Brainstack truth correctly
   - falsification:
     - prove the same principal has the relevant durable style rows in stored Brainstack state
   - repair decision:
     - if not falsified, repair write-path / promotion logic at the shared seam
     - if falsified, do not patch ingest

2. retrieval seam
   - hypothesis:
     - the rules are stored durably but are not selected after reset
   - falsification:
     - prove the same post-reset retrieval packet already contains the missing style truth
   - repair decision:
     - if not falsified, repair retrieval/ranking/selection at the bounded seam
     - if falsified, move forward to contract assembly

3. contract-assembly seam
   - hypothesis:
     - the right rows are retrieved but the active communication contract is built from the wrong subset
   - falsification:
     - prove the final contract already includes the missing rules before generation
   - repair decision:
     - if not falsified, repair assembly/composition, not storage
     - if falsified, move forward to application / override

4. application seam
   - hypothesis:
     - a valid communication contract exists but host defaults override it at generation time
   - falsification:
     - prove the final prompt/input path does not contain the correct contract to begin with
   - repair decision:
     - if not falsified, repair precedence / application, not retrieval
     - if falsified, return to the earliest still-open seam

5. stale runtime-path seam
   - hypothesis:
     - the source-of-truth may be correct while the deployed runtime/reset path bypasses it
   - falsification:
     - prove runtime parity and the exact deployed reset boundary against the current source
   - repair decision:
     - if not falsified, fix deployment/runtime parity before touching Brainstack logic
     - if falsified, keep the repair inside the proven Brainstack seam

## stop conditions before any patch

- stop if there is no single accepted failing trace
- stop if the deployed runtime path is still ambiguous
- stop if the team is about to patch a later seam before falsifying an earlier one
- stop if the proposed fix creates a second owner for personal-memory truth

## workstreams

### 1. live reproduction and evidence capture

- reproduce the failing path on the deployed Bestie runtime
- capture one clean trace where:
  - identity survives
  - Humanizer / communication rules do not
  - the runtime chooses a wrong recovery behavior
- record the exact reset boundary used in the failing path

exit gate:

- one accepted failing trace exists
- the trace is clean enough to follow end-to-end

### 2. durable truth audit

- inspect where communication-contract facts are supposed to be written
- verify whether the Humanizer rules entered durable Brainstack state
- compare durable identity storage vs durable style-rule storage for the same principal

exit gate:

- we can say whether the rules exist durably, partially exist, or never landed

### 3. recall and contract-assembly audit

- inspect retrieval outputs for the same principal after reset
- inspect how the active communication contract is assembled from recalled truth
- verify whether identity and style rules diverge before or during assembly

exit gate:

- the project can point to retrieval loss vs contract-assembly loss explicitly

### 4. runtime application and fallback-detour audit

- inspect the final runtime prompt / guidance composition
- verify whether a valid communication contract is being overridden by default host behavior
- inspect why wrong recovery behaviors are still available on this axis:
  - skill detour
  - local search detour
  - net lookup detour

exit gate:

- the project can point to an application/override failure or rule it out

### 5. seam verdict and repair decision

- produce a single seam verdict:
  - ingest
  - retrieval
  - assembly
  - application
  - or stale runtime path
- define the smallest correct repair surface
- define explicit regressions that must pass after the fix

exit gate:

- one named seam
- one bounded repair direction
- no prompt-only bandaid approved without deeper falsification

## final execute findings

- confirmed:
  - the live failure was not explained by missing durable data alone
  - the communication-contract loss was already visible before generation, at the scoped retrieval / contract boundary
  - the reset path also had a second correctness failure because Tier-2/session-end promotion crashed on `response_format`
- repair shipped locally and installed into the deployed docker target:
  - bounded fix A:
    - deterministic open-time backfill for legacy unscoped principal-scoped `identity` / `preference` rows when transcript evidence resolves to one unique principal scope
  - bounded fix B:
    - Tier-2 caller fix so the LLM JSON-object request flows through `extra_body` instead of an unsupported direct keyword
- post-repair proof:
  - live DB backfill produced:
    - `15` scoped profile keys
    - `12` scoped preference keys
    - `3` scoped identity keys
  - a fresh deployed-path `AIAgent._build_system_prompt()` on the real docker runtime now contains:
    - `Refer to yourself as Bestie when naming yourself.`
    - `Put each new thought on its own line.`
    - `Capitalize Én, Te, and Ő when you use them.`
    - `Always respond in Hungarian.`
    - the broader Humanizer lines
  - isolated runtime provider proof now flushes a scoped preference row successfully through `on_session_end()`
- ruled out:
  - a pure prompt-strength explanation
  - a pure application-only explanation for the already-missing contract lines
  - a theory where the communication data never existed durably
- explicitly deferred:
  - principal-model drift between older `default/hermes/numeric-id` scope and the newer naming shape
  - this remains an adjacent boundary issue, not the minimum correct repair for Phase 29

## regression net

- identity recall must not regress while fixing style-rule recall
- Brainstack-only ownership must remain intact
- procedural skill learning must not become the storage path for personal style memory
- reset/resume/session boundary behavior must remain Brainstack-aware
- no broader donor or host rewrite is authorized unless the seam proof forces it

## required working style

- correctness-first, not capability-first
- no "probably" explanations when a trace can be gathered
- no overengineering if one bounded seam explains the failure
- user-facing architecture checkpoints only if the execute path reaches a real fork

## recommended execute order

1. reproduce on deployed Bestie path
2. trace durable storage for the same principal
3. trace recall packet and contract assembly
4. trace final runtime application / override
5. only then choose repair scope
