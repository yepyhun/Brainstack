# Phase 20.5 Implementation Contract

## Purpose

Convert the `20.4` forensic findings into a reliable restoration baseline.

This phase must improve both:

- proof trustworthiness
- live retrieval behavior

without collapsing back into `20.2`-style compensatory overlay tuning.

## Required Workstreams

### Workstream A. Retrieval-Judge Calibration

This workstream comes first.

It must:

- inspect the fixed-clock `20.4` split report row by row where needed
- reduce obvious retrieval-judge false negatives
- preserve a hard distinction between:
  - direct evidence present
  - derived / integrated evidence
  - answer-model recovery without sufficient retrieval support

It must not:

- silently promote derived/integrated cases into retrieval success
- redefine retrieval success so broadly that the metric becomes meaningless

### Workstream B. Fact-Path Parity Restoration

This workstream must explain and improve the remaining gap between:

- the earlier `20.2` decomp-off `10 / 15`
- the post-`20.4` fixed-clock answer-only `5 / 15`

It must:

- document the exact runtime environment of the earlier `20.2` decomp-off `10 / 15` run before comparing it to the current benchmark path
- compare packet shape, rendering, routing behavior, and runtime assumptions
- isolate what still differs materially after the clock confounder was controlled
- produce bounded fixes that improve the actual live fact path

It must not:

- write off the remaining gap as “just benchmark noise”
- reopen the old decomposition or overlay bundle by default

### Workstream C. Route Activation Coverage

This workstream must address the fact that route activation remained minimal after the `401` fix.

It must:

- measure which real questions should plausibly route away from plain fact mode
- improve activation coverage in a bounded, language-agnostic way
- verify that newly activated routes do not silently bypass the currently load-bearing priority logic, or else introduce an explicit route-appropriate equivalent
- keep routing fail-open

It must not:

- reintroduce the old decomposition gate under a new name
- build a multilingual keyword table
- treat wider activation as a success if it only adds noise

### Workstream D. Graph Yield Recovery

This workstream must start from the measured `20.4` fact:

- graph sparsity already appears at the `SQLite` layer

It must:

- inspect why graph yield stays sparse in the benchmark path
- check yield / admission / publication timing in the seeding flow
- improve graph yield only through bounded, donor-respecting fixes

It must not:

- declare victory from `Kuzu` counts alone
- replace donor graph logic with shell-side reimplementation

### Workstream E. Comparable Proof Reruns

This workstream must finish the phase.

It must:

- rerun bounded answer-only comparability
- rerun bounded split-proof comparability
- report answer score, retrieval score, and both-correct score together

It must not:

- collapse the proof back to one headline number
- hide suspicious pass cases

## Protected Boundaries

### Proof Boundary

- the retrieval judge may be recalibrated
- but the meaning of retrieval success must stay explicit and narrow enough to be useful
- manual relabeling support is allowed for auditability
- silent metric laundering is forbidden

### Restoration Boundary

- fact-path restoration is allowed
- reopening `20.2` bundle tuning is not

### Routing Boundary

- route activation may be improved
- but routing must remain fail-open, bounded, and language-agnostic

### Donor Boundary

- graph-yield fixes must respect donor-first architecture
- no shell-side donor replacement logic

## Minimum Proof Required Before Calling Phase 20.5 Done

Before `20.5` can be called done, the phase must show:

- a calibrated retrieval-judge story with named corrected cases
- a measured live fact-path result that is better explained or improved over `20.4`
- measured route activation behavior after the phase changes
- measured graph-yield behavior after the phase changes
- comparable rerun outputs in both answer-only and split-proof modes
- if calibrated `retrieval_correct` remains at or below roughly the current floor and answer-only parity remains at or below roughly the current `20.4` level, the phase must explicitly report that restoration was still insufficient rather than claim partial success
