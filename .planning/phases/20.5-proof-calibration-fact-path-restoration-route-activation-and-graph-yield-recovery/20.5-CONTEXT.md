# Phase 20.5 Context

## Why This Phase Exists

Phase `20.4` closed the broad uncertainty around the live benchmark path, but it did not restore the system.

What is now measured rather than guessed:

- fixed-clock answer-only comparability improved from `3 / 15` to `5 / 15`
- therefore runtime / prompt instability was a real confounder
- but the earlier `20.2` decomp-off `10 / 15` parity still remains far away
- the injected direct route resolver removed the previous live `401`
- but the current route gate still requested a non-default route in only `1 / 15`
- `Chroma` is dense in `15 / 15`
- `SQLite` graph yield is nonzero in only `3 / 15`
- `Kuzu` matches `SQLite` when graph data exists, so the graph problem is sparse yield, not a proven `Kuzu`-only publication failure
- the current split-proof `retrieval_correct=2 / 15` is a judge-underestimate
- the strongest current retrieval-judge false negatives are:
  - `6a1eabeb`
  - `e9327a54`

Therefore the next phase cannot start from raw benchmark scores alone.
It must first calibrate the proof metric, then restore the live fact path, then increase structural route activation and graph yield.

## What This Phase Is

This is a restoration phase built on top of the finished `20.4` forensics.

It has four connected jobs:

1. calibrate the retrieval judge so the split-proof metric becomes trustworthy
2. restore fact-path parity toward the earlier `20.2` decomp-off baseline
3. improve route activation coverage without reopening the old decomposition mistake
4. improve graph yield in the benchmark path without violating donor-first boundaries

## What This Phase Is Not

This phase is **not**:

- a reopening of `20.2` bonus / wording / decomposition tuning
- a collapse back to raw answer score as the primary truth metric
- a claim that retrieval is already “fine” because some split rows were judge false negatives
- a license to grow new shell-side donor logic
- a permission slip for unbounded fallback widening
- a proof phase that skips restoration work and jumps straight to new benchmarks

## Entry Conditions

Phase `20.4` finished with these explicit pending gaps:

- judge calibration
- live fact-path parity gap
- structural route activation coverage
- sparse graph yield in the benchmark path

Those are now the direct inputs to `20.5`.

## Target End State

By the end of `20.5`, the project should have:

- a calibrated retrieval-judge path with fewer obvious false negatives
- a live fact path whose behavior is closer to the earlier strong baseline
- a routing path that activates structural modes where they are actually needed
- better graph yield visibility and, if fixable within donor-first constraints, better graph yield itself
- bounded reruns whose metrics are interpretable without hand-waving
