# Phase 20.5 Plan

## Goal

Turn the finished `20.4` forensics into a trustworthy restoration baseline by:

- calibrating the retrieval judge first
- restoring live fact-path parity as much as the current architecture allows
- increasing structural route activation coverage in the real benchmark path
- improving sparse graph yield where the benchmark path currently under-produces graph data

## Scope

### In

- retrieval-judge calibration and bounded manual relabeling support
- fact-path parity restoration work on the live benchmark path
- route activation coverage work that respects the `20.3` routing boundaries
- benchmark-path graph-yield diagnosis and bounded fixes
- comparable answer-only and split-proof reruns

### Out

- reopening `20.2` overlay tuning
- donor rewrites
- new broad architecture speculation beyond the measured `20.4` gaps
- claiming restoration success without calibrated proof

## Planning Assumption

The current best working assumption is:

- the raw split-proof `retrieval_correct=2 / 15` is too pessimistic
- but the system is still materially below the earlier `20.2` decomp-off parity level
- therefore this phase must improve both:
  - metric trustworthiness
  - actual live retrieval behavior

## Execution Steps

1. Calibrate the proof first.
   Revisit the fixed-clock `20.4` split rows and make the retrieval-judge path less prone to obvious false negatives.
   The phase must preserve the distinction between:
   - direct retrieval success
   - derived / integrated evidence
   - answer-model recovery despite thin retrieval

2. Restore fact-path parity next.
   First document the exact runtime environment of the earlier `20.2` decomp-off `10 / 15` run, then compare the current live fact path against that baseline and isolate what still diverges after the fixed-clock correction.
   This must be treated as a real restoration task, not as a reporting exercise.

3. Improve route activation coverage.
   The route resolver is now live-safe, but the gate barely triggers.
   This phase must widen useful activation without reintroducing the old over-broad decomposition behavior, and must verify that newly activated routes do not silently drop the currently load-bearing priority behavior.

4. Improve graph yield in the benchmark path.
   Start from the fact that graph sparsity already appears in `SQLite`.
   Fixes must target graph yield / admission / publication timing, not just `Kuzu` plumbing guesses.

5. Re-run bounded proof in comparable modes.
   Use:
   - answer-only comparability
   - split-proof comparability
   and interpret them together rather than collapsing back to one score.

## Implementation Rules

- Do not let judge calibration drift into answer-score laundering.
- Do not let fact-path restoration reopen quote/digit/decomposition overlay experiments.
- Do not let route activation become a disguised multilingual keyword table.
- Do not let graph-yield work become a shell-side replacement for donor graph logic.
- Do not call the phase done if the proof remains ambiguous.

## Output Shape Required From Phase 20.5

The phase output must clearly state:

- which split rows were reclassified and why
- what changed in the live fact path
- whether route activation coverage improved in measured runs
- whether graph yield improved at the `SQLite` layer, the `Kuzu` layer, or both
- whether the reruns actually closed part of the parity gap

## Exit Gate

`20.5` can be called done only if all of these are true:

- retrieval-judge calibration is explicit and auditable
- the fact-path parity story is materially clearer than after `20.4`
- route activation is no longer effectively dead on the benchmark subset
- graph-yield findings are either improved or tightly explained
- bounded reruns produce metrics that can be interpreted without hand correction as the primary story
- if calibrated retrieval remains near the current floor and answer-only parity remains near the current `20.4` level, the phase summary explicitly says restoration stayed insufficient rather than claiming partial success
