# Phase 30.1 Implementation Contract

## phase type

Roadmap-shaping architecture phase.

## required final architecture

- chat-time memory kernel and offline long-form knowledge pipeline are explicitly separated
- provenance and evidence requirements are defined before implementation

## minimum evidence required before calling the phase done

- there is a clear boundary document
- the offline target shape is concrete enough for later execution
- no ambiguity remains about ownership between live chat memory and offline KG work
- the bounded pilot proves documents, chunks, claims, evidence spans, and conflict candidates without mutating the live kernel stores
- the installed `final` payload can import and execute the offline pilot without creating live provider / retrieval wiring

## explicitly disallowed outcomes

- sneaking offline KG work into ordinary chat-memory phases
- vague "later we will graph the books" planning without target shape

## cleanup requirement

- retire any misleading roadmap wording that suggests the current live graph extractor already solves the book/KG problem

## compatibility requirement

- must preserve the chat-memory and behavior-policy roadmap sequence

## code hygiene requirement

- if any prototype appears, it must not land half-wired into the live kernel
## recommended model level

`xhigh`
