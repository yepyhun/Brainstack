# Phase 29.8 Context

## problem statement

The current compact communication contract is too small to represent many high-value user behavior rules without flattening nuance away.

## why this phase exists

If the compiled policy remains too coarse, the raw-vs-compiled split will still underperform.

This phase expands the machine-facing policy ontology so it can carry more real behavior detail while staying token-bounded.

## accepted evidence entering this phase

- current compact slots are useful but sparse
- the external review correctly identified a mismatch between a 27-rule pack and a tiny slot family
- this must be solved by schema and compiler design, not by raw prompt dumping

## explicit non-goals

- no giant regex rule table
- no per-language hand-built prompt patching
- no unbounded growth of the always-on policy block
## recommended model level

`xhigh`
