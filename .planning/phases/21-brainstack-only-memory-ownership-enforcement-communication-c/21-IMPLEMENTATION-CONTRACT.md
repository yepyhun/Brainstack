# Phase 21 Implementation Contract

## Objective

Repair the Brainstack-only personal-memory stack at the shared architectural seams rather than papering over one conversation failure.

## System Doctrine This Phase Must Preserve

- Brainstack is the intended owner of personal-memory truth on this axis.
- Host persona/context mechanisms may remain only as bounded compatibility shell, not as silent competing owner.
- Native host capabilities may remain for their own domain, but they must not become a second persistence/retrieval channel for personal-memory truth.
- The fix must restore one coherent behavioral stack:
  - capture
  - injected contract
  - live behavior
  - clean durable memory

Protected rule:

- do not trade architectural clarity for one green conversation rerun

## Workstream A: Ownership and Prompt Truth Before Repair

- Establish the exact active path for:
  - Brainstack-only ownership
  - legacy memory disablement
  - SOUL/context-file loading
  - Brainstack system-prompt block injection
  - forbidden file/skill/tool routes
- No shallow patch is allowed before this truth pass is complete.

Required artifact:

- one ownership/prompt truth table that names the exact active module/file per layer

## Workstream B: Shared Ownership-Seam Repair

- Repair the lowest practical seam that still allows personal identity/style memory to spill into the wrong path.
- Treat the observed tool misuse as evidence of a class, not a single incident.
- Check sibling routes driven by the same bad ownership logic.
- Explicitly include automation detours and secondary-memory API detours if they target personal-memory truth.

Repair rule:

- patch the policy or shared fallback seam before adding any per-tool restriction
- keep the repair axis-bounded; do not broadly disable native features that remain legitimate outside this ownership boundary

## Workstream C: Contract-to-Behavior Bridge Repair

- Brainstack’s active communication contract must not remain a passive fact list.
- It must become reliable live behavior after reset/session restart.
- Repair the lowest shared layer that explains the repeated drift.

Protected rule:

- do not solve this primarily by strengthening `persona.md`
- do not rely on dead side writes under `/root/.hermes` as product truth

Required artifact:

- one behavior-bridge matrix showing capture -> injection -> reset survival -> live obedience for each rule under test

## Workstream D: Memory Hygiene Repair

- User facts and preferences may be stored.
- Assistant speculation about architecture must not be promoted to durable truth.
- Tool narration and guessed system explanations must not become profile/graph truth unless they are independently grounded.

Protected rule:

- if one speculative class is filtered, inspect adjacent similar classes in the same ingest path before calling the work done

Repair rule:

- prefer class-based ingress/promotion constraints at normalization/reconciliation seams over one-off exclusions

## Workstream E: Proof

- Verify in live conversation-shaped scenarios that:
  - Brainstack-only ownership holds
  - style rules are applied without manual tool detours
  - identity/nickname rules survive reset
  - durable memory no longer contains the same class of speculative pollution
  - automation and secondary-memory sibling detours are either closed or honestly called out
  - empty `current_state_pairs` is either repaired or explicitly accepted as a non-blocking representation difference because profile/contract/injection/behavior remain aligned
- Inspect persisted records directly after the run.

Required proof shape:

- one manual scenario rerun artifact
- one persisted-state inspection artifact
- one adjacent-similar-failure check result

## Protected Boundaries

### Architectural Boundary

- Stay donor-first.
- Preserve modularity and upstream updateability.
- Do not convert this into a large host rewrite unless the truth audit proves the shared seam is unavoidably host-side.
- Do not create a second mutable source of personal-memory truth just to force behavior compliance.

### Anti-Band-Aid Boundary

- No local patch without checking the deeper layer.
- No fix to one symptom without looking for adjacent similar failures.
- No “prompt stronger” answer unless the deeper ownership and injection truth says prompt precedence is the real shared seam.
- No execute improvisation that skips the truth artifacts and jumps straight to code.

### Truth Boundary

- Do not accept assistant self-reports as architectural truth.
- Prefer code path truth, runtime truth, and persisted-state truth.
- If the system still needs host persona fallback for some part, document that honestly instead of pretending Brainstack fully displaced it.
- If a native host feature remains useful outside the personal-memory axis, preserve it and document the narrower ownership boundary instead of over-displacing it.

### Product Integrity Boundary

- This phase is not merely about stylistic polish.
- It is about whether remembered user-specific contract can be trusted to govern live behavior.
- A green result counts only if ownership, behavior, and hygiene line up together.

## Minimum Evidence Required Before Calling Phase 21 Done

- explicit ownership/prompt-layer truth artifact
- explicit behavior-bridge truth artifact
- explicit memory-hygiene truth artifact
- live rerun of the manual scenario family
- persisted-state inspection after rerun
- explicit adjacent-similar-failure result
- explicit statement of whether general host capabilities remained intact while competing personal-memory detours were closed
- clear statement of whether Brainstack now truly owns this axis, or which residual host dependency remains
