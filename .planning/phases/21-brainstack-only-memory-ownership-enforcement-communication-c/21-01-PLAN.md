# Phase 21 Plan

## Goal

Fix the Brainstack-only personal-memory behavior stack at the correct architectural depth:

- personal identity/style/preference memory must stay owned by Brainstack
- Brainstack’s active communication contract must reliably shape live behavior
- assistant guesses about system internals must stop polluting durable memory

## Scope

### In

- ownership-path audit before code changes
- prompt-layer audit before code changes
- durable-memory hygiene audit before code changes
- bounded repairs at the deepest shared seam that explains the observed failures
- adjacent-similar-failure sweep for the same class of problem
- targeted live and persistence verification using the same scenario family that exposed the issue

### Out

- benchmark tuning
- large host architecture rewrite
- broad personality framework redesign
- generic “make the prompt stronger” hand tuning as primary strategy
- unrelated graph/corpus capability work

## Concrete Repair Method

Before code changes, Phase 21 must produce three explicit truth artifacts.

1. Ownership matrix.
   - For each relevant layer, record:
     - module / file
     - intended owner
     - actual observed owner
     - whether the path is valid, shadow, or forbidden
   - Minimum layers:
     - Brainstack-only mode gates
     - legacy memory path
     - SOUL/context-file loading
     - `/root/.hermes` side-write path
     - tool / skill / ad hoc code detours

2. Behavior bridge matrix.
   - For each communication rule, record:
     - where it is captured
     - where it is injected
     - whether it appears in live prompt assembly
     - whether it survives reset
     - whether live output obeys it
   - Minimum rules:
     - Hungarian response
     - Humanizer tone
     - new-line formatting
     - `Én / Te / Ő`
     - Bestie naming

3. Hygiene matrix.
   - For each persisted statement class, record:
     - source speaker / source channel
     - normalized class
     - whether promotion is allowed
     - where filtering or promotion currently occurs
   - Minimum classes:
     - user fact
     - user preference
     - assistant commitment
     - assistant architecture speculation
     - tool narration
     - temporal/date statement

## Repair Decision Rules

Phase 21 should choose the fix according to the first seam that actually explains the failure class.

- If the failure comes from ownership routing, fix the routing/policy seam and hard-block sibling detours in the same pass.
- If the failure comes from prompt composition or precedence, fix prompt assembly or contract placement, not `persona.md`.
- If the failure comes from ingest promotion, fix normalization / reconciliation / promotion rules at the shared seam, not by deleting one bad row class ad hoc.
- If a host-side dependency remains necessary, document it explicitly instead of pretending Brainstack fully displaced it.

## Hypothesis -> Falsification -> Repair Decision

This phase should not move from symptom to code directly. Each seam needs one explicit hypothesis, one falsification check, and one repair decision.

1. Ownership seam.
   - Hypothesis:
     - personal-memory behavior is escaping Brainstack-only ownership through a shared host/tool fallback path
   - Falsification:
     - prove the observed detour is dead, non-authoritative, and cannot influence live behavior or persisted truth
   - Repair decision:
     - if falsified, do not patch ownership routing
     - if not falsified, repair the shared routing/policy seam and close sibling detours in the same class
     - preserve native host features that remain legitimate outside the personal-memory axis; do not convert this into blanket native-feature displacement

2. Behavior seam.
   - Hypothesis:
     - the Brainstack communication contract is captured and injected, but loses precedence or salience before live generation
   - Falsification:
     - prove the contract is absent, truncated, or not actually part of live prompt assembly
   - Repair decision:
     - if absent, repair injection/composition
     - if present but weak, repair precedence/contract rendering rather than adding a second prompt source

3. Hygiene seam.
   - Hypothesis:
     - durable-memory pollution happens because normalization/reconciliation promotes assistant self-explanations as if they were grounded user truth
   - Falsification:
     - prove the bad rows come from a narrower one-off path that does not generalize to sibling classes
   - Repair decision:
     - if not falsified, patch promotion rules at the shared ingest seam
     - if falsified, make the smallest class-complete repair at the real narrower seam

## Stop Conditions Before Any Patch

No code change is allowed until all of these are true:

- the ownership matrix identifies the active and shadow paths
- the behavior bridge matrix shows where enforcement is lost
- the hygiene matrix shows where speculative text becomes durable truth
- at least one adjacent similar failure candidate has been checked for each seam

## Validation Ladder

Validation must move from lowest-risk truth to highest-risk product proof.

1. Code-path truth.
   - verify the real active modules and prompt assembly path
2. Runtime truth.
   - verify container/home/config reality
3. Persisted-state truth.
   - inspect what was actually stored
4. Controlled live scenario proof.
   - rerun the same conversation family
5. Adjacent-similar proof.
   - confirm at least one sibling failure class is also closed, or explicitly still open

## Regression Net By Workstream

Each workstream must carry its own regression net so the phase cannot "look fixed" while moving the break elsewhere.

1. Workstream A regression net.
   - Re-check:
     - Brainstack-only mode still disables legacy personal-memory paths
     - host context loading truth is documented, not guessed
   - Failure here blocks all later code changes.

2. Workstream B regression net.
   - Re-check:
     - no personal-memory save/retrieval detours through file/skill/tool routes
     - no sibling fallback route remains open after the repair
     - general host capabilities like scheduling remain usable when they do not target personal-memory truth

3. Workstream C regression net.
   - Re-check:
     - Hungarian preference still applies
     - Humanizer still applies
     - new-line formatting still applies
     - `Én / Te / Ő` capitalization still applies
     - Bestie naming still survives reset

4. Workstream D regression net.
   - Re-check:
     - user preferences still persist
     - assistant promises do not become architectural truth
     - architecture speculation and tool narration do not re-enter through sibling ingest paths
     - date conflict handling does not regress unrelated temporal capture

5. Workstream E regression net.
   - Re-check:
     - live rerun result
     - persisted-state result
     - one adjacent similar class result
   - Phase 21 is not done unless all three are explicitly reported.

## Execution Steps

1. Workstream A: ownership and prompt-layer truth audit.
   - Trace the exact personal-memory ownership path in deployed Bestie:
     - Brainstack-only mode gates
     - legacy memory disabling
     - file/skill/tool detour blockers
     - SOUL/context-file loading
     - Brainstack system-prompt block injection
   - Produce an explicit truth table:
     - expected owner
     - actual owner
     - allowed path
     - forbidden path
   - Check whether the observed `persona.md` writes are:
     - dead side writes
     - active prompt inputs
     - or partially active shadow behavior
   - Name the exact modules/files that own each layer of the path.
   - Do not patch yet; establish the full chain first.
   - Compatibility note:
     - if host persona/context loading still participates in the final behavior stack, record it honestly as a residual compatibility dependency rather than silently designing against it.

2. Workstream B: deeper-layer ownership hardening.
   - Fix the shared ownership seam that still allows personal style/identity memory to leak into:
     - `read_file`
     - `write_file`
     - `patch`
     - `skill_view`
     - `skill_manage`
     - ad hoc code execution
     - `cronjob` automation detours
     - secondary-memory API detours invoked from ad hoc code
   - Do not fix only the exact observed tool path.
   - Check sibling paths using the same bad ownership logic and harden them in the same pass.
   - Preferred repair order:
     - policy/gating seam
     - shared fallback seam
     - only then per-tool guard, if still needed
   - Donor-risk note:
     - prefer a thin ownership policy correction over rewriting broad host tool behavior unless the truth pass proves the host seam is the only real owner.
     - do not break legitimate host automation; only block it when it acts as a competing persistence/retrieval channel for personal-memory truth.

3. Workstream C: communication-contract application hardening.
   - Audit how Brainstack’s active communication contract is built and injected.
   - Determine why the contract can be present in memory but fail to govern live style after reset.
   - Repair the shared bridge at the lowest practical layer:
     - prompt composition
     - contract rendering
     - prompt precedence
     - or response-policy shaping
   - Do not accept a fix unless it explains why the same remembered rule can be recalled yet not obeyed after reset.
   - Validate not only Humanizer, but also:
     - new-line formatting
     - `Én / Te / Ő` capitalization rule
     - Bestie naming preference
   - Donor-risk note:
     - do not create a second independent persona channel; the repair should strengthen the existing Brainstack contract path or honestly preserve a necessary host compatibility shell.

4. Workstream D: durable-memory hygiene hardening.
   - Audit the ingestion path so assistant self-explanations do not become durable truth just because they were uttered confidently.
   - Separate:
     - user-stated fact/preference
     - assistant promise
     - assistant speculation about internals
     - tool narration
   - Check adjacent similar classes:
     - architecture speculation
     - memory-system explanations
     - tool-result self-interpretation
     - temporal normalization conflicts like Laura’s date drift
   - Repair at the reusable ingestion / normalization / reconciliation seam where possible.
   - Prefer class-based promotion rules over one-off row deletions.
   - Donor-risk note:
     - keep ingestion filtering narrow enough that factual capture remains rich; this is a hygiene hardening phase, not a memory amputation phase.

5. Workstream E: proof and regression.
   - Re-run the manual scenario family with reset boundaries.
   - Verify:
     - no personal-memory file detours
     - no automation or secondary-memory detours trying to store or recover personal-memory truth
     - no false `persona.md` / loading claims
     - correct passive recall
     - correct live application of style contract
     - no speculative internals promoted to durable memory
     - explicit result for at least one adjacent similar failure check
     - explicit result for whether empty `current_state_pairs` is a real contract gap or only a non-blocking representation difference
   - Inspect the stored Brainstack records after the run, not only the chat output.

## Required working style for this phase

- If a bug appears, ask first:
  - is this the shallow symptom or the shared seam?
  - is another module affected by the same logic?
  - are there sibling failures from the same ownership or hygiene pattern?
- Do not accept a local patch until those checks are done.
- Prefer one correct deeper fix over several shallow patches.

## Output Shape Required From Phase 21

- one ownership/prompt truth artifact
- one behavior-bridge truth artifact
- one explicit repair decision for the deepest shared seam(s)
- one regression/proof artifact from the manual scenario family
- one persistence audit artifact showing what was and was not stored
- one honest next-step recommendation

## Exit Gate

- personal-memory ownership is once again unambiguous in Brainstack-only mode
- the known manual style/identity failures are fixed without file/skill/tool detours
- assistant speculative internals no longer enter durable memory through the same path
- at least one adjacent similar failure class was explicitly checked, not assumed absent
- the phase outcome is architecture-led, not reactive patchwork
