# Phase 31 Plan

## goal

Freeze the accepted root-cause model for the post-`30.5`/`30.6` live regression so the next hotfix targets concrete breakpoints rather than vague “memory got worse” narratives.

## scope

### in

- reproduce and pin the failing live interaction shape from Discord/session traces
- inspect the active canonical `behavior_contract` revision and the compiled policy it feeds
- inspect whether explicit multi-message rule teaching converges into one canonical contract or fragments across turns
- inspect whether Brainstack-only tool blocking applies on every live tool-execution path
- inspect whether `session_search` is still being used as a personal-memory fallback in the rule-recall flow
- produce one accepted, evidence-backed root-cause reading

### out

- shipping the hotfix itself
- broad behavior-policy redesign
- broad routing redesign outside the proven failing seams

## core principles this phase must preserve

- follow the canonical immutable project principles:
  - [IMMUTABLE-PRINCIPLES.md](../../IMMUTABLE-PRINCIPLES.md)
- especially:
  - donor-first
  - truth-first / no "good enough"
  - fail-closed upstream compatibility
  - zero heuristic sprawl

## architecture decision

### accepted target model

- the regression is modeled as a three-seam failure, not one fuzzy product complaint:
  - a partial canonical behavior-contract revision can still become active authority
  - explicit multi-message rule teaching does not converge into one durable canonical revision
  - Brainstack-only tool blocking is patched on the concurrent path but not guaranteed on the sequential path
- `session_search` misuse is treated as a supporting symptom of the same authority/runtime problem, not a separate root cause

### rejected model

- “the model forgot the rules”
- “30.5 itself was a mistake”
- “the provider auth incident explains the later Brainstack behavior”

## workstreams

### 1. live trace capture

- capture the concrete session trace showing:
  - partial rule recall
  - `session_search` use for style-contract recall
  - `skill_manage` use for personal-memory storage

### 2. canonical contract inspection

- inspect live `behavior_contracts` and `compiled_behavior_policies`
- prove whether the active revision is partial, who wrote it, and what it superseded

### 3. admission and convergence inspection

- trace explicit style-contract capture on:
  - `prefetch`
  - `sync_turn`
  - session-end/Tier-2 paths
- determine why multi-message user teaching fails to produce one good canonical revision

### 4. host runtime path inspection

- prove whether Brainstack-only blocking is wired identically on:
  - concurrent tool path
  - sequential tool path

### 5. accepted root-cause synthesis

- record one final accepted reading that reconciles:
  - live Discord behavior
  - runtime session artifacts
  - current source and installed host code

## validation ladder

1. live/session trace proves the bad behavior shape
2. DB inspection proves what the active canonical behavior-contract revision actually contains
3. provider/host code inspection proves where multi-message convergence fails
4. host execution-path inspection proves whether tool blocking is uniform or half-wired
5. final forensics note names the smallest correct hotfix surface

## anti-goals

- do not patch code in this phase
- do not blur the findings with speculative architecture changes
- do not accept an explanation that cannot be pointed to in:
  - session trace
  - DB row
  - concrete code seam
