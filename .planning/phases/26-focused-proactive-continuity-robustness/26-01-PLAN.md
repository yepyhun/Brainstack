# Phase 26 Plan

## Goal

Close the remaining proactive continuity residual by hardening reset-time plan continuation under stricter filters, without turning the work into a broad capability expansion.

## Scope

### In

- focused diagnosis of the remaining proactive continuity miss
- comparison between:
  - passing proactive runs
  - failing proactive runs
- analysis of over-recall / token-waste risk in proactive answers
- seam-level fix only if the cause is proven
- stricter validation gates for:
  - event-frame restoration
  - no-detour proactive continuation
  - selective recall / non-wordy packaging
  - whole-path diagnosis beyond the memory kernel alone
- deployed-path proof after the fix

### Out

- broad retrieval redesign
- SHIBA-style Tier-2 uplift
- reopening Phase 22 or Phase 24 by default
- broad host cleanup
- unrelated conversation-quality cleanup

## Validation method

The phase should behave like a narrow robustness repair, not a feature sprint.

The order is:

1. reproduce the proactive miss under the stricter filters
2. compare passing and failing traces
3. identify the deepest shared seam that explains the miss
4. patch only that seam
5. rerun the stricter proactive proof
6. rerun a bounded broader safety net so the fix does not regress into new drift

## Decision rules

1. If the miss is mostly answer-surface variance with no clear underlying seam, do not invent a large architectural fix.
2. If a seam is proven, fix the seam once rather than layering prompt band-aids around it.
3. If the fix improves event restoration but worsens “no-detour” behavior, it is not acceptable.
4. If the fix improves recall but becomes token-wasteful or overly verbose, it is not acceptable.
5. If a proposed fix starts pulling in broad retrieval or Tier-2 redesign, stop and classify it as a later phase instead.

## Hypothesis -> Falsification -> Repair decision

1. Event-frame weakness hypothesis.
   - Hypothesis:
     - the remaining miss comes from weak event-frame restoration in continuation synthesis, not from total memory absence.
   - Falsification:
     - show that the necessary event frame is absent from the supplied memory evidence itself.
   - Repair decision:
     - if falsified, repair memory supply
     - otherwise repair the continuation synthesis seam

2. No-detour weakness hypothesis.
   - Hypothesis:
     - when memory support is sufficient, the system still lacks a strong enough rule to continue directly instead of re-asking or detouring into tools/web lookup.
   - Falsification:
     - show that the system only detours when memory support is genuinely insufficient.
   - Repair decision:
     - if falsified, leave the detour behavior alone
     - otherwise harden the continuation behavior rule

3. Selective-packaging hypothesis.
   - Hypothesis:
     - a good proactive answer should restore the plan efficiently, not replay the maximum available memory.
   - Falsification:
     - show that better proactive quality in this residual really requires broader, denser memory replay rather than better answer shaping.
   - Repair decision:
     - if falsified, avoid over-tightening packaging
     - otherwise preserve only the minimal useful proactive payload

4. Thin-fix hypothesis.
   - Hypothesis:
     - the remaining proactive residual can be improved with a bounded seam-level fix.
   - Falsification:
     - show that solving it correctly would require broad retrieval or reasoning redesign.
   - Repair decision:
     - if falsified, stop and reclassify it as a future capability phase instead of forcing a bad local patch

## Stop conditions before any patch

No product code change is allowed until all of these are true:

- at least one passing and one failing proactive trace have been compared
- the event-frame gate has been evaluated explicitly
- the no-detour gate has been evaluated explicitly
- the selective-recall gate has been evaluated explicitly
- the diagnosis has considered more than the memory kernel alone
- the proposed fix is classified as:
  - synthesis seam
  - memory-supply seam
  - whole-path packaging / routing seam
  - or future capability thread

## Validation ladder

1. Stricter proactive canary definition
2. Passing vs failing trace comparison
3. Whole-path seam diagnosis
4. Bounded seam fix
5. Stricter proactive rerun
6. Negative-control / no-detour rerun
7. Selective-recall / token-waste check
8. Bounded broader safety check

## Compatibility / donor-risk checkpoints

1. Boundary stability
   - do not reopen the Phase 22 coexistence decision while fixing proactive continuity

2. Correctness stability
   - do not reopen the Phase 24 profile-isolation seam while fixing proactive continuity

3. No-detour discipline
   - do not let the fix improve memory phrasing by silently increasing unnecessary tool behavior

4. Maintenance safety
   - no new subsystem, no broad planner layer, no new permanent complexity unless the evidence is unusually strong

## Execution steps

1. Workstream A: stricter proactive canary.
   - Define the stricter pass criteria:
     - event frame must be restored
     - key details must be restored
     - no unnecessary re-ask / browse / tool detour first move
     - no unnecessary over-recall or token-heavy replay of stale context

2. Workstream B: trace comparison.
   - Compare:
     - one or more passing proactive runs
     - one or more failing proactive runs
   - Look at:
     - packet composition
     - routing / policy snapshot
     - persisted state
     - answer shape
     - answer length / payload selectivity

3. Workstream C: seam diagnosis.
   - Decide whether the miss lives in:
     - memory supply
     - continuation synthesis
     - routing / policy
     - response packaging
     - or a broader future capability gap

4. Workstream D: bounded fix.
   - Patch only the proven seam.
   - Keep it local and readable.

5. Workstream E: stricter proof.
   - Rerun the proactive scenario under the stricter filters.
   - Then rerun a small safety net to ensure the fix does not create:
     - fabricated event framing
     - needless user re-asking
     - needless tool detours
     - needless verbose replay / token waste

## Regression net

1. Event-frame regression net.
   - the system must not keep only venue/constraint fragments while dropping the plan frame

2. No-detour regression net.
   - sufficient memory evidence must not degrade into first-move re-asking or browsing

3. Selective-recall regression net.
   - proactive help must not bloat into always replaying the full memory packet

4. Boundary regression net.
   - no Phase 22 boundary reopen

5. Correctness regression net.
   - no Phase 24 profile-isolation regressions

## Required working style

- Prefer deeper seam diagnosis over conversational prompt patching.
- Prefer one local, durable fix over multiple reactive nudges.
- Prefer strict pass/fail language over “good enough” if the user explicitly wants to chase this residual.
- Prefer the smallest useful proactive answer over verbose context replay.
- If the seam is not cheaply fixable without broad redesign, stop honestly and say so.
