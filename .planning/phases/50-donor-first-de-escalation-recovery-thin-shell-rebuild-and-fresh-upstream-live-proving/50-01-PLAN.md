This phase is a de-escalation of host control, not a de-escalation of memory intelligence.

## workstreams

1. Fresh-upstream baseline and shell map
- freeze `/home/lauratom/Asztal/ai/finafina` as the only code baseline for this recovery
- map the smallest extension seams that can host Brainstack as a shell:
  - `agent/memory_provider.py`
  - `agent/memory_manager.py`
  - `plugins/memory/*`
  - `run_agent.py`
  - `gateway/run.py`
  - `gateway/session.py`
  - `tools/cronjob_tools.py`
- classify each current customization as:
  - keep
  - thin-wire rewrite
  - remove

2. Keep / move / remove matrix
- keep:
  - donor/provider memory intelligence
  - ownership boundaries
  - cross-store consistency
  - provider-side recall / write / reconciliation logic
- move:
  - any memory-specific “smartness” that currently lives in generic host reply delivery back behind provider or memory-manager seams
- remove:
  - host-level hard behavior gating on ordinary replies
  - generic blocked fallback chat replies
  - host-side rule-governance logic that turns memory support into chat control
  - cross-layer glue that makes the shell own donor behavior instead of orchestrating it

3. Host de-escalation
- remove ordinary-chat dependence on host-level Brainstack output blocking
- remove generic fallback replies like `Írd meg újra.`
- move any remaining output validation to bounded advisory / local repair roles
- ensure the shell cannot become a second behavior engine
- ordinary chat must remain chat-first:
  - working reply first
  - memory support second
  - donor-owned intelligence behind the seam, not in the host loop

4. Thin-shell donor wiring
- keep Brainstack only where it acts as:
  - donor orchestrator
  - ownership boundary
  - cross-store consistency layer
- push memory behavior back toward donor/provider seams rather than run-loop governance
- require native Hermes files to stay thin where possible

5. Clean runtime/profile proving setup
- use a fresh runtime/profile baseline for proving
- do not trust historically drifted Bestie memory state as the recovery oracle
- build one small but realistic live proof surface:
  - greeting
  - ordinary chat request
  - style recall
  - explicit memory recall
  - reminder flow

6. Rolling live test and rethink loop
- after the bulk simplification lands, run the live proof on the simplified product
- if failures remain, only apply bounded shell-level corrections that preserve the de-escalated architecture
- rerun the whole live proof after each correction
- stop when further fixes would reintroduce host-level rule creep

## decision rule

When a design choice is unclear, prefer the option that satisfies all three:
- ordinary chat delivery does not depend on host-level Brainstack rule gates
- memory intelligence remains in donor/provider seams
- the shell stays responsible only for orchestration, ownership, and consistency
