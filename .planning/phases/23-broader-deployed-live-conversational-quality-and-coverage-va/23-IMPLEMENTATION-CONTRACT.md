# Phase 23 Implementation Contract

## Objective

Read broader deployed-live conversational quality and coverage honestly on top of the now-settled Brainstack/native boundary, without turning validation into stealth feature development.

## System doctrine this phase must preserve

- Brainstack remains the owner of durable personal-memory truth.
- Phase 22 boundary decisions remain the default baseline.
- Native coexistence points are not to be reopened casually.
- This phase is primarily about product truth, not new capability growth.
- The project guardrails remain active and must be used explicitly:
  - donor-first
  - modularity / upstream updateability
  - truth-first
  - fail-closed on the owned axis
  - no benchmaxing
  - no overengineering

## Workstream A: Broader scenario matrix

- Define a broader real-use scenario set across:
  - ordinary conversation quality
  - continuity and carry-through
  - long-range memory and relation recall
  - larger knowledge-body use
  - bounded coexistence checks where relevant

Required artifact:

- one scenario matrix with category coverage, rationale, and expected evidence shape

## Workstream B: Real deployed-live execution

- Execute the scenario matrix on the real deployed-live path.
- Record outcomes by quality class, not only binary pass/fail.

Protected rule:

- do not let benchmark convenience replace real product-shaped conversation checks

## Workstream C: Failure / anomaly inspection

- For weak, failing, or contradictory outcomes, inspect:
  - packet composition
  - persisted state
  - provider / harness health

Protected rule:

- do not blame Brainstack first if the evidence points to provider variance or a harness seam

## Workstream D: Residual classification

- Convert the results into a usable residual map:
  - product bug
  - provider variance
  - harness issue
  - acceptable residual
  - future capability phase

Protected rule:

- no vague “needs improvement” bucket
- every important miss must be classified concretely

## Workstream E: Bounded action if forced

- Only perform a code or harness change if:
  - the evidence is clear
  - the change is thin
  - the change is needed for truthful interpretation of broader live quality

Protected rule:

- do not hide a new build phase inside this validation phase

## Protected boundaries

### Anti-overengineering boundary

- no new large subsystem
- no broad host rewrite
- no opportunistic architecture expansion just because broader validation found a weak spot

### Boundary-stability / donor-risk boundary

- Phase 22’s ownership/coexistence decisions remain in force unless broader live evidence directly falsifies them
- no reopening of `session_search` or `cronjob` boundary decisions without clear causal evidence

### Truth boundary

- read product behavior from live evidence first
- separate memory-supply failure from answer-synthesis weakness
- separate provider failure from product failure

## Minimum evidence required before calling Phase 23 done

- one broader deployed-live scenario matrix
- one broader live result set
- supporting packet/state inspection for important fails or anomalies
- one residual map with explicit classification
- one honest next-thread recommendation:
  - no new phase needed
  - bounded fix needed
  - or future capability phase justified
