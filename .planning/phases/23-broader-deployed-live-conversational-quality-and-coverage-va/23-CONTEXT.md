# Phase 23 Context

## Why this phase exists

Phase 21 restored Brainstack ownership of the durable personal-memory /
communication-contract axis.

Phase 22 then clarified the long-term coexistence boundary:

- Brainstack owns durable personal memory
- `session_search` may coexist as explicit transcript forensics / browsing
- `cronjob` remains native automation outside the owned axis
- the runtime/docs memory orchestration story is now clearer

That means the next correct move is no longer another ownership micro-phase.

The next correct move is broader deployed-live product truth:

- how good is the system now in ordinary real conversations?
- where is the quality already strong?
- where are the remaining weak spots?
- do the restored boundaries actually help the user-visible experience?

## What is already known

### Strong signals already established

- Phase 21 proved the personal-memory / communication-contract axis on the real deployed path
- Phase 22 removed a likely over-displacement and preserved useful native coexistence
- earlier broader product reads showed that the system can already produce strong live results when the runtime path is healthy

### What is still unknown

- whether the quality remains strong outside the narrower proofs already used
- whether communication-contract behavior stays good across more varied live conversations
- whether long-range recall, continuity, and large-knowledge use stay coherent when the conversations are less benchmark-shaped
- whether the new Brainstack/native coexistence boundary creates better user-visible behavior, not just cleaner architecture

## Core doctrine for this phase

This is a product-truth phase.

It is not a feature-building phase by default.

The purpose is:

- measure broader deployed-live conversational quality
- read residuals honestly
- separate product weakness from provider variance, harness error, and one-off noise
- only permit thin repairs if the measurement itself is broken or a newly exposed blocker is too clear to ignore

## Project guardrails with refreshed emphasis

These remain active and should be used explicitly during Phase 23:

- donor-first
  - do not react to every quality miss by inventing a new Brainstack subsystem
- modularity / upstream updateability
  - prefer bounded fixes and honest residual classification over rushed feature accretion
- truth-first
  - one red result is not automatically a new feature phase
  - but one clean blocker should not be hidden behind “overall looks good”
- fail-closed on the owned axis
  - no reopening of personal-memory shadow owners during validation
- no benchmaxing
  - this phase is about broader real product behavior, not benchmark score-shaping
- no overengineering
  - do not turn validation findings into architecture expansion unless the evidence is unusually strong

## Product targets this phase must read against

- coherent continuous conversation
- proactive stateful continuity
- long-range accurate recall and relation-tracking
- usable storage of large bodies of knowledge
- meaningful token savings

## Target reading after Phase 23

After this phase, we should have a sharper product-level picture:

- what is already strong enough to trust
- what is weak but acceptable as residual
- what is truly blocking quality
- what is provider-specific noise rather than Brainstack architecture failure

The ideal outcome is not “no failures”.

The ideal outcome is:

- honest broader product truth
- a cleaner residual map
- and a justified next thread, if another thread is still needed

## What this phase is not

- not a reopen of Phase 21 ownership work
- not a reopen of Phase 22 boundary work
- not a broad host cleanup
- not a benchmark campaign
- not a pretext for feature-building under the label of validation

## End-state invariants

1. Boundary invariant.
   - Phase 22’s ownership/coexistence decision remains the baseline.

2. Product-truth invariant.
   - broader conversational quality is measured against real use, not only prior narrow proofs.

3. No-feature-creep invariant.
   - new feature work is not silently pulled into this phase unless the evidence forces it.

4. Honest residual invariant.
   - unresolved misses must be classified clearly:
     - provider / model variance
     - harness problem
     - product bug
     - or next-phase capability gap

5. Maintenance invariant.
   - no new maintenance-heavy subsystem is justified by this phase alone.
