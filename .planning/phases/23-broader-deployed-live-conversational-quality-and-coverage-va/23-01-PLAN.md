# Phase 23 Plan

## Goal

Measure broader deployed-live conversational quality and coverage on top of the now-stabilized Brainstack/native boundary, so the next project move is guided by honest product truth rather than narrow proof momentum.

## Scope

### In

- broader deployed-live validation across the main product targets:
  - coherent continuous conversation
  - proactive stateful continuity
  - long-range recall / relation-tracking
  - larger knowledge-body use
  - token / packet usefulness
- scenario families that go beyond the narrower canary-style proofs already completed
- packet/state inspection for:
  - failures
  - surprising wins
  - ambiguous or contradictory outcomes
- thin harness corrections only if the measurement path itself is proven misleading
- residual classification and next-thread recommendation

### Out

- broad feature-building
- reopening Phase 21 ownership decisions by default
- reopening Phase 22 boundary decisions by default
- benchmark-shaping work
- broad host cleanup

## Validation method

This phase should behave like a product-truth read, not a reactive repair sprint.

The order is:

1. define the scenario matrix
2. run the deployed-live matrix on the real path
3. inspect state/packet truth where results are unclear
4. classify outcomes
5. only then decide whether any bounded fix is justified

## Decision rules

1. If a result is weak but explainable as provider variance or noisy conversational surface behavior, do not escalate it into an architecture phase immediately.
2. If a result shows a clear repeatable product bug on the live path, classify it explicitly rather than hand-waving it as noise.
3. If the measurement harness is misleading, fix the harness first before reading product truth from it.
4. If a finding would require broad new implementation to resolve, that belongs in a later dedicated phase, not inside Phase 23.

## Hypothesis -> Falsification -> Repair decision

1. Broader live quality hypothesis.
   - Hypothesis:
     - after Phases 21 and 22, the deployed-live product behavior is materially stronger than the earlier narrow proofs alone would suggest.
   - Falsification:
     - show that broader live scenarios collapse outside the previously curated proof shapes.
   - Repair decision:
     - if falsified, classify the specific blocking quality class rather than inventing a generic “quality is weak” story

2. Boundary-benefit hypothesis.
   - Hypothesis:
     - the clarified Brainstack/native boundary improves or at least preserves user-visible conversational quality.
   - Falsification:
     - show that the restored coexistence points or clarified ownership now create confusion, regressions, or quality loss.
   - Repair decision:
     - if falsified, reopen only the specific boundary that caused the regression

3. Measurement-truth hypothesis.
   - Hypothesis:
     - the broader live evaluation path now reads product behavior honestly enough to support next-step decisions.
   - Falsification:
     - show that the matrix is still being distorted by harness error, provider misconfiguration, or another measurement seam.
   - Repair decision:
     - if falsified, repair the measurement seam first and rerun before drawing product conclusions

## Stop conditions before any patch

No product code change is allowed until all of these are true:

- the broader scenario matrix has been run at least once on the real deployed path
- the surprising failures and wins have supporting packet/state inspection where needed
- any proposed fix is clearly classified as:
  - harness fix
  - bounded product fix
  - or future phase candidate

## Validation ladder

1. Scenario matrix definition
2. Real deployed-live execution
3. Failure / anomaly packet-state inspection
4. Residual classification
5. Thin correction only if measurement or product truth demands it
6. Final quality/coverage verdict

## Compatibility / donor-risk checkpoints

1. Boundary stability
   - do not casually reopen the Phase 22 coexistence decision during validation
   - only revisit it if a real live regression points directly at that boundary

2. Ownership stability
   - do not reintroduce personal-memory shadow owners while adding or adjusting validation scenarios

3. Validation realism
   - prefer ordinary user-like conversations over synthetic benchmark-shaped probes wherever possible

4. Maintenance safety
   - do not “just add one more system” to explain or mask a quality miss

## Execution steps

1. Workstream A: scenario matrix.
   - Build a broader deployed-live scenario set covering:
     - ordinary social continuity
     - personal preference carry-through
     - longer memory / relation recall
     - larger knowledge-body retrieval
     - explicit transcript forensics coexistence where relevant
   - Keep the set broad enough to be meaningful, but not so wide that it becomes a benchmark campaign.

2. Workstream B: real-path execution.
   - Run the matrix on the actual deployed-live path.
   - Capture outcome classes, not just pass/fail:
     - strong pass
     - weak but acceptable
     - ambiguous
     - clear fail

3. Workstream C: packet/state inspection.
   - For fails or suspicious results, inspect:
     - packet composition
     - relevant persisted memory state
     - any provider/harness anomalies
   - Separate “the model answered badly” from “the memory stack failed to supply the right truth”.

4. Workstream D: residual map.
   - Produce a clean classification of broader deployed-live residuals:
     - product bug
     - provider variance
     - harness issue
     - acceptable residual
     - future capability phase

5. Workstream E: bounded action only if justified.
   - If a thin fix is clearly warranted and does not derail the validation phase, perform it.
   - Otherwise close the phase with an honest next-thread recommendation instead of sneaking in new build work.

## Regression net

1. Ownership regression net.
   - no personal-memory shadow owner reopens during the broader validation work

2. Boundary regression net.
   - `session_search` coexistence remains bounded and does not silently become a second durable-memory owner

3. Measurement regression net.
   - no major product conclusion is drawn from a broken or misconfigured harness

4. Scope regression net.
   - the phase stays a validation phase unless a bounded fix is clearly forced by evidence

## Required working style

- Prefer broader truth over narrow green slices.
- Prefer honest residual classification over speculative fixing.
- Prefer thin corrections over new systems.
- If the phase surfaces a real product blocker, say exactly what it is.
- If the phase does not justify new implementation, close it cleanly instead of forcing more code work.
