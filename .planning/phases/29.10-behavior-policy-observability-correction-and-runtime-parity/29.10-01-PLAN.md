# Phase 29.10 Plan

## goal

Add inspectable behavior-policy observability, bounded correction, doctor checks, and runtime-parity gates so the active compiled policy is provable in live deployed use.

## scope

### in

- per-turn behavior-policy trace
- bounded inspection surface for active compiled policy, raw archival contract, and compiler coverage
- bounded correction / supersession flow
- doctor/install/live parity checks for behavior policy

### out

- broad admin UI
- broad second-brain operating context work
- offline document / KG pipeline work

## core principles this phase must preserve

1. donor-first
2. modular, thin integration
3. zero heuristic sprawl
4. multimodal-ready
5. token discipline
6. single durable owner per truth class
7. second-brain / proactive-agent fit
8. no dead code / no half-wired paths

## architecture decision

### accepted target model

- every ordinary-turn answer can expose which compiled behavior policy revision/hash was active
- the system can show:
  - current active compiled policy
  - current raw archival contract
  - compiler coverage / inactive or conflicting clauses
- runtime/install mismatches fail closed and are visible through doctor or parity gates

### rejected model

- black-box behavior policy with no traceability
- relying on benchmark or packet success to imply deployed runtime parity
- silent stale-policy injection

## workstreams

### 1. policy trace

- define a per-turn trace payload for active behavior policy state

### 2. inspection surface

- define bounded ways to inspect active compiled policy, raw contract, and coverage outcomes

### 3. correction flow

- define the minimum correction/update contract so rules can be superseded without full blind retraining

### 4. doctor and runtime parity

- add behavior-policy-specific doctor/install/live parity checks
- parity must explicitly name source-of-truth, shimmed Hermes worktree, and `hermes-final`
- active docker/runtime checks must use the same interpreter that the live gateway process actually runs
- bare container `python3` probes are not authoritative if the gateway runs from a project `.venv`

## validation ladder

1. trace payload proof
2. inspection snapshot proof
3. correction proof
4. doctor/runtime parity proof
5. stale-surface mismatch fail-closed proof
6. gateway-interpreter parity proof

## anti-goals

- no giant admin subsystem
- no verbose always-on debug spam in ordinary user answers
- no silent stale-policy state

## recommended model level

`high`
