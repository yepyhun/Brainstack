# Phase 29.10 Context

## Why this phase exists

By the time `29.6` through `29.9` succeed, Brainstack may have the right behavior-policy architecture and answer-level obedience gates, but it still will not be `10/10` product-grade unless the runtime can prove what policy was active on a given answer and whether the deployed Hermes/Bestie surface is actually using it.

The accepted architectural gap is:

- active compiled behavior policy must be inspectable
- stale / mismatched runtime state must be visible
- rule correction must be possible without blind full retraining
- install / doctor / live runtime parity must be fail-closed

This phase turns behavior policy from a hidden mechanism into an inspectable deployed runtime contract.

## Inputs

- `29.6` raw archival vs compiled policy split
- `29.7` synchronous policy teaching with fail-closed activation
- `29.8` compiler coverage / no-silent-drop guarantees
- `29.9` answer-level obedience harness

## Non-goals

- no broad UI-heavy policy manager
- no broad second-brain operating model changes
- no offline document/KG work
## recommended model level

`high`
