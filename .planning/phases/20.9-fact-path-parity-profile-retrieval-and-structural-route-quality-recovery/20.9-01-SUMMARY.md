# Phase 20.9 Summary

## Verdict
- `20.9` source-side retrieval fixes are real and locally validated.
- the first live `20.9` measurements were partially invalid because they ran against a stale Bestie runtime instead of the patched source parity
- after runtime parity was explicitly restored in the Bestie target runtime, the valid comparable answer-only rerun landed at `5 / 15`
- the full comparable split proof did **not** complete cleanly enough to be treated as a final phase-closing truth metric
- `20.9` therefore improved the project's epistemic position, but did **not** finish the phase mission as a trustworthy live restoration proof

## What Changed
- retrieval tokenization now removes broad question-noise terms before overlap ranking
- transcript/global retrieval now ranks overlap ahead of `same_session` so session locality no longer dominates obviously better matches
- semantic retrieval ordering now prefers semantic score before `same_session`
- deterministic route resolution was hardened against false-positive temporal cues such as:
  - `last time`
  - `before getting`
  - `before buying`
- stronger temporal patterns were added for true temporal-reasoning questions such as:
  - `how many days ... between`
  - `months ago`
  - `days had passed`
- structural route selection no longer hard-replaces fact support:
  - structural rows are blended with bounded fact support when the route has real support
- temporal route support now requires stronger non-recent anchors instead of treating `recent` rows as if they were true temporal evidence

## Key Live Findings
- runtime staleness was not a one-off problem
  - it reappeared again at `20.9`
  - this is now a workflow / install-pipeline reliability issue, not an anecdotal operator mistake
- valid patched-runtime answer-only truth is materially better than the stale-runtime `2 / 15`
  - but still far below the older stronger historical reference points
- some targeted live improvements are real:
  - `c8c3f81d` now routes correctly as `fact` and passes on the patched runtime
  - `6c49646a` now routes correctly as `aggregate` and passes on the patched runtime
- some important failures remain clearly real on the valid runtime:
  - `5d3d2817` remains a profile / identity retrieval miss
  - `0db4c65d` can activate `temporal` while still getting zero temporal candidates
  - `gpt4_7f6b06db` can activate `temporal` with nonzero candidates but still receive bad evidence quality
  - `f523d9fe` remains consistent with false-positive temporal routing
  - this means the deterministic classifier's documented live misroute family is now at least one case larger than the original `20.8` audit set

## Interpretation
- `20.9` should not be read as a clean restore / fail binary
- what it really proved is:
  - the source-side fixes were real
  - stale runtime was materially distorting the benchmark picture
  - route activation and some candidate-path improvements are real on the patched runtime
  - but the current benchmark loop is too slow and too operationally fragile to remain the main inner-loop diagnostic tool
- the dominant carry-forward problem is now two-part:
  - runtime / deployment truth must be verified before any benchmark run
  - retrieval quality must be diagnosed with faster harnesses than a full `15`-question live split run

## Proven vs Not Claimed
### Proven

- this phase does not claim profile / identity retrieval is recovered
