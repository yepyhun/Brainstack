# Phase 20.8 Plan

## Goal

Restore route-hint reliability as a bounded, standalone problem.

`20.8` should determine whether Brainstack can keep the current LLM route-hint path by tightening the output contract, or whether that path should be replaced with a simpler deterministic classifier.

## Scope

### In

- raw route-hint output audit on the benchmark question set
- route-hint parser / contract analysis
- explicit fix-vs-replace decision
- implementation of the chosen route-hint path
- isolated route-hint validation
- bounded comparable live reruns to prove route activation is no longer effectively dead

### Out

- broad fact-path parity restoration
- profile / identity retrieval recovery
- graph candidate quality recovery
- proof/judge recalibration
- decomposition revival
- generic prompt / wording retuning outside the route-hint contract itself

## Planning Assumption

`20.7` already proved that the current live collapse is no longer best explained by missing memory injection or fake route parsing.

`20.8` should therefore isolate route-hint reliability as its own gate before `20.9` reopens broader fact-path and profile retrieval quality work.

## Execution Steps

1. Audit raw route-hint outputs first.
   - capture the raw route-hint responses for the benchmark questions
   - record parser outcome, fallback cause, and whether the raw output is:
     - empty
     - truncated / malformed structure
     - explanatory prose
     - mixed multi-mode text
     - valid machine-usable mode output
   - do not start by changing the prompt blindly

2. Make the fix-vs-replace decision explicitly.
   - evaluate whether the current LLM contract can realistically be tightened enough to return only:
     - `fact`
     - `temporal`
     - `aggregate`
     - `unknown`
   - evaluate whether a simpler deterministic classifier would be more reliable and lower-risk
   - write down which path is chosen and why the other path is rejected for this phase

3. Implement the chosen route-hint path.
   - if the phase keeps the LLM route-hint:
     - tighten the contract
     - keep the parser strict enough to avoid fake route activations
   - if the phase replaces the LLM route-hint:
     - use a bounded deterministic classifier
     - keep the mode set small and explicit
     - do not reintroduce a multilingual keyword-table anti-pattern

4. Validate route-hint behavior in isolation.
   - run the route-hint harness across the benchmark question set
   - show route-source distribution and non-fact activation counts
   - measure route activation separately from downstream channel yield
   - for activated `temporal` / `aggregate` routes, report whether the intended channel actually returns candidates instead of only falling through to fact behavior
   - name the remaining failure class if route activation still collapses to fallback

5. Run bounded comparable live reruns.
   - answer-only
   - split-proof
   - the output must separate:
     - route activation improvement
     - temporal / aggregate channel candidate yield
     - downstream retrieval quality
   - `20.8` is not allowed to hide behind answer-score noise if route activation still fails

## Implementation Rules

- Do not let this phase drift into fact-path parity work.
- Do not assume prompt tightening is the only acceptable answer.
- Do not silently coerce explanatory prose into a route choice.
- Do not introduce benchmark-specific query whitelists or multilingual alias tables.
- Do not declare success if route-hint remains effectively dead.
- Do not let the chosen route-hint path weaken donor-first boundaries:
  - simpler deterministic replacement is acceptable
  - larger local heuristic sprawl is not

## Output Shape Required From Phase 20.8

- a raw-output audit summary for the benchmark questions
- an explicit fix-vs-replace decision
- the chosen implementation path and rejected alternative
- isolated route-hint validation results
- route activation vs channel-yield summary for activated `temporal` / `aggregate` routes
- bounded comparable live rerun results
- explicit carry-forward note that `20.9` remains the named fact-path parity / profile retrieval phase
  - and expands to temporal-channel candidate-yield recovery if `20.8` shows that activated temporal routing still has empty temporal-channel yield

## Exit Gate

- the dominant route-hint failure mode is named explicitly
- one route-hint path is chosen and implemented cleanly
- route activation is no longer effectively `15 / 15` fallback under comparable validation, or the phase explicitly declares failure
- the phase distinguishes route activation from temporal / aggregate channel candidate yield
- if activated temporal routing still yields zero temporal candidates, the phase must record that as a carry-forward requirement for `20.9`
- the phase explains whether the blocker was:
  - contract design
  - model behavior
  - parser brittleness
  - or replacement necessity
- `20.9` remains protected as the next fact-path parity / profile retrieval phase
