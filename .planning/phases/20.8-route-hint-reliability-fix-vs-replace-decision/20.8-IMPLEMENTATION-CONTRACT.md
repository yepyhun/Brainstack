# Phase 20.8 Implementation Contract

## Purpose

This phase exists to make the route-hint path trustworthy enough to activate structural routing safely, or to replace it with a simpler bounded alternative if the current LLM path remains unreliable.

## Required Workstreams

### Workstream A. Raw Route-Hint Output Audit

This workstream exists because `20.7` showed total fallback under the strict parser, but the exact raw output failure pattern still needs to be named cleanly.

It must:

- capture the raw route-hint outputs on the benchmark question set
- inspect both content-bearing fields actually returned by the target runtime
- record parser outcome and fallback cause per question
- classify the dominant output pattern explicitly

It must not:

- skip straight to prompt edits
- summarize the problem vaguely as “the model was unreliable”

### Workstream B. Fix-Vs-Replace Decision

This workstream exists because repeated route-hint fixes have changed the failure shape without closing the reliability problem.

It must:

- evaluate whether the current LLM route-hint contract can be made strict enough to emit only:
  - `fact`
  - `temporal`
  - `aggregate`
  - `unknown`
- evaluate whether a simpler deterministic classifier is now the lower-risk architecture
- name the chosen path explicitly
- name the rejected path explicitly

It must not:

- assume prompt tightening is mandatory
- leave both paths half-open at the end of the phase

### Workstream C. Chosen Route-Hint Path Implementation

This workstream exists to turn the fix-vs-replace decision into one clean implementation path.

If the chosen path is **fix**, it must:

- tighten the output contract
- keep the parser strict enough to reject fake route activations
- avoid silent coercion of explanatory prose into a mode

If the chosen path is **replace**, it must:

- stay bounded and deterministic
- avoid multilingual keyword-table sprawl
- keep the route space explicit and small
- remain donor-first-consistent rather than becoming a new heuristic subsystem

This workstream must not:

- reintroduce the old decomposition gate
- reopen broad fact-path recovery work

### Workstream D. Isolated Route-Hint Validation

This workstream exists because `20.8` must prove route-hint behavior directly before broader retrieval conclusions are drawn from it.

It must:

- run an isolated route-hint harness over the benchmark question set
- report mode outputs, route-source distribution, and fallback counts
- distinguish route activation from downstream channel yield
- for any activated `temporal` or `aggregate` route, report whether that route's intended channel actually returns candidates
- state plainly whether structural routing is still effectively dead

It must not:

- use answer score as the primary proxy for route-hint correctness
- hide failure behind “some live answers still passed”

### Workstream E. Comparable Live Reruns

This workstream exists because route-hint reliability has to survive the real live benchmark path, not only an isolated harness.

It must:

- run bounded comparable live reruns:
  - answer-only
  - split-proof
- report route activation metrics alongside answer metrics
- report temporal / aggregate channel candidate yield alongside route activation when those routes are activated
- separate route activation progress from downstream retrieval quality

It must not:

- claim full restoration if route activation remains effectively collapsed
- use `20.8` to smuggle in unrelated retrieval tuning

## Protected Boundaries

### Scope Boundary

`20.8` is a narrow route-hint reliability phase.

It must not absorb:

- fact-path parity restoration
- profile / identity retrieval recovery
- graph candidate quality recovery

Those belong to `20.9` unless route-hint work reveals a stricter dependency.

### Heuristic Boundary

`20.8` may choose a deterministic replacement only if it stays bounded and architecturally clean.

It must not:

- grow into a multilingual keyword/alias table
- add benchmark-specific query exceptions
- revive decomposition or other old local overlays

### Donor Boundary

Replacing the route-hint LLM call with a simpler deterministic classifier is allowed if it is more reliable and easier to reason about.

That replacement is considered donor-first-consistent because routing should be the simplest reliable executive seam, not the most repeatedly patched component.

### Proof Boundary

`20.8` must compare against the finished `20.7` truth, not against older pre-hardening runs alone.

The phase may report answer metrics, but route activation reliability is the primary success test.

## Minimum Proof Required Before Calling Phase 20.8 Done

- the raw route-hint output class breakdown is recorded explicitly
- the phase names whether the problem was primarily:
  - contract
  - model behavior
  - parser brittleness
  - or replacement necessity
- one route-hint path is chosen and implemented
- comparable validation proves that route activation is no longer effectively `15 / 15` fallback, or the phase explicitly records that the chosen path failed
- comparable validation distinguishes route activation from temporal / aggregate channel candidate yield
- if activated temporal routing still yields zero temporal-channel candidates, the phase explicitly carries that gap into `20.9`
- the summary states clearly that `20.9` remains the named next phase for fact-path parity and profile / identity retrieval recovery
