# Phase 30.2 Plan

## goal

Close the three live residual classes proven in Discord UAT after `v1.0.4`:
- cross-session task recall failure after reset
- lossy canonical style-contract storage
- missing same-session correction-to-policy semantics

## scope

### in

- task continuity recall contract across reset without violating donor-first task ownership boundaries
- canonical scoped style-contract source precedence and raw-content fidelity
- behavior-policy projection semantics:
  - no-compile-drop vs no-projection-drop
  - hard-rule omission visibility
  - degraded ordinary-turn status when projection drops active hard rules
- pre-answer activation for plain-chat structured behavior-policy teaching
- same-session user correction semantics for behavior-policy strengthening or correction
- source, shim, `hermes-final`, and live runtime verification surfaces for all three residuals

### out

- turning Brainstack into a shadow task/reminder manager
- broad second-brain or KG redesign
- regex-based output post-processing as the primary fix
- unrelated prompt-style polishing outside these three residual classes

## core principles this phase must preserve

- donor-first
- zero heuristic sprawl
- single durable owner per truth class
- no shadow task owner
- no silent data loss
- fail-closed parity across source, shim, `hermes-final`, and live runtime
- token discipline

## architecture decision

### accepted target model

- treat the residuals as three separate contracts:
  - `task continuity recall`
  - `style-contract fidelity`
  - `same-session correction semantics`
- add a fourth, tightly related contract:
  - `projection truthfulness`
- preserve the current owner boundary unless explicitly changed:
  - reminders, scheduling, and task truth do not silently become Brainstack-owned durable profile truth
- require exact evidence for each class:
  - DB row source and content
  - retrieval path and query-language behavior
  - ordinary chat correction path vs explicit correction path
  - compile coverage vs actual ordinary-turn projection coverage

### rejected model

- “it is all one prompt bug”
- “Brainstack forgot to save anything”
- “just make tasks profile memory too”
- “just strengthen the prompt and ship it”

## workstreams

### 1. task continuity recall contract

- lock what the product contract should be when the user asks for tasks after reset
- prove whether reset recall must come from:
  - same-session recent continuity
  - cross-session continuity search
  - some other bounded continuity projection
- explicitly address the proven Hungarian-query / English-summary mismatch
- preserve the owner boundary if Brainstack remains context-only for tasks
- execute shape:
  - keep Brainstack non-owner for tasks in `30.2`
  - close the proven Hungarian recall miss through bounded bilingual cross-session continuity query expansion
  - leave first-class task / commitment truth ownership to `30.4`

### 2. canonical style-contract fidelity

- audit why the live scoped `style_contract` row can still be `tier2_llm`-derived and lossy
- define source-precedence rules so a deterministic scoped contract cannot be silently shadowed by a lossy Tier-2 write
- require lossless raw rule storage for exact recall
- execute shape:
  - deterministic `prefetch:style_contract`, `memory_write:style_contract`, `sync_turn:user_style_contract`, and explicit correction writes outrank `tier2_llm`
  - weaker Tier-2 style-contract writes may not silently overwrite a stronger deterministic canonical row

### 3. projection truthfulness and no-silent-drop semantics

- split compile truth from ordinary-turn projection truth
- make hard-rule omission visible and non-green
- define the active/degraded semantics when a compiled policy cannot fit all active hard rules into the ordinary projection
- require traces and snapshots to stop overclaiming “all active” when budget omission occurs
- execute shape:
  - keep compile coverage visible as its own truth surface
  - add distinct ordinary projection completeness semantics
  - mark ordinary-turn policy `degraded` when active rules are omitted from the bounded projection

### 4. plain-chat pre-answer activation and same-session correction semantics

- make structured plain-chat behavior teaching activate before the next reply
- do not rely on post-answer `sync_turn()` alone for structured long-form rule packs
- define what ordinary chat correction should mean for active policy state

- define what should happen when the user says a normal-chat correction such as:
  - `Amit írtál az első mondatban az nem em dashjel?`
- decide whether that should:
  - trigger explicit correction machinery
  - create bounded session-local reinforcement
  - or stay chat-only while the assistant stops promising stronger behavior than the system supports
- make this behavior explicit and inspectable
- execute shape:
  - structured plain-chat rule packs activate in `prefetch()` before the first post-teach answer
  - ordinary correction chat stays non-durable in `30.2`
  - but it may create bounded session-local reinforcement for the immediate reply when it clearly calls out an active policy drift

### 5. live closure criteria

- reproduce the Discord-style scenarios directly:
  - exact rule recall
  - dash-rule obedience after correction
  - task recall after reset
  - same-session task recall
- projection truth checks:
  - compile coverage equals raw rules
  - projection coverage is explicit
  - hard-rule omissions cannot hide behind green active status
- require live proof, not just packet or route-mode proof

## validation ladder

- inspect live DB rows and metadata, not only route outputs
- verify all four surfaces:
  - source
  - shim
  - `hermes-final`
  - live runtime
- verify task recall with Hungarian user queries, not only English probes
- verify canonical `style_contract` row source and raw content before calling fidelity fixed
- verify plain-chat structured teaching activates before the first post-teach answer
- verify projection truth and hard-rule omission semantics in snapshots and traces

## anti-goals

- do not hide residuals behind “policy active” trace output
- do not add a second silent owner for tasks
- do not verify only exact recall while ordinary obey still drifts
- do not accept a fix that only works for English continuity summaries
- do not call compile coverage “done” if ordinary-turn projection still drops active hard rules
