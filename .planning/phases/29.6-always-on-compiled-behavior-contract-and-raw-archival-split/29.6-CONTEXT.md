# Phase 29.6 Context

## problem statement

The current style-memory shape is still split in the wrong place for the user's real product goal.

The system can preserve a detailed archival contract and can sometimes recall it explicitly.

But ordinary-turn behavior still depends on a compact communication contract that is too lossy for a strict multi-rule user behavior policy.

## why this phase exists

The validated external review was right on the most important architectural point:

- explicit user behavior rules are not only "memory to recall later"
- they also need an always-on execution form

So Brainstack needs two first-class layers:

- a raw archival behavior contract for exact recall and auditing
- a compiled always-on behavior policy for ordinary turns

## accepted live evidence entering this phase

- the detailed canonical `preference:style_contract` lane is a valid archival owner
- ordinary turns intentionally exclude the long-form style contract
- Tier-2 delay and route/runtime fragility make the current behavior path too indirect
- the current compact communication contract is useful, but too small to carry all important nuance

## key architectural reading

This phase is not a prompt-policing phase.

It is a data-model and runtime-policy phase.

The core move is to stop treating all user behavior rules as if they were only retrievable memory rows.

## explicit non-goals

- no persona-file fallback
- no skill-file fallback
- no output regex enforcement
- no second shadow owner outside Brainstack
- no broad KG redesign in this phase
## recommended model level

`xhigh`
