# Phase 70 Plan

## purpose

Add a focused Brainstack-owned model-facing memory tool surface so Hermes agents can recall, inspect, and understand memory state first, with write-like tools gated behind the later explicit durable capture contract.

## core question

How can Brainstack gain Mnemosyne/CerebroCortex-level agent usability while preserving Brainstack shelf authority, provenance, principal scope, and the memory-kernel boundary?

## GSD planning enforcement check

This phase must preserve:

- donor-first: borrow provider/tool-surface patterns, not foreign architecture ownership
- modularity: add a Brainstack tool adapter seam instead of spreading tool logic across shelves
- non-governor boundary: no scheduler, executor, approval, messaging, or hidden runtime command tools
- no benchmaxing: tools must improve ordinary memory workflows, not a synthetic task
- zero heuristic sprawl: no phrase-farm live capture hidden inside tool handlers
- multilingual/multimodal by design: tool schemas must carry typed evidence fields, not English-only prompts
- inspectability before confidence: every tool call must be visible through Phase 65 inspect/doctor

## shared north star and real-world proof gate

North star for Phase 70-77: Brainstack must become easier and safer for Hermes agents to use as a memory kernel without expanding Brainstack into runtime execution, workflow governance, or fake autonomy.

Every phase closeout must prove one of these:

- Hermes can use Brainstack more reliably in a real memory workflow
- or the phase is prerequisite work and proves no regression while naming the exact later phase that must deliver the user-visible improvement

Minimum proof shape:

- targeted tests for the changed surface
- at least one end-to-end Hermes/Brainstack interaction path or Brainstack packet/inspect proof using real provider/store code
- evidence ID/source/provenance proof where memory is involved, not only generated answer text
- no-regression check against Phase 66 golden gates once they exist
- bounded token/output sanity for any new tool, MCP, inspect, corpus, or trace surface

## primary goals

- expose Brainstack memory tools through the provider boundary
- make memory recall/inspect/stats available to the model without raw DB access
- define a strict tool allowlist and risk split so model-callable tools cannot include operator-only or unsafe mutation surfaces
- keep write-like tool schemas disabled until Phase 72 owns durable capture semantics
- keep all tools typed, scoped, inspectable, and shelf-authority aware
- prove tool behavior through Phase 66 golden evidence gates

## explicit scope model

### Brainstack-owned scope

- tool schema export
- provider tool dispatch
- typed tool input validation
- typed shelf read delegation
- typed shelf write delegation only after Phase 72
- tool call trace/receipt output
- integration with Phase 65 inspect/doctor

### out of scope

- runtime execution tools
- approval tools
- scheduling tools
- agent-to-agent messaging
- broad MCP tool inventory
- direct transcript-to-truth promotion

## tool risk split

Tools must be classified before export:

- model-facing safe: bounded read/inspect/status tools safe for ordinary agent use
- model-facing gated: explicit memory writes only after Phase 72 semantics and receipts exist
- operator/debug-only: rebuild, backfill, maintenance dry-run/apply, migration, and diagnostic tools that must not be automatically model-callable
- forbidden: execution, scheduling, approval, messaging, uncontrolled file ingest, or hidden workflow coordination

Provider export and optional MCP export must preserve this split. A tool being available to an operator does not mean the model may call it in normal conversation.

- `brainstack_consolidate`, only as a bounded maintenance trigger after Phase 73

## architecture decisions to freeze before execution

### 0. read-only first decision

The first accepted execution cut for this phase is read-only:

- `brainstack_recall`
- `brainstack_inspect`
- `brainstack_stats`
- optional exported schema visibility for future write tools

Write-like tools (`remember`, `supersede`, `invalidate`) must remain disabled or non-executable until Phase 72 has landed the explicit durable capture and supersession contract. This avoids giving the model a write surface before durable write semantics are safe.

### 1. authority routing

Every write tool must declare or derive:

- shelf target
- principal scope
- authority class
- source/provenance
- evidence id
- validity/supersession intent

### 2. receipt contract

Every tool call must return a compact receipt with:

- accepted/rejected/no-op status
- record ids or evidence ids
- shelf touched
- authority class
- inspect trace id
- rejection reason when applicable

### 3. tool visibility

Doctor/inspect must report exported tool names, enabled/disabled state, and degraded reasons.

### 4. allowlist enforcement

The provider must have an explicit allowlist for model-callable tools. Unknown tools, disabled tools, operator-only tools, and forbidden tool classes must return typed refusal receipts rather than falling through to generic execution.

## workstreams

1. Inventory current provider tool export and runtime handoff tool behavior.
2. Define Brainstack memory tool schema contracts.
3. Add provider tool dispatch adapter with typed validation and allowlist enforcement.
4. Wire tools into shelves without bypassing store authority.
5. Add receipts and inspect/doctor trace output.
6. Add targeted tests and Phase 66-style golden cases.

## execution sequencing

1. Map current provider tool surface.
2. Implement read-only `inspect/stats` tools first.
3. Implement `recall` tool with no writes.
4. Export future write-tool schemas as disabled capabilities only if that helps operator visibility.
5. Implement executable `remember`, `supersede`, or `invalidate` only after Phase 72 has landed.
6. Gate maintenance trigger behind Phase 73 availability.

## deliverables

- provider tool schemas
- tool dispatch adapter
- typed memory tool contracts
- tool allowlist/risk classification
- tool call receipts
- inspect/doctor tool visibility
- tests for accepted, rejected, no-op, and degraded tool paths

## phase-specific real-world proof examples

Closeout must include at least:

- a Hermes/Brainstack provider proof where the model can call read-only recall/inspect/stats and receive bounded evidence-backed output
- a proof that disabled write-like tools cannot mutate durable memory before Phase 72
- a proof that tool output includes evidence IDs or explicit no-hit/degraded reasons
- a no-regression proof that ordinary Brainstack packet construction still works without using the tool surface

## acceptance criteria

- Brainstack exposes memory tools that are useful to the agent without runtime/governor leakage
- every tool result is inspectable and evidence-backed
- no tool can silently write unscoped durable truth
- operator/debug-only MCP tools are not model-callable in normal Hermes turns
- executable write/supersede/invalidate tools are absent or disabled until Phase 72 completes
- tool behavior passes targeted tests and relevant Phase 66 golden cases
