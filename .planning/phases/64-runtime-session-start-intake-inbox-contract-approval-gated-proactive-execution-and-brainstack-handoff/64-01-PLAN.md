# Phase 64 Plan

## purpose

Define the runtime-side complement to the Brainstack state machine so Hermes can behave more proactively without pretending that the memory kernel itself is a scheduler, executor, or infinite loop.

## core question

How should Hermes consume Brainstack-authored state and tasks at session start so the agent feels entity-like and proactive, while still staying approval-bounded, non-heuristic, and architecturally clean?

## proactive definition

For this phase, `proactive` means bounded initiative at session start or explicit wake time:

- inspect explicit typed pending work
- surface or execute what is already approved
- notify on blocked or unfamiliar work

It does **not** mean pretending there is a continuous hidden daemon mind doing arbitrary work between sessions.

## primary goals

- define a strict JSON inbox-task contract between Brainstack/state producers and the Hermes runtime
- define the session-start intake order for inbox, roadmap, evolver outputs, and current operating state
- define an approval-gated execution policy for unfamiliar or explicitly blocked domains
- define the runtime writeback contract so execution results become durable state instead of transcript residue
- keep the whole design useful beyond this single Bestie use case

## explicit scope model

### runtime-owned scope

- inbox task ingestion and ordering
- session-start recovery workflow
- approval-gated execution
- bounded proactive wake behavior
- execution result writeback hooks

### Brainstack-owned boundary

- state authority
- policy memory
- operating context
- current live-state recall
- durable task or operating records

### multimodal rule

The intake contract must stay compatible with multimodal future tasks. The runtime may consume task envelopes that point at files, wiki pages, logs, or other non-chat artifacts. The contract must not collapse into plain-text-only assumptions.

## anti-goals

- do not turn Brainstack into the scheduler or executor
- do not solve this by reintroducing transcript scraping or query heuristics
- do not make cron the primary intelligence layer
- do not build a fake “always on” daemon narrative when Hermes still runs session-bounded agents
- do not make this Bestie-only by baking in local folder names as the architectural truth

## accepted evidence

- Brainstack now has enough state authority to provide `live_system_state`, `operating_context`, typed task/operating memory, and local typed understanding
- the remaining gap is on the runtime side: how those durable structures are consumed at session start and turned into bounded proactive execution
- prior live evidence showed that narrative claims about evolver, pulse, or proactive behavior were unreliable when no explicit intake/execution contract existed

## architecture direction to freeze before execution

### 1. ownership decision

Brainstack produces or projects authoritative state. Hermes runtime consumes that state and decides whether to execute now, defer, or ask for approval.

### 2. inbox contract decision

Inbox items must be explicit JSON envelopes with typed fields. The runtime must not infer task type or risk class from loose natural language at intake time.

### 3. approval decision

Approval gating must be metadata-driven. The runtime should rely on explicit domain/risk/approval fields from the task envelope and durable policy state, not keyword farms over arbitrary text.

### 4. session-start decision

Session start should run a bounded recovery sequence before normal answering. That sequence should be deterministic, idempotent, and cheap enough to run every session.

### 5. writeback decision

Execution outcomes should write back into a typed state surface through a Brainstack-owned seam so the next session sees completed, blocked, in-progress, or failed work without having to reconstruct it from transcripts.

### 6. cron decision

Cron remains a wake or reminder mechanism. It may create or schedule explicit task envelopes, but it should not become the hidden owner of memory truth or arbitrary execution semantics.

## workstreams

### 1. contract freeze

Define the inbox task schema, required fields, optional fields, status fields, and provenance fields.

### 2. runtime intake design

Define the exact session-start sequence and ordering:

- current live state
- inbox tasks
- roadmap or execution queue
- evolver outputs
- response to the user

Freeze a proof target for this routine as bounded and cheap startup work, not an unbounded recovery crawl.

### 3. approval gate design

Define how the runtime decides:

- execute now
- block and ask user
- defer
- noop because already complete or stale

### 4. writeback design

Define the typed status or artifact writeback path so execution leaves reusable durable state.

### 5. proof design

Define the UAT and runtime proofs for:

- proactive feeling without fake autonomy
- blocked-domain safety
- session-start usefulness
- no transcript-only reconstruction
- bounded startup cost

## execution sequencing inside the phase

Do not split the roadmap unless implementation proves the scope is still too large. Execute this phase in two internal slices:

- `64a`: inbox contract freeze plus bounded session-start intake
- `64b`: approval gate plus typed writeback through the agreed Brainstack seam

## execute cut chosen

This phase was executed as the bounded `64a` slice first:

- runtime-side read-only policy consumer
- typed session-start inbox intake
- Brainstack-seam mirroring for typed handoff state

The broader runtime executor/governor behavior remains explicitly out of this cut unless later proven necessary as separate runtime work.
