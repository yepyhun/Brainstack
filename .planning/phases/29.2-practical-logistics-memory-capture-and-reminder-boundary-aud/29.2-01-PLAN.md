# phase 29.2 plan

## goal

Establish the correct Brainstack shelf mapping and reminder boundary for practical logistics memory, then define the smallest durable-capture repair for the classes that should not remain transcript-only.

## scope

### in

- stable address / provider facts
- appointment occurrences and scheduled times
- reminder handoff semantics
- short practical todo items
- current live storage-path audit across transcript, continuity, profile, and graph

### out

- communication-contract hardening
- broad personal profile expansion
- new planner / task-manager product work
- full reminder-system rewrite

## validation method

- one live audit of where the observed logistics facts land today
- one source-path audit of extraction candidates and current shelf ownership
- one class-by-class ownership map
- one bounded repair proposal only for classes that truly deserve durable storage
- one explicit no-persist verdict for ephemeral classes where that is the correct answer

## decision rules

- if a fact is stable, reusable, and likely to be queried later, transcript-only is not enough
- if a fact is one-off and operational, it may belong in temporal/reminder lanes rather than profile
- if a reminder system executes the scheduling, Brainstack still needs only the minimum durable truth necessary to answer later queries coherently
- if a class has no clear durable value, do not invent persistence for it
- if two subsystems would become co-owners of the same reminder truth, reject the design

## north star

Practical logistics memory should be explicit and boring:

- stable logistics facts become durable in the right shelf
- ephemeral instructions do not pollute durable memory
- reminder execution and durable memory truth have a clean boundary

## core principles this phase must preserve

1. donor-first
   - prefer existing Brainstack shelves and settled temporal/entity patterns before inventing a new logistics subsystem
2. modular, thin integration
   - keep any repair on the smallest correct Brainstack seam
   - do not spread logistics handling across Brainstack, Chron, host wrappers, and runtime glue without proof
3. no heuristic sprawl
   - do not create a guessy reminder or todo classifier full of special-case text rules
   - only bounded class-level rules with clear evidence are acceptable
4. single durable owner
   - Brainstack owns only the durable truth it truly should own
   - Chron must not become a hidden canonical memory store
   - Brainstack must not become a second scheduler
5. native-system synergy, not replacement
   - Chron keeps reminder creation, timing, and delivery
   - Brainstack complements Chron with durable truth only where later recall truly needs it
6. durable only where durable value is real
   - stable provider/place facts and maybe bounded appointment truth may deserve durability
   - same-day noise and transient errands must not be promoted by default

## anti-goals

- no "everything to profile"
- no "everything to continuity"
- no giant reminder-memory blob
- no second-brain bloat for same-day noise
- no host-side hidden state replacing Brainstack memory truth

## hypothesis -> falsification -> repair decision

1. stable-fact omission seam
   - hypothesis:
     - stable real-world facts such as provider/location details are currently omitted from durable storage even though they should survive
   - falsification:
     - prove they already land durably in a clean, queryable shelf
   - repair decision:
     - if not falsified, define the canonical durable shelf and capture path

2. appointment-boundary seam
   - hypothesis:
     - one-off appointments are not mapped to the right temporal or structured memory lane
   - falsification:
     - prove they already land in a durable temporal/typed structure with clean retrieval semantics
   - repair decision:
     - if not falsified, define the bounded temporal representation

3. reminder-handoff seam
   - hypothesis:
     - reminder scheduling currently succeeds operationally without leaving enough Brainstack truth for later recall
   - falsification:
     - prove the minimal reminder context is already durably reconstructable later
   - repair decision:
     - if not falsified, define the minimum durable handoff residue

4. transient-todo seam
   - hypothesis:
     - same-day short todos are being under- or over-persisted
   - falsification:
     - prove the current behavior already matches the intended product boundary
   - repair decision:
     - if not falsified, define an explicit non-durable or bounded-durable rule

## file-level execution order

1. live runtime DB audit
   - transcript / continuity / profile / graph
2. `brainstack/tier2_extractor.py`
   - what candidate types can exist today
3. temporal / typed-entity path
   - current representation options
4. retrieval path
   - can the current shelves answer future logistics queries coherently
5. reminder boundary
   - installer/runtime hooks only if the audit proves the boundary belongs there

## protected boundaries

### ownership boundary

- Brainstack owns durable memory truth
- reminder/cron owns scheduling execution

### native Chron boundary

- Chron owns:
  - reminder creation
  - reminder timing
  - reminder triggering
  - reminder delivery
- Brainstack may own:
  - stable provider/place truth
  - bounded appointment truth if later recall value is real
  - minimum reminder residue only if coherent later recall needs it
- Brainstack must not own:
  - full scheduler state
  - execution metadata as a shadow reminder database
  - every short-lived todo or same-day errand by default

### noise boundary

- do not let transient errands flood durable memory

### retrieval boundary

- durable storage must be queryable later without requiring raw transcript luck

## stop conditions before any patch

- stop if the phase starts assuming every class should become durable
- stop if reminder scheduling becomes a second hidden memory database
- stop if the only proposed fix is "store the whole user message"
- stop if the design cannot explain how later retrieval would use the new data

## regression net

- no breakage to current reminder scheduling
- no pollution of profile with one-off noise
- no conflict with communication-contract or identity hardening
- no second owner for reminder truth

## compatibility / donor-risk notes

- prefer existing Brainstack shelves:
  - profile
  - temporal events
  - typed entities
  - continuity
- avoid introducing a new shelf unless the audit proves the existing ones are structurally wrong for this class

## validation ladder

1. prove current live shelf placement
2. define class-by-class ownership
3. identify missing representations
4. choose bounded repair or explicit no-op per class
5. execute only the justified subset

## required working style

- no blind persistence
- no product fantasy
- no overengineering
- preserve the core principles above exactly, not approximately
- if the best answer is "only stable provider/place facts deserve durability now", stop there

## recommended model level

- `high`
- reason:
  - this is a boundary and ontology phase, not a trivial bugfix
  - it still has a bounded decision surface and should stay smaller than a general productivity-system rewrite
