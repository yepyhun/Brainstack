# phase 29.2 context

## title

practical logistics memory capture and reminder boundary audit

## problem

New live evidence shows that practical logistics facts are currently surviving only in transcript and continuity, not in a settled durable memory shape.

The observed examples include:

- a service-provider address
- an appointment time
- reminder instructions
- same-day short todo items

Today, these facts appear in:

- `transcript_entries`
- `continuity_events`

But they do not appear in:

- `profile_items`
- graph shelves

So the gap is not total forgetting.
The gap is missing shelf ownership and durable-shape policy for practical logistics memory.

## north star

Brainstack should know, for each practical logistics fact class, what the right owner is:

- stable contact/place fact
- scheduled appointment occurrence
- reminder handoff / notification intent
- transient short todo

The target is not to make everything durable.
The target is to make the ownership boundary explicit, correct, and auditable.

## why this matters

If everything stays only in transcript/continuity, useful real-world logistics memory remains weak and difficult to retrieve reliably later.

If everything is shoved into profile, the system becomes noisy and semantically wrong.

If reminder scheduling becomes a second hidden memory owner, the product gets harder to reason about.

## settled constraints

- Brainstack remains the owner of durable memory truth
- reminder/cron systems may own execution scheduling, but not become the hidden canonical memory store
- do not blindly promote every task/address/time into profile rows
- do not treat continuity-only capture as a complete solution when the fact class is clearly durable
- do not solve this with host-only hacks outside Brainstack unless the audit proves the boundary belongs there

## anti-goals

- do not stuff all logistics into `profile_items`
- do not leave all logistics in `continuity_events` just because it is easy
- do not create a second owner where reminder scheduling and Brainstack disagree
- do not merge this with communication-contract hardening
- do not introduce a broad planner/task system as part of this phase

## investigated findings

### live evidence

- the massage address conversation is present in `transcript_entries`
- the reminder setup conversation is present in `continuity_events`
- the same facts were not found in `profile_items`
- the same facts were not found in graph shelves

### current reading

This is not the same seam as Phase 29 or 29.1.

The stronger reading is:

- no settled durable ontology for practical logistics memory
- no explicit shelf policy for:
  - durable place/provider facts
  - one-off appointments
  - reminder handoff state
  - transient short todos

## likely candidate seams

- extraction seam:
  - these fact classes are not being recognized as durable candidates when they should be
- shelf-mapping seam:
  - there is no canonical target shelf for them yet
- reminder-boundary seam:
  - scheduled reminders may succeed operationally without feeding any durable memory truth back into Brainstack

## required outcome

This phase is complete only when the project can say, for each practical logistics class:

- should it be durable or not
- if durable, in which shelf
- if scheduled externally, what minimal Brainstack truth should still remain

## final verdict shape

The closeout for this phase should end in one explicit mapping:

- fact class
- canonical owner
- fallback owner
- what is intentionally not persisted
