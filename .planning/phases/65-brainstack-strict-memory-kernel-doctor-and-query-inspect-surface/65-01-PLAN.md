# Phase 65 Plan

## purpose

Make Brainstack prove its own memory-kernel state before any further retrieval improvement work. The kernel must be able to explain what is active, what is degraded, what was written, what was retrieved, what was suppressed, and why.

## core question

How can Brainstack expose strict read-only health and query-inspection evidence so a strict reviewer can distinguish real memory behavior from silent fallback, empty extraction, lexical-only recall, or packet-budget loss?

## GSD planning enforcement check

This phase must explicitly preserve the immutable rules:

- Brainstack is memory/state/policy authority, not scheduler, executor, approval governor, or hidden runtime replacement.
- No heuristic sprawl, cue farms, locale phrase lists, or benchmark-specific rescue logic.
- No bandaid patching: health must reveal root-cause state instead of masking it with optimistic status.
- Token discipline: inspect output must be bounded and structured, not a raw dump of every row.

## shared north star and real-world proof gate

North star for Phase 65-69: Brainstack must become a more reliable memory kernel in user-visible terms, not merely gain more internal machinery.

Every phase closeout must prove one of these:

- the memory kernel is measurably better for a real recall/state/policy scenario
- or the phase is pure prerequisite work and proves no regression while naming the exact later phase that must deliver the user-visible improvement

Minimum proof shape:

- targeted tests for the changed surface
- at least one end-to-end Brainstack packet or inspect proof using real store/retrieval code
- evidence ID/source/provenance proof where memory is involved, not only answer text
- no-regression check against Phase 66 golden gates once they exist
- bounded token/output sanity for any new inspect, doctor, or trace artifact

## accepted evidence

- backend capability can currently be requested but later degraded or disabled while the system still continues
- graph/corpus status exists in parts of the store, but there is no strict kernel-wide doctor that makes requested-vs-active capability impossible to miss
- query packet behavior is hard to audit because route decisions, selected evidence, dropped evidence, and final render shape are not exposed as one inspectable artifact
- later PR/PullPhase retrieval work would be unsafe if it can hide missing writes or inactive backends behind improved ranking

## primary goals

- add a strict Brainstack doctor surface for memory-kernel capability truth
- add a bounded query inspect surface for route/channel/evidence/packet traceability
- expose requested capability, active capability, degraded reason, row counts, last writes, and Tier-2 schedule/result state
- make degraded state explicit without changing runtime execution behavior
- create proof tests that fail if health claims are green while requested capabilities are inactive

## explicit scope model

### Brainstack-owned scope

- memory-store health aggregation
- backend requested-vs-active capability reporting
- Tier-2 schedule/result status reporting
- query inspect trace construction
- evidence selection and suppression trace reporting

### runtime-owned scope

- container restart behavior
- Discord response UX
- scheduler execution
- approval enforcement
- any automatic remediation after a health failure

## anti-goals

- do not fix recall quality in this phase
- do not add route or capture heuristics
- do not auto-enable Kuzu, Chroma, or Tier-2 as a side effect of doctor
- do not make doctor trigger writes, restarts, migrations, or execution
- do not produce unbounded debug dumps that blow context or tokens

## architecture decisions to freeze before execution

### 1. doctor contract

Define a stable structured health object that reports:

- configured backend request
- active backend state
- degraded/unavailable reason
- row counts per shelf
- last write timestamps per shelf
- last Tier-2 run/schedule/parse/write status
- strict verdict for requested-but-inactive capabilities

### 2. inspect contract

Define a stable query-inspect object that reports:

- query principal and scope
- requested route and applied route
- route resolution source
- search channels attempted
- selected evidence IDs and shelves
- suppressed or dropped candidate reasons
- final packet item counts and render budgets

### 3. strict vs permissive mode

Doctor may support permissive and strict modes, but strict mode must not allow requested capability degradation to appear healthy.

### 4. read-only guarantee

Doctor and inspect must be side-effect free. They may read current state and run retrieval planning, but they must not mutate memory, scheduler state, inbox state, or runtime policy.

## workstreams

### 1. current health surface inventory

Inventory existing store/provider/backend health methods and identify missing pieces:

- graph backend status
- corpus backend status
- Tier-2 state
- row counts
- last writes
- route/packet trace availability

### 2. doctor data model

Create the internal data contract and serialization shape for strict health reporting. Keep it typed and bounded.

### 3. query inspect trace model

Instrument the existing retrieval/control-plane packet path enough to produce a structured trace without changing selection behavior.

### 4. public read-only surface

Expose the doctor and inspect surface through the narrowest Brainstack-owned seam that fits current architecture, such as a provider tool, CLI helper, or module API.

### 5. proof suite

Add tests for:

- requested graph backend missing -> strict degraded verdict
- requested corpus backend missing -> strict degraded verdict
- healthy SQLite-only permitted configuration -> honest non-strict verdict
- query inspect returns route, channels, evidence IDs, and packet counts
- doctor/inspect produce no writes

## execution sequencing

1. Read current health and retrieval trace code.
2. Freeze data contracts in code/tests before wiring output.
3. Implement doctor aggregation.
4. Implement query inspect trace.
5. Add focused tests.
6. Run targeted Brainstack tests first, then broader memory-provider regression ring.

## deliverables

- strict memory-kernel doctor surface
- query inspect trace surface
- focused tests for degraded capability truth and read-only inspect behavior
- execution result document recording what doctor can and cannot prove

## lightweight process refinements

### terminology lock

Before implementation changes behavior, freeze the exact meanings of these terms in the doctor/inspect contract:

- `requested`
- `active`
- `degraded`
- `unavailable`
- `selected`
- `suppressed`
- `dropped`
- `no-op`
- `failed`

This is a short glossary only. Do not turn it into a new framework.

### phase handoff gate

Closeout must include a short handoff block:

- proven by Phase 65
- still red after Phase 65
- blocked next phase, if any
- safe next phase to start
- exact inspect/doctor artifacts later phases must consume

## acceptance criteria

- strict doctor cannot report success when requested graph/corpus/Tier-2 capabilities are inactive or failed
- inspect can explain why a known query used or did not use each memory shelf
- inspect output includes evidence IDs or explicit no-hit/no-selection reasons
- doctor/inspect are side-effect free
- no heuristic route/capture logic is added
- Brainstack remains an observer/authority, not an executor or governor
