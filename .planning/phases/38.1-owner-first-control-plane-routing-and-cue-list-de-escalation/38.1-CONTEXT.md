# Phase 38.1 Context

## problem statement

The current control plane still relies too centrally on cue lists for preference, continuation, temporal, and task-like packet shaping.

That makes the shell more heuristic than the project principles allow.

## why this is the correct next phase

Packet quality improved in Phase 36, but the routing shell still has a cue-first ceiling.

Before continuing graph work, the control plane should become more owner-first so later phases are not compensating for the wrong shell behavior.

## accepted sharpened reading

- the problem is not that the control plane has any fallback heuristics at all
- the problem is that cue lists still sit too close to the primary decision path
- the fix must demote them without destabilizing retrieval quality

## phase boundary

This phase is about:

- owner-first packet shaping
- cue-list de-escalation
- policy shaping based on stronger signals

This phase is not about:

- rewriting retrieval itself
- adding more parsers
- building a planner

## expected proof shape

The proof should show:

- preference queries still avoid starvation
- operating and task queries still shape correctly
- temporal and high-stakes surfaces remain safe
- cue lists are no longer the main packet-shaping driver
