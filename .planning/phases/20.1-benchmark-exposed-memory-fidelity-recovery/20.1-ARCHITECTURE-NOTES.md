# Phase 20.1 Architecture Notes

## Why this phase exists

Phase `20` already proved that the donor-first recovery track fixed the large architectural drift:

- `L1` is now an executive retrieval layer
- `L2` uses a donor-aligned graph backend seam
- `L3` uses a donor-aligned corpus backend seam
- the shell coordinates `SQLite`, `Kuzu`, and `Chroma` without collapsing

The remaining blocker is not “architecture still wrong”.
It is **memory fidelity** on the real benchmark path.

Phase `20.1` exists to close the concrete retrieval-quality gap that still kept the final-boss result flat at `3 / 15`.

## What this phase is not

- not another shell redesign
- not another donor-selection debate
- not benchmark-gaming
- not handwritten multilingual heuristic stuffing
- not a fallback to “good enough in real life”

## Benchmark-exposed root causes now recorded

### 1. Conversation history is still semantic-blind

The current semantic leg only searches corpus documents.
Continuity and transcript history are still retrieved lexically.

That means the benchmark can ask from conversation history, while the strongest semantic retrieval path is pointed somewhere else.

This is why questions like:

- “food delivery services” vs `Uber Eats` / `Grubhub`
- exact prior shopping / activity / trip details

can still miss even though the donor-aligned corpus backend exists.

### Required correction

- publish continuity summaries and/or transcript turns into the semantic backend too
- keep session / turn / date metadata attached
- make the semantic leg work on real conversational history, not only on corpus docs

This is the biggest corrective lever.

### 2. Temporal evidence is not rendered explicitly enough

Even when related conversational evidence is retrieved, the rendered prompt does not yet present temporal information in the clearest possible way for:

- “which happened first?”
- “how many days passed?”
- “what was the order?”

The store may know timestamps, but the model still needs readable temporal evidence in the prompt.

### Required correction

- retrieved conversational evidence must surface explicit session/date labels when available
- temporal questions must receive date-rich evidence, not only generic continuity snippets

This is a major lever, but it depends on the right records being retrieved first.

### 3. Legacy graph ingress still pollutes truth context

The benchmark prompt evidence showed graph rows such as:

- `role=defense mechanism`
- `role=community association`
- `role=must-see`

These are not legitimate graph truth entries.
They come from the legacy regex-style graph ingress path still being too permissive on raw user text.

This pollution hurts in two ways:

- it steals token budget from real facts
- it can actively mislead the model

### Required correction

- add a graph extraction quality gate before truth rows can enter the rendered graph context
- stop random sentence fragments from becoming graph truth
- if needed, narrow or disable the legacy regex ingress path rather than “cleaning it up” with more ad hoc heuristics

This is a real corrective lever, but smaller than fixing conversational semantic retrieval.

### 4. Fixed trimming and shallow evidence packing lose exact detail

The current rendering trims many evidence rows around the `220` character mark.
That can cut off:

- numbers
- names
- exact thresholds
- specific titles

So even a correctly retrieved turn can still lose the answer-bearing detail before it reaches the model.

This is especially dangerous for:

- points / rewards thresholds
- pace / time / distance values
- named shops, shows, or venues

### Required correction

- move from flat shallow clipping to adaptive evidence packing
- prefer fewer, better, longer snippets when retrieval confidence is higher
- preserve answer-bearing detail instead of spreading budget across too many mediocre snippets

This is a fidelity lever, not the first retrieval lever.

### 5. The prompt does not yet express evidence priority clearly enough

The current memory preamble tells the model to use recalled memory silently and not mention internal memory blocks.
It does **not** yet clearly tell the model what to do when:

- a recalled memory fact is specific
- the model also has generic world knowledge
- and the two do not match

That leaves room for the model to “correct” memory with generic priors even when the recalled memory is more relevant to the user-specific question.

### Required correction

- add an explicit evidence-priority rule
- but do **not** use a blind “always trust memory” instruction

The correct rule is narrower:

- specific recalled facts with clear provenance and no open conflict should outrank generic prior knowledge
- conflicted or low-confidence memory should **not** be treated as automatically superior

This is acceptable as a fidelity rule.
It must **not** become benchmark-gaming prompt bias.

### 6. Transcript evidence is too capped and too lossy

The current transcript path is still highly constrained:

- provider default `transcript_match_limit = 1`
- provider default `transcript_char_budget = 280`
- control-plane policy still keeps transcript contribution very narrow

That means the system often relies on continuity summaries instead of the original user wording.
But continuity summaries are lossy, and exact details often live in the raw turn text:

- numbers
- names
- thresholds
- exact titles

### Required correction

- raise transcript availability enough that raw answer-bearing detail can survive
- do not flood the prompt with transcript spam
- pair this with adaptive packing so the extra transcript room goes to the best exact-detail evidence, not to noise

This is a real corrective lever and should sit above pure char-budget tuning in priority.

## Corrective priority order

Phase `20.1` should treat the corrective levers in this order:

1. conversational semantic indexing into the semantic backend
2. temporal/date-rich rendering for retrieved conversational evidence
3. graph extraction quality gate / legacy ingress tightening
4. transcript evidence widening for exact-detail access
5. adaptive evidence packing and detail-preserving char budgets
6. explicit evidence-priority instruction in the prompt contract

Why this order:

- first the system must retrieve the right conversation evidence
- then it must make temporal relationships legible
- then it must stop graph noise from stealing context
- then it must make raw exact-detail evidence actually available
- then it must stop truncating useful detail from otherwise-correct hits
- finally it should clarify how specific recalled fact beats generic model prior when there is no conflict

## Hard rules for Phase 20.1

- no benchmark-only prompt hacks
- no multilingual regex growth as a substitute for donor strength
- no rollback of the donor-first `L1` / `L2` / `L3` architecture
- no pretending that corpus-only semantics solves conversation-history questions
- no pretending that junk graph truth is acceptable just because the graph backend itself is healthy
- no “more snippets” strategy unless it preserves answer-bearing details better than the current packing
- no blind “trust memory no matter what” prompt instruction

## Quality bar

Phase `20.1` is only successful if it produces a **materially stronger rerunnable Phase `20` verdict path**.

That means:

- real improvement on benchmark-exposed failure families
- not only nicer architecture notes
- not only nicer harnesses
- not only bounded real-world passes

The phase must improve:

- conversation-history retrieval fidelity
- temporal ordering fidelity
- exact-detail carry-through
- contamination resistance
- evidence-priority behavior when specific recalled fact conflicts with generic prior knowledge

without regressing the donor-first architecture.

## Confidence rule

The project may aim for a large benchmark jump, but `20.1` must **not** promise a specific final score in advance.

That means:

- a significant improvement target is legitimate
- donor fidelity restoration is the right direction
- but `14 / 15` or `15 / 15` must remain an aspirational outcome until the corrected mini-proof actually shows it

The phase should therefore optimize for:

- the right corrective levers
- clean attribution
- rerunnable evidence

not for prematurely claiming a number.
