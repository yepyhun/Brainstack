# Phase 15.1 Architecture Notes

Date: 2026-04-11

## Why This Phase Exists

Live Discord testing after Phase `15` showed that Brainstack storage and recall are materially better, but the host runtime still leaks personal preference capture into the wrong tool families.

This is not a Layer-2 graph problem.
This is not a `skill_manage` existence problem.
This is a host-side ownership and guidance problem.

## Confirmed Problem Family

- Brainstack-only mode already disables legacy `memory` and `session_search`.
- Brainstack-only mode also blocks some `skill_manage` personal-memory payloads.
- But the model can still be nudged toward:
  - `skill_view`
  - `skill_manage`
  - `read_file`
  - `write_file`
  when the user is really just stating a personal communication preference.

## Architectural Decision

- Do **not** amputate skills globally.
- Do **not** solve this by growing Tier-1 heuristics.
- Do **not** jump ahead to Phase `16`.

Instead:

1. tighten Brainstack-only host guidance
2. narrow personal-memory tool routing boundaries
3. keep procedural skill usage intact
4. re-run live smoke on real Discord behavior

## Likely Fix Surface

- host/runtime guidance around memory and skills
- Brainstack-only tool boundary helpers
- installer-carried host patch, not a live-only local hack
- doctor coverage for the new boundary if possible

## What Success Looks Like

- User style/profile statements go straight into Brainstack-owned memory behavior.
- The model stops trying to solve those by reading/writing ad hoc notes files.
- The model stops trying to inspect or mutate skills for personal-style capture.
- Procedural workflows can still use skills normally.
