# Phase 15.1 Context

## Why This Phase Starts Now

Phase `15` closed the bounded usefulness / telemetry layer, but live Discord testing immediately showed a different remaining problem:

- Brainstack can already store and recall the right personal facts much better than before
- yet the host runtime can still steer personal preference capture into the wrong tool families

So this is not an L2 redesign step and not a benchmark step.
It is a host/runtime correctness step that must be finished before Phase `16`.

## What Problem This Phase Solves

In Brainstack-only mode, the system already:

- disables legacy `memory`
- disables legacy `session_search`
- blocks some personal-memory `skill_manage` payloads

But that is still not enough.

Live evidence showed that personal style/profile requests can still drift into:

- `skill_view`
- `skill_manage`
- `read_file`
- `write_file`

That creates the wrong ownership behavior:

- personal identity/style memory belongs to Brainstack
- procedural workflows may still use skills
- generic notes/files should not become a side memory system for user preferences

## Architectural Constraints

- no new runtime
- no second memory owner
- no global skill amputation
- no growth of Tier-1 lexical heuristics
- no benchmark-driven changes
- no language-specific fix strategy that only works for English or Hungarian

## Long-Term Tradeoff Rule

The fix should prefer **ownership and routing boundaries** over content heuristics.

Good direction:

- tighten Brainstack-only host guidance
- narrow clearly wrong tool detours for personal-memory capture
- carry the fix through the installer and doctor
- keep procedural skill workflows alive

Bad direction:

- add more keyword rules for every language
- hide the bug with prompt fluff only
- turn Brainstack into a tool-heavy memory UX

## Multilingual Constraint

This system may be used in many languages.
So the Phase `15.1` design should be as language-neutral as possible.

That means:

- prefer boundary rules based on ownership and tool intent
- prefer explicit runtime guidance about where personal memory belongs
- if any detection is unavoidable, keep it minimal and tied to concrete tool targets or existing structured signals
- do not build a new pile of language-specific preference patterns
