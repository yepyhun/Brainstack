# Phase 10.2 Context

## Goal
Stop Brainstack from writing obvious junk into durable memory while preserving the existing single-provider architecture and leaving multilingual semantic extraction to the later Tier-2 path.

## Why This Phase Exists
- Practical Discord smoke exposed a real correctness gap, not just a missing future feature.
- `profile_items` currently contain real user preferences mixed with pasted technical analysis and quoted transcript/doc noise.
- The current graph/corpus underfeeding is real, but it should not be patched by growing language-specific regex coverage.
- The immediate requirement is to stop the bleed first, so later Phase 11/12 work starts from a cleaner ingest boundary.

## Live Bug Truth
- The profile extractor currently runs only on user messages, not on assistant messages.
- The current noise comes from user messages that contain pasted technical analysis, uploaded document wrappers, or mixed quoted transcript blobs.
- The current graph extractor is still too narrow for everyday multilingual use, but that is not the primary 10.2 fix target.
- Tier-0 hygiene must be language-neutral and must not become another benchmark-shaped heuristic pile.

## Architecture Decision
Add a **Brainstack-owned Tier-0 admission guardrail** in front of durable writes.

### Chosen Direction
- Keep the existing three-layer Brainstack storage model unchanged.
- Treat Tier-0 as an ingest gate, not as a new memory layer.
- Reuse the strong donor shape from:
  - `/home/lauratom/Asztal/ai/hermes-agent-port/agent/stable_memory_guardrails.py`
- Expand that guardrail for Brainstack-specific noise forms without adding language-specific semantic extraction.
- Leave multilingual meaning extraction to Phase 12.

### Rejected For This Phase
- Expanding Hungarian or other language-specific graph regexes.
- Introducing Tier-2 inference early just to clean up current noise.
- Turning the guardrail into a semantic classifier.
- Adding a second runtime or separate memory owner.

## Phase Boundary
This phase is only about **language-neutral durable-write hygiene**.

In scope:
- add a reusable admission-decision object and guardrail function for durable writes
- gate profile writes and any equivalent stable-memory writes through that guardrail
- preserve raw transcript and continuity evidence even when durable profile admission is denied
- prove live Discord usage no longer keeps writing obvious technical junk into the profile shelf

Out of scope:
- multilingual semantic extraction
- richer graph filling
- corpus ingestion improvements
- temporal supersession policy
- confidence-weighted Tier-2 reconciliation

## Required Tier-0 Guardrail Families

### 1. Markdown Table Filtering
Reject profile admission for text that looks like structured markdown tables or copied matrix-style analysis.

This includes patterns such as:
- multiple `|`-delimited cells
- repeated table header/separator structure
- matrix-like rows copied from planning or comparison output

Expected behavior:
- the text may still remain in raw transcript/continuity evidence
- it must not become a durable profile fact

### 2. Code Fence Filtering
Reject profile admission for code-heavy or fenced snippets.

This includes patterns such as:
- triple backticks
- long indented code-like blocks
- obvious command/script payloads

Expected behavior:
- code examples must never become durable preferences just because they contain words like `prefer`, `always`, or `never`

### 3. Quoted Transcript / Dump Filtering
Reject profile admission for mixed transcript dumps that embed role-prefixed conversation history.

This includes patterns such as:
- `User:` / `Assistant:` line prefixes
- multi-turn pasted conversation text
- quoted chat-dump style content

Expected behavior:
- direct user self-statements are still allowed
- pasted conversation transcripts are not promoted into stable profile memory

### 4. Uploaded Document Wrapper Filtering
Reject profile admission for Hermes/Gateway wrapper text around uploaded files or cached documents.

This includes patterns such as:
- `[The user sent a text document: ...]`
- `[Content of ...]`
- `/opt/data/cache/documents/...`
- similar wrapper scaffolding around file contents

Expected behavior:
- the uploaded document can still live in transcript/corpus paths later
- the wrapper blob must not be interpreted as the user's personal preference

### 5. Agent Analysis / Technical Blob Filtering
Reject profile admission for copied planning, audit, or diagnostic analysis that is clearly not a direct stable user fact.

This includes patterns such as:
- phase-plan style sections
- requirement/traceability blocks
- comparative technical analysis
- noisy shell/stacktrace-like diagnostic text

Expected behavior:
- “the user copied an analysis blob” stays transcript evidence only
- it does not become durable identity/preference/shared-work memory

## Required Precision Rules
- Do not block short, direct, first-person self-statements just because they are technical.
- Do not turn Tier-0 into a semantic intent engine.
- Prefer conservative rejection of obvious noise over clever guessing.
- Every rejection path must expose a concrete reason code, not a vague boolean.
- The guardrail must remain small enough to stay reviewable and stable.

## Existing Donor Fit
`stable_memory_guardrails.py` is not the whole solution, but its shape is correct:
- `StableMemoryAdmissionDecision`
- explicit `allowed` plus `reason`
- small deterministic checks

Brainstack should adopt that pattern and extend it only for its known ingest-noise forms.

## Anti-Half-Wire Requirements
- The guardrail must be used by the live Brainstack provider path, not only by tests.
- The source Brainstack repo and the installed Bestie runtime must stay aligned after reinstall.
- A passing unit test alone is not enough; live DB evidence must show that new junk rows stop appearing.
