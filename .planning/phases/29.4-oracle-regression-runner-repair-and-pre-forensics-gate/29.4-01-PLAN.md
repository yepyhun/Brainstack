# Phase 29.4 Plan

## goal

Repair the bounded oracle regression runner so it can answer one precise question before any further recall debugging:

- did the recent `29.1` / `29.2` / `29.3` work regress earlier Brainstack proof behavior?

## scope

### in

- `scripts/run_brainstack_longmemeval_subset.py`
- only the runner and proof-harness seams needed to restore the bounded oracle regression path
- one bounded rerun of the old oracle retrieval-only regression check
- one explicit verdict on whether recent work regressed prior proof
- one explicit handoff note for the next recall-forensics phase

### out

- no new Brainstack product memory features
- no new 27-rule recall fixes
- no prompt-behavior tuning
- no broad benchmark reruns
- no donor reshaping

## north star

The next recall investigation must start from a trustworthy baseline.

That means the project must first regain a bounded oracle probe that is:

- runnable
- auditable
- not blocked by harness-only wiring mistakes
- and specific enough to tell product regression from runner regression

## core principles this phase must preserve

1. donor-first
   - do not turn the oracle runner into a separate product logic fork
   - preserve Brainstack and Core2 as the real systems under test

2. modular, thin integration
   - keep changes inside the benchmark/oracle harness seam
   - do not smear benchmark-only conditions into product runtime code

3. no heuristic sprawl
   - no new retrieval tricks
   - no benchmark-row-specific hacks
   - no answer-shaping shortcuts just to make the oracle run green

4. correctness-first
   - the first job is to restore a truthful regression gate
   - not to improve the product score

5. bounded proof, not benchmaxing
   - use one bounded `oracle 60` retrieval-only rerun
   - do not expand into a broad benchmark sweep

6. honest attribution
   - the final verdict must separate:
     - runner failure
     - product regression
     - unchanged product with broken harness

## architecture decision

### accepted target model

The oracle runner must support a bounded retrieval-only proof path that:

- does not require remote generation/judge wiring when the proof mode does not need it
- does not force API-key gating merely because other runner branches do
- still seeds and measures the same Brainstack retrieval surfaces that earlier proof relied on

### rejected model

- patching Brainstack product code just to satisfy the old runner
- faking oracle success with cached artifacts
- skipping the rerun and guessing from recent local tests

## oracle 60 protocol

The required regression gate is:

1. use the old LongMemEval subset runner
2. run `--oracle-seed --retrieval-only`
3. use bounded sample size around `60`
4. record:
   - runtime sync result
   - memory context presence
   - backend population nonzero
   - zero-backend cases
   - empty-text batches
   - route families by question class
5. compare the new result against the previously recorded `70`-item oracle spillover notes, without pretending they are identical runs

This gate is passed only if the run itself is trustworthy and the comparison can be interpreted honestly.

## file-level execution order

1. `scripts/run_brainstack_longmemeval_subset.py`
   - remove runner-global API-key enforcement from retrieval-only oracle mode if that mode should not need it
   - isolate which branch truly requires remote providers

2. any directly touched helper in the same runner path
   - only if needed to make retrieval-only oracle mode runnable again

3. no product file changes unless the runner proves they are truly required for parity

## workstreams

### 1. runner gating audit

- identify exactly where retrieval-only oracle mode is incorrectly forced through remote API-key gating
- document whether this is:
  - CLI validation
  - seed path design
  - route resolver injection
  - direct Tier-2 extraction branch

### 2. bounded runner repair

- make the smallest architecturally correct harness change so retrieval-only oracle mode can run again
- keep answer-only / full generation branches behaviorally intact

### 3. oracle regression rerun

- run the bounded `oracle 60` retrieval-only check
- capture result artifact
- classify the outcome:
  - no regression
  - partial regression
  - major regression

### 4. verdict and next-thread handoff

- write a precise handoff for the next phase:
  - if regression exists, localize which product seam must be reopened
  - if no regression exists, open the next exact recall-forensics phase on the live 27-rule behavior

## validation ladder

1. local runner tests or targeted script checks pass
2. retrieval-only oracle mode runs without false API-key gating
3. bounded `oracle 60` regression artifact is produced
4. result is compared against prior recorded oracle notes
5. a final written verdict attributes the problem correctly

## anti-goals

- do not "fix" the benchmark by weakening what it measures
- do not sneak recall/product changes into a runner phase
- do not treat a non-runnable runner as evidence of product failure
- do not overfit the runner to one 27-rule scenario
