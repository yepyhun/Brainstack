# Phase 29.4 Implementation Contract

## phase type

Correctness-first benchmark/oracle harness repair phase.

This is a gate-restoration phase, not a product-feature phase.

## required final architecture

The final state must have:

1. one bounded oracle regression path that is runnable again
2. no product-code contamination from benchmark-only fixes
3. one explicit artifact proving whether recent Brainstack work preserved earlier proof behavior

## minimum evidence required before calling the phase done

1. a concrete explanation of why retrieval-only oracle mode was blocked
2. the exact runner change that restored it
3. one bounded `oracle 60` retrieval-only regression report
4. one written comparison against the old recorded oracle proof notes
5. one explicit verdict:
   - regressed
   - not regressed
   - inconclusive because of still-unclosed harness noise

## explicitly disallowed outcomes

- "fixed" because the script now accepts an API key quietly but the underlying mode is still wrong
- "passed" because a different benchmark mode ran instead
- changing Brainstack product behavior before the oracle gate is restored
- broad benchmark reruns used as a substitute for the bounded proof

## cleanup requirement

If any temporary runner workaround is introduced during diagnosis, it must be removed before phase close.

No dead code, no alternate hidden path, no debug-only permanent branch.

## compatibility requirement

The repaired runner must preserve:

- full-generation benchmark paths
- answer-only benchmark paths
- retrieval-only oracle paths

The retrieval-only fix must not silently damage the other two.

## regression-audit requirement

The phase is incomplete unless it explicitly answers:

- which recent changes were genuinely helpful
- which recent changes were irrelevant to the failing oracle gate
- whether any recent change measurably harmed prior proof behavior

This must be written down, not implied.

## code hygiene requirement

- no benchmark-row-specific hacks
- no cue hacks
- no oracle-shaped product drift
- no hidden dependency on secrets for a mode that should be local and bounded

## handoff requirement

The phase must end with one of two next steps only:

1. reopen product-path regression work on a precisely named seam
2. open the next exact recall-forensics phase for the 27-rule behavior

No vague "investigate more" closeout is acceptable.

## recommended model level

`xhigh`
