# Phase 29.4 Context

## problem statement

The current Brainstack product-path issue is still unresolved:

- after reset, Bestie can acknowledge that a `27`-rule communication contract exists
- but it cannot reliably recall the detailed rules from Brainstack durably and correctly

Before any further product-path fix work, we need to answer a different question cleanly:

- did the recent `29.1` / `29.2` / `29.3` changes break any earlier bounded oracle proof path?

Right now that question cannot be answered honestly, because the old oracle runner is itself not trustworthy:

- `scripts/run_brainstack_longmemeval_subset.py` currently forces an API key even in retrieval-only oracle mode
- the retrieval-only oracle seed path still routes through remote Tier-2 flush wiring
- that means the old regression harness can fail before it measures the thing we actually want to check

So the first required move is not another Brainstack memory patch.

It is to repair the oracle regression harness so it can serve as a real pre-forensics gate again.

## why this phase exists

If we skip this gate, the next recall investigation will be ambiguous:

- we will not know whether the product regressed
- we will not know whether the benchmark harness regressed
- and we risk treating runner noise as product truth

That would violate the project doctrine:

- correctness first when kernel behavior is in question
- live behavior has priority over benchmark shaping
- but benchmark/oracle regressions still need a trustworthy calibration probe

## current evidence

Verified locally:

- the current `29.3` style-contract lane now has a canonical durable row
- live runtime checks show the style row can be written and retrieved explicitly
- but the user-facing 27-rule recall behavior is still not correct end-to-end

Verified on the runner path:

- the old bounded oracle regression script still demands `COMET_API_KEY` / `COMETAPI_API_KEY`
- retrieval-only oracle seed is not treated as a no-key local proof mode
- therefore the requested `oracle 60` regression check cannot yet run as a clean 0-step gate

## intended phase outcome

This phase is complete only if:

1. the bounded oracle regression path is runnable again
2. the regression result is recorded honestly
3. we know whether recent Brainstack work preserved or damaged earlier proof
4. only then do we open the next exact forensics phase for the 27-rule recall failure

## this phase is not

- not another Humanizer/style architecture redesign
- not another prompt-behavior tweak
- not another recall fix guess
- not benchmark-maxing
- not broad benchmark reruns

It is a trust-restoration phase for the proof harness.
