## Important Technical Correction

During Docker rebuild, the runtime failed because the build context included mutable runtime files under `hermes-config/` with restrictive ownership from the running container.

This was corrected by excluding runtime/config/state artifacts from the Docker build context in:

- `/home/lauratom/Asztal/ai/memory-repo-bakeoff/hermes-agent-bestie-latest/.dockerignore`

This is the right fix:

- runtime config should be mounted into the container
- runtime state should not be baked into the image
- secrets and mutable state should stay out of the image build context

The Docker healthcheck also needed correction:

- the original check used `pgrep`, but the image did not contain `pgrep`/`ps`
- this produced a false `unhealthy` status even while Hermes Gateway was running
- the compose healthcheck was updated to inspect `/proc/1/cmdline` for `hermes gateway run --replace`

## Practical Result

The Docker/Discord Bestie runtime is no longer just “fresh Hermes gateway”; it now contains Brainstack as the selected external memory provider.

That means the next real smoke step can legitimately test Brainstack behavior in Discord instead of accidentally testing a Brainstack-free runtime.

## Remaining Caveat

This phase does **not** prove every future upstream Hermes release will remain compatible automatically.

What it does prove is:

- the update workflow is now repeatable
- compatibility is checked explicitly
- the install path fails closed instead of silently producing a half-wired runtime
