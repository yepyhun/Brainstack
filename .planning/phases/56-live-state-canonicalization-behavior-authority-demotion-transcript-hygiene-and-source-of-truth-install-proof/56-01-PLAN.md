# Phase 56 Plan

## goal

Fix the remaining inspector-blocking defects by making the deployed state match the intended explicit-truth contract, removing active behavior-authority residue from native explicit rule packs, preventing internal runtime status text from entering transcript memory, and proving the result from the `Brainstack-phase50` source-of-truth repo through wizard install onto a fresh Hermes checkout and the final `finafina` runtime.

This phase is not a new governor phase.
This phase is not a new extraction-policy phase.
This phase is a source-of-truth recovery, deployment-proof, and live-state repair phase.
This phase is not just dirty-state cleanup; it is authority-lane cleanup as well.

## scope

### in

- `Brainstack-phase50` as the sole code source of truth for all fixes
- wizard/install path correctness for a fresh Hermes checkout
- deployed-state canonicalization for native explicit user/profile truth
- removal or demotion of explicit-rule-pack `behavior_contracts` / `compiled_behavior_policies`
- transcript hygiene so internal status/runtime lines do not persist as assistant memory
- end-to-end proof that the installed runtime in `finafina` comes from the source-of-truth repo rather than ad hoc local edits
- installed runtime / provider-path proof after the canonicalization and residue cleanup
- real Discord UI proof on the installed runtime after the canonicalization and residue cleanup

### out

- hand-maintaining two divergent implementations in `Brainstack-phase50` and `finafina`
- new behavior or style governor logic
- locale-specific parsing, regex farms, or user-specific special casing
- prompt-only masking of degraded stored truth
- fresh-state-only proof while long-lived deployed state remains dirty
- editing `finafina` as if it were the source repo instead of a deployment target
- understating the issue as “just dirty state” when active authority residue still exists

## workstreams

### 1. Live-state defect audit and canonicalization plan

- compare the intended Phase 54/55 explicit-truth contract with the actual deployed state:
  - `USER.md`
  - `USER_PROFILE_INDEX.json`
  - Brainstack `profile_items`
  - Brainstack `behavior_contracts`
  - Brainstack `compiled_behavior_policies`
  - transcript storage
- classify which live artifacts are:
  - canonical
  - degradations of canonical truth
  - authority residue
  - internal-status contamination
- implement a bounded canonicalization path that repairs existing deployed state instead of proving only on fresh temp homes

### 2. Explicit-pack authority cleanup

- stop native explicit rule packs from producing active Brainstack behavior authority artifacts
- keep only owner-correct artifacts:
  - native explicit truth in Hermes
  - bounded Brainstack mirror and/or archival truth where justified
- ensure explicit packs do not become:
  - active `behavior_contracts`
  - active `compiled_behavior_policies`
  - ordinary-turn governor projections
- keep the fix generic:
  - no rule-pack-specific logic
  - no language-specific slot taxonomy

### 3. Transcript hygiene and internal-status exclusion

- trace how internal runtime status strings can become assistant transcript entries
- block or filter internal runtime/status surfaces before transcript persistence
- prove that strings like:
  - `Operation interrupted: waiting for model response ...`
  - other internal lifecycle/status surfaces
  do not persist into transcript memory or later recall
- keep this fail-closed:
  - internal status must be excluded from durable transcript memory, not merely hidden from the user

### 4. Source-of-truth wizard install proof

- audit the wizard/install path in `Brainstack-phase50`
- ensure a fresh Hermes checkout can be brought to the corrected state by install/copy from source of truth
- prove the exact files that must land in the target runtime are provided by the source repo
- eliminate any manual runtime-only fix that the wizard cannot reproduce

### 5. Final installed-runtime proof

- deploy from `Brainstack-phase50` into `finafina` by copy/install, not manual duplicate implementation
- verify the installed runtime state is canonical
- verify the Brainstack DB no longer contains active explicit-rule-pack behavior authority residue
- verify transcript memory excludes internal runtime status text
- run installed-runtime / provider-path proof against the installed runtime
- run separate real Discord UI proof against the same installed runtime
