# Phase 66 Plan

## purpose

Define and build the golden proof harness for Brainstack memory behavior: write evidence, recall evidence, prove the returned evidence ID/source/provenance, and record current gaps honestly.

## core question

How do we prove that Brainstack memory works across profile, task, operating, graph, corpus, temporal, and cross-session scenarios without relying on answer vibes, live provider luck, or hidden transcript fallback?

## GSD planning enforcement check

This phase must preserve:

- source-of-truth discipline: tests must assert durable Brainstack evidence, not only assistant text
- no heuristic sprawl: golden cases must not justify hardcoded query phrases or benchmark hacks
- no governor boundary violation: tests verify memory behavior only, not runtime execution
- truth-first closeout: current failures must be reported as gaps, not hidden by loosened assertions

## shared north star and real-world proof gate

North star for Phase 65-69: Brainstack must become a more reliable memory kernel in user-visible terms, not merely gain more internal machinery.

Every phase closeout must prove one of these:

- the memory kernel is measurably better for a real recall/state/policy scenario
- or the phase is pure prerequisite work and proves no regression while naming the exact later phase that must deliver the user-visible improvement

Minimum proof shape:

- targeted tests for the changed surface
- at least one end-to-end Brainstack packet or inspect proof using real store/retrieval code
- evidence ID/source/provenance proof where memory is involved, not only answer text
- no-regression check against Phase 66 golden gates once they exist
- bounded token/output sanity for any new inspect, doctor, or trace artifact

## accepted evidence

- simple identity recall can currently succeed through continuity/transcript evidence while profile recall remains under-proven
- paraphrase recall can fail even when durable profile evidence exists
- existing unit tests protect invariants, but they do not fully prove user-visible write-to-recall quality with evidence IDs
- later semantic index and graph recall phases need a stable target suite to prevent regression and benchmark gaming

## primary goals

- create a deterministic golden memory eval harness
- cover write-to-recall scenarios across all durable shelves
- assert evidence IDs, source shelves, principal scope, provenance, and authority class
- separate baseline observations from enforced regression gates
- make future recall changes measurable before they are merged

## explicit scope model

### Brainstack-owned scope

- deterministic store setup
- typed fixture writes
- query packet or inspect execution
- evidence attribution assertions
- baseline/gate reporting

### out of scope

- improving recall ranking
- adding semantic index behavior
- changing Tier-2 extraction semantics
- changing Hermes runtime startup or approval behavior
- using live LLM/provider calls as required test dependencies

## anti-goals

- no answer-string-only tests
- no tests that pass when the correct evidence was not retrieved
- no query-specific rescue code added to make a golden case pass
- no hidden reliance on live Discord transcript state
- no broad benchmark suite before the kernel can explain individual cases

## golden scenario set

The first suite must cover at least:

- profile identity: write name/preference, recall by direct and paraphrased query
- profile supersession: newer preference or identity state wins over stale state
- task memory: explicit task write, active task recall, completed/expired task suppression
- operating state: active work/current state recall with provenance
- graph state/relation: entity/state/relation write, direct and paraphrased recall
- corpus/document: bounded document ingest, recall with document ID and section/source metadata
- temporal validity: stale temporal truth does not surface as current truth without grounding
- cross-session continuity: durable evidence survives a new store/session boundary
- negative memory: unsupported question does not invent memory-only truth

## architecture decisions to freeze before execution

### 1. fixture strategy

Use deterministic typed writes and fake Tier-2 seams where needed. Do not require external provider calls for the core suite.

### 2. assertion model

Every positive case must assert:

- retrieved evidence ID
- source shelf
- principal/session scope
- provenance class
- authority/currentness class where applicable

### 3. baseline vs gate

Cases that represent known current defects may initially be recorded as baseline failures or expected gaps. Once a later phase fixes a case, it becomes a non-negotiable regression gate.

Each expected-red case must carry an owning follow-up phase:

- Phase 67 for typed semantic/paraphrase recall gaps
- Phase 68 for Tier-2 scheduling, parse, promotion, or write-count gaps
- Phase 69 for graph recall mode, alias, state, or relation gaps

No expected-red case may remain ownerless.

### 4. integration with Phase 65

Use Phase 65 doctor/inspect outputs when available so failures explain whether the issue is write absence, retrieval absence, route/channel suppression, or packet-budget loss.

## workstreams

### 1. scenario specification

Write the scenario matrix with inputs, durable writes, queries, expected evidence, and expected current behavior.

### 2. harness utilities

Build reusable helpers for temporary Brainstack stores, principals, sessions, typed writes, fake Tier-2 outputs, and query inspect execution.

### 3. evidence assertions

Add assertion helpers that fail when answer text is plausible but evidence attribution is missing or wrong.

### 4. baseline report

Run the suite against current behavior and record known failures without weakening the suite semantics.

### 5. regression gate split

Mark currently supported behavior as hard gates and leave known structural gaps tied to Phase 67-69 remediation.

## execution sequencing

1. Create the scenario matrix before writing test code.
2. Add harness utilities.
3. Implement positive and negative cases shelf by shelf.
4. Run baseline and record current gaps.
5. Convert supported cases into hard regression tests.
6. Link each remaining expected gap to Phase 67, 68, or 69.

## deliverables

- golden recall scenario matrix
- deterministic test harness
- positive and negative evidence-attribution tests
- baseline failure/gap report
- regression gate split for currently supported cases

## lightweight process refinements

### expected-red expiry

The baseline report must include a compact table of expected-red cases with:

- case ID
- failing behavior
- owning phase
- reason it is not gated yet
- condition that promotes it to a hard regression gate

This is a planning guard only. Do not build a separate lifecycle system around it.

### phase handoff gate

Closeout must state:

- which cases are hard green gates now
- which cases remain expected-red
- which owning phase is responsible for each expected-red case
- whether Phase 67 can start safely

## acceptance criteria

- golden cases prove evidence attribution, not just generated answer text
- suite runs without mandatory external LLM/provider calls
- current gaps are visible and mapped to later phases
- no code-level heuristic is added to make a case pass
- the suite gives Phase 67-69 concrete measurable targets
