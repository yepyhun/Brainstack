# Phase 76 Plan

## purpose

Turn large knowledge storage into a product-grade Brainstack corpus subsystem with stable source adapters, sectioning, hashing, idempotent re-ingest, citations, and bounded recall across local knowledge bodies.

## core question

How can Brainstack make large knowledge storage practical without creating prompt-injection shortcuts, raw file stuffing, duplicate documents, or a new source-of-truth owner?

## GSD planning enforcement check

This phase must preserve:

- usable storage of large bodies of knowledge
- meaningful token savings through bounded corpus recall
- donor-first source adapters instead of one-off file hacks
- no bandaid patching: wiki/environment notes must become normal corpus sources
- inspectability: corpus ingest and recall must expose document ids, hashes, sections, and citations

## shared north star and real-world proof gate

North star for Phase 70-77: Brainstack must become easier and safer for Hermes agents to use as a memory kernel without expanding Brainstack into runtime execution, workflow governance, or fake autonomy.

Every phase closeout must prove one of these:

- Hermes can use Brainstack more reliably in a real memory workflow
- or the phase is prerequisite work and proves no regression while naming the exact later phase that must deliver the user-visible improvement

Minimum proof shape:

- targeted tests for the changed surface
- at least one end-to-end Hermes/Brainstack interaction path or Brainstack packet/inspect proof using real provider/store code
- evidence ID/source/provenance proof where memory is involved, not only generated answer text
- no-regression check against Phase 66 golden gates once they exist
- bounded token/output sanity for any new tool, MCP, inspect, corpus, or trace surface

## architecture decisions to freeze before execution

### 1. source adapter contract

Each source must provide:

- stable source key
- document key
- title/path/page identity
- content hash
- sections/chunks
- provenance and owner

### 2. citation contract

Corpus recall must be able to cite document id, section id, and source metadata.

### 3. re-ingest rule

Re-ingest must be idempotent and must update/delete stale sections without duplicate spam.

### 4. corpus fingerprint and drift rule

Corpus documents and derived search/index rows must record:

- source adapter version
- parser/normalizer version
- sectioner/chunker version
- embedder identity/config
- corpus schema version
- source content hash

If any relevant fingerprint changes, doctor/inspect must report the corpus index as stale or needing rebuild. Stale corpus hits must not be presented as fresh authoritative evidence.

## phase-specific real-world proof examples

Closeout must include at least:

- ingest and re-ingest proof for the same local knowledge source with no duplicate document spam
- stale document replacement proof where old sections stop winning after update/delete/reindex
- fingerprint drift proof where changed parser/embedder/sectioner state marks the derived corpus index stale instead of green
- corpus recall proof with citation/evidence ID/section metadata
- token proof that raw documents do not enter the prompt unless selected by bounded retrieval

## acceptance criteria

- corpus documents can be ingested and re-ingested without duplicates
- recall returns evidence/citation ids
- raw documents do not enter prompts unless bounded retrieval selects them
- stale documents can be replaced safely
- stale corpus indexes are explicit and cannot silently masquerade as fresh recall
