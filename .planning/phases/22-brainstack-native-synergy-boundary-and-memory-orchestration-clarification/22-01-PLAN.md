# Phase 22 Plan

## Goal

Decide and, where justified, repair the Brainstack/native Hermes capability boundary so that:

- Brainstack owns the durable personal-memory axis
- native Hermes capabilities remain where they provide better explicit utility
- the runtime layering is clear enough that future work does not create duplicate ownership or avoidable maintenance burden

## Scope

### In

- capability-boundary truth audit for:
  - Brainstack durable personal memory
  - `session_search`
  - native automation / `cronjob`
  - host context shell (`SOUL.md` / context-file loading)
  - legacy built-in memory vs plugin memory orchestration
- narrow runtime or guidance changes only if the truth pass shows a real boundary mistake
- documentation/runtime reconciliation where architecture drift would otherwise mislead future work

### Out

- broad host-monolith refactor
- replacing all native memory-adjacent features with Brainstack
- new retrieval capabilities
- benchmark work

## Concrete repair method

This phase should behave like a boundary-clarification phase, not a replacement campaign.

The order is:

1. classify the capability
2. decide whether it is the same ownership axis or a different capability class
3. only then decide:
   - Brainstack owner
   - bounded coexistence
   - or explicit native owner

## Decision rules

1. If a native feature persists or retrieves the same user-specific truth that Brainstack already owns, it is a competing owner and should be constrained.
2. If a native feature provides a different capability class, such as explicit transcript forensics, it should not be displaced just because it is memory-adjacent.
3. If a clarity problem can be solved by a thin orchestration or documentation correction, do not perform a broad refactor.
4. If a fix would create a second durable source of truth, reject it.

## Hypothesis -> Falsification -> Repair decision

1. `session_search` boundary.
   - Hypothesis:
     - `session_search` is being over-hidden because it was treated like durable personal memory even though it is really explicit transcript browsing.
   - Falsification:
     - prove that `session_search` actually behaves as a competing personal-memory owner rather than a separate transcript-forensics tool.
   - Repair decision:
     - if falsified, keep it hidden and document why
     - if not falsified, restore bounded coexistence with explicit guidance and guardrails

2. Memory orchestration clarity.
   - Hypothesis:
     - the runtime/documentation layering between legacy built-in memory and plugin memory is unclear enough to create future boundary bugs.
   - Falsification:
     - prove the runtime and docs already describe the same orchestration model clearly enough for future work.
   - Repair decision:
     - if not falsified, reconcile the layering at the thinnest useful level
     - prefer explicit boundary cleanup over broad memory-framework redesign

3. Native coexistence doctrine.
   - Hypothesis:
     - there are other memory-adjacent native features being displaced more broadly than necessary.
   - Falsification:
     - prove the current guards are already axis-bounded and do not suppress different capability classes.
   - Repair decision:
     - if not falsified, narrow the guard to the actual ownership axis rather than keeping blanket suppression

## Stop conditions before any patch

No code change is allowed until all of these are true:

- there is an owner/coexistence matrix for the candidate capabilities
- `session_search` has been classified by capability, not by name similarity
- the orchestration drift is demonstrated in real code paths, not inferred from docs alone

## Validation ladder

1. Code-path truth
2. Runtime/config truth
3. Capability classification matrix
4. Narrow repair if justified
5. Proof that personal-memory ownership remains protected
6. Proof that native coexistence remains intact where intended

## Compatibility / donor-risk checkpoints

1. `session_search`
   - do not restore it in a way that creates a second durable personal-memory owner
   - if coexistence is chosen, keep it explicitly scoped to transcript forensics / session browsing

2. Native automation
   - do not weaken the current cron boundary by allowing automation-based personal-memory persistence back in
   - do not broadly disable native cron behavior outside that axis

3. Memory orchestration clarity
   - prefer a thin orchestration clarification over a broad memory-stack rewrite
   - only touch the deeper layering if the truth pass proves the current split is causing real boundary confusion

4. Host compatibility shell
   - `SOUL.md` and other host context shell inputs may remain as bounded shell
   - do not silently let them become the owner of personal-memory truth again

## Execution steps

1. Workstream A: capability-boundary truth audit.
   - Build an explicit matrix for:
     - Brainstack durable personal memory
     - `session_search`
     - `cronjob`
     - host context shell
     - legacy built-in memory
     - plugin memory orchestration
   - Classify each one as:
     - owner
     - compatibility shell
     - explicit transcript forensics
     - automation
     - competing owner

2. Workstream B: `session_search` decision.
   - Read the actual `session_search` path and usage contract.
   - Decide whether it is:
     - a competing owner
     - or a valid coexistence capability
   - If coexistence is justified:
     - restore it narrowly
     - keep explicit guidance that it is transcript browsing, not durable personal-memory ownership
   - If coexistence is not justified:
     - keep it hidden
     - document the real conflict clearly

3. Workstream C: orchestration clarity repair.
   - Audit the real relationship between:
     - legacy built-in memory in `run_agent.py`
     - plugin memory behind `MemoryManager`
   - Repair only the thinnest useful layer needed to make runtime and docs tell the same story.
   - Do not attempt a broad memory stack rewrite unless the truth pass proves it is unavoidable.

4. Workstream D: proof.
   - Verify:
     - Brainstack still owns personal-memory truth
     - cron/automation still works as native capability
     - `session_search` is either correctly restored or correctly kept hidden
     - the runtime/docs boundary story is now coherent

5. Workstream E: user decision checkpoints during execute.
   - Before any boundary decision is locked, present the live finding back to the user:
     - in simple language
     - without programmer jargon
     - with a short explanation of why the recommended option is better
   - Use these checkpoints for decisions like:
     - keep native
     - coexist with guardrails
     - or let Brainstack own the axis
   - Do not silently choose the final architecture if the phase reaches a real product tradeoff.

## Regression net

1. Personal-memory regression net.
   - No file/skill/code/automation detours reopen personal-memory shadow ownership.

2. Native-capability regression net.
   - Native automation is not accidentally broken by personal-memory hardening.
   - Any restored `session_search` path does not become a durable profile owner.

3. Architecture-clarity regression net.
   - Docs and runtime continue to describe the same ownership model after the repair.

4. Decision-process regression net.
   - major architecture choices made during execute are reflected back to the user in plain language before final lock-in

## Required working style

- Prefer boundary decisions over new abstractions.
- Prefer thin fixes over new frameworks.
- If a native feature is already good at its own job, keep it.
- Do not use "memory-adjacent" as a reason to let Brainstack absorb it.
- When the phase reaches a real architecture fork, stop and discuss the options with the user in simple language before finalizing the direction.
