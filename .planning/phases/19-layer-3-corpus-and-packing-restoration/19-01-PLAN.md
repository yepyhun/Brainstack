# Phase 19 Plan

## Goal

Restore Layer 3 so Brainstack can honestly claim MemPalace-style corpus strength on an embedded backend, with `Chroma` as the default target, while preserving the thin-shell architecture and turning the currently degraded semantic corpus leg into a real donor-backed path.

## Scope

### In

- introduce a real L3 backend seam with `Chroma` as the default implementation target
- move active corpus retrieval strength out of the current SQLite-only path
- restore donor-shaped raw corpus retrieval as the primary L3 strength
- restore bounded packing as a secondary quality layer over stronger retrieval candidates
- extend the existing store-agnostic publish journal from shell↔`Kuzu` to shell↔`Kuzu`+`Chroma`
- make L1’s semantic corpus leg genuinely live through the existing executive retrieval contract
- define a bounded corpus eval ladder that proves usefulness rather than backend novelty

### Out

- L1 conceptual redesign
- L2 backend redesign or graph contract reset
- full benchmark chasing as the main driver
- service-zoo drift or separate document-platform ambitions
- rebuild loops unless a bounded runtime smoke is actually needed

## Planning Assumption

`19` comes after `18`.

That means this phase must do four things at once:

1. replace the local SQLite corpus center with a donor-aligned embedded corpus path
2. extend the journal from one active donor target to two without rewriting its core
3. activate the currently degraded semantic corpus leg in L1
4. improve packing because retrieval is stronger, not because local trimming gets fancier

This is not a cosmetic corpus cleanup.
It is the real L3 restoration step.

## Execution Steps

1. Define the active L3 backend contract.
   - Introduce an explicit corpus backend seam that can host `Chroma`.
   - The contract must cover:
     - document registration
     - chunk / section publication
     - semantic retrieval
     - lexical retrieval where still useful
     - bounded result metadata for later packing
   - The seam must support later donor refresh work without another architecture reset.

2. Move corpus write ownership behind the backend seam.
   - Re-route active corpus publication away from direct SQLite centrality.
   - Keep Brainstack shell ownership only for:
     - publish / retry / partial-failure coordination
     - deterministic canonical chunk identity
     - journal state transitions
   - Do not let `corpus.py` remain the place where donor corpus strength is simulated.

3. Replace the active corpus retrieval center.
   - Corpus results returned to L1 and the final prompt path must become materially stronger than the current SQLite FTS plus `LIKE` path.
   - Improve:
     - long-document recall
     - semantic retrieval quality
     - relevance under large corpus size
     - continuity of useful raw passages across follow-up questions
   - Preserve lexical / FTS support only as a complement, not as the hidden primary path.

4. Restore bounded packing on top of stronger retrieval.
   - Packing must become a second-stage quality layer over better candidates.
   - Improve:
     - token-budget awareness
     - passage usefulness density
     - duplicate / overlap reduction
     - stable ordering under constrained budgets
   - Do not let packing become the place where weak retrieval is cosmetically rescued.

5. Thin down the local corpus approximations.
   - `brainstack/corpus.py` must stop being the effective corpus-intelligence core.
   - `brainstack/db.py` must stop being the effective corpus-retrieval engine.
   - `brainstack/retrieval.py` must not become the place where weak L3 recall is cosmetically rescued.
   - `brainstack/donors/corpus_adapter.py` must become a real seam, not a nominal wrapper around local logic.

6. Extend cross-store ingest consistency to `Chroma`.
   - Reuse the existing store-agnostic publish journal core.
   - Add `Chroma` as a named publish target.
   - Preserve:
     - `pending` vs `published`
     - partial-failure visibility
     - idempotent replay
     - resumable retry after interruption
   - Plan for:
     - SQLite -> `Chroma` bootstrap
     - incremental publication
     - failure without duplicate chunk publication
     - clear visibility when `Kuzu` and `Chroma` are out of sync
   - This remains Brainstack shell responsibility, not a new subsystem.

7. Make the L1 semantic corpus leg honestly live.
   - `executive_retrieval.py` must receive real semantic corpus results through the existing contract.
   - The phase must end with the semantic corpus leg no longer degraded-by-design.
   - If the semantic leg is still effectively fake after the phase, the phase fails.

8. Build the bounded Phase 19 corpus eval ladder.
   - Gate A:
     - fast corpus backend contract tests
     - bootstrap and journal tests
   - Gate B:
     - corpus usefulness scenarios
     - must prove better long-document retrieval and packing
   - Gate C:
     - small benchmark-derived corpus subset as a regression guard
   - Gate D:
     - bounded live/runtime smoke only if source-side proof leaves a real doubt

## Implementation Rules

- donor-first means MemPalace-shaped corpus behavior, not MemPalace-flavored naming
- `Chroma` is the default target unless execution evidence clearly disproves it
- no SQLite-only corpus center as the accepted end state
- no packaging-only success criteria
- no compression-first drift if raw retrieval remains the stronger donor path
- no service-zoo drift
- no hidden L1 redesign inside the corpus phase
- no hidden L2 redesign inside the corpus phase
- no benchmark-gaming
- no new handwritten multilingual heuristics as a substitute for semantic corpus restoration

## Output Shape Required From Phase 19

The implementation must leave behind:

1. a real L3 backend seam with a `Chroma`-default implementation path
2. thinner local corpus modules
3. materially stronger corpus retrieval results for both L1 and final prompt packaging
4. donor-faithful bounded packing over stronger retrieval candidates
5. a journal extension that adds `Chroma` as a named publish target without rewriting the journal core
6. explicit proof that the L1 semantic corpus leg is genuinely live
7. a bounded corpus eval ladder

## Exit Gate

- L3 is materially stronger in real long-document / corpus behavior, not just tidier in presentation
- the active corpus center is no longer the old SQLite local path
- raw corpus retrieval is meaningfully stronger and bounded packing improves because candidates are better
- the semantic corpus leg in L1 is no longer degraded-by-design
- the journal core remains store-agnostic while now coordinating both `Kuzu` and `Chroma`
- the result is strong enough that calling the phase “MemPalace-recentered” is defensible
