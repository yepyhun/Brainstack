# Phase 06.1.1 Context

## Goal
Create a mandatory anti-vibecoding audit gate that catches half-wired Brainstack integrations, hidden old-path dependence, silent fallbacks, and product-goal drift before later phases build on them.

## Why This Phase Exists
- The user explicitly asked for a separate step that audits common AI coding failure modes instead of trusting implementation at face value.
- Recent work repeatedly showed the same risk pattern:
  - a new path exists, but the host does not really use it
  - the old path remains partially alive in parallel
  - code, docs, or tests change without a real runtime takeover
  - the implementation drifts toward a proxy or benchmark-friendly answer instead of the actual product goal
- Phase 06.1 proves replacement coverage at the native-memory contract level, but it does not, by itself, formalize a reusable anti-half-wire / anti-goal-drift gate for future Brainstack work.

## Failure Classes This Phase Must Audit
- `half_wired`
  - producer and consumer do not meet on a real live runtime path
- `shadow_path`
  - the legacy/native path is still active beside the new path
- `silent_fallback`
  - failures quietly revert to old behavior instead of failing closed or surfacing the issue
- `surface_only_change`
  - docs, names, or tests changed while runtime ownership did not
- `dead_hook`
  - a provider method, hook, or interface exists but the host never actually calls it
- `config_illusion`
  - a flag or config switch exists but does not materially change behavior
- `goal_drift`
  - the implementation optimizes a proxy, benchmark, or local shortcut instead of the intended product behavior

## Required Standard
Every audited responsibility or path must end in an explicit state with evidence:
- live and owned by Brainstack
- intentionally retired / blocked
- intentionally deferred with a named blocking reason

Anything else is a failure.

The audit must produce evidence for:
- ownership path
- live caller path
- old-path retirement
- fail-closed behavior
- negative-path tests
- product-goal alignment

## How Graph MCP Should Be Used
- Use `code-review-graph` with full postprocess when it is stable enough, because caller/callee, affected flows, and file summaries are high-value evidence for detecting half-wire and dead-hook problems.
- Treat graph evidence as support, not as the only proof.
- Final verdicts must still be backed by code references and executable tests.

## Non-Goals
- Do not turn this phase into broad refactoring or architecture redesign.
- Do not add product features.
- Do not re-open benchmark-specific tuning unless it directly proves goal drift or false confidence in the current layer.
