# Phase 61 Plan

## purpose

Make Brainstack behave like a real memory kernel for recent work after restart.

When the user asks broad recap questions such as:
- what were we just doing
- what did we just fix
- did the previous phase land
- do you remember the earlier Brainstack work

the answer should come from Brainstack-owned durable operating truth first, not from an expensive transcript-search bailout.

This phase is not a session-search tuning phase. It is not a generic Hermes resume phase. It is a Brainstack phase whose job is to make recent work survive restart in a compact, truthful, recallable form.

## core question

How should Brainstack preserve and recall recent active work and major completed outcomes so that ordinary restart recap questions are answered from Brainstack durable state before transcript search is even considered?

## primary goals

1. make recent work and major completed outcomes survive restart as compact Brainstack operating truth
2. make broad recap questions route to that operating truth instead of falling through to empty recall
3. keep transcript/session search as a secondary exact-detail tool, not the first-line memory kernel
4. preserve token discipline and avoid transcript dumping or heuristic sprawl

## explicit scope model

### Brainstack-owned scope

In scope:
- missing or under-populated durable representation of recent active work
- missing or under-populated durable representation of major completed outcomes
- missing or under-populated durable representation of discarded/superseded work when that state matters to future recap correctness
- recap-recall routing that fails broad recent-work questions and leaves Brainstack state unused
- cross-session recent-work retrieval that is too literal, too query-fragile, or too tied to current-session boundaries
- operating-context projection that fails to surface restart-relevant recent work even when the DB has enough evidence
- Brainstack-side query expansion, evidence ranking, and projection for recent-work recap

### host-boundary scope

Out of scope unless a thin seam is strictly required for Brainstack correctness:
- generic `session_search` speed tuning as the primary solution
- generic `/resume` UX
- generic gateway reset messaging
- generic session-store mechanics
- generic Discord adapter behavior

### sidecar rule

This phase must not assume that a missing sidecar is the root cause.

Current live evidence says:
- Brainstack plugin is active
- `sidecars.rtk.enabled` is false
- the core failure is still a Brainstack recall failure, not merely "forgot to enable a sidecar"

If a sidecar later appears relevant, the phase must prove why the main Brainstack kernel cannot satisfy the recall contract without it.

## anti-goals

- no `session_search` timeout patch presented as the main fix
- no exact-phrase rescue for `phase 60`
- no user-id, thread-id, or one-language special casing
- no raw transcript stuffing into working memory
- no giant recap blobs that blow the context window
- no second memory system for "recent work" beside Brainstack
- no host-wide resume rewrite hidden inside a Brainstack phase
- no keyword farm for every imaginable way to ask "what were we doing"

## accepted evidence

The current live evidence already proves:

1. Brainstack is active and configured, with the plugin mounted and live.
2. Relevant `phase 60` material exists in Brainstack-managed tables:
   - `continuity_events`
   - `transcript_entries`
3. The same live DB currently contains no useful `operating_records` for that recent work.
4. A real restart recap question currently produces an effectively empty Brainstack recall packet:
   - no continuity rows
   - no operating rows
   - no task rows
   - no transcript rows
   - only one profile row
5. Transcript FTS can find relevant rows for simpler normalized queries like:
   - `phase 60`
   - `phase 60 brainstack`
   - `brainstack`
6. Therefore the problem is not "there is no stored data at all"; it is that the Brainstack recap route fails to convert stored evidence into restart-usable working memory.

## phase hypotheses to test

1. Brainstack currently lacks a strong recent-work operating-summary path, so important work remains trapped in continuity/transcript lanes.
2. Broad recap questions are not recognized as a first-class recall class, so retrieval stays too literal and under-matches.
3. Restart recap currently over-depends on the current session boundary even when the relevant evidence is in earlier sessions within the same principal scope.
4. `session_search` is compensating for a Brainstack recap-lane defect rather than merely adding optional transcript precision.

## architecture direction to freeze before execution

Phase 61 execution must freeze a compact architecture decision for all of the following:

### 1. recent-work durable lane

Prefer existing Brainstack structures first:
- `operating_records`
- `operating_context`
- already existing continuity evidence

Do not invent a new storage lane unless existing operating truth cannot carry the recent-work contract cleanly.

### 2. recap query class

Define a bounded "recent-work recap" retrieval class for queries like:
- what were we just doing
- what did we fix
- did the previous phase land
- what happened before restart

This must be:
- semantically broader than exact transcript matching
- narrower than a generic fuzzy-everything heuristic farm
- language-robust enough not to collapse on minor wording changes
- architecturally separate from the existing `task_like` route, even if both routes later reuse the same operating-truth substrate

The recap route must not be implemented by stretching the `task_like` lane until it also tries to answer:
- what we were doing
- what landed
- what was discarded

Those are different recall semantics from structured obligation/task lookup.

### 3. evidence order

For recent-work recap:
1. operating truth
2. compact recent continuity
3. bounded transcript evidence
4. transcript-search tools only if the Brainstack packet is still genuinely insufficient

### 4. truth contract

The recap lane must preserve:
- active work
- major completed outcomes
- explicit abandonment/discard decisions when they matter

But it must not preserve:
- raw scratch logs
- every intermediary step
- assistant self-congratulation
- transcript-shaped noise

### 5. promotion timing

The phase must freeze whether recent-work operating truth is written:
- continuously during active work
- only at session end
- or in a bounded hybrid

Current planning presumption:
- session-end consolidation is useful
- but session-end-only promotion is too weak for restart/interrupt-heavy reality
- the likely correct shape is bounded ongoing promotion plus session-end consolidation

## workstreams

### 1. evidence freeze

- freeze the exact live evidence that shows:
  - relevant DB rows exist
  - Brainstack packet is empty for a real recap question
  - transcript FTS can still find simpler matches

### 2. current-lane audit

- inspect how recent work currently flows into:
  - `operating_records`
  - `continuity_events`
  - `transcript_entries`
- explain why the recent-work lane is currently under-populated or under-projected

### 3. recap-route design

- define a dedicated recap-recall route
- specify what query cues, route semantics, and fallback order are allowed
- reject designs that require giant locale-specific phrase catalogs

### 4. durable promotion design

- decide exactly which recent-work facts become durable operating truth
- require a compact schema and explicit provenance expectations
- include negative examples of what must remain transcript-only
- explicitly decide when those records are written:
  - during active work
  - at session end
  - or both
- do not assume session-end flush alone is sufficient

### 5. projection design

- ensure restart recap questions actually surface recent work in the ordinary Brainstack packet
- keep the packet compact and bounded
- ensure recall does not silently regress into transcript dumping

### 6. boundary validation

- document what remains outside this phase:
  - generic session_search performance
  - generic `/resume`
  - generic gateway reset UX
- justify any thin host seam only if Brainstack cannot otherwise capture restart-relevant work state

## acceptance criteria

- the phase explicitly treats this as a Brainstack memory-kernel failure, not merely a session-search timeout
- the planned fix is centered on durable recent-work recall, not transcript-search mitigation
- the plan makes it impossible to declare success while the Brainstack packet for a broad recap query is still empty
- the plan explicitly forbids implementing recap by extending the task-like route
- the plan explicitly treats `operating_records` as principal-scoped truth with session provenance, not as an open scoping question
- the plan forbids user/thread/phrase-specific rescue logic
- the later execute step will have a clear proof target:
  - after restart, a broad recap ask must produce non-empty Brainstack recall without needing `session_search`

## canonical principle reference

This phase must remain consistent with:
- truth-first
- donor-first
- fail-closed behavior
- zero heuristic sprawl
- modularity / upstream survivability
- token-discipline instead of transcript bloat
