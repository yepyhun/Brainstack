# Phase 61.1 Context

## why this phase exists now

`Phase 61` removed active task/operating cue tables and replaced them with `structured_understanding.py`. That move aligned with the project's no-heuristic rule, but the current live evidence shows the new seam is carrying too much ordinary-kernel authority too early.

This phase exists because the correct conclusion from that evidence is not:

- roll back to heuristic routing
- or keep shipping `v1.0.17` as if it were already stable

The correct conclusion is:

- freeze feature churn
- stabilize the current architecture in place
- reduce the availability blast radius of structured understanding

## validated facts

### source facts

- `structured_understanding.py` was introduced in `v1.0.17`
- current source routes task lookup, operating lookup, task capture, operating capture, and route analysis through that seam
- `prefetch()` is intentionally read-only
- `sync_turn()` still performs automatic task/operating capture

### live facts

- recent live logs show repeated Brainstack structured-understanding warnings
- the failures include:
  - timeouts
  - `400 Invalid input`
  - `402` credit-related failures
- therefore the structured-understanding seam is not only a theoretical risk; it is a current production instability source

## accepted root-cause model

- the core problem is not that `v1.0.17` removed heuristics
- the core problem is that too much ordinary Brainstack correctness/availability now depends on fresh auxiliary understanding success
- because the current seam is networked and failure-prone in the live environment, the kernel becomes fragile in ordinary operation

## rejected root-cause models

- "the answer is to restore the old phrase farms"
- "the answer is only to increase timeouts"
- "the answer is to pin a better provider and call it done"
- "the answer is to skip to Phase 62 anyway"

## execution implication

This phase must not become a compromise that secretly reintroduces code-level heuristics.

Instead it must produce:

- a bounded degraded mode
- a narrower use of structured-understanding in ordinary kernel paths
- stronger reliance on explicit typed durable records and compact operating truth

## relation to Phase 62

`Phase 62` is still needed, but only after this stabilization is done.

Reason:

- `Phase 62` needs a trustworthy kernel to represent current live-system authority
- it should not be executed on top of an unstable understanding seam that can still fail ordinary Brainstack operation before the live-state problem is even addressed
