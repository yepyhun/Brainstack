# Phase 61.1 Plan

## purpose

Stabilize the `v1.0.17` Brainstack kernel transition without rolling back to heuristic routing, so ordinary Brainstack behavior no longer depends on successful live `structured_understanding` calls on every relevant turn.

## core question

How do we keep the no-heuristic, structured, multimodal-compatible direction of `v1.0.17` while removing its current availability blast radius and correctness fragility?

## primary goals

- keep the `structured_understanding` direction; do not revert to cue lists or phrase farms
- stop ordinary Brainstack operation from becoming unavailable or misleading when structured-understanding calls timeout, error, or return unusable payloads
- narrow where structured-understanding is mandatory, especially on ordinary read/prefetch surfaces
- define a strict degraded mode that remains correct, bounded, and non-heuristic
- freeze a release-stabilization discipline before any new Brainstack capability work continues

## explicit scope model

### Brainstack-owned scope

- `structured_understanding.py` availability/correctness role in query routing and capture understanding
- Brainstack read-path behavior when structured understanding is unavailable
- Brainstack write/capture behavior when structured understanding is unavailable
- projection and retrieval behavior so the kernel can still answer safely in degraded mode
- release discipline needed to stop shipping new capability churn on top of an unstable kernel seam

### host-boundary scope

- Hermes auxiliary transport behavior may be used as evidence
- provider outages, OpenRouter credit state, and generic runtime timeouts may be documented as evidence
- this phase does not become a generic Hermes auxiliary-client rewrite

### multimodal rule

- the design must not depend on locale-specific text cues or exact wording
- the design must not regress toward text-only phrase routing
- structured typed records and provenance remain the preferred authority model because they can stay modality-agnostic

## anti-goals

- no rollback to `v1.0.16`
- no restoration of task / operating cue tables, phrase detectors, or locale-specific routing farms
- no provider-specific `if kimi` / `if nous` rescue logic in the Brainstack kernel
- no `session_search` tuning presented as the kernel fix
- no broad host/runtime patch sweep disguised as Brainstack stabilization
- no new capability scope from Phase 62 mixed into this phase

## accepted evidence

- source-of-truth diffs between `v1.0.16` and `v1.0.17`
- live runtime logs proving structured-understanding timeouts / 400 / 402 / fallback churn
- current source-of-truth use sites for query routing, task lookup, operating lookup, task capture, and operating capture
- current Brainstack regression and smoke tests

## validated current reading

- the criticism that `v1.0.17` became too fragile too quickly is materially valid
- the stronger claim that automatic capture is entirely gone is false:
  - `prefetch()` is read-only
  - `sync_turn()` still performs automatic task/operating capture
- the stronger claim that everything in Brainstack depends on an LLM call every turn is too strong
- the live evidence still shows a real production problem:
  - recent live logs contain repeated structured-understanding warnings including timeouts, `400`, and `402`

## architecture direction to freeze before execution

### 1. no rollback decision

- `v1.0.16` is not the recovery target
- the project must not reintroduce code-level heuristics as the safety net

### 2. degraded-mode decision

- if structured understanding is unavailable or unusable, the kernel must degrade to a bounded fact-oriented mode
- degraded mode must stay safe:
  - no fake task/operating claims
  - no transcript dump masquerading as understanding
  - no tool spiral presented as memory

### 3. call-site narrowing decision

- structured understanding may remain the preferred typed-understanding seam
- but it must no longer be a mandatory success path for all ordinary read behavior
- execution must explicitly identify which current call sites remain mandatory and which can be narrowed, deferred, or bypassed safely

### 4. write-authority decision

- ordinary durable truth should prefer:
  - explicit native write seams
  - typed operating truth consolidation
  - already-established durable records
- execution must justify any remaining automatic capture dependency on live structured-understanding success

### 5. release discipline decision

- no new Brainstack feature release is justified until this stabilization phase is proven

## workstreams

### 1. evidence freeze

- freeze the exact `v1.0.17` fragility claims against source and live evidence
- separate valid criticism from overstatement

### 2. call-site audit

- enumerate every mandatory `structured_understanding` read/write call site
- classify:
  - required
  - can degrade safely
  - should be removed from ordinary path

### 3. degraded-mode contract

- define what the Brainstack packet and answer path are allowed to do when structured understanding is unavailable

### 4. authority rebalance

- move more authority onto typed durable records and compact operating truth where possible
- reduce ordinary-turn dependence on fresh semantic classification

### 5. verification design

- define proof that the kernel still works:
  - without heuristics
  - without pretending to understand tasks/operating state when it does not
  - without falling into transcript-search or tool churn as the primary answer path

## deliverables

- a stabilization implementation contract
- hard gates that explicitly forbid rollback to heuristics
- a source-of-truth implementation that narrows structured-understanding blast radius
- proof from tests and live evidence that degraded mode is safe and bounded

## acceptance criteria

- the phase preserves the no-heuristic architecture direction
- the phase does not revive cue tables or phrase routing in Brainstack code
- a structured-understanding failure no longer causes ordinary Brainstack behavior to collapse into misleading or expensive fallbacks
- the resulting kernel remains compatible with the already-planned Phase 62 work
