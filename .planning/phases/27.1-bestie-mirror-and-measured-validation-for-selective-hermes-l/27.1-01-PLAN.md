# Phase 27.1 Plan

## Goal

Mirror the landed Phase `27` Brainstack slices into Bestie from Brainstack source truth, then measure whether they produce real Bestie-side value instead of only cleaner internal architecture.

## Scope

### In

- mirror only the landed Phase `27` slices:
  - bounded snapshot provenance
  - thin lifecycle/frontier state
- Bestie-side validation and measured proof
- explicit comparison between:
  - wiring correctness
  - measurable product effect
- honest verdict if the measured gain is weak or absent

### Out

- new donor slices
- bounded expand/search implementation
- session filtering implementation
- fresh local redesign in Bestie
- broad host refactor
- vague “feels cleaner” acceptance

## Concrete implementation method

- first map the exact Phase `27` source changes onto the Bestie mirror seam
- mirror only what already landed in Brainstack source
- then validate in three layers:
  - mirror correctness
  - runtime availability
  - measured product effect
- if the mirror is correct but measurable gain is weak, close with that truth instead of forcing a second patch round

## Decision rules

- do not rewrite by hand where the Brainstack source truth can be copied or minimally mirrored
- do not widen scope beyond the two landed slices
- do not claim token or retrieval wins without explicit before/after evidence
- if the new data only improves auditability/diagnostics, say exactly that
- if the Bestie mirror forces design changes instead of narrow mirroring, stop and surface the mismatch

## Hypothesis -> Falsification -> Repair decision

### 1. Bestie mirrorability

- hypothesis:
  - the landed Phase `27` source slices can be mirrored into Bestie cleanly
- falsification:
  - Bestie mirror seams have drifted too far and require a local redesign
- repair decision:
  - if false, stop the phase and report mirror-drift truth instead of hand-writing a second design

### 2. Runtime availability

- hypothesis:
  - after the mirror, Bestie actually emits/stores the new provenance and lifecycle data
- falsification:
  - the mirror compiles but the integrated runtime path does not produce the new data
- repair decision:
  - if false, fix only the narrow wiring seam

### 3. Measured product value

- hypothesis:
  - the mirrored slices measurably improve at least one of:
    - token discipline
    - retrieval usefulness / targeting
    - auditability / diagnosability
- falsification:
  - the mirror is correct but there is no measurable gain
- repair decision:
  - if false, close honestly as “correct mirror, no proven product lift yet”

## Stop conditions before any patch

- do not patch Bestie before mapping the exact Brainstack source diff
- do not patch if the mirror requires new design not present in Brainstack source
- do not patch if the measurement method is vague or hand-wavy
- do not expand into new donor slices

## Validation ladder

1. source-to-mirror mapping truth
2. Bestie mirror correctness
3. Bestie runtime persistence truth
4. measured before/after comparison
5. honest product-value verdict

## Compatibility / donor-risk checkpoints

- keep Brainstack as source of truth
- keep Bestie as mirror / validation target only
- preserve the settled Phase `22` boundary
- preserve the settled Phase `27` deferred decisions
- preserve ordinary-turn token discipline

## Execution steps

1. Map the exact landed Phase `27` Brainstack diff onto the Bestie mirror seam.
2. Mirror only those two landed slices into Bestie.
3. Prove the mirrored data is actually emitted/stored in Bestie runtime.
4. Define a small measured validation harness:
   - before/after token impact
   - retrieval targeting / usefulness effect
   - auditability / diagnostics effect
5. Run the measured comparison.
6. Close with one of three explicit verdicts:
   - measured win
   - correct mirror, no proven product lift yet
   - not worth keeping

## Regression net

- no new donor slice may sneak in
- no Bestie-local redesign drift
- no ordinary-turn token inflation without proof
- no regression to the settled Brainstack/native boundary
- no fake-green “architecture improved therefore product improved” reasoning

## Required working style

- keep the phase narrow
- measure before claiming value
- prefer honest neutral verdict over embellished success
- if the mirrored slices help only diagnostics, say that directly
