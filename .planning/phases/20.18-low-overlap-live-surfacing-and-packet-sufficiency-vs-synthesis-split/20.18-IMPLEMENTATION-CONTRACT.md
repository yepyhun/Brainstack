# Phase 20.18 Implementation Contract

## Objective

Resolve the post-`20.17` live residual more precisely:

1. improve the bounded low-overlap same-principal surfacing failures on the active degraded stack
2. separate those from failures where the packet is already sufficient and the model still answers weakly

## Workstream A: Residual-Class Separation

- Begin from the latest isolated live eval and the `20.17` packet-localization artifact.
- The phase must explicitly classify:
  - `temporal_order`
  - `coffee_order`
  - `gift_context`
- The phase must not collapse all three into one bucket without evidence.

### Expected starting interpretation

- `temporal_order`: low-overlap same-principal surfacing / coverage
- `gift_context`: packet-sufficient but answer-weak
- `coffee_order`: mixed, with real modifier-loss from low-overlap surfacing

## Workstream B: Bounded Low-Overlap Surfacing Fix

- Fix only the structurally under-surfaced class.
- Preferred generic directions:
  - same-principal session-summary promotion
  - bounded low-overlap evidence carry-through
  - bounded same-principal evidence preference when current-session rows are weak or self-referential
- Content-keyword hacks are not allowed.
- Broad global widening is not allowed.

## Workstream C: Optional Control Rerun

- If the `20.17` score movement still reads as ambiguous, one bounded control comparison is allowed.
- Its purpose is only:
  - patch regression vs provider variance
- Do not let this become a second evaluation campaign.

## Workstream D: Faithful Live Proof

- Re-run the isolated deployed-live reset eval after the bounded fix.
- Compare against the latest repeated `20.17` baseline:
  - `2 / 5`
- Record:
  - per-scenario packet sufficiency
  - per-scenario final answer outcome
  - whether the low-overlap class moved

## Workstream E: Honest Next-Phase Decision

- If low-overlap surfacing is improved but synthesis still dominates, say so directly.
- If synthesis is no longer the dominant blocker, say that directly too.
- If graph/corpus live enablement becomes the highest-value next thread, call it explicitly instead of silently continuing degraded-stack tuning forever.

## Protected Boundaries

### Product Boundary

- coherent continuous conversation and correct recall outrank broader capability expansion
- live deployed truth outranks oracle intuition

### Architectural Boundary

- preserve donor-first seams
- do not reopen principal isolation or route policy without new evidence
- do not smuggle graph/corpus enablement into this phase

### Anti-Benchmaxing Boundary

- no scenario-specific lexical rules
- no tuning against named benchmark IDs
- no gains that only exist in oracle-only conditions

## Minimum Evidence Required Before Calling Phase 20.18 Done

- one explicit residual split for `temporal_order`, `coffee_order`, and `gift_context`
- one bounded low-overlap surfacing fix or one explicit proof that the residual is no longer primarily surfacing
- one repeated isolated live eval artifact after the phase work
- one explicit next-phase recommendation
