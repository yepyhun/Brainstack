# Phase 74 Plan

## purpose

Add memory-safe session/procedure support so Brainstack can recall how work is normally done and what session state exists, while Hermes runtime remains responsible for execution, scheduling, and approval.

## core question

How can Brainstack adopt the useful part of CerebroCortex's agent-work helper model without becoming a workflow engine or hidden governor?

## GSD planning enforcement check

This phase must preserve:

- synergistic Hermes boundary: runtime executes; Brainstack recalls state/procedure evidence
- donor-first modularity: do not replace Codex skills, Hermes policies, or runtime task systems
- no live-case distortion: procedure memory must generalize beyond one workflow
- inspectability: every session/procedure recall needs evidence ids and authority labels

## shared north star and real-world proof gate

North star for Phase 70-77: Brainstack must become easier and safer for Hermes agents to use as a memory kernel without expanding Brainstack into runtime execution, workflow governance, or fake autonomy.

Every phase closeout must prove one of these:

- Hermes can use Brainstack more reliably in a real memory workflow
- or the phase is prerequisite work and proves no regression while naming the exact later phase that must deliver the user-visible improvement

Minimum proof shape:

- targeted tests for the changed surface
- at least one end-to-end Hermes/Brainstack interaction path or Brainstack packet/inspect proof using real provider/store code
- evidence ID/source/provenance proof where memory is involved, not only generated answer text
- no-regression check against Phase 66 golden gates once they exist
- bounded token/output sanity for any new tool, MCP, inspect, corpus, or trace surface

## explicit scope model

### Brainstack-owned scope

- procedure memory records
- session state records
- read-only projections
- recall and packet evidence
- version/provenance/scope handling

### out of scope

- scheduling
- execution
- approval enforcement
- task queue ownership
- replacing skills
- multi-agent messaging

## architecture decisions to freeze before execution

### 1. procedure vs policy vs skill

Procedure memory may describe how work is usually done. It cannot override canonical policy or replace executable skills.

### 2. session state expiry

Session records must have owner, scope, current/stale status, and expiration or supersession rules.

### 3. read-model-only rule

Brainstack can project recallable session/procedure context. It cannot initiate work.
