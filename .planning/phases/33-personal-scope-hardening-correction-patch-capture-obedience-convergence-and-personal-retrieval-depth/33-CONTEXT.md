# Phase 33 Context

## problem statement

The immediate `32` hotfix addresses concrete regressions:

- canonical contract degradation
- missing multi-message convergence
- sequential-path tool blocking

That still leaves a broader second-brain quality gap. Brainstack can store personal truth correctly and still feel brittle, forgetful, or inconsistent if:

- personal truth is tied too tightly to exact runtime scope
- short user corrections do not become durable canonical patches
- canonical personal truth does not reliably influence ordinary turns
- personal/preference routes starve themselves of the very evidence needed for continuity

This phase exists to improve Brainstack as a general continuous agent, not to optimize one transcript-shaped failure.

## why this is the correct next phase

The most useful accepted reading from the broader audit is not “fix this one test harder.” It is:

- harden personal-scope continuity
- treat short explicit corrections as first-class durable updates where appropriate
- shrink the gap between authoritative personal truth and visible behavior
- stop self-imposed starvation on personal/preference recall

Those are general kernel-quality improvements. They help many flows, not just the recent rule-recall incident.

## phase boundary

This phase is broader than `32`, but still tightly bounded:

- it is about personal second-brain quality
- it is not about broad graph redesign
- it is not about broad multimodal redesign
- it is not about benchmark hacking

## accepted sharpened reading

The four important broader items that should anchor this phase are:

1. personal-scope hardening / fallback
2. short explicit correction -> canonical patch path
3. canonical personal truth -> ordinary-turn obedience convergence
4. personal/preference retrieval starvation reduction

These are the core of `33`, not side notes.
