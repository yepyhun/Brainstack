# Phase 62 Plan

## purpose

Make Brainstack act like an authoritative memory kernel for live autonomous system state, not a transcript scrapbook.

The current live Discord use exposed two linked failures:

1. Brainstack cannot reliably answer whether previously established autonomous mechanisms such as heartbeat, pulse, and evolver are actually active now.
2. A scheduled cron run claimed it had no filesystem-write capability, but the cron scheduler surface does not obviously prohibit the relevant tool families, and the session artifact shows no tool attempt at all.

This phase exists to fix the Brainstack-owned half of that failure and to pin down the cron capability-truth boundary precisely.

## core question

How should Brainstack preserve and project authoritative live-system state so that questions like "is heartbeat/evolver still working?" are answered from current operating truth first, while stale assistant narration and transcript residue are demoted?

## primary goals

1. make current autonomous-system state durable and recallable as Brainstack operating truth
2. stop stale transcript/continuity narration from acting like current system truth
3. ensure ordinary questions about running mechanisms do not collapse into empty recall plus tool flailing
4. prove whether the cron file-write failure was a real capability restriction or a false capability claim

## scope model

### Brainstack-owned scope

In scope:
- missing or under-populated operating truth for live autonomous mechanisms
- missing or weak projection of "currently active / currently absent / previously established but no longer live" system state
- retrieval precedence that lets transcript residue answer live-state questions ahead of authoritative operating truth
- durable-state design for heartbeat / pulse / evolver status when that state matters across restart or session boundaries
- Brainstack-side demotion of historical assistant claims that were never re-grounded against current state

### boundary scope

The following may be documented and, if needed, requested as thin seams, but must not take over the phase:
- generic cron engine rewrite
- generic Hermes tool-surface redesign
- generic filesystem permission cleanup
- generic Discord or gateway timeout cleanup

### capability-truth rule

The cron file-write incident belongs in this phase only to the extent that it reveals a truth/authority gap:
- if cron truly lacks write tools, Brainstack must not store or project the opposite
- if cron has write tools but the model falsely claims it does not, that is a host capability-truth defect that must be documented as a boundary

## anti-goals

- no generic Hermes cron rewrite disguised as a Brainstack phase
- no phrase-matching rescue for `heartbeat`, `evolver`, or `pulse`
- no transcript-first "it once said it worked, so it must still work" logic
- no session-search bailout presented as system-state memory
- no user-specific or thread-specific patching
- no heuristic table for every way to ask whether something is still running

## accepted evidence

The current live evidence already proves:

1. the canonical communication contract exists in `USER.md`
2. the active cron store can be empty while the assistant still talks as if previous autonomous systems are live
3. the live Brainstack DB contains old heartbeat / evolver / pulse claims primarily in transcript / continuity style lanes
4. the cron session `session_cron_d0c8894058fb_20260423_170000.json` responded with a no-write-tools claim without making any tool call
5. the cron scheduler construction currently disables:
   - `cronjob`
   - `messaging`
   - `clarify`
   - `hermes-discord`
   but does not, on current evidence, explicitly disable terminal-style file-writing tools

6. validated adjacent runtime findings that matter to authority modeling:
   - cron uses fresh session ids and `skip_memory=True`
   - `brainstack_pulse.py` only reads tasks, prints markers, and appends to a log file
   - `native_moa_research.py` is not cleanly standalone because `hermes_tools` is unavailable in direct container execution
   - Evolver CLI is installed but no active loop process is running
   - `pip` is absent, but `uv` exists, so "dependencies are impossible to install" is too strong

## expected execution shape

Phase 62 should proceed in this order:

1. model the authoritative-state defect in Brainstack terms
2. query the live Brainstack DB for where autonomous-system truth currently lives
3. decide what operating-record shape should carry live autonomous-state truth
4. verify whether a thin host seam is required to feed that state into Brainstack
5. separately close the cron capability-truth investigation with a precise ownership verdict
6. explicitly reject overstated runtime conclusions that are not supported by current evidence
