# Phase 28 Plan

## Goal

Audit the latest upstream donor deltas against the settled Brainstack baseline, then adopt only the small slices with clear product value and low maintenance burden.

## Scope

### In

- latest-delta audit for:
  - Hindsight
  - MemPalace
  - Graphiti
- selective adoption only where:
  - ROI is real
  - boundary fit is clean
  - maintenance cost stays bounded
- Brainstack-side implementation only
- explicit no-op / defer verdicts where adoption is not justified
- donor refresh truth update if the audit changes the adoption reading

### Out

- blanket upstream syncing
- donor vanity adoption
- Bestie implementation work
- new memory-owner layers
- broad feature expansion under the label of “update”
- SHIBA / Tier-2 uplift smuggling

## Decision rule

This phase is successful if it reaches one of these honest outcomes:

- selective win:
  - one or more thin donor deltas land cleanly
- audit-only no-op:
  - the current local code already covers the useful parts
- explicit defer:
  - a donor delta looks valuable in theory but is not worth the cost right now

## Hypothesis -> Falsification -> Decision

### 1. Hindsight retrieval-budget candidate

- hypothesis:
  - the latest Hindsight recall-budget idea can improve Brainstack retrieval/token discipline through a thin local policy
- falsification:
  - Brainstack already has enough control through existing packet/route limits, and a new layer would only duplicate that control
- decision:
  - if false, record explicit no-op and do not add a second budget abstraction

### 2. MemPalace backend-boundary candidate

- hypothesis:
  - latest MemPalace backend discipline still reveals a local thin hardening opportunity
- falsification:
  - all local Chroma usage is already correctly confined to the backend seam
- decision:
  - if false, close as audit-only and do not manufacture work

### 3. Graphiti immediate delta candidate

- hypothesis:
  - latest Graphiti changes include a concrete graph-truth delta worth immediate adoption
- falsification:
  - latest changes are repo/ops noise or otherwise non-material to current Brainstack graph behavior
- decision:
  - if false, explicitly defer and keep Graphiti out of this phase

### 4. Refresh-truth tooling candidate

- hypothesis:
  - donor refresh reporting or registry truth needs a bounded update after the latest audit
- falsification:
  - current tooling already describes actual adoption truth accurately
- decision:
  - if false, do not touch the reporting path

## Execution order

1. Write a donor-delta matrix for the latest upstream heads:
   - donor
   - latest relevant delta
   - local overlap
   - ROI
   - portability
   - decision
2. Audit Hindsight against:
   - `brainstack/control_plane.py`
   - `brainstack/executive_retrieval.py`
3. Audit MemPalace against:
   - `brainstack/corpus_backend.py`
   - `brainstack/corpus_backend_chroma.py`
   - any direct Chroma touchpoints
4. Audit Graphiti only enough to prove adopt-now vs no-op
5. Land only the justified thin slices
6. Update donor refresh truth only if the audit changed it
7. Verify with bounded tests and before/after packet discipline checks

## Stop conditions before any patch

- stop if a candidate requires broad host/runtime redesign
- stop if a candidate requires new donor runtime adoption
- stop if a candidate adds more config surface than value
- stop if the candidate only improves “architectural neatness” without measurable or operational benefit
- stop if the candidate would be implemented differently in Brainstack and Bestie

## Regression net

- no regression to settled Phase `22` boundary
- no regression to settled Phase `24/25` correctness baseline
- no regression to settled Phase `27/27.1` host-level donor reading
- no ordinary-turn token bloat without proof
- no new handwritten heuristic sprawl to imitate donor behavior
- no donor delta may become a second memory-runtime owner

## Verification ladder

1. donor-delta truth is written down explicitly
2. changed code stays inside narrow Brainstack-owned seams
3. targeted tests pass
4. packet footprint / token-discipline check stays bounded
5. final verdict is one of:
   - selective win
   - audit-only no-op
   - explicit defer

## Required working style

- prefer deletion, non-adoption, or no-op over speculative glue
- prefer thin policy adaptation over new architecture
- prefer explicit “not worth it” over donor-chasing
- preserve donor-first updateability without treating every donor move as sacred
