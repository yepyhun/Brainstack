# Phase 20.11 Context

## Why This Phase Exists

Phase `20.10` deliberately stopped being a retrieval-fix phase and instead delivered the fast-feedback ladder:

- runtime-sync verification
- retrieval-quality harness
- fixed `5`-question canary
- deterministic route-harness cleanup

That means `20.11` must **use** those instruments before changing retrieval code.

## What This Phase Is

This is a **harness-first retrieval-recovery phase**.

It should:
- measure patched-runtime canary retrieval truth first
- then repair the highest-signal retrieval classes exposed by that baseline
- only use larger answer-only proof after the canary and retrieval harness show real improvement

## What This Phase Is Not

It is **not**:
- another benchmark-validation phase
- another route-hint reliability phase
- another full `15`-question inner-loop phase
- another decomposition / bonus / wording tuning phase
- broad aggregate counting optimization as the main goal

## Evidence Standard

The first artifact of `20.11` must be a **pre-change retrieval-harness baseline** on the fixed canary with runtime-sync passing.

No retrieval code change should be treated as valid before that baseline exists.
