# Phase 20.11 Plan

## Goal

Use the `20.10` fast-feedback ladder to recover real retrieval quality on the fixed canary, starting with fact-path parity and the named structural/profile failure classes, without returning to slow blind full-benchmark loops.

## Scope

### In

- patched-runtime `--retrieval-only --canary` baseline reading before any retrieval code change
- fact-path parity recovery on the fact / profile canary cases
- structural route quality recovery on the temporal / aggregate canary cases
- explicit deterministic misroute re-audit, including `f523d9fe`
- canary answer-only proof after retrieval changes

### Out

- full `15`-question split/answer reruns as inner loop
- route-hint replace-vs-fix reopening
- decomposition / bonus / wording tuning
- broad aggregate-counting optimization as the main target

## Execution Steps

1. Workstream A: baseline instrument reading first.
   - Run runtime-sync verification.
   - Run patched-runtime `--retrieval-only --canary`.
   - Persist the baseline route, channel, candidate, and packed-context truth.
   - Do not change retrieval code before this artifact exists.

2. Workstream B: fact-path parity recovery.
   - Use the canary fact/profile cases to inspect why the live fact path remains weaker than the older stronger reference.
   - Focus on:
     - candidate ordering
     - candidate inclusion / exclusion
     - packing relevance
   - Keep `5d3d2817` named explicitly as a profile / identity miss class.

3. Workstream C: structural route quality recovery.
   - Re-audit deterministic misroutes before reading structural-route quality numbers.
   - Keep false-positive temporal and false-negative temporal separate.
   - Inspect temporal / aggregate route quality at three layers:
     - route selection
     - candidate quality / channel yield
     - evidence packing
   - Keep `f523d9fe` as an explicit route-audit case even though it is outside the canary.

4. Workstream D: bounded proof ladder.
   - Re-run the retrieval harness on the canary after changes.
   - Then run canary answer-only.
   - Only if the canary shows material improvement should the phase consider a larger live proof run.

## Implementation Rules

- No retrieval code change before the baseline harness artifact exists.
- Do not skip runtime-sync verification.
- Do not mutate the fixed canary.
- Do not let structural-route work collapse into generic “retrieval is bad” language.
- Do not use full `15`-question live runs as the default debugging loop.
- Do not let aggregate counting absorb disproportionate effort relative to its payoff.

## Output Shape Required From Phase 20.11

- one pre-change canary retrieval-harness baseline artifact
- explicit diagnosis for:
  - fact-path parity
  - profile / identity miss
  - structural route selection vs candidate quality vs packing
- one post-change canary retrieval-harness artifact
- one post-change canary answer-only artifact
- clear statement whether larger proof is justified or should wait

## Exit Gate

- baseline patched-runtime canary retrieval artifact exists
- at least one named retrieval class has a concrete code-level explanation or fix
- post-change canary retrieval artifact is comparable to the baseline
- post-change canary answer-only result exists
- the phase can say whether larger proof is justified without guessing
