# Phase 20.14 Implementation Contract

## Objective

Protect the `20.13` upstream generation win while resolving the next real blocker: temporal chain coverage compression in final selection after the desired events are already available.

## Workstream A: Restored-Event Baseline

- Start from the strongest current bounded proof:
  - oracle-only `session_boundary + 900` on `gpt4_7f6b06db`
- Persist:
  - generated `temporal_event` rows
  - selected `matched` rows
  - selected `recent` rows
  - final selected temporal block
  - effective route bounds / caps
- No selection-layer implementation work may begin before these artifacts exist.

## Workstream B: Selection-Layer Localization

- Treat these as distinct possible blockers:
  - cap arithmetic
  - diversity failure
  - dedupe collapse
- Explicitly verify whether the current temporal caps:
  - `TEMPORAL_CONTINUITY_CAP`
  - `TEMPORAL_RECENT_CAP`
  over-compress the restored chain.
- Explicitly verify whether one trip consumes multiple slots while another receives none.
- Explicitly verify whether `_dedupe_rows` / `_candidate_key` collapses distinct trip evidence that should survive as separate temporal chain items.
- The phase must name which mechanism is dominant if the evidence is mixed.
- This workstream must not silently drift back into:
  - scorer tuning
  - generation-path work
  - trip-name heuristics

## Workstream C: Bounded Selection Fix

- Implement one bounded, generic selection-layer fix.
- Allowed intervention classes:
  - temporal cap rebalance
  - temporal diversity constraint
  - dedupe correction
  - bounded combination if justified by the artifacts
- Preferred generic diversity signals for any temporal diversity constraint include:
  - source turn-number distance
  - source session origin
  - content embedding distance
  - temporal-anchor / observed-at distance
- Content-keyword matching must not be used as the diversity signal.
- The fix must generalize to temporal continuity handling beyond the named handoff case.
- Preserve:
  - the bounded semantic rerank layer
  - the env-gated external scorer hook
  - the bounded `session_boundary` benchmark-path option

## Workstream D: Bounded Proof

- Re-run the targeted oracle-only `session_boundary + 900` retrieval-only proof on `gpt4_7f6b06db`.
- Re-run the fixed canary retrieval-only proof afterward.
- Do not escalate to broader answer-only proof unless the retrieval-only artifacts show real chain-coverage improvement instead of noise.

## Protected Boundaries

### Anti-Benchmaxing Boundary

- No trip-name hardcoding.
- No manual slot rules that only make sense for the named benchmark case.
- No lexical cue patches.
- No reopening of already rejected micro-tuning loops.

### Architectural Boundary

- Do not reopen whether a semantic reranker exists; `20.12` already answered that.
- Do not reopen whether upstream generation exists; `20.13` already answered that under bounded proof.
- Do not turn `session_boundary` into a production retrieval policy.

### Scope Boundary

- The phase is about final temporal chain coverage after restoration, not about upstream extraction capability.
- The phase must explain whether the winning fix improved:
  - coverage
  - diversity
  - or both
- If dedupe is part of the blocker, the phase must say so explicitly rather than folding it into vague “selection quality” language.

## Minimum Evidence Required Before Calling Phase 20.14 Done

- one baseline targeted oracle-only retrieval artifact
- one post-change targeted oracle-only retrieval artifact
- one canary retrieval-only regression artifact
- explicit statement of the dominant selection-layer blocker
- explicit statement whether the final selected temporal block now preserves better distinct trip coverage
