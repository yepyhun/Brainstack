# Phase 20.14 Summary

## Verdict

Phase `20.14` is execution-complete at gate.

It resolved the `20.13` handoff without reopening scorer or generation work. The dominant compression mechanism was not pure upstream absence and not row-level dedupe collapse. It was a bounded selection-path problem:

- temporal coverage was being over-compressed by the current caps **plus** same-bucket reuse inside the temporal selection path
- the temporal route could spend multiple limited slots on near-duplicate rows from the same temporal bucket
- the `recent` and `matched` temporal sections could also reuse the same rows, wasting coverage

The winning fix was therefore a bounded combination:

- temporal diversity by bucket inside temporal selection
- and no reuse of already-selected temporal rows between `recent` and `matched`

No cap increase was required to prove the named chain.

## What 20.14 Implemented

- refactored chronological helpers in `brainstack/executive_retrieval.py` so temporal selection can reason over the full chronologically sorted set instead of only a flat slice
- added a bounded temporal diversity helper
  - bucket key uses `session_id + observed_at day` when available
  - within one bucket, the most recent row wins
  - remaining slots are only filled from leftover rows after distinct buckets are represented
- changed temporal priority selection to use that diversity helper instead of plain chronological slicing
- changed temporal `matched` selection so it excludes rows already chosen for temporal `recent`
  - this prevents the same temporal row from consuming slots in both sections
- mirrored the same selection change into the Bestie runtime plugin copy
- added regression coverage proving:
  - distinct temporal buckets are preferred
  - temporal `recent` / `matched` do not silently reuse the same rows
