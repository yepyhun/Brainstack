# Phase 20.14 Plan

## Goal

Restore enough distinct temporal chain coverage in the final selected block after `20.13` upstream restoration by fixing the smallest generic selection-layer compression issue: cap arithmetic, diversity, dedupe, or a bounded combination of them.

- temporal selection-layer accounting for:
  - `TEMPORAL_CONTINUITY_CAP`
  - `TEMPORAL_RECENT_CAP`

  - `_dedupe_rows` / `_candidate_key`

### Out

## Execution Steps

1. Workstream A: freeze the restored-event baseline truth.
   - Start from the strongest current bounded proof:
     - oracle-only `session_boundary + 900` on `gpt4_7f6b06db`
   - Persist the pre-change truth for:
     - generated `temporal_event` rows
     - selected `matched` temporal rows
     - selected `recent` temporal rows
     - final selected temporal block
     - route bounds and effective temporal caps
   - Explicitly record how many distinct trip-bearing rows exist before selection and how many survive into the final block.
   - Keep the fixed canary ready for post-change regression checking.

2. Workstream B: localize the dominant selection-layer compressor.
   - Treat the following as separate suspects:
     - cap arithmetic
     - diversity failure
     - dedupe collapse
   - Verify whether current temporal caps are too small for the restored chain:
     - `TEMPORAL_CONTINUITY_CAP`
     - `TEMPORAL_RECENT_CAP`
   - Verify whether one trip is consuming multiple slots while another trip receives none.
   - Verify whether `_dedupe_rows` / `_candidate_key` removes distinct trip evidence that should survive as separate chain items.
   - If multiple mechanisms contribute, the phase must name the dominant one and explain why it deserves the first bounded fix.

3. Workstream C: implement one bounded generic selection fix.
   - Implement the smallest generic intervention that restores better distinct temporal chain coverage.
   - Allowed fix classes:
     - temporal cap rebalance
     - temporal diversity constraint
     - dedupe correction
     - a bounded combination if one change alone is clearly insufficient
   - If a temporal diversity constraint is chosen, prefer generic structural signals:
     - source turn-number distance
     - source session origin
     - content embedding distance
     - temporal-anchor / observed-at distance
   - Do not use content-keyword matching as the diversity signal.
   - Prefer interventions that generalize to other temporal continuity questions, not just the named handoff case.
   - Preserve:
     - the bounded temporal semantic rerank layer
     - the `20.13` upstream generation fix
     - donor-first boundaries

4. Workstream D: bounded proof.
   - Re-run the targeted oracle-only `session_boundary + 900` retrieval-only proof on `gpt4_7f6b06db`.
   - Re-run the patched-runtime fixed canary retrieval-only proof afterward.
   - Do not escalate to broader answer-only proof unless retrieval-only artifacts show a real chain-coverage improvement instead of noise.

## Implementation Rules

- Do not reopen scorer-vs-availability confusion.

- Do not turn `session_boundary` into a product-path retrieval rule.
- Keep LongMemEval in measurement role only.

- Prefer the smallest generic selection-layer fix that restores distinct temporal chain coverage.
- If the phase finds that caps are not the dominant blocker after all, it must say so explicitly rather than forcing a cap-only narrative.
