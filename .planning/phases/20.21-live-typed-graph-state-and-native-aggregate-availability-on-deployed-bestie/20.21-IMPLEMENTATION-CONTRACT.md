# Phase 20.21 Implementation Contract

## Objective

Convert the current vague “active backends but no live native aggregate evidence” state into a truthful diagnosis and, if possible, one bounded production-side repair on deployed Bestie.

## Workstream A: Live Typed Graph-State Availability Diagnosis

- Start from the `20.20` closeout artifact set:
  - `reports/phase20/brainstack-20.20-live-debug.json`
  - `reports/phase20/brainstack-bestie-live-reset-eval-isolated.json`
- Produce one explicit end-to-end chain read for a live aggregate scenario:
  - raw Tier-2 LLM output
  - `json_parse_status` and raw preview/tail
  - normalized `typed_entities`
  - reconciler actions / graph-state writes
  - Kuzu native aggregate query result
- Confirm where the live aggregate chain breaks:
  - typed extraction written or not
  - graph persistence populated or not
  - aggregate planner sees required live state or not
  - graph backend query returns candidates or not
  - candidate capture reaches packet debug or not
- Use the known live aggregate cases first:
  - `aggregate_total`
  - `native_total_distance`
- Do not assume strict Pydantic/schema rejection by default.
  - The current implementation uses custom JSON extraction and normalization.
  - Only call it a schema-validation blocker if the raw evidence actually proves that.

## Workstream B: Bounded Availability-Side Repair

- Only land a repair if the missing seam is local, donor-aligned, and production-relevant.
- Preferred repair shapes:
  - carry-through of typed metrics into live graph writes
  - alignment of live write/finalize path with planner prerequisites
  - bounded visibility repair if graph-native candidates exist but disappear before debug capture
- If the live chain shows structurally close but lossy Tier-2 output, a minimal normalization or prompt-tightening repair is allowed.
- Do not jump to database or planner redesign before the chain read proves the data reaches that seam.
- Do **not** respond to this phase by inventing a new aggregate planner or a second parallel native-aggregate path.

## Workstream C: Explicit Native Aggregate Live Proof

- Produce one proof artifact that answers this question directly:
  - does deployed Bestie now surface graph-native aggregate evidence on a known aggregate scenario?
- The proof must report:
  - routing
  - active channels
  - graph candidate count
  - selected `graph_rows`
  - selected `corpus_rows`
  - presence or absence of `native_aggregate`
- A correct final answer is not sufficient proof by itself.

## Workstream D: Post-Fix Isolated Live Re-Baseline

- Only if Workstream B lands a meaningful repair:
  - rerun the isolated deployed-live reset eval
- Compare against current baseline:
  - `4 / 5`
- Explicitly call out whether the aggregate path improved for the right reason and whether unrelated paths stayed stable.

## Workstream E: Honest Stop Boundary

- If the live chain still fails before graph-native candidate production, close honestly with that seam.
- If the required repair widens into a larger architectural availability project, stop and defer instead of stretching `20.21`.
- Do not claim “live graph/native aggregate solved” if `graph candidate_count` remains zero on the known live aggregate proofs.

## Protected Boundaries

### Product Boundary

- Keep the work tied to deployed Bestie product truth, not synthetic benchmark-only evidence.
- Preserve the current isolated live wins unless a regression is explicitly justified by stronger product truth.

### Architectural Boundary

- Stay donor-first and modular.
- Prefer fixing the real live production seam over adding fallback heuristics above it.
- Avoid schema-heavy redesign unless the existing live path is proven insufficient.
- Do not overfit to one model’s quirks without chain evidence; prefer the smallest normalization or carry-through repair that generalizes.

### Anti-Benchmaxing Boundary

- Do not turn this into a benchmark aggregate-expansion phase.
- Use the named aggregate cases only as live diagnostics for the production seam.

## Minimum Evidence Required Before Calling Phase 20.21 Done

- a written diagnosis of the first missing live aggregate-production seam
- either:
  - a bounded repair plus one explicit live native-aggregate proof
  - or an honest blocker closeout with no fake success claim
- if code changed meaningfully, one isolated post-fix live re-baseline
