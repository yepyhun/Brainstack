# Phase 20.21 Plan

## Goal

Diagnose and, if possible, repair the deployed Bestie seam where active graph/corpus backends still produce zero live graph/native aggregate presence on aggregate scenarios.

## Scope

### In

- live typed graph-state availability diagnosis on deployed Bestie
- native aggregate production diagnosis on the live aggregate path
- bounded availability-side fixes if the seam is local and donor-aligned
- one explicit live proof on a known aggregate case
- one post-fix isolated live re-baseline if a real availability fix lands

### Out

- broader aggregate planner expansion
- degraded-stack transcript/temporal heuristics
- new semantic reranking strategy
- deploy packaging redesign
- speculative benchmark-path tuning

## Execution Steps

1. Workstream A: live aggregate availability diagnosis.
   - Start from `20.20` truth:
     - channels are active
     - aggregate route can preserve corpus support in source proofs
     - but live aggregate scenarios still show `graph candidate_count = 0`
     - no selected `native_aggregate` row is present in focused live debug
   - Trace one full live aggregate chain end-to-end:
     - raw Tier-2 output
     - `json_parse_status` plus raw preview/tail
     - normalized `typed_entities` count and content
     - reconciler actions for typed entities / graph state writes
     - Kuzu native aggregate query result
   - Localize the first missing seam on deployed Bestie:
     - typed entity / metric extraction not written
     - graph persistence not populated
     - native planner not triggered despite aggregate route
     - graph query returns empty against real live state
     - candidate creation happens but dies before packet capture
   - Treat “silent drop” as a hypothesis to verify, not an assumption:
     - the current code path is custom JSON extraction plus normalization/reconciliation
     - do not assume a strict Pydantic validation failure unless raw evidence proves it

2. Workstream B: bounded availability-side repair.
   - If the seam is local and donor-aligned, land the smallest repair that restores actual graph-native candidate production.
   - Preferred bounded directions:
     - typed entity / metric persistence carry-through
     - live graph write path alignment
     - native aggregate planner precondition alignment
     - candidate debug capture for graph/native evidence where currently invisible
     - minimal normalization or prompt tightening only if the end-to-end chain shows live Tier-2 output is structurally close but lost before graph persistence
   - Do not widen into new aggregate planner classes here.

3. Workstream C: explicit native aggregate live proof.
   - Re-run at least one known aggregate scenario that should produce graph-native evidence:
     - `aggregate_total`
     - and preferably the stronger `native_total_distance`
   - The proof must distinguish:
     - correct answer by transcript arithmetic
     - correct answer with actual surfaced `native_aggregate` or graph-native evidence
   - If the proof still fails, say exactly where the live chain stops.

4. Workstream D: post-fix isolated live re-baseline.
   - If a real availability fix lands, re-run the isolated deployed-live reset eval.
   - Compare against the current baseline:
     - `4 / 5`
   - Read specifically:
     - whether aggregate truth is now correct for the right reason
     - whether earlier wins stay green
     - whether new graph/corpus availability destabilizes unrelated recall paths

5. Workstream E: honest next-phase decision.
   - If live native aggregate evidence now truly surfaces, say so directly.
   - If the aggregate path still depends on transcript arithmetic, say so directly.
   - If the blocker sits deeper than this phase can fix boundedly, close honestly with a localized seam, not a fake “graph live is done” claim.

## Implementation Rules

- Stay donor-first.
- Prefer production-side availability diagnosis over synthetic benchmark abstractions.
- Treat “channel active” and “candidate available” as different truths.
- Treat raw Tier-2 output, normalized typed entities, reconciler actions, and graph query results as separate checkpoints in the same chain.
- Do not claim success from answer correctness alone; require evidence of graph-native presence.
- Preserve the current `4 / 5` isolated live gains unless the new evidence justifies a tradeoff explicitly.

## Output Shape Required From Phase 20.21

- one explicit diagnosis of where live graph/native aggregate production disappears
- one bounded repair, or an explicit blocker if bounded repair is not honest
- one proof artifact showing whether `native_aggregate` or graph-native evidence is materially present
- one post-fix live re-baseline if code changed meaningfully
- one evidence-led next-thread recommendation

## Exit Gate

- there is direct evidence that the live aggregate path either:
  - now produces materially surfaced graph-native aggregate evidence
  - or is blocked at a specifically localized upstream seam
- `aggregate_total` is no longer treated as sufficient proof by answer correctness alone
- if a repair lands, one isolated deployed-live re-baseline exists after it
- the next-thread recommendation is based on live production truth, not packet-bridge guesswork
