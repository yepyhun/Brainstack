# Phase 35 Plan

## goal

Strengthen Brainstack where a real second brain should add the most value:

- what is active in the user's world and work
- what is still open
- what has been committed
- what should happen next
- what belongs to an external owner

This phase should make Brainstack better at operating truth, not just better at answer or behavior truth.

It should explicitly correct the current maturity imbalance:

- behavior / response truth has matured faster
- operating / world truth is still too derived, too thin, and too continuity-shaped

## scope

### in

- first-class operating truth:
  - add a very small bounded operating-truth layer instead of relying mainly on continuity-derived summaries
  - target candidate record families such as:
    - `active_work`
    - `open_decision`
    - `current_commitment`
    - `next_step`
    - `external_owner_pointer`
  - explicitly define which of these are:
    - user-world truth
    - project/workflow truth
    - external ownership truth
- operating context maturation:
  - make the operating context render real operating records when present
  - reduce over-reliance on continuity `tier2_summary` as a stand-in for current work state
  - stop treating empty external-owner pointers and continuity-derived summaries as acceptable long-term substitutes for first-class operating state
- owner-first retrieval:
  - move retrieval closer to deciding likely authoritative owner first
  - use surface cues more for budget shaping than for deciding what truth class is consulted at all
  - make query-shape cues advisory, not the main judge of what the system is allowed to inspect
- truth class integration:
  - reduce the current “good mini-systems” problem where task truth, graph truth, behavior truth, continuity, and operating context still feel too separate
  - ensure operating truth becomes the substrate that ties these truth classes together without collapsing their ownership boundaries
- narrow typed evidence extension:
  - keep typed evidence honest and bounded
  - extend only where it materially helps operating truth rather than building a broad ontology
  - prioritize claim families that improve actual second-brain continuity over claim families that merely increase graph surface area

### out

- another behavior/compliance-only phase
- broad KG redesign
- broad ontology project
- broad multimodal rebuild
- benchmark-shaped special casing
- unrelated auth or runtime cleanup
- pretending continuity summaries plus heuristics are “good enough” operating truth

## core principles this phase must preserve

- donor-first
- modularity / upstream updateability
- fail-closed upstream compatibility
- no benchmaxing
- truth-first / no "good enough"
- zero heuristic sprawl
- multimodal-first architecture

## architecture decision

### accepted target model

- Brainstack should not only remember how to answer; it should know more about what is going on
- operating truth gets a narrow first-class substrate instead of being mostly reconstructed from continuity summaries
- retrieval should become more owner-first and less query-shape fragile
- typed evidence remains conservative, small, and explicit
- the system should improve world/work continuity without turning into a giant ontology or planner platform
- operating context should progressively render committed operating records first, and derived continuity summaries second
- external ownership should become representable truth, not a permanently empty placeholder lane

### rejected model

- doubling down on style/behavior work while the operating kernel remains thin
- solving operating truth with ever larger cue lists
- pretending continuity summaries are equivalent to first-class operating records
- building a huge generalized knowledge graph to fix a still-bounded kernel problem
- using operating-context prose as a substitute for first-class operating ownership and records

## workstreams

### 1. operating-truth class design

- decide the smallest correct set of first-class operating records
- define ownership, storage, rendering, and retrieval boundaries for them
- keep the set small enough to stay donor-first and maintainable
- define explicit precedence between:
  - operating-truth records
  - task truth
  - continuity-derived summaries
  - graph truth
  - transcript evidence
- specify when an operating record is authoritative and when it is only advisory support

### 2. operating-context uplift

- refactor operating-context assembly so explicit operating records take priority
- keep continuity summary as support, not the primary substitute for operating truth
- surface external-owner pointers where they exist instead of leaving that lane empty
- make the rendered operating context answer these questions cleanly:
  - what are we actively working on
  - what is blocked or undecided
  - what is currently committed
  - what should happen next
  - what belongs to another owner

### 3. owner-first retrieval

- inspect where current routing is too cue-first or query-shape fragile
- move toward a model where likely authoritative owner is selected before aggressive budget pruning
- preserve token discipline without fake forgetfulness
- define a bounded owner-selection flow that prefers likely truth owners before surface-query heuristics prune transcript, graph, corpus, or operating channels
- make it harder for superficial wording differences to change the consulted truth class too early

### 4. narrow typed evidence expansion

- extend typed evidence only for bounded operating claims that materially improve second-brain value
- likely claim families include:
  - works on / active work
  - is at / location-ish state
  - current status
  - promised / committed
  - decision pending / decision made
  - external ownership pointer
- keep extraction intentionally conservative:
  - better to miss weak claims than to inflate the operating substrate with noisy pseudo-facts
  - every new claim family must correspond to a product-valuable operating record or owner signal

### 5. proof

- add proof that the system is better at current-world / current-work continuity, not just rule recall or answer shaping
- validate on source, `hermes-final`, and rebuilt live runtime
- use proof cases where the expected win is:
  - less query-shape brittleness
  - stronger active-work continuity
  - clearer open-decision surfacing
  - less dependence on continuity summaries as the only operating context

## validation ladder

- source:
  - targeted tests for operating record truth, owner-first retrieval, and operating-context rendering
  - tests for precedence between explicit operating records and continuity-derived fallback summaries
- `hermes-final`:
  - parity proof plus direct interpreter proof for the new operating substrate

  - rebuilt runtime smoke showing the system can surface active work / open decisions / current commitments more truthfully and with less query-shape brittleness
  - proof that external owner pointers and next-step style operating records survive ordinary live use when grounded in committed truth
