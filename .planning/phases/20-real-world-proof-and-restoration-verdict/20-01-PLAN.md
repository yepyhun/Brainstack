# Phase 20 Plan

## Goal

Produce a hard final proof and hard restoration verdict for the donor-first Brainstack recovery track.

This phase must determine whether the recovered `L1 + L2 + L3` stack is genuinely closer to the original `big 3 + thin glue` ambition, or whether the project still falls short despite the recovery work.

## Scope

### In

- end-to-end proof across the already-restored `L1`, `L2`, and `L3`
- bounded reruns of the `17`, `18`, and `19` eval ladders as regression gates
- real conversational proof of:
  - smarter follow-up behavior
  - better relation surfacing
  - better reuse of larger prior material
- explicit cross-store consistency and degraded-read proof for:
  - shell state
  - `Kuzu`
  - `Chroma`
- one bounded final-boss benchmark-style run through the real Brainstack path
- a written verdict that routes either to milestone closeout or corrective follow-up

### Out

- new donor architecture redesign
- new memory features unrelated to proof
- benchmark-specific heuristics or data shaping
- hidden `L1`, `L2`, or `L3` rewrites
- milestone closeout without explicit verdict evidence

## Planning Assumption

Phases `17`, `18`, and `19` are now individually restored and verified.

That means Phase `20` is the first phase that must prove the combined system, not an isolated layer.

This phase succeeds only if it can say something stronger than:

- each layer looked better on its own

It must prove:

- the combined system feels stronger in real use
- the combined system stays coherent across its multiple active stores
- the final verdict is hard enough to support or block milestone closeout honestly

## Execution Steps

1. Freeze the proof target.
   - Treat the current recovered architecture as the target under judgment.
   - Record the exact source and runtime surfaces under test.
   - Do not let Phase `20` slide into fresh donor-restoration work unless a correctness blocker is uncovered.

2. Build the evidence matrix.
   - Define explicit evidence buckets:
     - everyday conversational smartening
     - everyday graph usefulness
     - everyday corpus usefulness
     - cross-store resilience / consistency
     - bounded regression ladder reruns
     - final-boss benchmark evidence
   - Each bucket must produce concrete pass/fail evidence, not impressions only.

3. Re-run the bounded layer gates as regression proof.
   - Re-run the fast and medium gates from Phases `17`, `18`, and `19`.
   - Confirm that the combined stack has not regressed any restored layer while assembling the final system.
   - If a gate now fails only because of a Phase `20`-local proof harness issue, fix the harness.
   - If a gate fails because the recovered system itself regressed, route to corrective follow-up.

4. Prove real-world conversational usefulness.
   - Run bounded but realistic multi-turn scenarios that exercise:
     - continuity and “agent got smarter” feel
     - relation and graph usefulness in ordinary questions
     - larger prior material reuse without answer collapse
   - Split the proof so attribution remains clean:
     - Stream A baseline:
       - explicitly disable or degrade graph and corpus channels
       - prove `L1` smartening from continuity + temporal + keyword/lexical behavior only
     - Stream B delta:
       - re-enable graph contribution
       - prove the added usefulness attributable to `L2`
     - Stream C delta:
       - re-enable corpus contribution
       - prove the added usefulness attributable to `L3`
   - Require evidence that the system:
     - carries the important thread better
     - pulls in relevant connected prior knowledge
     - does not become chaotic just because more memory surfaces are live

5. Prove cross-store consistency and degraded behavior.
   - Exercise the publish journal and recovery path with multiple active publish targets.
   - Confirm:
     - pending vs published visibility
     - idempotent replay
     - partial-failure visibility
     - usable degraded reads when a donor backend is unhealthy
   - The goal is not perfect invisibility; the goal is coherent, fail-closed behavior.

6. Run the final-boss benchmark through the real Brainstack path.
   - Use the real Brainstack/Hermes runtime path, not a direct model-only shortcut.
   - Run it only after the cheaper evidence gates pass.
   - Treat the benchmark as verdict evidence:
     - it may confirm the recovery
     - it may block a full restoration verdict
   - Do not tune behavior specifically for the benchmark.

7. Write the restoration verdict.
   - End with one explicit verdict:
     - restored
     - mixed / corrective phase required
     - not restored
   - Support it with evidence from all required buckets.
   - If the verdict is not “restored”, create or route to the corrective phase instead of soft-closing the milestone.

## Implementation Rules

- Phase `20` may fix bounded correctness blockers, but may not reopen donor architecture.
- No new handwritten multilingual heuristics may be added in the name of proof.
- Benchmark-driven prompt or retrieval gaming is forbidden.
- The final benchmark must run through the actual Brainstack path.
- Real-world usefulness evidence is mandatory; benchmark-only proof is invalid.
- Stream A evidence must be isolatable from graph and corpus contribution.
- Stream B and Stream C must show additive usefulness rather than reusing the same blended proof.
- A pretty narrative without hard evidence is invalid.

## Output Shape Required From Phase 20

The implementation must leave behind:

1. a bounded proof harness / evidence set for the combined stack
2. explicit regression results for the restored layer gates
3. explicit cross-store consistency evidence
4. explicit real-world conversational proof notes
5. one final benchmark-style result through the real Brainstack path
6. one written restoration verdict with routing to either closeout or corrective follow-up

## Exit Gate

- the combined stack is judged against the original donor-first ambition, not a weaker baseline
- real-world use shows stronger smartening, graph usefulness, and corpus usefulness together
- cross-store consistency and degraded-read behavior are proven, not assumed
- the final benchmark has been run through the real Brainstack path
- the phase ends with a hard verdict, not a soft “better than before”
- if restoration is incomplete, the phase opens a corrective route instead of pretending closure
