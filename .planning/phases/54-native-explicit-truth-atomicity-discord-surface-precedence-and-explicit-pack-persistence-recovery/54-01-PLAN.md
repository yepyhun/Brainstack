# Phase 54 Plan

## goal

Fix the remaining deep Discord-surface defect family by making native explicit truth writes atomic and first-class, separating transport identity from addressing authority, and persisting explicit multi-rule packs as durable truth without reintroducing Brainstack governance or heuristic schema drift.

This phase is not a behavior-policy phase.
This phase is not a new schema-design phase.
This phase is a host/native-truth contract recovery phase.

## scope

### in

- native explicit write atomicity on the Hermes host seam
- Discord-surface precedence between transport handle and explicit addressing truth
- first-class persistence for explicit multi-rule packs taught in one or two user messages
- one-way native-write -> Brainstack mirror correctness
- live Discord proof of:
  - first greeting / addressing precedence
  - explicit rule-pack persistence
  - reset-safe recall
  - no user-surface leak
- source and runtime fixes only on the narrow host/native-truth surfaces directly involved

### out

- new behavior policy layers
- new ordinary-turn communication-governor logic
- field enums or canonical personal-attribute taxonomies
- locale-specific or language-specific parsing as the main solution
- free-chat style-slot mining as an acceptance shortcut
- reply-time post-processing hacks that mask a weak underlying truth path

## workstreams

### 1. Native explicit truth write contract audit

- identify exactly where explicit user truth is written today:
  - native memory tool
  - host review/write loop
  - Brainstack mirror bridge
- classify the current write path for:
  - atomic single-fact writes
  - multi-fact writes
  - multi-rule / named pack writes
- determine where write completeness is currently being left to model best-effort instead of host contract
- recover a generic host-owned atomic write path that supports:
  - one explicit entry
  - multiple explicit entries in one call
  - one explicit multi-line pack entry
  without introducing field-specific schemas or language-specific parsing

### 2. Discord surface identity and addressing precedence recovery

- separate:
  - transport identity (`LauraTom`-style handle / platform metadata)
  - explicit addressing truth (`Tomi`-style durable preference or identity truth)
- make the user-facing surface prefer explicit addressing truth whenever it exists
- ensure the platform handle remains metadata, not a rival truth owner
- remove rival transport-handle wording from the user-facing prompt path where it is not required for correctness
- prove first-greeting and ordinary-addressing behavior on the running Discord product

### 3. Explicit pack persistence as first-class truth

- make explicit multi-rule teaching land as durable explicit truth instead of:
  - partial best-effort memory writes
  - covert free-chat style mining
  - transcript residue
- preserve genericity:
  - the mechanism must support arbitrary explicit packs, not just style rules
  - the mechanism must not assume a text-only personal schema
- prefer generic atomic native writes over secondary extraction passes, transcript mining, or pack-specific heuristics
- ensure Brainstack mirrors this path without becoming the authority owner

### 4. Narrow code recovery and proof

- limit code changes to the narrowest surfaces that actually own this defect family
- prioritize:
  - `tools/memory_tool.py`
  - `agent/prompt_builder.py`
  - `run_agent.py`
  - Discord/gateway surface wiring only where ownership or metadata precedence truly lives
- keep the fix generic, durable, and inspectable

### 5. Live proof and regression closure

- prove on the running Discord surface:
  - explicit addressing truth wins over handle metadata
  - explicit rule-pack persistence survives reset
  - no tool trace / blocker leak appears
  - Brainstack mirror stays one-way
- rerun the relevant `53` ladder slices that this phase can affect
