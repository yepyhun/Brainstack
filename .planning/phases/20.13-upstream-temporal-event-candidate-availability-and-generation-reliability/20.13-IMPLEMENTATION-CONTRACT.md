# Phase 20.13 Implementation Contract

## Objective

Protect the `20.12` scorer-capability win while resolving the next real blocker: upstream temporal-event candidate availability / generation reliability before reranking.

## Workstream A: Baseline Upstream Reading

- Start with a runtime-sync-passing patched-runtime retrieval-only read on `gpt4_7f6b06db`.
- Persist the pre-change truth for:
  - relevant trip-window Tier-2 generation output
  - persisted `temporal_event` continuity rows
  - bounded temporal-event candidate pool composition before reranking
  - final reranker input composition
- Immediately after the baseline retrieval-only artifact, run one direct continuity-events presence check for the named handoff trip chain.
- Any such DB audit must be scoped to the current proof session or another explicit run identity.
- Mixed-run residue must not be treated as valid evidence.
- This direct check is allowed as a diagnostic shortcut for the explicit handoff case.
- It must not be converted into production benchmark-row logic.
- Keep the fixed canary available for later regression checks.
- No upstream implementation work may begin before these baseline artifacts exist.

## Workstream B: Dominant Dropout Localization

- Use the direct continuity-events presence check as the first branching shortcut:
  - if the expected trip chain is absent from persisted `temporal_event` continuity rows, generation / persistence becomes the first dominant branch to prove or fix
  - if the expected trip chain is present in persisted `temporal_event` continuity rows, surfacing / bounded candidate-pool assembly becomes the first dominant branch to prove or fix unless later evidence disproves it
- Explicitly verify whether dedupe collapses distinct trip evidence before reranking:
  - inspect `_dedupe_rows` / `_candidate_key` behavior on the relevant `temporal_event` candidates
  - distinguish “present in DB” from “available as distinct candidates”
- The phase must explicitly determine whether the desired trip chain is lost at:
  - generation
  - persistence
  - cross-session surfacing
  - bounded candidate-pool assembly
  - pre-rerank filtering
- The decision must be justified against concrete artifacts rather than intuition.
- If multiple layers contribute, the phase must still name which layer is dominant and why it deserves the first bounded fix.
- This workstream must not silently drift into scorer retuning.

## Workstream C: Bounded Upstream Reliability Fix

- Implement one bounded, generic upstream fix against the dominant dropout point.
- Prefer interventions that improve:
  - temporal-event generation reliability
  - continuity persistence / surfacing reliability
  - bounded pool assembly reliability
- Preserve:
  - the bounded temporal semantic rerank layer
  - the env-gated external scorer hook
  - the current donor-first boundaries
- The phase must not convert itself into:
  - a full embedder migration
  - a broad retrieval-stack rewrite
  - a benchmark-specific heuristic patch

## Workstream D: Bounded Proof

- Re-run the targeted patched-runtime retrieval-only proof on `gpt4_7f6b06db`.
- Re-run the fixed canary retrieval-only proof afterward.
- Do not escalate to broader answer-only proof unless the retrieval-only artifacts show material upstream improvement rather than noise.

## Protected Boundaries

### Anti-Benchmaxing Boundary

- No benchmark-row-specific lexical cues.
- No trip-name hardcoding.
- No “solve the LongMemEval question” prompt shaping.
- No reopening of already rejected micro-tuning loops just because they are cheap.
- No DB-audit conclusions from mixed-session or mixed-run residue.

### Architectural Boundary

- Do not reopen the question of whether a semantic reranker exists; `20.12` already answered that.
- Do not treat Jina v5 TEI as a hard architectural requirement.
- It is acceptable to carry Jina v5 TEI forward as the preferred external candidate, but this phase is about upstream reliability, not scorer preference.

### Scope Boundary

- Aggregate counting and typed-entity expansion remain separate tracks unless the new evidence makes them the dominant upstream blocker.
- The phase must leave behind a reusable upstream reliability improvement, not a benchmark-specific patch.
- Explanations must stay layer-correct:
  - if the fix improved visibility only, say so
  - if it improved availability to the reranker, say so
  - if the remaining blocker moved again, say so explicitly
- If dedupe is part of the blocker, the phase must say so explicitly rather than folding it into vague “pool quality” language.

## Minimum Evidence Required Before Calling Phase 20.13 Done

- one baseline targeted retrieval-only artifact
- one post-change targeted retrieval-only artifact
- one canary retrieval-only regression artifact
- explicit statement of the dominant upstream dropout point
- explicit statement whether the reranker now receives a meaningfully better temporal-event pool
