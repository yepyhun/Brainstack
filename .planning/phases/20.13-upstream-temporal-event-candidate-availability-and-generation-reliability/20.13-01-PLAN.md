# Phase 20.13 Plan

## Goal

Make the residual post-`20.12` temporal miss legible and improvable at the upstream layer by proving where desired `temporal_event` chains disappear before reranking, and by landing one bounded generic fix if the failure point is clear enough.

## Scope

### In

- patched-runtime retrieval-only baseline on the named temporal handoff case
- explicit source-of-truth accounting for:
  - temporal-event generation
  - temporal-event persistence
  - temporal-event pool assembly before reranking
- one bounded upstream implementation if the evidence points clearly enough
- fixed-canary regression protection
- anti-benchmaxing and donor-first guards

### Out

- route-hint redesign
- lexical cue tuning for one benchmark question
- re-litigating whether a semantic reranker exists
- full corpus-wide embedder swap by default
- broad aggregate-counting redesign
- full `15`-question live reruns as inner-loop tools

## Execution Steps

1. Workstream A: freeze the upstream baseline truth.
   - Start with a runtime-sync-passing patched-runtime retrieval-only read on `gpt4_7f6b06db`.
   - Persist the pre-change truth for:
     - generated `temporal_event` rows in the relevant trip windows
     - persisted continuity rows that correspond to those events
     - bounded temporal-event candidate pool composition before reranking
     - final reranker input pool composition
   - Immediately after the baseline retrieval-only proof, run one direct continuity-events existence check for the named handoff trip chain.
   - Any direct DB audit for the named handoff case must be session-clean:
     - scope the query to the current proof session or other explicit run identity
     - do not treat cross-run residue as valid evidence
   - This direct read is diagnostic only for the explicit handoff case; it must not turn into production trip-name logic.
   - Keep the fixed canary available for later regression checks.

2. Workstream B: localize the upstream dropout point explicitly.
   - Use the direct continuity-events existence check as the first discriminator:
     - if the expected trip chain is absent from persisted `temporal_event` continuity rows, treat generation / persistence as the dominant upstream branch first
     - if the expected trip chain is present in persisted `temporal_event` continuity rows, treat surfacing / bounded pool assembly as the dominant upstream branch first unless later evidence disproves it
   - Explicitly test whether dedupe collapses distinct trip evidence before it reaches the reranker:
     - inspect `_dedupe_rows` / `_candidate_key` effects on persisted `temporal_event` rows
     - do not assume “present in DB” means “available as distinct candidates”
   - Determine whether the desired trip chain is lost at:
     - generation
     - persistence
     - cross-session surfacing
     - bounded pool assembly
     - pre-rerank filtering
   - Treat this as a structural accounting problem first, not a tuning problem.
   - If the evidence points to multiple layers, the phase must name the dominant one and explain why.

3. Workstream C: implement one bounded upstream reliability fix.
   - Implement the smallest generic intervention that addresses the dominant upstream dropout point.
   - Prefer:
     - candidate availability / assembly reliability
     - continuity surfacing reliability
     - generation-path robustness
   - Preserve the already working bounded semantic rerank layer and env-gated external scorer hook.

4. Workstream D: bounded proof.
   - Re-run the targeted patched-runtime retrieval-only proof on `gpt4_7f6b06db`.
   - Re-run the fixed canary retrieval-only proof afterward.
   - Do not escalate to broader answer-only proof unless retrieval-only artifacts show material improvement instead of noise.

- Do not reopen failed shortlist, query-shaping, or pool-widening ideas.
- Do not write benchmark-row-specific cues or exceptions.
