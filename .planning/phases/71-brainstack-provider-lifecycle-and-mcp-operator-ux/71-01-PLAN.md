# Phase 71 Plan

## purpose

Make Brainstack feel like a practical, drop-in Hermes memory provider by hardening lifecycle hooks, operator health/status UX, and optional MCP/CLI access without increasing host patch surface.

## core question

How can Brainstack adopt Mnemosyne/CerebroCortex provider ergonomics while keeping Hermes runtime ownership and Brainstack's donor-first memory-provider boundary intact?

## GSD planning enforcement check

This phase must preserve:

- donor-first and upstream updateability: reduce patch-heavy host behavior, do not fork Hermes semantics
- synergistic with Hermes: Hermes owns runtime lifecycle; Brainstack exposes memory state and hooks
- fail-closed compatibility: no green provider state when lifecycle hooks or capabilities are inactive
- inspectability: every lifecycle hook must report active/degraded state
- quality priority: recall lifecycle latency must stay live-safe; slow work moves to explicit async/session-end paths

## shared north star and real-world proof gate

North star for Phase 70-77: Brainstack must become easier and safer for Hermes agents to use as a memory kernel without expanding Brainstack into runtime execution, workflow governance, or fake autonomy.

Every phase closeout must prove one of these:

- Hermes can use Brainstack more reliably in a real memory workflow
- or the phase is prerequisite work and proves no regression while naming the exact later phase that must deliver the user-visible improvement

Minimum proof shape:

- targeted tests for the changed surface
- at least one end-to-end Hermes/Brainstack interaction path or Brainstack packet/inspect proof using real provider/store code
- evidence ID/source/provenance proof where memory is involved, not only generated answer text
- no-regression check against Phase 66 golden gates once they exist
- bounded token/output sanity for any new tool, MCP, inspect, corpus, or trace surface

## architecture decisions to freeze before execution

### 1. lifecycle hook inventory

Define which hooks are supported:

- provider init
- system prompt block
- pre-turn recall
- post-turn sync
- session-end maintenance request
- explicit tool export
- health/doctor probe

### 2. optional MCP stance

MCP may expose Brainstack memory operations for operator/dev use, but it cannot become a second runtime authority.

### 2.1 shared-state concurrency rule

If the provider and optional MCP/operator surface can read or write the same Brainstack state, the phase must define:

- one storage owner model per shelf
- write serialization or transaction boundary
- stale-cache/index-resync behavior
- lock/fingerprint/degraded-state reporting where applicable
- failure behavior when concurrent writes or stale derived indexes are detected

The goal is not to copy CerebroCortex's exact SQLite/igraph implementation, but to preserve its useful lesson: a provider and an MCP process sharing storage must not silently diverge.

### 3. host seam contract

Every Hermes integration point must be classified as:

- stable seam
- temporary compatibility patch
- host-owned behavior
- rejected patch
