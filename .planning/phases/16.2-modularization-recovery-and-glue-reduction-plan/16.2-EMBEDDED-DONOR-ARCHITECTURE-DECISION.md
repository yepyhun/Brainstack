# Embedded Donor Architecture Decision

## Status

Accepted architectural decision after the donor re-centering audit.

## Final Direction

Brainstack will **not** pursue either extreme:

- not an all-SQLite memory engine where Brainstack locally reimplements donor intelligence
- not a multi-service donor zoo with several heavyweight servers and Docker dependencies

The accepted direction is:

- Brainstack remains the thin shell
- donor strength is restored with **embedded donor-aligned backends**
- the full stack should run inside the Hermes host process or as tightly embedded local libraries, not as a sprawling service mesh

## Final Role Split

### Brainstack shell

Brainstack-owned responsibilities are:

- orchestration
- **cross-store ingest consistency**
- host boundaries
- safety and guardrails
- recall packaging
- installer / doctor / update-safe integration
- small local shell state:
  - session state
  - profile
  - transcript
  - shell/control-plane metadata

### Layer 1

Target shape:

- **Hindsight / TEMPR-style executive retrieval intelligence**
- semantic + keyword/FTS + graph + temporal fusion
- this is the smartening layer
- this is not a separate long-term storage engine

Implementation rule:

- restore donor-faithful retrieval/fusion behavior
- do not invent a new local retrieval engine if Hindsight already defines the behavior
- do not fall back to handwritten language heuristics

### Layer 2

Target shape:

- **Graphiti-shaped truth / time / relation layer**
- embedded graph backend target: **Kuzu**

Why:

- stronger multi-hop traversal than local SQLite graph approximation
- embedded deployment model
- avoids service-zoo overhead while preserving graph-native behavior

Rule:

- explicit truth remains primary
- current / historical / conflict / inferred separation remains mandatory
- Graphiti-level graph usefulness is the target, not just safe local approximation

### Layer 3

Target shape:

- **MemPalace-style raw corpus retrieval**
- embedded vector/corpus backend target: **Chroma**

Why:

- the strongest proven MemPalace result is raw verbatim retrieval, not AAAK
- L3 should restore real long-horizon corpus retrieval advantage

Rule:

- raw corpus retrieval comes first
- packing/compression is secondary
- do not turn L3 into an overdesigned compression project

### SQLite

SQLite is no longer the target engine for all memory intelligence.

SQLite stays only where it is naturally strong:

- session state
- profile
- transcript
- shell metadata
- local safety/control-plane bookkeeping

It must **not** remain the implicit default for:

- graph-native L2 traversal
- large-scale semantic corpus retrieval
- donor-strength substitution by local glue

## Cross-Store Consistency Rule

This is a first-class shell responsibility, not an implementation footnote.

If ingest touches multiple stores:

- L3 corpus store
- L2 graph store
- Brainstack shell state

then Brainstack must own:

- durable ingest journal
- idempotent writes
- partial-failure visibility
- retry / resume behavior
- explicit published-vs-pending state

The system must not silently claim a memory is fully ingested when only one donor-backed store succeeded.

## Bulk Ingest Rule

Large corpus ingest will require significant L2 extraction work.

Examples:

- many books
- many repos
- long document collections

This likely means many LLM extraction calls for entity / relation creation.

This is accepted, but it must be handled as part of the ingest/orchestration layer:

- batching
- rate limiting
- resumable offline execution
- retry after partial failure

This is **not** a reason to add a new standalone component.
It belongs under cross-store ingest consistency.

## Refresh / Upstream Rule

The long-term refresh model is:

- pinned donor SHAs
- donor contracts per layer
- differential verification
- one-click refresh **workflow**

This does **not** mean magical zero-maintenance sync.
The realistic target is:

- one-click diff + verify
- fail closed if the donor contract is broken

## Non-Negotiable Rule

The recovery track must not settle for:

- “roughly donor-shaped”
- “good enough for now”
- “mostly local but inspired by donor ideas”

The standard is:

- donor-maximal layer restoration
- thin shell around it
- deployable on the target machine
- no handwritten multilingual heuristic fallback as the core intelligence path
