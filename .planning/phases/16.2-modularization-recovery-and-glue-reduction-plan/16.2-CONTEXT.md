# Phase 16.2 Context

## Why This Phase Exists

Phase `16.1` proved that Brainstack is currently closer to:

- a strong integration shell

than to:

- the original donor-first “big 3 plus thin glue” design

That means the next move cannot be another feature.
It must be a module-by-module recovery contract.

This phase exists to answer one concrete question:

- which modules should remain Brainstack-owned shell
- which modules should be thinned
- which modules must be re-centered around donor strength
- which growth patterns must stop entirely

## What This Phase Is Not

This is not:

- a code rewrite phase
- a donor transplant phase
- a benchmark phase
- a “cleanup later” note

It is the hard contract that later restoration phases must obey.

## Recovery Principle

Brainstack is still allowed to own:

- ownership
- orchestration
- safety
- packaging
- host integration

Brainstack is not allowed to keep expanding as the place where donor-strength gaps get hand-patched forever.

## Concrete Pressure Points

From `16.1`, the dangerous middle has become visible:

- `tier1_extractor.py` contains real heuristic growth
- `control_plane.py` carries a lot of local ranking and routing logic
- `retrieval.py` carries a lot of local contract and packaging logic
- donor adapters are thin, but the donor-shaped strength behind them is still under-realized

The modularization problem is therefore not abstract.
It is file-level.

## Target Outcome

By the end of this phase, the project must have a concrete module contract:

- keep local
- thin down
- donor-recenter
- remove / stop growing

This contract must be strict enough that `17`, `18`, and `19` can be executed without drifting back into local glue expansion.

## Hard Rules

- no new feature work inside this phase
- no new heuristics as “temporary help”
- no extra abstraction layer for its own sake
- prefer reducing glue over adding wrappers
- every module kept local must defend itself as:
  - ownership
  - orchestration
  - safety
  - packaging
  - host integration

## Success Criteria

This phase succeeds only if:

- the module contract is concrete
- the contract is donor-first
- the contract is anti-MVP
- the contract reduces future ambiguity for `17/18/19`
