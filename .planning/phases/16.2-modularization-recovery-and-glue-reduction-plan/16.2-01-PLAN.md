# Phase 16.2 Plan

## Goal

Turn the `16.1` donor audit into a concrete modularization recovery contract.

The output of this phase is not new behavior.
It is a binding file-by-file responsibility map for later restoration work.

## Scope

### In

- classify major Brainstack modules
- define what remains Brainstack-owned
- define what must be thinned
- define what must be donor-recentered in later phases
- define which growth patterns are explicitly forbidden
- make `17/18/19` inherit non-MVP, donor-first constraints

### Out

- no direct donor restoration coding yet
- no new benchmark loop
- no new feature work
- no speculative abstraction layer

## Execution Steps

1. Enumerate the major module groups.
   - kernel / store
   - orchestration / routing
   - retrieval / packaging
   - extraction
   - donor seams
   - host integration

2. Classify each module under one of:
   - keep local
   - thin down
   - donor-recenter
   - remove / stop growing

3. Explain the reason for each classification.
   - what real value does it provide
   - what risk does it create
   - what later phase owns its correction

4. Freeze forbidden growth patterns.
   - more regex growth in L1
   - more local ranking glue pretending to be donor power
   - more nominal corpus shelf growth without real corpus strength

5. Write the module recovery contract.
   - explicit enough that `17/18/19` cannot reinterpret it loosely

## Required Output Shape

For each major module or module group:

1. Current role
2. Why it exists
3. Classification
4. Keep / reduce / restore rule
5. Which later phase owns the follow-up

## Hard Rules

- “good enough for now” is not an allowed outcome
- modules may be kept only if they clearly protect Brainstack’s shell role
- donor-strength replacement logic is presumed guilty until justified
- if a module is weak but structurally necessary, this phase must still say how it gets thinned or re-scoped

## Exit Gate

- there is a concrete module-by-module recovery contract
- `17/18/19` inherit explicit donor-first non-MVP constraints
- the plan prefers reduction and scope correction over extra indirection
