# Phase 20.2 Plan

## Goal

Reduce the remaining post-`20.1` final-boss failures by improving **high-precision conversational fact retrieval** without reopening the donor-first architecture.

## Scope

### In

- exact-detail preservation in continuity / transcript evidence
- better answer-bearing conversational turn selection
- stronger user-specific fact grounding where memory is explicit and non-conflicted
- stronger preference for fresher corrected exact values
- bounded multi-entity decomposition for truly complex questions
- a targeted eval ladder on the residual fail class

### Out

- any rollback of `Kuzu` / `Chroma` / donor-first recovery
- broad control-plane redesign
- multilingual regex growth
- new tool-based memory surface
- broad benchmark-only hacks

## Planning Assumption

The current plateau is not evidence that the architecture is wrong.
It is evidence that the current retrieval pipe is still too weak at selecting and preserving exact conversational facts.

This means the phase should prefer:

- precision work over architecture churn
- isolated levers over vague “smarter retrieval” claims
- explicit attribution over one giant bundled patch

## Execution Steps

1. Implement packing fidelity first.
   - make transcript / continuity packing sentence-aware
   - stop cutting words and answer-bearing suffixes into unusable fragments
   - prefer dropping a lower-value row over clipping the best row into garbage
   - scaffold the residual eval ladder immediately for the packing-sensitive fail class so the first lever is measured before later changes blur attribution

2. Implement fact-seeking conversational ranking.
   - improve ranking toward turns that contain answer-bearing details
   - prefer result / outcome turns over planning/setup turns where the question asks for a final number, title, or exact fact
   - prefer turns with numbers / names / explicit values when the query clearly asks for exact recall

3. Implement user-fact grounding.
   - make the rendered memory contract slightly stronger when a concrete recalled fact is present
   - prefer user-stated fact rows above generic assistant restatement where appropriate
   - keep this narrow; no blind “always trust memory” rule

4. Implement exact-value supersession preference.
   - improve preference for newer corrected numeric or exact-value facts
   - ensure the retrieval path can favor the fresher value when old and new values both exist
   - explicitly audit whether the existing graph / reconciler path already captures numeric corrections strongly enough
   - if it does not, strengthen that existing path instead of inventing a second update system

5. Implement bounded decomposition last.
   - only for queries whose residual error class is genuinely multi-entity / multi-event
   - use a single bounded auxiliary step that both decides and emits capped sub-queries
   - the cap is `3` sub-queries maximum
   - fuse sub-query results back into the existing executive retrieval flow

6. Add a targeted residual eval ladder.
   - expand and finalize the ladder that began alongside step 1
   - rerunnable checks for the known fail classes:
     - multi-entity ordering
     - exact title / occupation
     - final amount vs planning context
     - generic-prior override
     - stale value vs fresher corrected value
   - keep a small, explicit fail-set slice in addition to the broader final-boss rerun

## Implementation Rules

- do not make query decomposition the whole story
- do not spend the phase budget on the highest-latency lever first
- do not introduce multilingual alias lists or regex decomposition logic
- do not hide weak retrieval behind more prompt text alone
- do not create a second hidden memory-update path outside the existing graph / reconciler architecture

## Output Shape Required From Phase 20.2

The phase should produce:

- a more faithful packing path
- a more exact-fact-oriented conversational ranking path
- a narrower but stronger grounding rule
- a clearer fresher-value preference path
- an optional bounded decomposition path for real multi-entity questions
- a targeted residual eval slice plus rerunnable broader proof

## Exit Gate

All of the following must be true:

1. at least one residual fail class is improved by packing fidelity without architecture side effects
2. exact-value/result questions prefer answer-bearing turns over nearby planning/setup turns
3. user-specific recalled facts are grounded strongly enough to reduce generic-prior override cases
4. fresher corrected values are preferred more reliably than older stale values
5. if decomposition exists, it uses one bounded auxiliary step rather than a two-call chain or heuristic sprawl
6. the phase reduces the residual error class without reopening the donor-first architecture

If any of these is missing, the phase is not done.
