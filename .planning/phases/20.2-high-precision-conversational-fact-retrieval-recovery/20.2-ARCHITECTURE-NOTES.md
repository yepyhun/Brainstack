# Phase 20.2 Architecture Notes

## Why this note exists

Phase `20.1` moved the final-boss benchmark from `3 / 15` to `9 / 15`.

That confirms the donor-first recovery direction is working.
It also proves the remaining problem is no longer architecture restoration in the broad sense.

The remaining gap is now a narrower but harder class:

- high-precision conversational fact retrieval
- exact-detail carry-through
- update/supersession preference for newer values
- stronger attribution between multi-entity query decomposition and single-fact precision fixes

This note records that residual class so it does not get lost before `20.2` planning.

## What changed after 20.1 rerun

The rerun evidence now says:

- the post-`20.1` final-boss path reached `9 / 15`
- this is a real improvement over the flat `3 / 15`
- therefore the donor-first shell plus `L1/L2/L3` recovery is not the main blocker anymore

The residual failures are narrower:

- some questions still retrieve the right topic but not the exact answer-bearing turn
- some retrieve an older value rather than the fresher corrected one
- some still let generic model prior knowledge override a user-specific recalled fact

## Residual error classes now recorded

### 1. Multi-entity conversational queries still need decomposition

Some failures are not simple single-topic misses.
They ask for the order or relation between multiple distinct events or items.

Examples recorded in discussion:

- book-order questions such as `Hate U Give` vs `Nightingale`
- multi-event shopping / savings chains such as `ShopRite`, `Walmart coupon`, `Ibotta`

The current retrieval pipe is still mostly single-query and limited-budget.
That means it can miss one or more of the sub-events even when each one exists in memory.

Recorded corrective idea:

- decompose the user query before prefetch into sub-queries
- run retrieval on each sub-query
- then fuse the results back into a single evidence set

Recorded implementation options:

- LLM-based decomposition
- NER / entity extraction heuristic path
- hybrid path: decompose when multiple named entities or events are detected

Important recorded correction:

- query decomposition is **not** the answer to every remaining fail
- it appears to directly target the true multi-entity misses
- it should therefore be treated as one lever inside `20.2`, not as a universal cure

### Bounded decomposition rule now recorded

The word `bounded` must not remain vague.

For `20.2`, the acceptable decomposition rule is:

- no two-step `classify first, decompose second` LLM design
- no multilingual regex / named-entity heuristic fallback as the main gate
- if decomposition is used, it should be a **single bounded auxiliary call** that:
  - decides whether decomposition is needed
  - returns up to a small capped number of sub-queries

This is recorded as the preferred multilingual-safe shape because:

- the project cannot drift into language-dependent entity heuristics
- two separate LLM calls would add cost and latency too easily
- a single short auxiliary call keeps the decomposition step constrained enough to fit the existing prefetch path

The decomposition step should therefore remain:

- optional
- capped
- sparse
- and only active on clearly complex retrieval questions

not a universal prefetch default.

The cap is now recorded explicitly:

- one auxiliary decomposition step at most
- at most `3` emitted sub-queries

### 2. Fact-seeking conversational ranking is still too weak

Some failures are not query decomposition problems.
They are precision problems:

- retrieval finds the right topic
- but surfaces planning / setup turns instead of result turns
- or surfaces a related transition without the exact old fact

Recorded examples from discussion:

- charity total: plan/setup context surfaces, but not the final raised amounts
- previous occupation: the system finds the topic transition, but not the exact previous occupation title

This means the remaining retrieval problem is also:

- answer-bearing conversational turn ranking
- especially for turns containing:
  - numbers
  - names
  - explicit titles
  - explicit outcome/result statements

Recorded corrective direction:

- prefer answer-bearing user turns over generic assistant turns
- prefer turns containing exact quantities / named facts when the query asks for an exact value
- distinguish planning/setup evidence from outcome/result evidence

### 3. Packing fidelity still matters after retrieval improves

Even after `20.1`, the packing path can still lose precision if the wrong part of a turn is clipped.

Recorded follow-up idea:

- sentence-boundary-aware truncation
- never cut a word in half
- protect answer-bearing suffixes where numbers or names often occur
- prefer dropping a lower-value turn over clipping the best turn into uselessness

This remains a distinct lever from retrieval itself.

### Execution-priority correction now recorded

The discussion corrected the initial ordering.

Even if query decomposition remains an important residual lever, it is **not** the best first implementation step.

The recorded execution priority is now:

1. packing fidelity
2. fact-seeking ranking
3. user-fact grounding
4. exact-value supersession preference
5. bounded query decomposition

The residual eval ladder should be scaffolded alongside step `1`, then expanded as later workstreams land, so attribution is not lost.

Why this order is now preferred:

- packing fidelity is the cheapest deterministic win
- fact-seeking ranking affects more than one remaining fail class
- grounding is small but real
- supersession is deeper but still bounded
- decomposition is the highest-latency / highest-complexity residual lever and should come later, not first

This priority order is about implementation sequence, not about denying that decomposition remains strategically important.

### 4. User-content priority remains a real lever

The discussion recorded an important asymmetry:

- user turns often contain the fact
- assistant turns often restate, generalize, or drift into generic knowledge

Therefore a follow-up lever is:

- when scoring conversational evidence, prefer user-authored fact turns above assistant paraphrase turns

This is not a donor rollback.
It is a fidelity rule on top of the existing donor-first retrieval path.

### 5. Knowledge-update / supersession for exact values is still incomplete

The `37 -> 38 coins` example shows a remaining update-priority problem:

- the older numeric fact can still win retrieval
- the fresher corrected value does not always outrank it

Recorded corrective direction:

- strengthen exact-value update tracking
- make newer corrected numeric facts supersede older ones more reliably
- ensure Tier-2 / graph-state update logic can mark these corrections explicitly enough for retrieval preference

## Clarification recorded from discussion

The earlier overstatement:

- “query decomposition fixes three misses”

was corrected during discussion.

The more precise recorded view is:

- query decomposition primarily addresses the real multi-entity misses
- single-topic failures like charity totals are better explained by weak fact-seeking ranking
- therefore the residual class should be named **targeted exact-fact error recovery**, not just “add query decomposition”
- bounded decomposition must be concretely defined as a single gated auxiliary call, not as hand-wavy future magic
- the implementation order should start with the cheapest deterministic wins before the highest-latency decomposition work

This wording is important because it prevents `20.2` from turning into a one-lever phase.

## Candidate split structure retired

The earlier candidate split numbering is now explicitly retired.

These had been recorded only as thought-structure, not as committed roadmap phases:

- `20.3` packing fidelity / sentence-boundary clipping protection
- `20.4` user-content priority in conversational evidence ranking
- `20.5` exact-value supersession / knowledge-update preference

After the full `20.2` forensic cycle, that numbering is no longer the right forward structure.

The correct next move is now a **single clean structural phase**:

- `20.3` Structural Retrieval Routing For Temporal, Aggregate, And Fail-Open Intent-Aware Queries

Why the renumbering is now simplified:

- the next work is no longer more `20.2`-style local overlay tuning
- the residual gap is now understood as structural retrieval architecture work
- keeping the old candidate-slice numbers would falsely imply that the next steps are still local tuning shards of the same family

The underlying ideas are still preserved, but they now belong as inputs to the new unified structural `20.3` phase rather than as pre-assigned separate phase numbers.

## Hard truth recorded

The residual `9 / 15` plateau should currently be treated as:

- not a donor-architecture failure
- not proof that `20.1` was insufficient in direction
- but proof that the current pipe is still too weak at **exact conversational fact recovery**

That means the right next move is:

- targeted retrieval-precision recovery
- not another broad architectural rewrite

## Post-execute forensic conclusions

Phase `20.2` did not end as a clean net win.

The real-path rerun moved from:

- post-`20.1`: `9 / 15`
- post-`20.2`: `7 / 15`

That means the `20.2` bundle cannot currently be treated as the final corrective answer.

### What is now proven

- the regressions are real, not just previously failing questions staying failed
- the stopword/token cleanup was not the main reason the regressed questions became decomposition-eligible
- the current decomposition gate is broad enough to fire on all of the benchmark questions that were under discussion
- some exact-fact fixes were still real:
  - Sephora grounding improved
  - the `37 -> 38` coins case improved
  - one compare/order book case improved
- bounded decomposition was the primary regression lever in the initial `20.2` bundle
- the decomposition-off forensic baseline is a real net improvement over post-`20.1`:
  - `9 / 15` -> `10 / 15`
- quote removal alone is materially harmful in that decomposition-off configuration:
  - `10 / 15` -> `5 / 15`
- `digit` removal alone is also materially harmful in that decomposition-off configuration:
  - `10 / 15` -> `7 / 15`
- removing quote and `digit` together is catastrophically harmful:
  - `10 / 15` -> `3 / 15`
- the bonus-forensics matrix is now closed:
  - decomp off, quote on, `digit` on -> `10 / 15`
  - decomp off, quote off, `digit` on -> `5 / 15`
  - decomp off, quote off, `digit` off -> `3 / 15`
  - decomp off, quote on, `digit` off -> `7 / 15`
- the best currently proven configuration is therefore:
  - decomposition off
  - quote on
  - `digit` on
  - `user-led` on

### What is still only a strong hypothesis

- the stronger `authoritative` grounding wording may be an over-strong coupling factor
- but it is still not proven as the primary collapse source without a wording-only A/B
- quote and `digit` being load-bearing does not prove they are principled long-term design
- it only proves that removing them in the current retrieval/grounding stack is highly destructive

Therefore the right immediate next move is not more bundled tuning.
It is split-pass isolation.

## Retrieval problem classes now explicitly separated

The failure discussion now has to distinguish at least these classes:

### 1. Simple exact-fact retrieval

Examples:

- favorite running-shoe brand
- previous occupation

Primary levers:

- ranking
- packing fidelity
- grounding

### 2. Knowledge-update / supersession

Examples:

- `37 -> 38` coins

Primary lever:

- graph-state update preference
- fresher exact values outranking stale ones

### 3. Aggregation questions

Examples:

- total distance across four road trips
- total charity amount

Primary implication:

- these are not just ranking problems
- they need broader recall of multiple relevant events, not only top-K “best” rows

### 4. Temporal ordering / time-difference questions

Examples:

- which book finished first
- trip ordering
- days between two events

Primary implication:

- these are not just ranking problems
- they need timestamp-aware retrieval and rendering, and in some cases explicit ordering logic

## Retrieval architecture conclusion now recorded

The current one-path mental model is too coarse.

It is no longer good enough to say “make the ranking smarter”.

The retrieval layer should now be thought of as at least three modes:

1. relevance retrieval
   - top-K / fused evidence
   - best for straightforward exact-fact recall

2. exhaustive recall
   - broader bounded recall across all relevant events
   - needed for aggregate questions

3. timestamp-ordered retrieval
   - explicitly ordered by event/session time
   - needed for temporal questions

Bounded multi-entity decomposition then becomes a fourth helper lever that can feed one of those retrieval modes when a query truly contains several distinct items/events.

This does **not** mean a donor-first rollback.
It means the `L1` executive retrieval layer needs query-intent-aware routing rather than one universal RRF-shaped answer to every query class.

## Compensatory-stack interpretation now recorded

An additional forensic interpretation is now locked in:

- all four local exact-fact levers proved load-bearing in the current stack:
  - quote bonus
  - `digit` bonus
  - stronger grounding wording
  - keeping decomposition disabled
- but that should **not** be misread as “the local stack is now strongly architected”

The more defensible reading is:

- these overlays are compensating for each other's weaknesses and for deeper retrieval limitations
- they are not evidence that the current local overlay layer is fundamentally strong by itself

This is visible in the score arithmetic:

- post-`20.1` baseline: `9 / 15`
- best post-`20.2` forensic baseline: `10 / 15`
- difference: only `+1`

So even though removing individual levers is destructive inside the current stack, the net gain over the earlier baseline is still small.

This matters because it strengthens the routing conclusion:

- the remaining gap is genuinely structural
- more overlay tuning is now unlikely to buy much beyond the current `10 / 15`

## Clean-slate routing requirement now recorded

The next structural retrieval phase must **not** inherit the current bounded decomposition implementation as-is.

The forensics proved that the existing decomposition gate/path was harmful enough to be disabled in the best current baseline.

Therefore the next retrieval design must start from a clean sheet:

- `temporal` query class -> timestamp-ordered retrieval path
- `aggregate` query class -> exhaustive / widened recall path
- simple fact lookup -> relevance retrieval path without the current decomposition gate
- decomposition may return only as a narrowly justified helper for truly multi-entity queries

And even then:

- it must not reuse the current gate logic by default
- it must not become the universal prefetch path again

So even if the exact numerical ceiling is not yet proven, it is now recorded that:

- ranking-only tweaks are unlikely to be the full path to a stable `14 / 15`
- temporal and aggregate query handling likely need distinct retrieval treatment

## Immediate forensic route now locked

The next diagnostic step should be:

1. rerun the real benchmark with `query_decomposer=None`
2. compare against the `9 / 15` post-`20.1` rerun
3. if the score snaps back up, decomposition is the primary `20.2` regression lever
4. if it does not, ablate the new candidate-priority bonuses next

This is the correct short-term route because:

- it isolates the most likely regression lever first
- it avoids more bundled guesswork
- it keeps the donor-first architecture stable while the retrieval strategy is being corrected

## Epistemic correction about grounding now recorded

The discussion after the `3 / 15` collapse added an important correction:

- it is now legitimate to say the current system is dangerously **coupled**
- it is legitimate to say strong grounding plus weaker retrieval can amplify error rather than reduce it
- it is **not** yet legitimate to say the stronger `authoritative` wording is the proven primary cause of the collapse

That last claim remains only a hypothesis because no grounding-only A/B has been run yet.

Other plausible contributors still exist inside the same bundle, including:

- packing side effects
- exact-value auto-supersession side effects
- bonus interactions unrelated to wording alone

Therefore the grounding conclusion must now be phrased carefully:

- “stronger grounding is a plausible part of the dangerous coupling”
- not:
  - “authoritative wording is already proven to be the main culprit”

Immediate consequence:

- keep the grounding question open as a live hypothesis
- do not collapse the current forensic route into a wording-only diagnosis
- run the quote-off-only third pass first, because it is still the cleanest next isolation step

## Final interpretation to preserve

Both of these statements are now recorded as simultaneously true:

- short-term: run split-pass forensics and isolate the lever that caused `9 -> 7`
- long-term: a stable `14 / 15` path likely requires query-intent-aware retrieval modes rather than a single universal ranking path

This distinction is important.
Without it, future discussion will collapse two different questions into one:

- “what regressed right now?”
- “what architecture is needed to sustain much higher benchmark fidelity later?”

## Benchmark role and anti-benchmaxing correction

The current benchmark discussion is now explicitly anchored to the real product goal:

- the target is still a more accurate, more useful everyday kernel
- the benchmark is a diagnostic lens, not the optimization target

Therefore the following distinction is now recorded:

- structural fixes such as better packing, better routing, better timestamp handling, and better update handling are legitimate capability work
- benchmark-calibrated overlays such as arbitrary numeric bonuses or overly strong prompt wording are suspect unless they can be justified outside the benchmark itself

This means `20.2` should now be interpreted with more nuance than “all fixes were good” or “all fixes were benchmark gaming.”

## Bonus calibration distinction now recorded

Not every `20.2` ranking bonus should be treated as equally suspect.

The current recorded distinction is:

- `digit` bonus and quote bonus are the first candidates for ablation because they look the most benchmark-calibrated and the least donor-grounded
- `user-led` priority is not in the same bucket:
  - it has a real epistemic basis because user-authored turns are often closer to source fact than assistant paraphrase
  - however, the current value may still be too strong and should be reduced rather than automatically deleted if the second A/B pass reaches that point

This distinction matters because “remove every bonus” is too blunt and would lose a potentially real everyday-memory signal.

## Donor-first boundary correction now recorded

The donor-first discussion now distinguishes three cases:

1. orchestration-layer changes that remain legitimate:
   - boundary-aware packing/rendering
   - bounded decomposition at the `L1` orchestration layer
   - retrieval-mode routing decisions

2. overlay behavior that may be useful but must be treated skeptically:
   - ranking bonuses layered above donor-shaped retrieval
   - stronger grounding text

3. shell-side donor-logic drift that should not become permanent:
   - exact-value auto-supersession living in `db.py`

The last item is now recorded as **temporary technical debt**, not as accepted final architecture.

Short-term decision:

- keep the current auto-supersession heuristic for now because removing it immediately would knowingly regress correctness

Long-term decision:

- move supersession responsibility toward the Tier-2 / reconciler path when that path can emit explicit update intent in a future `v2.1`-style follow-up

## Immediate forensic order tightened further

The short-term split-pass sequence is now recorded more strictly:

1. rerun the real benchmark with `query_decomposer=None`
2. keep every other `20.2` lever unchanged in that first rerun
3. only if the score does not recover, ablate the `digit` and quote bonuses next
4. only after that, decide whether the `user-led` bonus should be reduced
5. only after those A/B passes should grounding wording be softened from the current stronger form toward a milder “prefer strongly” style

This ordering is important because otherwise multiple changes move at once and the regression source becomes ambiguous again.

## Long-term routing nuance now recorded

The longer-term routing direction is now recorded more precisely than a simple “intent classifier.”

It should be:

- query-intent-aware
- fail-open
- retrieval-mode aware

It should **not** be:

- a hard exclusive classifier that forces exactly one path and suppresses all fallback
- a regex-heavy multilingual branch table

The desired future shape is:

- temporal-first when the query clearly needs ordering or date difference
- aggregate-widened when the query clearly needs totals across several events
- relevance-first for simpler exact-fact recall
- bounded multi-entity decomposition only as a helper lever
- generic relevance fallback remains available so a wrong intent guess does not collapse the answer path
