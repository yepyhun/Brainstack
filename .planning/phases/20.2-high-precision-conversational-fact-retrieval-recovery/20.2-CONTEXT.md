# Phase 20.2 Context

## Why this phase exists

Phase `20.1` fixed the broad benchmark-exposed fidelity regressions and the final-boss rerun improved from `3 / 15` to `9 / 15`.

That result is large enough to confirm the donor-first recovery direction.
It is also clear enough to show what is still missing:

- the remaining failures are now concentrated in **exact conversational fact recovery**
- the problem is no longer broad architecture restoration
- the problem is now residual retrieval precision and exact-detail selection

## Current diagnosis

The remaining fail classes recorded in GSD are:

1. multi-entity single-query misses
2. weak fact-seeking conversational ranking
3. packing / clipping fidelity losses
4. user-specific fact grounding still not strong enough in some prompts
5. exact-value supersession / update preference still too weak

These are related, but not identical.

This phase must not collapse them into one vague “do query decomposition” story.

## What this phase is

Phase `20.2` is the first **residual exact-fact recovery** phase after `20.1`.

It is intended to:

- improve exact answer-bearing conversational retrieval
- improve preference for fresher corrected values
- improve fidelity of packed evidence reaching the model
- add bounded multi-entity decomposition only where it is truly justified

## What this phase is not

This phase is **not**:

- another donor-architecture rewrite
- an excuse to reopen `L1/L2/L3` ownership
- a benchmark-gaming prompt-hack pass
- a multilingual regex / alias-table growth phase
- a claim that query decomposition alone should solve the remaining plateau

## Architectural constraints carried forward

- Brainstack remains a thin shell around donor-aligned retrieval and storage layers
- `Kuzu` remains the active graph backend target
- `Chroma` remains the active semantic corpus backend target
- the shell does not become a new local memory brain
- changes must stay compatible with the existing executive retrieval contract

## Key planning decision recorded

The implementation order for `20.2` is **not** the same thing as the conceptual error taxonomy.

Execution priority should be:

1. packing fidelity
2. fact-seeking ranking
3. user-fact grounding
4. exact-value supersession preference
5. bounded query decomposition

This order is deliberate:

- cheapest deterministic wins first
- highest-latency / highest-complexity lever last

## Decomposition rule carried forward

If query decomposition is implemented in this phase, it must follow the already recorded hard rule:

- no separate classify-first then decompose-second LLM chain
- no multilingual regex / NER fallback as the primary path
- at most one bounded auxiliary call that both:
  - decides whether decomposition is needed
  - emits a capped sub-query list

## Success condition

The phase is successful only if it reduces the residual `9 / 15` error class by improving exact conversational fact recovery, without reopening the restored architecture.
