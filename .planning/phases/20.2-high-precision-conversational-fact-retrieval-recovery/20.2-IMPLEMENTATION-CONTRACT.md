# Phase 20.2 Implementation Contract

## Purpose

Phase `20.2` must close the residual **exact-fact conversational retrieval** gap that remained after the `9 / 15` post-`20.1` rerun.

This phase is explicitly about precision, not another architecture pass.

## Required Workstreams

### Workstream A. Packing fidelity

Required output:

- sentence-aware continuity / transcript clipping
- no half-word clipping in packed evidence
- better protection of answer-bearing suffixes such as names, numbers, and exact titles

Hard rule:

- this may not bloat the prompt by simply dumping more rows
- it must improve fidelity per row, not just increase text volume

### Workstream B. Fact-seeking conversational ranking

Required output:

- ranking that better prefers answer-bearing conversational turns
- explicit improvement for final-value / exact-title / exact-name / exact-number questions
- reduced tendency to surface planning/setup context when the answer lives in a later result turn

Hard rule:

- this may not become a giant keyword table
- the ranking logic must remain bounded and explainable

### Workstream C. User-fact grounding

Required output:

- stronger preference for concrete recalled user facts over generic assistant-world priors
- explicit preference for user-authored fact turns over assistant paraphrase turns where appropriate

Hard rule:

- no blind “always trust memory” rule
- no broad prompt inflation pretending to solve retrieval weakness

### Workstream D. Exact-value supersession preference

Required output:

- stronger preference for fresher corrected exact values
- clearer handling of cases where an older exact value and a newer exact value both exist
- an explicit audit of whether the existing graph / reconciler path already captures numeric corrections strongly enough
- if that path is insufficient, a strengthening of that existing path rather than a parallel side-path

Hard rule:

- this may not create a second hidden update system beside the graph/reconciler path
- older values must remain historically accessible even if retrieval now prefers the fresher one

### Workstream E. Bounded multi-entity decomposition

Required output:

- optional decomposition path for true multi-entity / multi-event questions
- one bounded auxiliary step that both:
  - decides whether decomposition is needed
  - emits a capped sub-query list
- the cap is `3` sub-queries maximum
- fused sub-query results fed back into the existing executive retrieval contract

Hard rule:

- no two-step LLM classifier + decomposer chain
- no multilingual regex / NER sprawl as the main path
- decomposition must remain sparse and capped
- this workstream may not become the universal retrieval path

### Workstream F. Residual eval ladder

Required output:

- the residual eval ladder must be scaffolded alongside Workstream A, not only after all implementation work is done
- explicit rerunnable checks for the residual fail families:
  - exact occupation/title
  - final amount vs planning context
  - stale numeric value vs corrected newer value
  - generic prior overriding user-specific recall
  - multi-entity ordering / compare questions
- a small targeted fail-set slice in addition to any broader rerun

Hard rule:

- the eval ladder must attribute which lever improved which class
- it may not collapse all gains into one opaque “seems better” verdict

## Protected Boundaries

### Architecture boundary

- `20.2` may not reopen the donor-first architecture
- `20.2` may not demote `Kuzu` or `Chroma`
- `20.2` may not turn the shell back into an all-local memory brain

### Heuristic boundary

- no multilingual regex growth as the main fix
- no giant alias table for entity decomposition
- no heuristic-only decomposition gate pretending to be language-agnostic

### Prompt boundary

- prompt guidance may sharpen grounding
- prompt guidance may not be used as a substitute for retrieval precision
- the phase may not paper over wrong retrieval with louder instructions

### Runtime boundary

- decomposition must be bounded in latency and invocation count
- no more than one auxiliary decomposition call per eligible query
- no more than `3` emitted sub-queries
- the heaviest residual lever must not become default for every trivial query
- the existing executive retrieval contract must remain stable

## Minimum Proof Required Before Calling Phase 20.2 Done

All of the following must be true:

1. packing fidelity measurably improves exact-detail carry-through
2. answer-bearing conversational ranking improves at least one result-vs-plan failure class
3. fresher exact values are preferred more reliably than before
4. user-specific recalled facts are grounded strongly enough to reduce generic-prior override
5. if decomposition is present, it is bounded exactly as specified above
6. the residual fail-set shrinks without architecture churn

If any one of these is missing, the phase is not done.
