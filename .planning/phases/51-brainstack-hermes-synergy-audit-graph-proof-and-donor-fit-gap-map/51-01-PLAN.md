# Phase 51 Plan

## goal

Determine whether Brainstack is truly synergistic with Hermes as a donor-first memory provider, or whether the current integration is only superficially thin while remaining opaque, oversized, or weakly inspectable in practice.

The target is not to justify the current design. The target is to judge it honestly.

## scope

### in

- full graph-based audit on `/home/lauratom/Asztal/ai/finafina`
- runtime seam audit across:
  - `agent/memory_provider.py`
  - `agent/memory_manager.py`
  - `plugins/memory/__init__.py`
  - `plugins/memory/brainstack/*`
  - `run_agent.py`
  - gateway/session surfaces where memory integration materially affects the host
- donor-fit judgment across:
  - seam thinness
  - host blast radius
  - provider optionality and failure behavior
  - inspectability
  - proof quality
  - ongoing maintenance risk
- one explicit verdict:
  - truly synergistic
  - partially synergistic with named gaps
  - or only paper-synergistic

### out

- feature work
- new retrieval capabilities
- new rule machinery
- “fixing” graph visibility with fake wrappers just to make the audit look cleaner
- softening findings to protect earlier decisions

## workstreams

1. Graph proof
- build the full code graph for `/home/lauratom/Asztal/ai/finafina`
- capture architecture overview, communities, hub nodes, bridge nodes, and knowledge gaps
- explicitly measure:
  - host hubs that any memory provider must avoid thickening
  - blast radius of `agent/memory_manager.py`
  - whether Brainstack plugin files are first-class visible in the graph or partly invisible

2. Runtime seam proof
- inspect the actual wiring path from:
  - provider discovery
  - provider loading
  - memory manager registration
  - run-loop system prompt use
  - prefetch
  - sync
  - session end / pre-compress hooks
- judge whether Brainstack acts as:
  - additive memory provider
  - thin orchestration shell
  - or hidden governance layer

3. Synergy criteria
- score the integration against these concrete questions:
  - does Hermes still function cleanly if Brainstack fails or is absent?
  - does Brainstack improve recall/context without thickening host control?
  - is the blast radius centered in the official memory seam instead of spilling into generic host control?
  - can an inspector actually see the integration in the graph and tests?
  - are the biggest remaining risks inside Brainstack itself or in the host path it touches?

4. Donor-fit gap map
- classify findings into:
  - confirmed synergy
  - weak synergy / paper synergy
  - direct anti-synergy
- produce a gap map with:
  - issue
  - why it matters
  - whether it violates a principle
  - whether it is blocking product claims

5. Next-phase recommendation
- if the verdict is good enough:
  - define only the minimum follow-up proof or decomposition work
- if the verdict is not good enough:
  - define the exact corrective phase needed
- do not mix this with feature expansion
