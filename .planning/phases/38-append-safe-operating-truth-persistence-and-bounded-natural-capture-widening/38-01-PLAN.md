# Phase 38 Plan

## goal

Close the next operating-memory correctness gap after `35` and the style-contract hotfix in `37`:

- operating truth currently risks silent overwrite for certain multi-item record types
- operating/task truth still depends too much on rigid structured blocks
- any improvement on that seam must stay bounded, fail-closed, and donor-aligned

This phase should make the operating substrate:

- append-safe where it should be plural
- singleton only where it should truly be singular
- cleaner at bounded structure-promotion from explicit chat material
- still donor-friendly and audit-safe

The target is not “more extraction everywhere” and not “understand every phrasing”.

The target is:

- correct persistence semantics
- more trustworthy operating memory
- a cleaner read/write boundary
- bounded structure-promotion hygiene where the signal is already explicit enough

## scope

### in

- append-safe persistence:
  - re-evaluate singleton vs appendable operating record types
  - remove silent overwrite risk for `current_commitment` and `next_step`
  - preserve dedupe only where the underlying content is genuinely the same
- stable-key correctness:
  - align stable-key construction with the semantic multiplicity of each operating record type
  - keep `active_work` singleton only if that still reflects product truth
  - make multi-item operating truth auditable and replay-safe
- bounded structure-promotion hygiene:
  - improve operating/task promotion only where recent user material already exposes donor-aligned structure strongly enough to map onto existing operating/task record types
  - preserve a hard fail-closed split where ambiguous free-form chat remains continuity-only
  - allow a middle path through bounded pending or non-committed candidates, not through broad free-form canonical inference
  - keep structured explicit capture as the strongest path
- read/write boundary cleanup:
  - re-evaluate the current `prefetch` shape where read-time packet assembly also attempts capture/upsert on some candidate types
  - separate:
    - evidence lookup
    - candidate inference
    - committed truth write
  - keep bounded proactive capture only where the acceptance rule is explicit, auditable, and donor-aligned
  - stop letting ordinary read-path assembly behave like an implicit write path by default
- proof:
  - prove multiple commitments and next steps survive one capture batch
  - prove bounded structure promotion works without introducing phrase farms or language-specific cue ladders
  - prove read-time retrieval no longer mutates state except through explicitly accepted bounded promotion paths
  - prove ambiguous free-form turns still fail closed into continuity-only handling
  - prove no cue farm or ontology sprawl was introduced

### out

- broad operating-kernel redesign
- giant schema expansion
- autonomous planner behavior
- loose implicit capture that writes canonical truth from weak hints
- language-specific phrase tables or regex farms as the main promotion mechanism
- broad natural-language intent routing added to `prefetch`
- broad routing rewrite

## architecture decision

### accepted target model

- only truly singleton operating truths should use singleton stable keys
- commitments and next steps must be able to coexist as multiple committed records
- explicit structured capture should remain the strongest write path
- any widening beyond heading blocks must come from bounded structure-promotion hygiene inside existing operating/task seams, not from free-form semantic phrase hunting
- continuity remains fallback/support, not a covert canonical write lane
- evidence retrieval and committed-memory writes should become more cleanly separated, and any surviving bridge must be explicit, narrow, and auditable

### rejected model

- keep overwriting multiple commitments because the render layer can show a list
- add broad language-specific phrase farms to infer operating truth
- treat `prefetch` as a general-purpose semantic write engine
- treat every chat sentence as operating truth
- solve the issue by pushing more prose into continuity while leaving persistence wrong

## workstreams

### 1. operating persistence audit

- map which operating record types are truly singleton vs appendable
- audit stable-key semantics against the current render expectations
- freeze the accepted multiplicity model before changing writes

### 2. append-safe key and write semantics

- update stable-key logic for plural operating record types
- preserve dedupe only for truly identical or superseded content
- keep writes auditable through metadata and provenance

### 3. bounded structure-promotion hygiene

- improve operating/task promotion only where recent user material already contains explicit enough donor-aligned structure
- keep structured inputs first-class
- keep ambiguous free-form material continuity-only
- avoid turning the parser into a heuristic language trap or broad intent router

### 4. read/write boundary cleanup

- inspect the current `prefetch` flow and identify where read-path packet assembly still attempts write-like upserts
- move toward a cleaner split between:
  - evidence packet construction
  - candidate generation
  - committed write execution
- preserve only the smallest justified proactive write bridge, and document its acceptance rule explicitly

### 5. proof

- add regressions for:
  - multiple `current_commitment` persistence
  - multiple `next_step` persistence
  - bounded structure-promotion acceptance
  - read-path non-mutation except through explicit bounded promotion seams
  - preserved non-promotion for weak implicit hints

## validation ladder

- `hermes-final`:
