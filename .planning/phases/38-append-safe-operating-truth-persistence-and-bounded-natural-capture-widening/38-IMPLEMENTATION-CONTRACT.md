# Phase 38 Implementation Contract

## invariant

Brainstack must not silently overwrite plural operating truths, and any improvement on the operating/task promotion seam must remain bounded, auditable, fail-closed on ambiguity, and free of heuristic sprawl.

## required implementation properties

- `current_commitment` and `next_step` must not silently overwrite distinct committed items
- stable-key semantics must match the real multiplicity of the underlying record type
- natural-chat improvement may come only through bounded structure-promotion hygiene inside existing operating/task seams, not through broad cue farms
- read-time evidence retrieval and committed-memory writes must become more cleanly separated on the operating/task seam
- structured explicit capture must remain the strongest path
- weak or ambiguous hints must still fail closed into continuity-only handling
- any surviving proactive write bridge must have an explicit, auditable acceptance rule
- the result must remain donor-first and upstream-safe

## prohibited outcomes

- singleton overwrite remaining hidden under list rendering
- heuristic language traps for every possible phrasing
- language-specific phrase tables or regex farms as the primary promotion mechanism
- autonomous planner drift
- broad ontology growth
- silent truth loss during migration to append-safe keys
- keeping hidden write-side mutation inside ordinary read-path assembly without explicit bounded justification

## likely implementation seams

- `brainstack/operating_truth.py`
- `brainstack/__init__.py`
- `brainstack/task_memory.py`
- `brainstack/operating_context.py`

## verify contract

- prove multiple commitments persist
- prove multiple next steps persist
- prove bounded structure-promotion acceptance improves without phrase-farm drift
- prove read-time retrieval no longer mutates state except through explicit bounded promotion seams
- prove weak hints still do not overcommit
- prove parity on source, `hermes-final`, and live runtime
