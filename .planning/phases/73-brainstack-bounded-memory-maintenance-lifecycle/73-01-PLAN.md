# Phase 73 Plan

## purpose

Add explicit, inspectable memory maintenance for dedupe, supersession cleanup, stale evidence handling, conflict hygiene, and bounded consolidation without adopting unbounded dream-engine behavior.

## core question

How can Brainstack gain the practical benefits of Mnemosyne sleep and neural/Cerebro consolidation while remaining deterministic, bounded, evidence-preserving, and non-governing?

## GSD planning enforcement check

This phase must preserve:

- truth-first: maintenance cannot make hidden destructive changes
- fail-closed compatibility: failed maintenance must be visible and non-green
- no benchmaxing: maintenance must improve real durability and recall quality
- no bandaid patching: cleanup rules must be reusable and shelf-aware
- inspectability: every maintenance decision needs dry-run/receipt visibility

## shared north star and real-world proof gate

North star for Phase 70-77: Brainstack must become easier and safer for Hermes agents to use as a memory kernel without expanding Brainstack into runtime execution, workflow governance, or fake autonomy.

Every phase closeout must prove one of these:

- Hermes can use Brainstack more reliably in a real memory workflow
- or the phase is prerequisite work and proves no regression while naming the exact later phase that must deliver the user-visible improvement

Minimum proof shape:

- targeted tests for the changed surface
- at least one end-to-end Hermes/Brainstack interaction path or Brainstack packet/inspect proof using real provider/store code
- evidence ID/source/provenance proof where memory is involved, not only generated answer text
- no-regression check against Phase 66 golden gates once they exist
- bounded token/output sanity for any new tool, MCP, inspect, corpus, or trace surface

## primary goals

- add bounded maintenance operations
- preserve evidence ids and provenance
- reduce duplicate/stale/conflicting memory
- expose dry-run and receipt outputs

## explicit scope model

### Brainstack-owned scope

- dedupe candidates
- supersession cleanup
- stale evidence downranking
- conflict hygiene
- safe summary/compaction candidates
- maintenance receipts

### out of scope

- autonomous hidden dream loop
- broad salience engine
- destructive summarization
- host-side scheduling
- assistant self-reflection promotion

## architecture decisions to freeze before execution

### 1. maintenance modes

Support at minimum:

- dry_run
- apply_bounded
- inspect_last_run

### 2. receipt shape

Every run reports:

- candidates scanned
- proposed changes
- applied changes
- skipped/no-op reasons
- affected evidence ids
- latency and errors

### 3. scheduling boundary

Brainstack may expose a maintenance request hook. Hermes runtime decides when to call it.

## workstreams

1. Inventory duplicate/supersession/stale/conflict patterns.
2. Define maintenance candidate model.
3. Implement dry-run first.
4. Add bounded apply mode.
5. Add tool/lifecycle integration and receipts.
6. Add regression tests proving no evidence loss.

## execution sequencing

1. Add dry-run candidate generation.
2. Add receipts and inspect reporting.
3. Add bounded apply for one maintenance class at a time.
4. Integrate Phase 70 maintenance trigger.
5. Run Phase 66 golden no-regression gates.

## deliverables

- maintenance candidate model
- dry-run and bounded apply modes
- maintenance receipts
- inspect/doctor maintenance state
- no-evidence-loss tests

## phase-specific real-world proof examples

Closeout must include at least:

- dry-run proof that duplicate, stale, or conflicting candidates are identified without mutation
- apply proof for one safe maintenance action with before/after receipts and preserved source evidence IDs
- failure proof where maintenance errors leave durable truth unchanged and visible as failed/degraded
- recall proof that maintained memory is no worse than before on relevant Phase 66 golden cases

## acceptance criteria

- maintenance is useful without hidden autonomy
- no durable truth is destroyed without trace
- duplicate/stale/conflict cleanup is measurable
- failures are visible and fail closed
