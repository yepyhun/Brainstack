## Runtime Reset Hardening
- The live smoke investigation uncovered a separate correctness gap: clearing Brainstack DB and state DB alone was not enough because Docker-persisted session replay files under `/opt/data/sessions/` could still survive and repopulate apparent memory.
- The Brainstack-generated Docker helper now owns a real conversational reset path:
  - `purge` stops the service and clears Brainstack DB, state DB, runtime status files, session replay files, and native-memory persistence directories
  - `reset` performs that purge and starts the service again
- Both destructive actions now require an explicit `DELETE` confirmation, and the desktop wrapper surfaces them as `WARNING DELETE EVERY MEMORY` options instead of quiet housekeeping actions.
- The cleanup runs through `docker compose run --rm ...` against the mounted `/opt/data` volume so it still works when container-owned permissions make host-side deletion incomplete.
- The generated helper contract is now covered by a source-side regression test in `/home/lauratom/Asztal/ai/atado/Brainstack/tests/test_install_into_hermes.py`.
