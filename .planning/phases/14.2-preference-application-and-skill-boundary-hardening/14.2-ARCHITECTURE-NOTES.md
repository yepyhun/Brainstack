# Phase 14.2 Architecture Notes

## Why This Phase Exists

The live Discord smoke proved that Brainstack durability improved, but two important correctness gaps remain:

- freshly stated communication preferences are not applied fast enough in the next replies
- `skill_manage` can still be used for user-profile style memory, which creates a parallel side channel instead of keeping Brainstack as the sole owner of personal memory

This is not a reason to remove Hermes skill learning entirely. The problem is routing, not the mere existence of skills.

## Ownership Contract

### Brainstack owns

- user identity facts
- user age and stable biography facts
- communication style preferences
- project context and shared work continuity
- temporal state changes
- graph truth and relations

### `skill_manage` may still own

- reusable procedures
- repeatable workflows
- skill-style playbooks for future tasks
- maintenance of existing skills when a real workflow changed

### `skill_manage` must not own in Brainstack-only mode

- user profile facts
- communication preferences
- identity or relationship facts
- personal style instructions like emoji / dash / jargon rules

## Required Fix Shape

- do not globally ban `skill_manage`
- add a scoped guard so user-profile style writes are rejected or rerouted in Brainstack-only mode
- strengthen prompt/runtime steering so communication preferences get a faster application path than the delayed Tier-2 durable path alone
- normalize duplicated identity and relation writes such as `User` vs `Tomi`

## Anti-Patterns

- no full `skill_manage` amputation
- no new memory runtime
- no new storage layer
- no prompt-only band-aid without runtime enforcement
- no acceptance of parallel personal-memory owners
