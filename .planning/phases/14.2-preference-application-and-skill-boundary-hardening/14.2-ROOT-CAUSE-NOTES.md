# Phase 14.2 Root Cause Notes

Date: 2026-04-11

## Why This Note Exists
- Live Discord testing after the nominal `14.2` execute showed that Brainstack now stores many of the right facts, but still fails to apply some of them cleanly in the next replies.
- A heuristic-expansion branch was started and then intentionally stopped.
- These findings must survive context compaction.

## Confirmed Live Symptoms
- Bestie stores user identity and communication preferences, but still replies with emoji, dash-like style, or over-friendly filler after those preferences are stated.
- Bestie can narrate internal memory scaffolding back to the user:
  - “BrainStack visszatolja nekem...”
  - “az emlékezet azt mondja...”
- Bestie can still reach for generic host tools around personal style capture:
  - `skill_view` / `skill_manage` on `humanizer`
  - `read_file` / `write_file` against `~/.hermes/notes/...`
- After reset, ephemeral session state was cleared correctly, but long-lived Brainstack facts still flowed back, which is expected.
- The visible problem is not “nothing is stored”; it is “stored facts are not being packaged and applied correctly”.

## What Was Confirmed Not To Be The Main Bug
- The desktop launcher path is correct.
- Reset of ephemeral session data worked when `sessions/`, `state.db`, and Brainstack data were cleared.
- The `skill_manage` path being open is intentional and must remain available for reusable procedures.
- The main bug is not “skill learning exists”.
- The main bug is not “Tier-1 extracted nothing”.

## Confirmed Root Causes

### 1. Backflow / return-path weakness
- Core style rules are written as memory facts, but they are not projected back as a short, active communication contract.
- The system stores the rule, but does not re-present it strongly enough as an immediate behavior requirement.

### 2. Internal memory privacy is too weak
- The host wraps recalled memory as generic background information.
- The wrapper does not say strongly enough that this is private internal context that must be applied silently.
- This allows the model to mention memory headings, policy scaffolding, or “the memory says...” language in user-visible replies.

### 3. Retrieval packaging is too literal
- Overlapping style/profile rows are fed back as multiple separate memory items.
- The current merge logic mostly deduplicates by exact `stable_key` or exact content.
- That is too weak for style and communication rules, where multiple rows can describe the same intent.

### 4. Continuity can absorb leaked internal narration
- Once the assistant narrates memory internals to the user, that text can itself be summarized into continuity.
- This creates a feedback loop where internal scaffolding becomes part of later recall.

### 5. Host tool guidance still over-invites generic tool use
- In Brainstack-only mode, legacy `memory` and `session_search` are gated, but the host still carries broad generic guidance about persistent memory, skill creation, and file/tool-first behavior.
- This leaves room for the model to solve a personal-preference request by reaching for:
  - `skill_view`
  - `skill_manage`
  - `read_file`
  - `write_file`
  instead of staying inside Brainstack-owned personal memory.

### 6. Ownership boundaries are stronger for `skill_manage` than for adjacent file workflows
- `skill_manage` has scoped blocking for personal/profile/style payloads.
- The same boundary does not currently extend to generic note-file workflows, so the model can still try to externalize personal style memory into ad hoc files.

### 7. Provider timeouts create interrupted half-actions
- Live logs showed repeated `Interrupted during API call` and `Connection to provider dropped (ReadTimeout)` events in the middle of tool-heavy preference-capture attempts.
- This is not the root cause of the ownership bug, but it makes the visible behavior noisier and can leave partially executed side paths.

## Important Architectural Decision
- Do **not** solve this by expanding the heuristic layer further.
- Existing Tier-1 bootstrap heuristics may remain for now.
- New heuristic growth was explicitly rejected in this session.
- The next correction should target:
  - read path
  - backflow
  - private recall packaging
  - retrieval-time collapse of overlapping style rules

## Explicitly Rejected Direction
- No new broad regex growth for communication/style semantics.
- No full `skill_manage` amputation.
- No new runtime.
- No prompt-only band-aid without runtime carry-through.

## Preferred Next Technical Direction

### A. Build an explicit current communication contract
- Derive a short, imperative “active communication contract” from current Brainstack style/profile facts.
- Feed that contract back in a compact, high-priority form.
- This should be current-state oriented, not a dump of all style rows.

### B. Make recalled memory private-by-default
- Strengthen the host recall wrapper so it clearly instructs the model:
  - this is internal recalled context
  - apply it silently
  - do not mention memory blocks, headings, conflict notes, or storage details
  - only discuss memory internals when the user explicitly asks about debugging or architecture

### C. Collapse overlapping style rows at retrieval time
- Prefer current canonical style facts over raw overlapping rows.
- Fix the packing stage rather than growing the write heuristics.

### D. Keep skills, but keep ownership boundaries
- `skill_manage` remains available for reusable procedures/workflows.
- Personal identity, communication style, and project-memory facts stay Brainstack-owned.

### E. Tighten adjacent host nudges around personal-memory capture
- Brainstack-only mode should not encourage the model to solve personal style/profile capture through generic notes or skill inspection/writes.
- The next fix should target host guidance and routing boundaries, not just Brainstack storage.

## Current View On The Existing Tier-1 Heuristics
- The presence of rules like `vibecoder / nem developer` is not itself the current blocker.
- The wrong direction would be to keep adding more such rules as the main fix.
- Do not expand this layer further unless there is a later explicit architectural decision to do so.

## Relation To The Hindsight Donor
- The Hindsight donor direction does **not** justify more hand-written preference regexes.
- Hindsight's retain path is documented as LLM extraction plus normalization of facts, temporal data, entities, and relationships.
- That means the current Brainstack hand-written Tier-1 bootstrap is our own bounded compromise, not something inherited from Hindsight.
- Architectural conclusion:
  - keep the existing bounded bootstrap for now
  - do not grow it further
  - move the next correction toward better packaging, current-contract projection, and private recall behavior
