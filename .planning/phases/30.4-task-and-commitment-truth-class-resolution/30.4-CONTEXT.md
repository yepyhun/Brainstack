# Phase 30.4 Context

## Why this phase exists

The product now behaves like a second brain in conversation, but task/commitment truth is still unresolved. Live evidence shows this gap clearly:

- explicit task-like user intent appears in conversation
- the assistant can speak as if it saved the tasks
- reset-era recall is not dependable
- the current architecture explicitly keeps task truth outside Brainstack, but the product surface does not yet honor that boundary truthfully

## Inputs

- `30.2` should settle what continuity/task recall can and cannot do
- `30.3` should settle truthful receipts, barriers, and lookup semantics
- this phase exists because the remaining issue is no longer just retrieval quality; it is ownership and truth-class design

## Non-goals

- no full productivity suite
- no giant general-purpose project system
- no vague backlog placeholder

## resolution snapshot

- the evidence from `30.2` and `30.3` resolved the ownership ambiguity:
  - continuity and session search are not enough for task truth
  - truthful receipts without a real task owner are still insufficient
  - the stronger architecture for this product slice is bounded Brainstack-owned task memory
- the remaining non-blocker after verify is not ownership ambiguity anymore; it is eager Chroma backend warmup in empty temp runtimes
- that residual stays named and separated from task-memory correctness

`xhigh`
