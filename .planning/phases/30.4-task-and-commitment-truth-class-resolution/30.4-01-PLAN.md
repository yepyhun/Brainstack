# Phase 30.4 Plan

## goal

Resolve the task / commitment memory gap with a first-class, durable, inspectable product contract: either Brainstack-owned task truth or a real external-owner bridge with durable pointers, receipts, and retrieval.

## scope

### in

- truth-class decision for:
  - daily tasks
  - commitments
  - optional tasks
  - date-scoped agenda items
- relative-date canonicalization for user timezone
- durable storage and retrieval semantics for the chosen architecture
- inspection and recall semantics for:
  - today
  - yesterday
  - explicit date
  - optional vs required tasks

### out

- full project/task-management product
- calendar UI
- generic workflow engine beyond the bounded memory contract

## core principles this phase must preserve

- donor-first
- single durable owner per truth class
- truthful ownership claims
- explicit date semantics
- no transcript-search masquerading as task truth
- inspectable committed state

## architecture decision

### accepted target model

- choose one of two defensible architectures:
  - Brainstack becomes the bounded durable owner for task/commitment memory
  - or Brainstack integrates a real external owner and keeps durable pointers plus retrieval/receipt semantics
- whichever model is chosen must support:
  - commit receipts
  - due-date normalization
  - optionality
  - status
  - structured lookup before transcript fallback

### rejected model

- pretending continuity summaries are enough
- pretending session search is task truth
- continuing to say “I saved it” without a real durable owner

## workstreams

### 1. truth-class choice

- choose bounded Brainstack-owned task truth or real external-owner bridge
- document why the rejected path is worse for this product

### 2. task data model

- define task/commitment fields that are unlikely to change:
  - title
  - due-date/date-scope
  - optionality
  - status
  - source turn/session
  - ownership metadata

### 3. retrieval contract

- define structured lookup semantics for:
  - `Mik a mai feladataim?`
  - `És tegnapra?`
  - `Tegnap előtt?`
- define fallback behavior when no structured record exists

### 4. product truthfulness

- align assistant language with the real owner
- forbid false durable-memory claims

## validation ladder

- prove explicit task capture
- prove reset persistence
- prove date-scoped lookup
- prove optional-task preservation
- prove truthful not-found semantics

## anti-goals

- do not leave task truth implicit
- do not hide owner ambiguity behind friendly assistant language
- do not rely on transcript FTS as the main retrieval path

## execution outcome

- chosen architecture:
  - Brainstack is now the bounded durable owner for explicit task and commitment truth
  - broader reminders / scheduling authority still remains outside Brainstack unless a future explicit owner bridge is introduced
- implemented contract:
  - explicit task capture commits pre-answer with truthful receipts
  - task rows survive reset
  - Hungarian date-scoped asks resolve through structured lookup first
  - optionality survives capture and recall
  - task-like retrieval no longer pretends transcript or corpus lookup is the primary owner

## verify findings

- source tests proved:
  - explicit capture writes durable task rows
  - reset persistence holds
  - structured lookup returns committed rows
  - corpus retrieval is disabled for task-like lookup
- installed `hermes-final` doctor proved:
  - shell `brainstack_doctor.py --json` no longer false-reds docker dependency checks
- rebuilt live runtime proved:
  - committed receipt status is `committed`
  - owner is `brainstack.task_memory`
  - task rows survive provider reset
  - task lookup is authoritative from structured rows

## named residual

- opening an otherwise empty temp provider in the container can still warm Chroma's ONNX cache
- this is explicitly not being misreported as a `30.4` task-memory failure
- fixing it cleanly would require a broader lazy corpus-backend lifecycle change
