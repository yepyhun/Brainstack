## Workstream Findings

### A. Fact-Path Parity

Measured scores on the same fixed-size subset:

- `20.2` decomp-off historical baseline: `10 / 15`
- `20.3` legacy comparability before fixed clock: `3 / 15`
- `20.4` answer-only comparability with fixed clock: `5 / 15`

What this proves:

- the old `10 / 15 -> 3 / 15` collapse was **not** fully attributable to Brainstack memory-context differences
- part of the drop was runtime/prompt instability
  - the fixed benchmark clock raised answer-only comparability from `3 / 15` to `5 / 15`
- but the parity gap is still real
  - `5 / 15` remains far below the earlier `10 / 15` decomp-off baseline

Therefore the correct `20.4` parity verdict is:

- measurement/runtime drift is a real part of the gap
- but not the whole gap
- a live fact-path restoration problem still remains open

### B. Route-Hint Reliability

Live route-hint findings after direct injected resolver:

- `route_resolver_injected=true`: `15 / 15`
- `route_source=fallback`: `0 / 15`
- `route_source=direct_benchmark_route_hint`: `1 / 15`
- `route_source=default`: `14 / 15`
- `requested_mode=fact`: `15 / 15`
- `applied_mode=fact`: `15 / 15`

What this proves:

- the previous live `401` route-hint failure was a benchmark-runtime configuration/wiring issue
- the injected direct resolver removed that blocker
- however, the benchmark subset still almost never requests a non-fact route under the current gate

Therefore the correct `20.4` route verdict is:

- route-hint runtime reliability is now materially repaired
- structural route activation coverage is still too low to move this subset in practice

  - `6a1eabeb`

  - `9ee3ecd6` remains outside clean retrieval success because the `100`-point answer is not stated directly in memory
  - `69fee5aa` remains outside clean retrieval success because `38` depends on integrating update evidence rather than simple recall

- a safer working estimate is closer to `~4 / 15` than to `2 / 15`
- but retrieval is still far from restored, and the split report must not be reinterpreted as “retrieval is basically fine”

What this proves:

- the split instrumentation remains necessary
- raw answer score still overstates what Brainstack memory itself reliably supplied
- the answer judge and retrieval judge are still not clean enough to collapse back into one metric

Therefore the correct `20.4` proof verdict is:

- proof reliability improved
- but judge calibration is still an open problem, not a solved one
- the first explicit workstream of the next phase should be retrieval-judge calibration and manual relabeling of the fixed-clock `20.4` split rows before any new retrieval score is treated as architectural truth
