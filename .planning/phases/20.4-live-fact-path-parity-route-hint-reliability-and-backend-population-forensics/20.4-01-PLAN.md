# Phase 20.4 Plan

## Goal

Explain and harden the live benchmark/runtime path after Phase `20.3`, with the first priority being fact-path parity:

- why the current `20.3` legacy runner fell to `3 / 15`
- why live route hinting fell back with `401`
- why `Kuzu` population is sparse while `Chroma` is dense
- how much of the remaining proof instability is retrieval versus judge behavior

## Scope

### In

- fact-path parity audit between:
  - `20.2` decomp-off best-known baseline
  - current `20.3` legacy answer-only runner
  - explicit documentation of the runtime environments behind both measurements
- live route-hint runtime/configuration audit
- benchmark-path graph publication / population audit
- proof/judge calibration based on the new split runner
- bounded remediation where the root cause is explicit and local
- bounded reruns to prove the fixes

### Out

- new route families
- new decomposition experiments
- new benchmark-shaped scoring overlays
- donor replacement
- broad architecture churn beyond the measured blockers

## Planning Assumption

The main planning assumption is now:

The current live failure surface is dominated by a runtime/reliability + benchmark-path parity problem, not by lack of another new routing idea.

That assumption is justified by:

- legacy runner on current `20.3` stack: `3 / 15`
- split runner on current `20.3` stack: `5 / 15`
- live route-hint fallback in `14 / 15`
- `Kuzu` sparsity with `Chroma` density
- the fact that structural routing was barely exercised live

## Execution Steps

1. Audit fact-path parity first.
   Compare the earlier `20.2` decomp-off winning path against the current `20.3` fact-fallback path at the packet/render/budget/runtime level.
   Document the exact runtime environment behind the earlier `10 / 15` baseline before treating the comparison as code-only parity.

2. Audit route-hint runtime reliability.
   Trace the auxiliary route-hint caller in the actual benchmark runtime and explain the `401` path concretely.

3. Audit benchmark-path graph publication.
   Trace how benchmark seeding is expected to publish graph data, where `Kuzu` counts are supposed to increase, and why they currently remain near-empty.
   Include whether graph publication depends on Tier-2 extraction completing inside the per-question seeding window or whether `TemporaryDirectory` teardown can happen before graph data is written.

4. Audit proof and judge calibration.
   Separate:
   - real retrieval absence
   - retrieval present but judge says no
   - suspicious answer judge yes-pass
   For suspicious answer passes, surface the actual memory context that reached the LLM so the phase can distinguish missing evidence from judge error.

5. Implement only bounded fixes that directly answer measured blockers.
   No speculative redesigns.

6. Re-run bounded proof under both:
   - legacy answer-only runner
   - current split runner
   so the parity and proof stories remain comparable.

## Implementation Rules

- do not reopen `20.2` tuning
- do not treat route-hint `401` as an architecture verdict
- do not treat `Kuzu` sparsity as a full retrieval verdict until publication/audit is traced
- do not let proof hardening blur into changing the product to satisfy the judge
- do not call the phase done with raw answer score alone
- do not accept “not broken” framing; each component must be:
  - correct for the current phase contract
  - explicitly pending
  - or wrong and fixed now

## Output Shape Required From Phase 20.4

This phase must produce:

1. a fact-path parity finding
2. a live route-hint runtime/configuration finding
3. a benchmark-path `Kuzu` publication/population finding
4. a proof/judge calibration finding
5. bounded rerun evidence showing whether the fixes changed:
   - legacy answer-only score
   - split answer score
   - retrieval-correct score
   - route application / fallback behavior

## Exit Gate

Phase `20.4` is planned correctly only if:

- fact-path parity is the first explicit workstream, not an afterthought
- route-hint `401` is treated as a runtime blocker to be explained
- `Kuzu` sparsity is treated as a benchmark-path publication question, not hand-waved
- proof/judge hardening is explicit and separate from retrieval changes
- later reruns are required to be comparable across both runner styles
- the phase stays forensic-and-hardening focused rather than drifting into new architecture invention
