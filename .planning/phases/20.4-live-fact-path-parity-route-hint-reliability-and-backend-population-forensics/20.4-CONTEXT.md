# Phase 20.4: Live Fact-Path Parity, Route-Hint Reliability, And Backend Population Forensics - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

## Why this phase exists

Phase `20.3` is now fully executed and its bounded forensic bundle is complete.

That bundle proved:

- the current `20.3` stack regressed on the legacy answer-only runner to `3 / 15`
- the current split-proof rerun on the same stack reached only `5 / 15`
- the live routing layer was not actually exercised:
  - `applied_mode=fact` in `15 / 15`
  - `route_source=fallback` in `14 / 15`
  - dominant fallback reason: route-hint `401` failure in the target runtime
- `Chroma` is dense in the benchmark path
- `Kuzu` is sparse in the benchmark path
- raw answer score still overstates Brainstack memory quality

This means the next phase cannot be another routing redesign or another overlay-tuning loop.
It must first explain the live runtime and parity failures that are now measured.

## What this phase is

Phase `20.4` is a bounded forensic-and-hardening phase focused on four concrete questions:

1. why did the fact path fall from the earlier `20.2` decomp-off `10 / 15` baseline to `3 / 15` on the current `20.3` legacy runner?
2. why is live route hinting failing with `401` in the benchmark runtime?
3. why is benchmark-path `Kuzu` population sparse while `Chroma` population is dense?
4. how strict and trustworthy are the new proof/judge layers?

## What this phase is not

This phase is **not**:

- a new structural routing redesign
- another `20.2` bonus / wording tuning loop
- a donor-first rollback
- a `Kuzu` / `Chroma` replacement phase
- a broad new benchmark-optimization shard

## Proven inputs carried forward

- the `20.2` best current exact-fact baseline remains `10 / 15`
- the `20.3` live fallback path did **not** preserve that level
- route metadata is already available from provider state
- backend population logging is already available in-run
- the real-path runner uses a per-question `TemporaryDirectory`
- benchmark evidence remains diagnostic, not the product goal

## The first-order open question

The first forensic question in `20.4` is explicit and non-negotiable:

- if live proof collapsed to fact fallback in almost every row, why did the fact path score only `3 / 15` on the legacy runner instead of something near the earlier `10 / 15` decomp-off baseline?

This is the fact-path parity question.
It must be answered before any claim that fixing route-hint auth or graph population alone will restore performance.

## Architectural constraints carried forward

- Brainstack remains a thin shell around donor-aligned layers
- no new shell-side donor logic growth wave is allowed as a “quick fix”
- route-hint reliability must be fixed as a real runtime/configuration path, not by silently disabling the feature and calling it solved
- `Kuzu` sparsity is currently a measured population fact, not yet a full retrieval-quality verdict
- judge reliability is now an explicit proof concern, not an afterthought

## Success condition

Phase `20.4` succeeds if it produces an evidence-based execution plan that:

- treats fact-path parity as the first forensic target
- makes route-hint runtime reliability an explicit execution stream
- audits graph publication/population in the benchmark seeding path
- hardens proof interpretation so retrieval quality and judge behavior are not conflated
- leaves `20.5` with concrete measured blockers instead of vague suspicion
