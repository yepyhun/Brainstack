# Phase 20.4 Implementation Contract

## Purpose

Turn the finished `20.3` forensic bundle into one bounded execution phase that explains the live benchmark/runtime failure surface before any further structural phase is opened.

## Required Workstreams

### Workstream A. Fact-path parity audit

Explain the gap between:

- the earlier `20.2` decomp-off best-known fact baseline: `10 / 15`
- the current `20.3` legacy answer-only fact-fallback result: `3 / 15`

Hard requirements:

- this is the first explicit forensic workstream
- it must compare packet shape, rendering, budgets, runtime environment, and benchmark seeding assumptions
- it must explicitly document the runtime environment behind the earlier `20.2` `10 / 15` baseline before treating the gap as a pure code-path regression
- it must not be collapsed into “probably the route-hint failure”

### Workstream B. Live route-hint runtime reliability

Trace the auxiliary route-hint path in the actual benchmark runtime.

Hard requirements:

- explain the observed `401` concretely
- determine whether the issue is:
  - wrong key
  - wrong endpoint
  - blocked / unfunded key
  - wrong runtime wiring
- fix the route-hint path or provide a bounded, explicit runtime-safe fallback
- any fallback must remain fail-open and bounded

### Workstream C. Benchmark-path graph publication and population audit

Audit why `Kuzu` remained sparse while `Chroma` was populated.

Hard requirements:

- inspect the benchmark seeding path, not only the steady-state store API
- distinguish:
  - no publication
  - failed publication
  - publication to the wrong path
  - structurally low graph yield from the seeded material
- explicitly check whether graph publication depends on Tier-2 extraction completing within the per-question seeding window, or whether `TemporaryDirectory` teardown can happen before graph data is written
- keep the conclusion at the right epistemic level:
  - population sanity
  - publication correctness
  - or retrieval adequacy

### Workstream D. Proof and judge hardening

Refine the proof interpretation layer around the split runner.

Hard requirements:

- preserve the retrieval-vs-answer split
- surface suspicious answer-judge passes more explicitly
- for `answer_recovered_despite_retrieval_gap` rows, surface the actual memory context that was sent to the answer model so the phase can tell apart missing evidence, judge over-strictness, and answer-model recovery
- check whether the retrieval judge is too strict for temporal / comparative questions
- do not treat judge rewriting as a substitute for retrieval fixes

### Workstream E. Route-level forensic visibility

Preserve and use the route metadata that `20.3` added.

Hard requirements:

- proof rows must continue to expose:
  - `requested_mode`
  - `applied_mode`
  - `source`
  - `reason`
  - `fallback_used`
- the execution phase must use these fields in diagnosis rather than inferring route behavior indirectly

### Workstream F. Bounded comparability reruns

The phase must end with bounded reruns that answer the parity question under comparable measurement.

Hard requirements:

- rerun the legacy answer-only path after fixes
- rerun the split-proof path after fixes
- interpret both together
- do not declare success from only one runner

## Protected Boundaries

### Architecture boundary

- `20.4` may not reopen the `20.3` routing design as if it were unplanned
- `20.4` may not become a new route-family invention phase
- `20.4` may not replace `Kuzu` or `Chroma`

### Donor boundary

- do not grow new shell-side donor logic to “paper over” route-hint or graph-publication issues
- do not normalize temporary shell heuristics as the long-term design

### Runtime boundary

- fixes must work in the target benchmark runtime, not only in the source repo environment
- because the runner uses a per-question `TemporaryDirectory`, backend evidence must stay in-run where needed

### Proof boundary

- raw answer score is never sufficient evidence by itself
- retrieval-correct, answer-correct, both-correct, and judge reliability remain separate proof dimensions

## Minimum Proof Required Before Calling Phase 20.4 Done

- fact-path parity has a concrete explanation
- route-hint `401` has a concrete explanation and bounded fix
- `Kuzu` sparsity has a concrete publication/population explanation
- proof/judge behavior is tightened enough that obvious suspicious passes are surfaced
- post-fix reruns exist in both legacy and split forms
- the resulting state leaves the next phase with measured blockers instead of ambiguous regression stories
