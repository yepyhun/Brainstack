# Phase 27 Plan

## Goal

Adopt only the `hermes-lcm` host-level donor ideas that materially improve the current system, and reject any donor slice that adds more maintenance cost or architectural headache than real product value.

## Scope

### In

- source-window / compaction provenance
- explicit lifecycle / frontier state
- bounded compacted-history expand/search ergonomics
- conditional ignored/stateless-session filtering, but only if evidence proves it is worth it
- preflight duplicate/overlap audit in Brainstack before any implementation

### Out

- full `LCMEngine` integration
- a new `ContextEngine` slot
- donor DAG/runtime transplant
- replacing the current host context compressor
- broad host cleanup
- any Brainstack memory-ownership change

## Concrete implementation method

- first build a portability and overlap truth map:
  - which donor slices are modular enough to import/adapt
  - which donor slices depend too heavily on the full `LCMEngine`
  - which parts already exist partially in Brainstack and should be extended instead of recreated
- then implement only the bounded slices that pass all of:
  - donor-first portability
  - utility-first ROI
  - no parallel runtime
  - bounded token effect
  - clear auditability or UX gain
- if a slice fails those gates, defer it instead of recreating it locally from scratch
- implementation source of truth is Brainstack only
- Bestie is for later validation and mirroring, not parallel double-writing

## Decision rules

- provenance comes first
  - it is the smallest, cleanest, highest-confidence win
- if preflight shows the needed seam already exists partially in Brainstack, extend that seam instead of importing a second structure
- lifecycle/frontier state comes second
  - it is allowed if it stays a thin host-state substrate
- bounded expand/search comes third
  - it is only allowed if token discipline is explicit and measurable
- session filtering is fourth and conditional
  - if noisy/stateless-session evidence is weak, do not implement it
- if any candidate requires a new host architecture slot, it is out of scope
- if any candidate starts to behave like a second compactor/runtime, it is out of scope
- if any candidate produces more operator burden or maintenance surface than product value, it is out of scope

## Hypothesis -> Falsification -> Repair decision

### 1. Source-window / compaction provenance

- hypothesis:
  - the donor's source-window bookkeeping can be adopted without architectural churn
- falsification:
  - the donor code depends on the full donor DAG runtime to stay coherent
- repair decision:
  - if false, keep only the minimal provenance schema pattern and stop there

### 2. Lifecycle / frontier state

- hypothesis:
  - a small explicit lifecycle/frontier state substrate can make host compaction behavior easier to reason about
- falsification:
  - the donor state model is too bound to `LCMEngine` rollover/session management to transplant cleanly
- repair decision:
  - if false, adopt only the smallest useful frontier/checkpoint concept, not the full donor model

### 3. Bounded expand/search ergonomics

- hypothesis:
  - compacted-history inspection can become more precise without increasing ordinary-turn token load
- falsification:
  - the only workable version requires broad new tools, broad prompt changes, or unbounded expansions
- repair decision:
  - if false, do not ship it in this phase

### 4. Session filtering

- hypothesis:
  - ignored/stateless-session filtering will reduce real host-side noise
- falsification:
  - our runtime does not actually suffer enough noisy stateless sessions to justify it
- repair decision:
  - if false, keep it out of the phase and leave it as a later candidate only

## Stop conditions before any patch

- do not patch before the donor portability matrix is written down
- do not patch before the duplicate/overlap audit is written down
- do not patch if the candidate feature needs a new host plugin slot
- do not patch if the candidate feature would create a second compaction runtime
- do not patch if the token cost cannot be bounded in plain language
- do not patch in Bestie first

## Validation ladder

1. donor portability truth
2. duplicate/overlap truth
3. host seam truth
4. bounded implementation proof
5. token-effect proof
6. no-boundary-regression proof

## Compatibility / donor-risk checkpoints

- do not fork the donor engine wholesale
- prefer isolated donor files or narrow donor patterns over local reinvention
- do not weaken the settled Brainstack/native boundary from Phase `22`
- do not let host expand/search become a shadow memory owner
- keep the implementation easy to diff against future donor updates
- do not create a Brainstack version and a separate Bestie version by hand

## Execution steps

1. Write the donor portability matrix for the four candidate slices.
2. Write the duplicate/overlap audit against the current Brainstack seams.
3. Map the current host compaction and compacted-history seams exactly.
4. Land source-window / compaction provenance if the portability and overlap gates pass.
5. Land explicit lifecycle/frontier state if the portability and overlap gates pass.
6. Evaluate bounded expand/search against utility, token, and boundary gates, then either implement or explicitly defer it.
7. Evaluate noisy/stateless-session evidence, then either implement filtering or explicitly defer it.
8. Validate the result in Bestie and mirror it there only if the Brainstack-side result proves out.
9. Prove no regression against:
   - host compaction behavior
   - Phase `22` ownership boundary
   - ordinary-turn token discipline

## Regression net

- compacted history must remain readable and bounded
- host compression must not become less stable
- Brainstack ownership must remain unchanged
- no new personal-memory shadow path may appear
- ordinary-turn token usage must stay flat unless the phase explicitly proves a bounded exception
- no duplicated implementation drift may appear between Brainstack and Bestie

## Required working style

- prefer isolated donor uptake over fresh local abstraction
- prefer evidence-backed deferral over “while we are here” expansion
- if the system is already “good enough” on a slice, stop instead of polishing for donor purity
- keep user-facing reasoning simple:
  - what was adopted
  - what was deliberately not adopted
  - why
