# Phase 30.5 Plan

## goal

Make operating-rule memory behave correctly under real chat conditions:

- long rule packs may arrive across multiple user messages
- the user may refine or correct rules after the first draft
- the assistant may restate the rules once during the conversation
- the session may reset immediately after that

The system must still converge those events into one canonical, durable operating-rule contract and preserve it across reset.

This phase now explicitly includes the architect decision that the operating-rule contract must stop living as a half-promoted truth object. The durable user-authored operating-rule contract must become a real first-class memory class with revisioned storage, not just a profile-lane row plus compiled policy plus slot fragments.

## scope

### in

- first-class behavior / operating-rule contract storage
- canonical multi-message style / operating-rule convergence
- dedicated operating-rule teaching admission path, separate from generic stable-memory guardrails
- durable canonical revisioning for explicit user rule corrections
- reset-safe commit barrier for confirmed operating-rule updates
- truthful distinction between:
  - user-authored canonical rule text
  - assistant paraphrase
  - session-local reinforcement
- exact post-reset recall of the current canonical operating-rule revision
- inspectable receipts / revision metadata for operating-rule updates

### out

- broader KG or document pipeline work
- new benchmark-oriented retrieval tuning
- unrelated donor redesign
- personality/content changes that are not needed to fix canonical rule durability

## architecture decision

### accepted target model

- the canonical operating-rule truth is user-authored, revisioned, and stored in a dedicated first-class behavior-contract seam
- assistant restatements may help the conversation, but they are not canonical truth
- session-local correction reinforcement may still exist for immediate obedience, but it must not impersonate durable canonical change
- explicit user corrections that clearly mean “change the rule set” must produce a committed canonical revision or a visible failure
- reset after a confirmed update must preserve the same canonical revision that the user just confirmed
- transcript-derived slot fragments and compiled policy remain allowed as derivations, but they must never compete with the canonical raw contract as the authority
- full rule recall must resolve from the committed canonical contract or fail explicitly; it must not be reconstructed from fragments

### rejected model

- treating one nice assistant summary as if it were the user's canonical rule source
- keeping the durable operating-rule contract as only a profile-item-shaped storage lane while treating it as if it were already first-class
- scattering rule truth across:
  - raw contract
  - assistant restatement
  - correction reinforcement
  - transient turn context
  without one authoritative durable revision
- silent fallback from durable revision to session-only memory

## workstreams

### 1. canonical multi-message convergence

- introduce a dedicated behavior / operating-rule contract storage seam with revision id, parent linkage, source hash, committed timestamp, and supersession semantics
- define how multiple user rule-teaching messages become one canonical draft / revision
- support real chat shape:
  - initial rule list
  - follow-up additions
  - corrections like “the 21st rule should also say ...”
  - later count correction like `27 -> 29`
- ensure the canonical row is built from the user-authored operating-rule content, not from an assistant-assembled echo

### 2. dedicated teaching admission

- explicit operating-rule teaching must use a dedicated validator / extractor path rather than the generic stable-memory durable-admission rules
- the path must accept real human teaching shapes:
  - long pasted rule packs
  - transcript-like quoted rule corrections
  - multi-message additions and clarifications
- fail-closed means:
  - committed canonical revision
  - or explicit visible failure / pending state
  not silent degradation into graph / continuity only

### 3. durable correction semantics

- define when a plain chat correction is merely session-local reinforcement
- define when it is a durable operating-rule update
- require durable updates to land as:
  - committed canonical revision
  - or visible failure / pending status
- preserve rule identity and inspectable revision history

### 4. reset-safe durability

- no assistant acknowledgement like “updated” or “I will remember it” without a committed canonical revision
- session reset must not erase a just-confirmed operating-rule change
- if reset would occur before commit, the system must fail closed or visibly disclose the pending state

### 5. truthful recall after reset

- post-reset “list the rules” must read from the canonical revision
- the returned rule count and content must match the durable canonical source
- if the canonical source is incomplete, the system must say so instead of pretending full recall
- deterministic contract-recall routing must prefer the committed raw contract directly
- partial slot fragments or compiled policy summaries must not be allowed to masquerade as full-contract recall

### 6. observability and live closure

- expose:
  - canonical revision id / hash
  - source of last committed update
  - whether the latest change was durable or session-local only
- prove the full live sequence:
  - teach across multiple messages
  - refine a rule
  - reset
  - recall the updated canonical rule set

## validation ladder

- source:
  - targeted regression tests for multi-message canonicalization and reset-safe durability
- shim:
  - plugin payload parity and direct interpreter smoke
- `hermes-final`:
  - reinstall and direct runtime smoke
- live runtime:
  - reproduce the real conversational failure pattern and close it there

## anti-goals

- do not solve this with brittle keyword farms
- do not let assistant paraphrase become the truth owner
- do not claim durable success when only session-local reinforcement happened
- do not use benchmark score movement as proof that this phase is solved
- do not let compiled policy or slot fragments serve as authoritative full-contract recall
- do not hide operating-rule teaching behind generic durable-admission heuristics that reject real user teaching shapes
- do not widen this into a broad extraction-pipeline redesign before the canonical behavior-contract seam is fixed
