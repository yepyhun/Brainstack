# Phase 30.5 Context

## problem statement

Live evidence now shows a narrower but more serious residual than generic “memory loss”.

The system can sometimes:

- echo a long operating-rule set once inside the same conversation
- obey parts of it temporarily
- even acknowledge a follow-up correction

but after reset it may fall back to only a partial subset of rules.

That means the failure is not just recall quality. It is canonical operating-rule durability:

- the durable operating-rule contract is still not fully first-class in storage terms
- multi-message user teaching is not reliably converging into one canonical raw contract
- assistant restatement can appear stronger than the true durable source
- plain chat corrections are often only session-local reinforcement
- reset can happen before a durable canonical revision exists
- the recall path can still fall back to fragmentary operational representations instead of one committed canonical source

## why this is the correct next phase

This is now a direct product failure in the main use case:

- the user teaches operating rules
- the system appears to accept them
- the user resets the session
- the system cannot truthfully recover the same rule set

That violates:

- donor-first product correctness
- truth-first status discipline
- coherent continuous conversation
- proactive stateful continuity
- long-range accurate recall

## phase boundary

This phase is specifically about canonical operating-rule durability.

It is not:

- a broader benchmark phase
- a KG phase
- a generic retrieval tuning phase
- a prompt-writing cleanup phase

## accepted sharpened reading

The current system already has important adjacent pieces:

- compiled behavior policy
- trace and inspection surfaces
- truthful write receipts
- task / commitment truth

Those were necessary but not sufficient.

The remaining gap is narrower:

- the user's operating rules still do not behave like a fully promoted first-class truth object end-to-end
- profile-lane storage, compiled policy, and slot fragments can still pull in different directions
- that is why the bug looks like “memory loss” from the outside while really being canonical authority failure

## expected proof shape

The closing proof for this phase should look like this:

1. User teaches a long rule set across multiple messages.
2. User corrects or extends one of the rules.
3. The system commits a canonical revision and can show that receipt.
4. The session resets.
5. The system recalls the same canonical revision and correct rule count.
6. The trace distinguishes durable canonical truth from session-local reinforcement.
7. If the canonical revision does not exist, the system fails explicitly instead of reconstructing a fake full rule list from fragments.
