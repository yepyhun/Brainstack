# Phase 20.22 Implementation Contract

## Objective

Turn the `20.21` narrow live breakthrough into a broader, honest product-quality read on deployed Bestie without reopening capability sprawl.

## Workstream A: Broader Deployed-Live Eval Matrix

- Construct a bounded, realistic scenario set on deployed Bestie.
- Use the now-working stack, not degraded-mode assumptions.
- Include scenarios that exercise:
  - coherent follow-through inside a conversation
  - remembered constraints after reset/session boundary
  - relation-tracking across multiple facts
  - longer factual recall chains
  - one larger knowledge-body interaction

## Workstream B: Residual-Class Read

- For every miss, classify the dominant failure type.
- Do not collapse all failures into “retrieval”.
- Distinguish:
  - retrieval / surfacing
  - packet sufficiency vs answer synthesis
  - continuity / proactivity miss
  - runtime variance

## Workstream C: Lightweight Token / Packet Truth

- Read packet-size and selected-row burden on the enabled stack.
- Keep this bounded:
  - no new telemetry platform
  - no broad instrumentation framework
- The aim is simply to know whether current live gains are reasonably efficient.

## Workstream D: Honest Stop Boundary

- If the broader live read shows one new dominant blocker, stop there and localize it clearly.
- If the broader live read is healthy, say so directly.
- Do not invent a new feature phase from speculative ideas that are not required by the evidence.

## Protected Boundaries

### Product Boundary

- Keep the work tied to deployed Bestie product truth.
- Preserve the live gains from `20.21`.
- Prefer realistic scenarios over synthetic benchmark pressure.

### Architectural Boundary

- Stay donor-first and modular.
- Do not widen into architecture redesign from inside a validation phase.
- Do not backslide into benchmark-first optimization.

### Anti-Benchmaxing Boundary

- Do not use LongMemEval as the primary decision-maker for this phase.
- Do not add new capability work just because a benchmark-shaped idea looks tempting.

## Minimum Evidence Required Before Calling Phase 20.22 Done

- one broader deployed-live evaluation artifact
- one explicit residual-class summary
- one token / packet overhead read
- one clear next-thread recommendation grounded in product truth
