# Phase 20.22 Plan

## Goal

Measure broader deployed-live product quality on the now-working Brainstack/Bestie stack, using real conversation-shaped scenarios rather than benchmark-shaped pressure.

## Scope

### In

- broader deployed-live validation on the enabled stack
- conversation-quality truth across:
  - coherent continuous conversation
  - proactive stateful continuity
  - long-range recall / relation-tracking
  - large-knowledge usability
  - token / memory-packet overhead truth
- bounded harness adjustments only if needed to make the read faithful
- one evidence-led next-phase recommendation

### Out

- new retrieval capability branches
- aggregate-expansion work
- benchmark-suite widening
- deploy packaging redesign
- broad prompt rewrites without live evidence

## Execution Steps

1. Workstream A: broader deployed-live eval matrix.
   - Build a bounded scenario matrix on deployed Bestie.
   - Prefer `8–12` realistic scenarios over a huge sweep.
   - Cover at least:
     - same-session coherence
     - reset-boundary continuity
     - proactive follow-through / remembered constraint use
     - long-range factual recall
     - relation-tracking across multiple facts
     - one “large body of knowledge” style scenario
   - Keep the matrix product-shaped, not benchmark-shaped.

2. Workstream B: live answer-quality run and residual classification.
   - Run the broader deployed-live eval.
   - Classify any remaining misses by product residual class, not by old benchmark categories.
   - Separate:
     - retrieval/surfacing miss
     - answer synthesis miss
     - continuity/proactivity miss
     - runtime instability / nondeterminism

3. Workstream C: token and packet overhead truth.
   - Measure the practical memory/prompt burden on the enabled stack.
   - Read:
     - packet size / selected row count trends
     - whether live wins are coming with unreasonable prompt inflation
   - Keep this lightweight and truthful; do not build a new telemetry subsystem here.

4. Workstream D: honest deployed-live quality verdict.
   - State clearly whether the current system is now “good enough to broaden with confidence” or whether one new dominant product blocker has replaced the old runtime-availability seam.
   - If a clear next blocker appears, localize it tightly.
   - If no new blocker dominates, say that too.

## Implementation Rules

- Stay donor-first.
- Keep the phase product-led, not benchmark-led.
- Prefer smaller, faithful live evaluation over large noisy sweeps.
- Do not create new capability work just because the eval reveals possible future ideas.
- Treat token savings as a first-class product target, but measure it lightly.
- Preserve the `20.21` live gains; do not trade them away casually.

## Output Shape Required From Phase 20.22

- one broader deployed-live eval artifact
- one residual-class breakdown across the evaluated scenarios
- one lightweight token / packet overhead read
- one honest product-quality verdict
- one evidence-led next-thread recommendation

## Exit Gate

- there is a broader deployed-live product-quality read beyond the narrow isolated reset harness
- the read covers continuity, recall, and at least one token/packet overhead truth
- the next-thread recommendation is based on product behavior, not technical momentum
- no new feature branch is started implicitly inside the validation phase
