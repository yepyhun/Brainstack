# Phase 20.12 Plan

## Goal

Turn the residual `gpt4_7f6b06db` temporal miss into a general memory-kernel capability problem and solve it with a bounded semantic retrieval / reranking layer over structured temporal continuity, without drifting into benchmark-specific hacks.

## Scope

### In

- patched-runtime retrieval-only proof on the named temporal handoff case
- explicit design choice between:
  - bounded hybrid over temporal-event candidates
  - dedicated semantic retrieval over `temporal_event` rows
  - stronger external semantic layer if the first two are not sufficient
- one bounded implementation path
- fixed-canary regression protection
- anti-benchmaxing and donor-first guards

### Out

- route-hint redesign
- lexical cue tuning for one benchmark question
- full corpus-wide embedder swap by default
- broad aggregate-counting redesign
- full `15`-question live reruns as inner-loop tools
- vendor lock before a project-local bakeoff

## Execution Steps

1. Workstream A: freeze the baseline truth for the capability decision.
   - Reuse the patched-runtime retrieval-only harness.
   - Freeze the current `gpt4_7f6b06db` temporal-event truth:
     - candidate pool
     - surfaced temporal-event rows
     - final selected block
   - Reconfirm canary non-regression expectations before capability work starts.

2. Workstream B: make the capability choice explicitly.
   - Compare the bounded options:
     - **bounded hybrid first**:
       - surface a generous but still bounded temporal-event pool
       - rerank that pool semantically in-memory before final selection
     - dedicated semantic retrieval over `temporal_event` continuity rows
     - stronger external semantic layer only if the bounded internal route is clearly insufficient
   - Judge them on:
     - genericity
     - donor-first fit
     - runtime cost
     - local deployability
     - minimal collateral risk to fact/profile/aggregate paths
   - Monitor one explicit risk while making the decision:
     - short factual `temporal_event` rows may score weakly against question-shaped queries under generic symmetric embedding backbones
   - If an external model family is considered, keep the decision capability-level first and brand-level second, but treat Jina v5 TEI as the preferred fast-track evaluation candidate rather than a random option.

3. Workstream C: implement one bounded temporal-event relevance layer.
   - Default to the bounded hybrid first unless Workstream B produces strong evidence against it.
   - Keep it targeted to temporal continuity, not the whole retrieval stack.
   - Preserve `20.11` structural wins.
   - If the chosen path needs a new semantic helper, keep it narrow and local rather than turning the phase into a full backend migration.
   - Instrument the new relevance layer so candidate provenance and score movement stay visible.

4. Workstream D: prove or reject the capability with the fast ladder.
   - Re-run targeted patched-runtime `--retrieval-only` proof on `gpt4_7f6b06db`.
   - Re-run the fixed canary retrieval-only loop to catch collateral regressions.
   - Only if retrieval-only evidence materially improves should the phase consider any answer-only proof.

## Implementation Rules

- Do not reopen route-hint work.
- Do not reopen failed temporal shortlist, query-shaping, or pool-widening ideas.
- Do not write benchmark-row-specific cues or exceptions.
- Keep LongMemEval in measurement role only.
- Treat the bounded hybrid as the preferred first implementation shape.
- Treat the Jina v5 TEI path as a preferred external evaluation candidate, not as a mandatory outcome.
- Prefer the smallest generic layer that materially improves structured temporal relevance.

## Output Shape Required From Phase 20.12

- one explicit capability decision with rationale
- before/after targeted retrieval-only artifacts for `gpt4_7f6b06db`
- canary retrieval-only regression artifact
- clear statement whether the new layer:
  - materially improved temporal-event relevance
  - failed cleanly
  - or needs a later, larger retrieval backbone phase

## Exit Gate

- the `20.12` capability choice is explicit and evidence-backed
- one bounded temporal-event semantic retrieval / reranking path exists in code
- targeted `gpt4_7f6b06db` retrieval-only proof is comparable before vs after
- canary retrieval-only regression check exists
- the phase can state whether broader answer-only proof is justified without guessing

## Recommended Effort

- `xhigh`, because this is the first phase that moves from cheap structural fixes into a new retrieval capability layer
