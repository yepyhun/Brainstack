# Phase 20.12 Context

## Why This Phase Exists

Phase `20.11` closed the cheap structural donor-first retrieval fixes that were still clearly worth doing:

- patched-runtime canary truth was established
- the continuity-to-transcript join bug was fixed
- deterministic route truth was restored on the live path
- compact Tier-2 schema, JSON repair, reasoning fallback, and explicit `temporal_event` surfacing materially improved temporal admission / persistence

Those wins narrowed the remaining temporal miss on `gpt4_7f6b06db` to a different class of problem:

- route activation is no longer the blocker
- the relevant trip window can now be admitted and written
- `temporal_event` rows can now be surfaced
- but the final selected block still does not surface the full trip chain reliably

That means the next step should no longer be another `20.11` micro-tuning round. It should be a dedicated capability phase for semantic retrieval / reranking over structured temporal continuity.

## Current Truth Entering 20.12

- `20.11` is execution-complete at gate.

- `e9327a54` is a confirmed generic fact-path packing win.

  - the remaining miss is relevance discrimination over already-structured temporal evidence

## What This Phase Is

This phase is a bounded memory-kernel capability phase:

- improve semantic relevance over `temporal_event` continuity rows
- decide whether the right bounded move is:
  - a **preferred bounded hybrid**:
    - broad enough temporal-event candidate surfacing first
    - then semantic reranking over that bounded pool
  - dedicated semantic retrieval over temporal events
  - or a different bounded hybrid if the first hybrid shape is wrong
- preserve donor-first behavior and genericity
- keep the fast canary / retrieval-only workflow from `20.10`

The default planning assumption for `20.12` is:

- try the bounded hybrid first
- keep full backend / embedder replacement out of the first implementation pass
- if a stronger external semantic layer is needed, treat Jina v5 via TEI as the current preferred evaluation candidate, but not as a hard rule
- if Jina v5 TEI is evaluated, prefer a practical CPU-first deployment sketch with:
  - explicit model cache volume
  - healthcheck
  - fixed retrieval model command
  - `last-token` pooling
  - and only stage GPU after basic service truth is stable

## What This Phase Is Not

This phase is **not**:

- a return to route-hint work
- a return to shortlist / query-shaping / pool-widening tweaks
- a benchmark-score chase
- a full-system embedder migration by default
- a vendor lock decision
- a LongMemEval-specific solver phase

LongMemEval remains only a calibration probe. The target remains a stronger Hermes memory kernel with coherent continuity, proactive state use, long-range usable recall, and meaningful token savings.

## Entry Conditions

- runtime-sync verification already exists and must continue to gate all live proof
- the fixed `5`-question canary remains the fast feedback set
- `20.11` negative evidence remains binding:
  - do not reopen route-hint tweaking
  - do not reopen lexical shortlist/query-shaping/search-limit tuning
  - do not treat “just raise max tokens” as an open hypothesis

## Target End State

- a bounded temporal-event relevance layer exists and is instrumented
- the project can say, with evidence, whether semantic retrieval / reranking over structured continuity materially improves the `gpt4_7f6b06db` class
- the phase leaves behind a generic capability path that can later help similar temporal continuity questions, not just one benchmark row
- if an external semantic layer is evaluated, the phase can say whether Jina v5 TEI was a good fit, an unnecessary detour, or a later-phase concern
- any named model family remains a preferred evaluation candidate until a project-local bakeoff actually justifies locking it in
