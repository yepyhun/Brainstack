# Phase 20.12 Implementation Contract

## Objective

Add a bounded, generic temporal-event semantic retrieval / reranking capability over structured continuity evidence, so the project can discriminate the right temporal event chain without falling back to benchmark-specific heuristics.

## Workstream A: Baseline Capability Reading

- Start with a runtime-sync-passing patched-runtime retrieval-only read on `gpt4_7f6b06db`.
- Persist the pre-change truth for:
  - temporal-event candidates
  - surfaced temporal-event rows
  - final selected block
- Keep the fixed canary available for later regression checks.
- No capability-layer code change should be called valid before this baseline artifact exists.

## Workstream B: Capability Decision

- The phase must choose explicitly between:
  - a **bounded hybrid first**:
    - bounded temporal-event candidate surfacing
    - then semantic reranking over that pool
  - dedicated semantic retrieval over `temporal_event` continuity rows
  - a stronger external semantic layer if the internal bounded route proves too weak
- The decision must be justified against:
  - genericity beyond one benchmark row
  - donor-first fit
  - local / embedded deployability
  - latency and cost
  - risk of collateral regressions
- The decision must explicitly check one known risk:
  - short factual `temporal_event` strings may underperform under generic symmetric semantic scoring against question-shaped queries
- If the phase names a brand-level candidate, it must still keep the swap point abstract enough that the project is not locked to one vendor by plan wording alone.
- If an external candidate is tested, Jina v5 TEI should be treated as the preferred first evaluation candidate unless local evidence rejects it quickly.
- If Jina v5 TEI is tested, prefer a pragmatic deployment order:
  - CPU-first service truth
  - explicit model cache volume
  - explicit healthcheck
  - fixed retrieval model command and pooling mode
  - GPU only after the basic path is stable, because GPU bring-up may be the fragile part rather than the retrieval logic itself

## Workstream C: Bounded Temporal Capability Implementation

- Implement one bounded relevance layer for temporal continuity.
- Prefer the bounded hybrid first unless Workstream B produces clear contrary evidence.
- Keep the intervention scoped to temporal-event retrieval / reranking, not a full retrieval-stack rewrite.
- Preserve the already working `20.11` paths:
  - profile/fact carry-through wins
  - route truth
  - Tier-2 compact schema and repair improvements
  - explicit `temporal_event` surfacing
- Add enough telemetry that later phases can see:
  - candidate provenance
  - pre/post score movement
  - what actually changed in the final selected block
- If a new semantic helper is added, it must stay narrow enough that the phase still counts as bounded capability work rather than a stealth infrastructure migration.

## Workstream D: Bounded Proof

- Re-run the targeted patched-runtime retrieval-only proof on `gpt4_7f6b06db`.
- Re-run the fixed canary retrieval-only proof afterward.
- Do not escalate to broader answer-only proof unless the retrieval-only artifacts show material improvement rather than noise.

## Protected Boundaries

### Anti-Benchmaxing Boundary

- No benchmark-row-specific lexical cues.
- No “solve the LongMemEval question” prompt shaping.
- No reopening of already rejected micro-tuning loops just because they are cheap.

### Architectural Boundary

- Do not convert the phase into a full embedder migration unless the evidence forces it.
- Do not treat a brand-level candidate such as Jina v5 TEI as a locked architectural requirement before local evidence exists.
- It is acceptable to be more permissive in evaluation order:
  - Jina v5 TEI may be the preferred external candidate
  - but it still must earn adoption through local evidence rather than plan wording alone

### Scope Boundary

- Aggregate counting and typed-entity expansion stay separate follow-up tracks unless they are strictly required to make the temporal capability truthful.
- Route-hint policy stays out of scope unless new evidence proves a fresh regression.

## Quality Standard

- The phase must leave behind a reusable capability, not a benchmark-specific patch.
- Explanations must stay layer-correct:
  - selection
  - semantic retrieval / reranking
  - packing
  - answer generation
- If the chosen capability path fails, the failure must be stated explicitly rather than buried under “partial improvement” language.

## Minimum Evidence Required Before Calling Phase 20.12 Done

- one baseline targeted retrieval-only artifact
- one post-change targeted retrieval-only artifact
- one canary retrieval-only regression artifact
- explicit decision record for the chosen capability path
- explicit statement whether the residual gap moved, stayed flat, or requires a larger future backbone phase
