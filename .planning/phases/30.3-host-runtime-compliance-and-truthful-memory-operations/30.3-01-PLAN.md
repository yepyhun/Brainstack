# Phase 30.3 Plan

## goal

Close the host/runtime quality gaps so Brainstack-backed behavior and memory operations become truthful, inspectable, reset-safe, and mechanically enforced where the rules are objective.

## scope

### in

- hard surface validation for mechanical final-answer constraints:
  - forbidden Unicode punctuation
  - emoji
  - bold markdown
  - explicitly forbidden phrases or surface forms
- truthful write-receipt semantics before the assistant claims durable memory success
- reset-time durable write barrier for explicit durable writes
- truthful null-result semantics for memory claims
- tool-routing and lookup-order corrections for task-like asks

### out

- broad semantic restyling engine
- regex output-policing as the primary behavior system
- deciding the long-term task truth class itself
- general-purpose workflow/task-management features beyond the identified memory contract

## core principles this phase must preserve

- donor-first
- truth-first
- fail-closed
- no fake success acknowledgements
- no hidden write races
- no mixed ownership claims
- bounded enforcement only for objective/mechanical constraints

## architecture decision

### accepted target model

- Brainstack memory truth and host/runtime compliance are separate planes
- this phase hardens the host/runtime plane around the existing memory surfaces
- objective constraints may use a thin final-answer validator
- soft stylistic behavior still belongs to the compiled policy and the model
- memory acknowledgements must follow committed receipts, not optimistic intent

### rejected model

- “policy in prompt is enough”
- “if session_search found nothing, we can say it was never stored”
- “the assistant can promise it saved something before a durable commit”
- “all enforcement should stay inside the model with no hard checks”

## workstreams

### 1. hard-surface validator

- define the exact mechanical constraints eligible for host/runtime final-answer validation
- ensure violations are caught deterministically
- define allowed repair behavior without turning the system into a post-hoc rewriting engine

### 2. truthful write receipts

- define committed write receipts for durable user-memory operations
- ban durable-success language without receipt-backed commit evidence
- make receipt provenance inspectable

### 3. reset-time write barrier

- identify explicit durable writes that must flush synchronously before reset/session teardown
- make reset behavior fail closed if those writes are still pending

### 4. null-result semantics and tool routing

- separate:
  - no record found
  - no durable owner exists
  - lookup path was only transcript/session search
  - lookup may have failed lexically
- define structured lookup precedence for task-like questions before fallback tools run

## validation ladder

- prove hard-surface validator on live-style examples
- prove receipt-backed acknowledgements
- prove reset barrier behavior on explicit durable writes
- prove truthful no-result wording for task-like retrieval misses
- verify source, shim, `hermes-final`, and live runtime surfaces

## anti-goals

- do not create a giant output-rewriter
- do not mix task-truth design into this compliance phase
- do not leave “saved” acknowledgements possible without commit evidence
