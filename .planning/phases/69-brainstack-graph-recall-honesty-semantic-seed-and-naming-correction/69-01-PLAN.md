# Phase 69 Plan

## purpose

Make Brainstack graph recall honest and useful by separating graph storage health from recall semantics, then either naming lexical recall accurately or adding a bounded semantic/alias seed before graph expansion.

## core question

How can Brainstack avoid overselling lexical Kuzu expansion as semantic graph memory while improving graph recall through typed semantic seeds that preserve source-of-truth and no-heuristic rules?

## GSD planning enforcement check

This phase must preserve:

- truth-first closeout: call lexical graph recall lexical if it remains lexical
- no heuristic sprawl: do not solve aliases/paraphrases with phrase farms
- source-of-truth discipline: Kuzu storage and typed rows remain traceable to Brainstack-owned evidence
- non-governor boundary: graph recall informs memory packets, not execution or approval decisions
- no donor transplant: do not import Graphiti/Zep-style architecture without a reviewed Brainstack-owned contract

## shared north star and real-world proof gate

North star for Phase 65-69: Brainstack must become a more reliable memory kernel in user-visible terms, not merely gain more internal machinery.

Every phase closeout must prove one of these:

- the memory kernel is measurably better for a real recall/state/policy scenario
- or the phase is pure prerequisite work and proves no regression while naming the exact later phase that must deliver the user-visible improvement

Minimum proof shape:

- targeted tests for the changed surface
- at least one end-to-end Brainstack packet or inspect proof using real store/retrieval code
- evidence ID/source/provenance proof where memory is involved, not only answer text
- no-regression check against Phase 66 golden gates once they exist
- bounded token/output sanity for any new inspect, doctor, or trace artifact

## accepted evidence

- current Kuzu-backed graph storage can exist while recall still starts from lexical `CONTAINS`-style seeds
- semantic graph usefulness cannot be proven if doctor only says the backend is active
- Phase 67 should provide a typed semantic evidence index that can act as a bounded seed source
- Phase 66 should provide graph recall golden cases for direct, paraphrased, alias, state, and relation queries

## primary goals

- expose graph storage health separately from graph recall mode
- name the current recall mode accurately when it is lexical
- integrate typed semantic/alias seeding if Phase 67 provides a suitable source
- keep graph expansion bounded and inspectable
- prove graph recall behavior through Phase 66 golden tests

## explicit scope model

### Brainstack-owned scope

- graph recall mode naming
- Kuzu storage health reporting
- graph seed source selection
- bounded graph expansion
- graph recall traces and evidence attribution

### out of scope

- replacing Brainstack with an external graph-memory system
- broad ontology design
- new runtime behavior
- graph-side keyword farms
- unbounded multi-hop expansion

## anti-goals

- no "semantic graph" label if the seed is still lexical only
- no alias phrase tables
- no Graphiti clone
- no hidden corpus/transcript stuffing to fake graph recall
- no untraceable graph ranking

## architecture decisions to freeze before execution

### 1. terminology decision

Define explicit graph recall modes:

- unavailable
- lexical_seeded
- semantic_seeded
- hybrid_seeded, only if both paths are genuinely active and traceable

### 2. storage vs recall separation

Doctor must report:

- Kuzu/storage backend active state
- graph row/node/relation counts
- recall mode
- seed source
- degraded reason

### 3. seed source rule

If semantic seeding is added, it must come from Phase 67 typed semantic evidence results or another typed Brainstack-owned substrate. It must not come from free-text keyword/phrase guessing.

### 4. expansion bound

Graph expansion must have explicit hop, row, and token bounds, and inspect must show seed IDs plus expanded graph evidence IDs.

### 5. fallback honesty

If semantic seed is unavailable, graph recall may still run lexical mode, but the system must label it as lexical and tests must reflect that capability honestly.

## workstreams

### 1. current graph path inventory

Map Kuzu backend storage, graph write paths, graph search/recall paths, and current seed behavior.

### 2. mode reporting

Add graph recall mode and seed-source reporting to doctor/inspect.

### 3. semantic seed integration

If Phase 67 is ready, use typed semantic evidence hits to seed entity/state/relation graph expansion.

### 4. bounded expansion trace

Expose seed hits, expansion hops, candidate evidence, suppression reasons, and final packet inclusion.

### 5. golden proof

Run Phase 66 graph cases and record which are solved by lexical mode, semantic seed mode, or remain unsupported.

## execution sequencing

1. Inventory current graph recall and naming.
2. Add mode reporting without behavior change.
3. Decide whether semantic seed integration is available from Phase 67.
4. Implement bounded semantic seed integration only if the substrate is ready.
5. Add graph inspect traces.
6. Run golden graph tests and update closeout honestly.

## deliverables

- graph storage vs recall-mode health reporting
- graph recall mode naming in doctor/inspect
- optional semantic/alias seed integration from typed semantic evidence
- bounded graph expansion trace
- graph golden eval proof and truth-first closeout

## lightweight process refinements

### terminology reuse

Use the Phase 65 status vocabulary and do not invent alternate graph-only status labels for the same state. Graph-specific terms are allowed only for recall mode names such as `lexical_seeded` and `semantic_seeded`.

### expected-red ownership

Phase 69 must consume the Phase 66 expected-red graph cases it owns and close each one as:

- fixed and promoted to hard regression gate
- honestly unsupported in current graph recall mode
- deferred only with an explicit future architecture dependency

### phase handoff gate

Closeout must state:

- graph storage health status
- graph recall mode
- graph golden cases fixed
- graph cases still unsupported
- whether PR/PullPhase candidate promotion is safe or still blocked

## acceptance criteria

- active Kuzu storage cannot be mistaken for semantic graph recall by itself
- graph recall mode is visible and accurate
- semantic graph improvements, if implemented, resolve to typed Brainstack evidence IDs
- graph paraphrase/alias cases are proven or explicitly left unsupported
- no graph-side keyword farm, ontology sprawl, or donor transplant is introduced
