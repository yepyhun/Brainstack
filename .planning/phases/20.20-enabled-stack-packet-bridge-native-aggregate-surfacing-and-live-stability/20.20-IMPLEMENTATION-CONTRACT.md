# Phase 20.20 Implementation Contract

## Objective

Turn the enabled graph/corpus runtime from a passive “active channel” fact into a real final-packet contributor on deployed Bestie, with explicit proof around native aggregate surfacing and honest live stability re-measurement.

## Workstream A: Enabled-Stack Packet Bridge Diagnosis

- Start from the `20.19` post-enablement truth:
  - `semantic` active
  - `graph` active
  - final packet still mostly transcript/continuity-dominant
- Diagnose where the enabled stack loses influence:
  - candidate generation
  - native aggregate surfacing inside `matched` versus the separate `graph_rows` / `corpus_rows` blocks
  - channel ranking
  - selected-row composition
  - final answer synthesis
- The phase must not treat “channel active” as equivalent to “capability actually surfaced”.

## Workstream B: Bounded Bridge Hardening

- Land the smallest donor-aligned change that lets enabled graph/corpus/native evidence matter in the final packet.
- Preferred bounded directions:
  - selected-row priority rebalance for enabled graph/corpus rows
  - aggregate-mode surfacing adjustments for native aggregate evidence
  - packet composition adjustments that preserve existing route structure
- The current aggregate-mode native prepend already exists; prefer proving or tightening that path before inventing any second bypass mechanism.
- Do not broaden into a new retrieval architecture.

## Workstream C: Native Aggregate Surfacing Proof

- Produce explicit evidence that the live `aggregate_total` path can surface native aggregate evidence.
- Acceptable proof forms:
  - selected packet includes a native aggregate row
  - candidate debug shows native aggregate evidence materially entering the final surfaced block
- The proof must state explicitly whether the native evidence surfaced through `matched` or through `graph_rows`; `graph_rows = 0` alone is not a disproof.
- A merely correct final number is not enough.

## Workstream D: Enabled-Stack Live Stability Re-Baseline

- Re-run the isolated deployed-live reset eval after the bridge changes.
- Compare against the enabled-stack baseline from `20.19`:
  - `3 / 5`
- Read specifically:
  - `aggregate_total`
  - `temporal_order`
  - `gift_context`
- If answers still swing, record that as stability truth rather than hand-waving it away.

## Workstream E: Honest Stop Boundary

- If the enabled-stack bridge proves larger or noisier than expected, the phase may close with:
  - diagnosed blocker
  - bounded packet-level progress
  - explicit deferral to `20.21`
- Do not claim that live graph/corpus integration is “done” if packets remain transcript-dominant and native evidence still does not surface materially.

## Protected Boundaries

### Product Boundary

- Preserve:
  - donor-first
  - modularity / upstream updateability
  - coherent continuous conversation
  - proactive stateful continuity
  - long-range accurate recall and relation-tracking
  - meaningful token savings

### Architectural Boundary

- Use the existing Brainstack graph/corpus path.
- No parallel ad hoc aggregate path.
- No schema redesign unless the current bridge is provably impossible without it.

### Anti-Benchmaxing Boundary

- Do not add case-shaped heuristics just to force `aggregate_total`.
- The work must generalize to enabled-stack live recall, not just the named evaluation case.

## Minimum Evidence Required Before Calling Phase 20.20 Done

- one explicit enabled-stack packet proof
- one explicit native aggregate surfacing proof
- one enabled-stack isolated live re-baseline
- one honest residual-class verdict if a miss remains
