# Phase 68 Plan

## purpose

Make Tier-2 extraction execution reliable, observable, bounded, and truth-preserving so durable promotion no longer depends on invisible "maybe it ran" behavior.

## core question

How can Brainstack prove when Tier-2 extraction was scheduled, run, parsed, emitted records, wrote records, no-oped, or failed, while preventing assistant-authored contamination and avoiding an always-on inference tax?

## GSD planning enforcement check

This phase must preserve:

- no bandaid fixes: do not just retry provider calls while leaving extraction state invisible
- no heuristic importance engine: promotion must remain typed and bounded
- provenance discipline: assistant-authored self-diagnosis cannot become user-owned durable truth
- non-governor boundary: Tier-2 may promote memory, not execute tasks or enforce runtime approval
- token discipline: extraction windows and summaries must be bounded

## shared north star and real-world proof gate

North star for Phase 65-69: Brainstack must become a more reliable memory kernel in user-visible terms, not merely gain more internal machinery.

Every phase closeout must prove one of these:

- the memory kernel is measurably better for a real recall/state/policy scenario
- or the phase is pure prerequisite work and proves no regression while naming the exact later phase that must deliver the user-visible improvement

Minimum proof shape:

- targeted tests for the changed surface
- at least one end-to-end Brainstack packet or inspect proof using real store/retrieval code
- evidence ID/source/provenance proof where memory is involved, not only answer text
- no-regression check against Phase 66 golden gates once they exist
- bounded token/output sanity for any new inspect, doctor, or trace artifact

## accepted evidence

- Tier-2 is the intended semantic promotion path for several durable memory classes
- current behavior can leave users unsure whether Tier-2 ran, failed, parsed empty, or wrote nothing
- Phase 65 will expose extraction status, and Phase 66 will expose which memory gaps depend on extraction reliability
- previous live defects showed request-contract, timeout, and stall failures can affect Tier-2-related paths

## primary goals

- add explicit Tier-2 run/schedule/result records
- surface parse status, emitted counts, write counts, no-op reasons, and failure reasons
- make idle/session-end/user-turn flush policy bounded and predictable
- preserve strict provenance and assistant-contamination controls
- prove extraction outcomes through Phase 65 inspect/doctor and Phase 66 golden cases

## explicit scope model

### Brainstack-owned scope

- Tier-2 scheduling state
- extraction result recording
- parse/write/no-op/failure instrumentation
- durable promotion boundaries
- provenance filtering and contamination prevention

### out of scope

- generic provider reliability work unrelated to Brainstack extraction
- runtime session-start execution
- broad natural-language heuristic extraction
- making Tier-2 mandatory on every turn
- using Tier-2 as behavior-policy governor

## anti-goals

- no always-on broad extraction tax
- no assistant-message durable truth promotion shortcut
- no silent fallback that reports success after failed parse/write
- no locale-specific extraction phrase farm
- no direct runtime-to-DB state mutation outside Brainstack-owned write paths

## architecture decisions to freeze before execution

### 1. run record model

Define a Tier-2 run record with:

- trigger reason
- input batch identity
- principal/session scope
- request status
- parse status
- emitted candidate counts
- accepted write counts
- rejected/no-op reasons
- error reason
- timestamps and duration

### 2. trigger policy

Define bounded trigger behavior for:

- idle flush
- session-end flush
- explicit flush
- batch-size flush
- disabled/degraded mode

### 3. provenance gate

Define which input evidence classes Tier-2 may promote. Assistant-authored self-diagnosis and reflection residue must stay excluded unless explicitly typed and approved by a later contract.

### 4. no-op semantics

Empty extraction is not automatically failure. It must carry a reason:

- no eligible user evidence
- extractor returned empty valid payload
- parse failed
- all candidates rejected by provenance/schema
- duplicate/superseded write

### 5. doctor/inspect integration

Phase 65 doctor must be able to show last runs and recent failures without dumping raw transcripts.

## workstreams

### 1. current Tier-2 path inventory

Map all current scheduling, idle, session-end, default caller, parser, and write sites.

### 2. run status persistence

Add bounded persistent or queryable run-result state with retention limits.

### 3. trigger policy hardening

Make flush triggers explicit, bounded, and testable.

### 4. promotion gate hardening

Ensure provenance/schema gates reject unsupported assistant-authored or speculative contamination.

### 5. proof and regression

Add tests for success, empty valid extraction, parse failure, provider failure, duplicate no-op, and rejected contamination.

## execution sequencing

1. Freeze run record schema and retention.
2. Instrument existing Tier-2 run path without changing extraction semantics.
3. Add explicit no-op/failure categories.
4. Harden trigger policy.
5. Harden provenance gate tests.
6. Connect status into Phase 65 doctor/inspect.
7. Run Phase 66 relevant golden cases.

## deliverables

- Tier-2 run/result status model
- bounded trigger policy
- parse/write/no-op/failure instrumentation
- provenance contamination tests
- doctor/inspect integration
- execution proof document with current extraction reliability facts

## lightweight process refinements

### terminology reuse

Use the Phase 65 meanings for `no-op`, `failed`, `degraded`, and `unavailable`. In particular, a valid empty extraction must not be labeled as failure unless the frozen glossary says so.

### expected-red ownership

Phase 68 must consume the Phase 66 expected-red cases it owns and close each one as:

- fixed and promoted to hard regression gate
- still red with explicit extraction or provider-boundary reason
- moved to a later phase only with a written dependency reason

### phase handoff gate

Closeout must state:

- last-run observability now available
- extraction failure modes now distinguishable
- expected-red cases closed or carried forward
- whether Phase 69 is blocked by missing Tier-2 evidence

## acceptance criteria

- a strict reviewer can tell whether Tier-2 ran, failed, parsed empty, wrote records, or no-oped
- Tier-2 failures do not disappear as generic warnings
- assistant-authored self-diagnosis cannot become user-owned durable truth through this path
- extraction remains bounded and does not become an always-on hot-path tax
- Phase 66 cases that depend on Tier-2 have traceable pass/fail reasons
