# Phase 16.1 Plan

## Goal

Produce an honest donor-first recovery audit for Brainstack.

This phase does not improve the product directly.
It exists to stop the project from continuing down the wrong architectural slope.

## Scope

### In

- audit `Hindsight`, `Graphiti`, and `MemPalace` against the current Brainstack implementation
- use the local donor repos as the evidence source
- use the current Brainstack source as the reality source
- treat the `3/15` Brainstack LongMemEval result as mandatory baseline evidence
- classify current Brainstack logic into:
  - keep local
  - thin down
  - donor-recenter
  - remove
- write a recovery scope strong enough to drive `16.2`

### Out

- no new memory features
- no benchmark tuning
- no new regex / keyword heuristics
- no donor transplant execution yet
- no L1/L2/L3 restoration coding yet

## Execution Steps

1. Lock the audit question.
   - this is not “what can we improve”
   - it is “what did we promise, what did we actually build, and what must now be restored”

2. Audit donor intent from the local donor repos.
   - `Hindsight`
   - `Graphiti`
   - `MemPalace`

3. Audit current Brainstack reality.
   - donor adapters
   - local extraction
   - local ranking
   - local packaging
   - local glue

4. Compare promise versus reality donor-by-donor.
   - original expected strength
   - actually realized strength
   - justified local value
   - accidental glue
   - restoration target

5. Freeze the recovery rules.
   - no more feature drift during this window
   - no heuristic growth as substitute for donor strength
   - no “good enough for now” fallback acceptance

6. Write the concrete output artifacts.
   - donor audit table
   - recovery scope summary
   - state update

## Required Output Shape

For each donor, the audit must answer:

1. Why was this donor chosen originally?
2. What is the strongest part of that donor today?
3. What part of that strength is actually present in Brainstack now?
4. What part is missing or diluted?
5. Which local Brainstack logic is justified and should remain?
6. Which local logic became accidental kernel glue and should be reduced?
7. What must later be restored in `17/18/19`?

## Hard Rules

- the `3/15` baseline must be treated as hard evidence
- the audit must not hide behind “the system still works”
- the audit must not confuse safety/integration quality with donor-memory strength
- local Brainstack logic that substitutes for donor power must be presumed suspect
- no MVP framing is allowed in this recovery track
- the output must be strong enough that later phases can be judged against it, not reinterpret it

## Exit Gate

- the project has an explicit donor-by-donor audit table
- a feature freeze rule is written down for the recovery window
- justified local value is clearly separated from accidental glue
- the output can directly drive `16.2` without another ambiguity round
