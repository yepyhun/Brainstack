# Phase 16.1 Architecture Notes

## Why this phase exists

The system is no longer in a simple "keep adding features" state.

Brainstack works, but it has drifted away from the original intended shape:

- the vision was donor-first:
  - Hindsight
  - Graphiti
  - MemPalace
- Brainstack itself was supposed to remain a thin, durable owner layer:
  - host ownership
  - orchestration
  - safety
  - packaging
  - update-safe integration

Instead, the current system contains materially more Brainstack-owned middle logic than planned. Some of that glue is justified and valuable. Some of it diluted donor strength and weakened the original "best-of-three" ambition.

This phase is the explicit strategic correction point.

## What this phase is not

- not a benchmark-maximization phase
- not a new feature phase
- not a donor transplant phase
- not a phase for introducing more heuristics
- not a phase for replacing the single Brainstack owner model

## Core questions

For each major donor line, answer:

1. What was the original expected strength?
2. What is actually present now?
3. What became Brainstack-owned glue?
4. Which local logic is justified and should stay?
5. Which local logic is accidental complexity and should be reduced?
6. What must be re-centered around donor-shaped logic in later phases?

## Expected donor-by-donor audit outputs

### Hindsight / Layer 1

- intended role:
  - continuity
  - recency
  - agent-smartening
- likely current gap:
  - continuity exists
  - "noticeably smarter agent" effect is weaker than intended

### Graphiti / Layer 2

- intended role:
  - relationships
  - temporal truth
  - strong knowledge graph
- likely current gap:
  - safety and truth separation improved
  - graph usefulness still not at the original ambition level

### MemPalace / Layer 3

- intended role:
  - strong corpus handling
  - high-quality bounded packing
  - large-document usefulness
- likely current gap:
  - corpus shelf exists
  - full corpus-engine strength is still not clearly realized

## Required output shape

This phase should produce a brutally honest table with columns like:

- donor
- original promise
- current reality
- strong local adaptation worth keeping
- accidental glue to reduce
- later restoration target

## Recovery rule

No new "clever" Brainstack logic should be added unless it clearly belongs to:

- ownership
- orchestration
- safety
- packaging
- update-safe integration

If a candidate change looks like a replacement for donor strength rather than a boundary around donor strength, it should be treated as suspect by default.

## Quality bar

This recovery track is not allowed to degrade into:

- MVP acceptance
- “good enough for now” glue preservation
- benchmark-gaming
- more local heuristics standing in for donor strength

The intended bar is:

- donor-first restoration
- hard module ownership boundaries
- fewer custom moving parts, not more
- a system that is defensibly closer to the original “big 3 plus thin glue” ambition

If a proposed recovery step makes the project merely more elaborate while leaving donor strength weak, it fails the spirit of this phase even if the code technically works.
## New hard evidence from benchmark baseline

- A Brainstack-specific 15-case LongMemEval subset now runs through the real Hermes/Brainstack path.
- Result: `3/15`, `0.20` accuracy, `392.4s`.
- This materially changes the status of the recovery track:
  - the current weakness is no longer only a qualitative suspicion
  - it is now backed by a hard, end-to-end baseline

### Interpretation

- Do **not** respond to this by benchmark-gaming.
- Do **not** add more Brainstack-owned heuristics just to raise the score.
- Do treat this as proof that the donor-first restoration thesis is directionally correct.

### What the score most likely means

- L1 is not delivering the level of agent-smartening originally expected from the Hindsight role.
- L2 is still below the intended Graphiti-like graph/truth strength.
- L3/corpus retrieval is not yet producing the intended big-memory advantage.
- The current custom glue burden is too high relative to the donor-strength actually recovered.

### Audit consequence

- `16.1` must explicitly compare:
  - promised donor strength
  - currently realized Brainstack strength
  - missing or diluted strength
  - custom glue that should be reduced instead of extended
