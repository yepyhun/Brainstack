# Phase 16.1 Donor Audit

## Hard Evidence

- Brainstack-specific LongMemEval subset result:
  - `3/15`
  - `0.20`
  - `392.4s`
- This is not treated as a benchmark target.
- It is treated as proof that donor-strength restoration is necessary.

## Feature Freeze Rule

Until `16.2` is written and accepted:

- no new memory feature work
- no new Brainstack-owned heuristics to “patch over” donor weakness
- no new benchmark loop
- no local glue growth unless it is strictly:
  - ownership
  - orchestration
  - safety
  - packaging
  - host integration

## Concrete Local Evidence

### Hindsight gap evidence

Donor evidence:

- `hindsight/skills/hindsight-local/SKILL.md`
- `hindsight/skills/hindsight-cloud/SKILL.md`

Current Brainstack evidence:

- `brainstack/donors/continuity_adapter.py`
- `brainstack/tier1_extractor.py`

What this shows:

- the donor expects rich retain from full context and server-side structured extraction
- current Brainstack continuity adapter mainly writes turn summaries and transcript snapshots
- current Brainstack still leans on explicit Tier-1 heuristics such as:
  - `vibecoder`
  - `nem developer`
  - `emoji`
  - `humanizer`

Conclusion:

- L1 is carrying useful continuity, but not the original Hindsight-level smartening burden

### Graphiti gap evidence

Donor evidence:

- `graphiti/README.md`

Current Brainstack evidence:

- `brainstack/donors/graph_adapter.py`
- `brainstack/retrieval.py`
- `brainstack/control_plane.py`

What this shows:

- the donor emphasizes temporal context graph, validity windows, episodes/provenance, and graph-aware retrieval
- current Brainstack graph adapter is very thin, delegating into local graph ingestion
- Brainstack now has good truth classes and safer packaging, but much of the usable graph behavior still depends on local retrieval and packaging glue

Conclusion:

- L2 is no longer shallow in structure, but it is still below the intended graph power

### MemPalace gap evidence

Donor evidence:

- `mempalace/README.md`
- `mempalace/benchmarks/BENCHMARKS.md`

Current Brainstack evidence:

- `brainstack/donors/corpus_adapter.py`
- `brainstack/retrieval.py`
- `brainstack/control_plane.py`

What this shows:

- the donor emphasizes corpus hierarchy, search power, and bounded packing as a real product strength
- current Brainstack corpus adapter mainly prepares stable keys and sections
- the L3 shelf exists, but not yet with enough restored retrieval/pacing power to justify the donor expectation

Conclusion:

- L3 exists as a clean seam, but not yet as a donor-strength advantage

## Recovery Consequence

The project should now be treated as:

- a strong integration shell
- with diluted donor memory power

That means the next phase must not ask:

- “what else can Brainstack add?”

It must ask:

- “which Brainstack middle logic should stay because it protects ownership and safety?”
- “which Brainstack middle logic should be reduced because it replaced donor strength without matching donor power?”

## What Must Later Stay Local

These remain justified Brainstack-owned responsibilities:

- host ownership
- orchestration
- safety
- packaging
- installer / doctor / update-safe integration

## What Must Later Be Restored Donor-First

- `17` = Hindsight-shaped continuity and smartening
- `18` = Graphiti-shaped graph and temporal knowledge strength
- `19` = MemPalace-shaped corpus retrieval and packing strength
