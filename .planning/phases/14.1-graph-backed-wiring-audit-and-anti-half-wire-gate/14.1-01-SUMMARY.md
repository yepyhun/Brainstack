- Closed a deeper runtime-truth bug family in the installed Bestie gateway:
  - Docker health had been process-only, so the container could report `healthy` while Discord was not actually ready
  - `gateway_state.json` could preserve stale failure fields because `write_runtime_status(..., exit_reason=None)` did not clear old values
- Added a readiness-aware gateway health helper in the installed runtime:
  - `/home/lauratom/Asztal/ai/memory-repo-bakeoff/hermes-agent-bestie-latest/scripts/hermes-gateway-healthcheck.py`
- Updated the installed runtime so gateway status now reflects real adapter state:

  - `/home/lauratom/Asztal/ai/memory-repo-bakeoff/hermes-agent-bestie-latest/gateway/run.py`
- Updated the installed Docker path so health and operator status now depend on readiness rather than mere process existence:
  - `/home/lauratom/Asztal/ai/memory-repo-bakeoff/hermes-agent-bestie-latest/docker-compose.bestie.yml`
  - `/home/lauratom/Asztal/ai/memory-repo-bakeoff/hermes-agent-bestie-latest/scripts/hermes-brainstack-start.sh`

## Core Phase-14.1 Decisions
- Treated the Discord symptom as a system-truth and wiring problem, not as a memory-only bug.
- Fixed the deeper readiness contract instead of adding another prompt or memory-side workaround.
- Kept the solution bounded:
  - no new runtime
  - no queue server
  - no extra memory owner
  - no giant audit framework
- Kept the carry-forward rule intact:
  - fix the live runtime
  - then move the same logic into the Brainstack installer and doctor path

## Anti-Half-Wire Proof
- Before the fix, the runtime could be simultaneously:
  - Docker `healthy`
  - `gateway_state: running`
  - and still effectively disconnected from Discord with stale failure state
- After the fix, the live runtime reports a truthful single-path state:
  - `gateway_state: running`
  - `exit_reason: null`
  - `platforms.discord.state: connected`
- The start helper now waits for actual readiness and prints the readiness result instead of implying success immediately after `docker compose up`.
- The integration kit dry-run and doctor both confirm that the same readiness-aware contract is carried into future Hermes installs instead of living only in one patched checkout.

## Outcome

- the system no longer looks healthy while secretly being disconnected
- operators now get truthful runtime/readiness signals
- the fix survives future Hermes refreshes because it was moved into the Brainstack integration kit
- the upstream Hermes issue now contains a tighter root-cause report instead of only the symptom
