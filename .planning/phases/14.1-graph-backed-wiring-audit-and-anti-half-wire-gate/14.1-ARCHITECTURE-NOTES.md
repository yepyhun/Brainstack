# Phase 14.1 Architecture Notes

Captured: 2026-04-11
Status: pre-planning note

## Why This Phase Exists
This phase is the late-stage copy of the earlier anti-half-wire / AI wiring audit pattern, but applied after the Brainstack feature path is already built and practically proven.

## Scope Rule
Keep this phase bounded.

It should not become:
- a giant new test framework
- a second proving milestone
- a 40-checkpoint documentation dump

It should be:
- graph-backed
- test-backed
- live-path-backed
- short enough to stay practical

## Intended Failure Patterns To Catch
- a new module exists but the old direct path is still live
- a seam exists in code but is never actually called
- a provider branch silently bypasses the intended write-policy path
- a new helper looks wired in tests but is absent from the real runtime path
- the architecture diagram is true on paper but false in the live process

## Priority Failure Families
Focus on the agentic-coding failure modes that create the most damage with the least visible surface area.

1. Path ownership drift
- the new Brainstack path exists, but an older direct path is still writing or reading live data
- two owners silently coexist, so behavior depends on branch order or runtime timing

2. Declared seam, dead seam
- a clean interface or helper was introduced, but no real caller reaches it in production
- the code looks modular in review, but the seam is decorative

3. Test-only wiring
- the intended path is exercised in unit or integration tests
- the real runtime still uses a different path, stub, or fallback

4. Write accepted, propagation missing
- a turn is logged or accepted at the entry point
- but it never reaches the intended downstream shelf, reconciler, or write-policy path

5. Config/runtime mismatch
- config says Brainstack owns the memory path
- but the live process still routes through an old provider, fallback, or hidden branch

6. Dead adapter / unused result
- a new adapter, helper, or donor-inspired module runs
- but its output is ignored, dropped, or superseded by an older call path

## What This Phase Should Explicitly Ignore
Do not spend time here on generic style review, micro-optimizations, or broad code-quality policing.

This phase is specifically for:
- silent bypasses
- duplicate owners
- decorative seams
- hidden fallbacks
- dropped outputs
- paper architecture drift

## Preferred Audit Formula
Use all three, but keep each bounded:

1. Graph evidence
- caller/callee or file-level path checks
- stale direct path detection
- seam visibility

2. A few targeted tests
- only the highest-value anti-half-wire cases
- no giant matrix

3. Live runtime checks
- practical provider/log/DB/path proof
- enough to confirm the real route is active

## Minimal Proof Shape
Each high-priority finding should ideally be supported by at least two of the following:

- graph evidence that shows the expected caller/callee or file-level route
- one targeted test that proves the intended path is the active one
- one live runtime proof such as a log, DB write, provider status, or observed single-path behavior

Do not expand this into a giant evidence pack.
The purpose is to catch the classic "looks integrated but is not" vibecoding failures with bounded effort.

## Important Constraint
Graph MCP is support evidence, not the only source of truth.

The phase should explicitly separate:
- graph-backed findings
- test-backed findings
- live-runtime findings

## Design Principle
This is a sanity gate against common agentic-coding mistakes.
It is not a new architecture phase and not a replacement for the earlier feature work.

More specifically, it is the late-stage bounded copy of the earlier anti-half-wire guard idea:
catch the small number of high-damage integration failures that AI coding regularly introduces while keeping the audit practical.
