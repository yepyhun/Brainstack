# Phase 29.5 Plan

## goal

Close the immediate live `27`-rule recall blocker with the newest evidence, then hand off cleanly into the larger always-on behavior-policy architecture sequence.

## scope

### in

- live route/runtime parity for explicit style questions
- the canonical detailed style-contract owner row
- correction / supersession behavior for later rule edits
- render/projection of the detailed lane into explicit recall
- live verify after reset

### out

- general retrieval-health redesign
- prompt-output policing
- ordinary-turn compiled behavior policy redesign
- new sidecar files or second owners
- benchmark score chasing

## north star

After `29.5`, the deployed runtime must be able to reach and surface the correct detailed style-contract owner when the user explicitly asks for the full rules.

This phase does not yet solve the broader "always obey the rules in every ordinary turn" architecture.

It solves the bridge problem cleanly enough that `29.6+` can build on truth instead of guesswork.

## core principles this phase must preserve

1. donor-first
   - reuse the existing Tier-2, canonical profile, and retrieval seams
   - do not create a parallel style-memory subsystem

2. modular, thin integration
   - fix only the ingest / canonicalization / supersession seam that actually drops information
   - do not smear the repair across unrelated retrieval or host code

3. no heuristic sprawl
   - no regex-driven rule mining
   - no language-specific style guessing
   - no post-hoc prompt patching to fake recall

4. single durable owner
   - one canonical detailed style-contract row remains the authoritative owner
   - compact operational communication slots may coexist, but must not become a second detailed owner

5. native-system synergy, not replacement
   - ordinary turns stay on the compact operational contract
   - explicit detailed recall stays on the detailed style-contract lane

6. durable only where durable value is real
   - preserve the full taught contract because it is stable, user-specific, and cross-session valuable
   - do not turn every transient formatting preference into a new artifact type

## lane decision matrix

| memory class | ordinary turn needed | explicit detailed recall needed | principal-scoped | canonical lane |
| --- | --- | --- | --- | --- |
| compact operational communication rules | yes | no | yes | profile compact slots |
| detailed taught style contract | no | yes | yes | canonical `preference:style_contract` row |
| raw teaching transcript | no | fallback only | yes | transcript evidence, non-authoritative |

## architecture decision

### accepted target model

- keep the current two-lane shape for now
  - compact operational communication contract
  - detailed canonical style-contract row
- prove the immediate live blocker precisely
- repair only the minimum deployed-path seam needed so explicit detailed recall works again
- leave the broader always-on policy architecture to `29.6+`

### rejected model

- declaring the problem solved by a local/live probe while Discord still fails
- returning to corpus as the detailed owner
- storing the contract in files, skills, or persona sidecars
- adding per-language heuristics to reconstruct dropped rules later

## workstreams

### 1. deployed-path blocker proof

- inspect the exact live path for:
  - explicit question
  - route selection
  - detailed profile match
  - final rendered block
- prove whether the remaining live failure is:
  - route/runtime dependency
  - render/projection
  - supersession
  - or deploy parity

### 2. bounded seam repair

- repair only the proven live seam
- keep the detailed owner canonical
- keep ordinary turns compact
- do not smuggle in the future compiled-policy design here

### 3. canonicalization hygiene

- enforce one canonical detailed owner row
- ensure no partial legacy row keeps winning after a richer corrected row exists
- verify later corrections overwrite or supersede correctly instead of partially merging

### 4. live verify

- confirm the deployed runtime uses the fixed path
- confirm explicit post-reset recall reaches the detailed canonical owner
- confirm the user can ask for the full rules without re-teaching
- confirm ordinary turns still do not inject the full long-form contract
- record any residual gap that belongs to `29.6+`, not `29.5`

## validation ladder

1. unit or probe proves the exact current deployed blocker
2. targeted seam test proves the repaired path
3. correction/supersession test still passes
4. live DB verify proves the canonical row is full and current
5. live post-reset explicit recall verify passes in the real runtime

## anti-goals

- no broad retrieval refactor
- no prompt-bloat
- no model-behavior overfitting
- no dead code from previous failed style-contract experiments

`xhigh`
