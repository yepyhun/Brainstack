# Phase 29.5 Context

## problem statement

The live system can now route explicit detailed-style questions into the canonical `preference:style_contract` lane.

The latest live evidence shows the situation is narrower and more concrete:

- the full detailed `preference:style_contract` row can exist in the DB
- but live explicit recall still fails in the deployed path

So the current problem statement is no longer "storage is always lossy".

It is:

- exact post-reset detailed recall is still not reliable in the real runtime
- and the remaining blocker must be localized cleanly before `29.6+` expands the architecture into compiled always-on behavior policy

## why this phase exists

Phase `29.4` already proved the bounded oracle retrieval-only gate is healthy again.

So the remaining problem is no longer “did the whole memory system regress?”

It is narrower:

- a user teaches a long-form multi-section style contract
- Brainstack may store the full contract in the canonical detailed lane
- later explicit recall still fails live
- therefore the remaining culprit is somewhere in:
  - route/runtime parity
  - render/projection
  - or deployed-path wiring

## live evidence already established

- the active canonical profile row exists:
  - `stable_key = preference:style_contract::principal_scope::...`
- newer local/live probes have shown full multi-section content in that row
- a separate short `preference:communication_rules` summary row also exists
- live failures have included a route-hint dependency break that caused explicit style questions to fall back to `fact`
- the older corpus style-contract document is inactive and therefore not the authoritative owner anymore
- the compact active communication contract still exists separately and should remain compact

## key distinction

This is no longer safely readable as a pure ingest-loss problem.

The strongest current evidence now points to a mixed deployed-path seam:

- storage may already be correct
- but explicit live recall can still miss the detailed lane because of route/runtime/projection issues

## likely seam candidates to prove or rule out

1. The deployed route/runtime path is still not reliably activating `style_contract` mode for explicit questions.
2. The live render/projection path still surfaces only the compact summary even when the detailed lane exists.
3. The canonical detailed row is still being overwritten or out-ranked by a shorter summary row.
4. Later corrections are not superseding cleanly, so the detailed owner remains semantically stale.

- do not broaden into general memory retrieval redesign
- do not add prompt-policing or output guardrails
