## Proven Findings

### 1. The live half-wire was real, but it was not the main score blocker

The installed Bestie plugin had a stale `executive_retrieval.py` compared to source.

That mismatch was real and was fixed.

But after fixing it, the full live reruns got worse, not better.

That means the stale-file half-wire was hiding the true restoration quality, not causing all of the retrieval collapse by itself.

### 2. The route-hint response-shape bug was real and is now fixed

The live route resolver previously failed because MiniMax frequently returned:

- empty `content`
- non-empty `reasoning_content`

with too-small route-hint token budgets.

After fixing parsing and increasing the route-hint token budget, live structural routing actually activated.

So the old ŌĆ£route activation is dead because the live resolver is brokenŌĆØ explanation is no longer valid.

### 3. Structural route activation is now real, but route quality is poor

Compared with `20.5`, `20.6` no longer stayed almost entirely on default fact mode.

Live route activation materially increased:

- answer-only non-fact modes: `6 / 15`
- split non-fact modes: `4 / 15`

But the newly activated routes did not improve quality enough.

This means the blocker moved from:

- route wiring not firing

to:

- poor route selection
- poor route-specific candidate quality
- poor evidence quality once the structural routes actually run

### 4. Backend population is no longer mostly empty

`20.5` made graph yield look almost empty.

The repaired `20.6` runs show a different truth:

- answer-only backend population:
  - `SQLite` graph nonzero: `14 / 15`
  - `Kuzu` graph nonzero: `14 / 15`
  - `Chroma` nonzero: `15 / 15`
- split backend population:
  - `SQLite` graph nonzero: `12 / 15`
  - `Kuzu` graph nonzero: `12 / 15`
  - `Chroma` nonzero: `15 / 15`

So the main remaining failure is no longer ŌĆ£the graph backend is empty in the benchmark path.ŌĆØ

The harder problem is now:

- bad route/candidate quality
- bad evidence selection
- and at least one execution-path anomaly

### 5. Fact-path parity is still broken, and now measured as worse than before

Best known older strong answer-path baseline:

- `20.2` decomp-off answer path: `10 / 15`

Later comparable live answers:

- `20.4`: `5 / 15`
- `20.5`: `4 / 15`
- `20.6`: `3 / 15`

So fact-path parity was not restored.

It got worse under the repaired live route-active stack.

### 6. The profile / identity retrieval miss class is real

`5d3d2817` remains a clean example:

- `requested_mode=fact`
- `applied_mode=fact`
- route actually ran
- retrieval still failed

This is not a dead-route artifact.

It is a genuine profile / identity retrieval miss class on the live fact path.

### 7. The grounding leak class is real

`d682f1a2` remains the clean bounded case:

- `retrieval_correct=true`

So there is still a real memory-to-answer handoff leak even when retrieval itself succeeds.

### 8. There is now an explicit execution-path anomaly class

`6a1eabeb` and `gpt4_7f6b06db` showed a different failure shape:

- `route_events` empty
- `requested_mode` blank

This is not a normal retrieval miss.

It is a separate execution-path anomaly that must be tracked explicitly instead of being flattened into generic retrieval failure.
