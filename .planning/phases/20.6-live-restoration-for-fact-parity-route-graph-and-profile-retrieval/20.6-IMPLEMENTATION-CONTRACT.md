# Phase 20.6 Implementation Contract

## Purpose

Restore the live benchmark-path retrieval stack after `20.5` without reopening the already-closed proof-calibration loop.

## Required Workstreams

### Workstream A. Fact-Path Parity Restoration

This workstream is first on purpose.

It must:

- document the exact runtime environment and effective path of the earlier stronger decomp-off baseline before comparing
- audit packet shape, rendering, budgets, and runtime assumptions on the current live fact path
- identify whether part of the collapse is inside the fact path even when routing stays on `fact`

It must not:

- collapse into “probably route activation”
- be deferred behind structural route work again

### Workstream B. Route Activation And Graph Yield Recovery

This workstream treats route activation and graph yield as one coupled restoration problem.

It must:

- improve route activation coverage in a bounded, language-agnostic way
- recover graph yield in the same execution slice
- verify that newly activated routes do not silently lose the currently load-bearing priority behavior without an explicit route-appropriate replacement
- measure the two together in the rerun output

It must not:

- reactivate the old decomposition gate
- build multilingual keyword routing tables
- widen route activation while leaving structural routes effectively empty
- claim graph recovery without showing live route use

### Workstream C. Profile / Identity Retrieval Miss Audit

This workstream exists because `5d3d2817` exposed a distinct miss class.

It must:

- audit identity-like / profile-like fact questions as a named retrieval class
- inspect corpus retrieval, ranking, and packing behavior for that class
- state explicitly whether the class reproduced in `20.6`

It must not:

- bury the class under generic retrieval failure language
- speculate if the sample does not reproduce

### Workstream D. Bounded Grounding Leak Audit

This workstream is bounded to known retrieval-correct / answer-wrong rows.

It must:

- audit cases like `d682f1a2`
- inspect whether the recalled evidence was visible and answer-bearing in the final prompt
- identify whether the leak is formatting, priority, prompt handoff, or model behavior
- state explicitly if the sample does not reproduce strongly enough for action

It must not:

- reopen broad judge redesign
- relabel weak retrieval into success
- drift into answer-score laundering

### Workstream E. Comparable Live Reruns

This workstream closes the phase.

It must:

- run comparable answer-only and split reruns
- report answer score, retrieval score, both-correct, route activation distribution, and graph-yield distribution
- clearly say whether live restoration improved or remained insufficient

## Protected Boundaries

### Proof Boundary

`20.5` already did the major proof-calibration work. `20.6` may use that instrumentation, but must not turn back into a proof-methodology phase.

### Restoration Boundary

This phase is about live benchmark-path restoration. It must not treat better diagnostics alone as a restoration win.

### Heuristic Boundary

This phase must not reopen `20.2`-style overlay tuning as the main strategy.

### Donor Boundary

This phase must stay donor-first:

- no donor replacement
- no shell-side reimplementation of donor internals as the main fix
- no benchmark-only logic that cannot be justified outside the benchmark

## Minimum Proof Required Before Calling Phase 20.6 Done

Before calling `20.6` done, the phase must provide:

- a concrete fact-path parity verdict
- a concrete combined route-activation / graph-yield verdict
- a concrete profile / identity miss verdict
- a concrete bounded grounding leak verdict
- final comparable reruns

If one of the smaller classes (`C` or `D`) does not reproduce strongly enough, the phase must say so explicitly rather than invent a fix.
