# Phase 63.1 Context

## why this phase exists

Recent live evidence exposed a durable-rule authority gap:

- explicit engineering doctrine can land as a skill
- the same doctrine can appear in transcript/continuity evidence
- native memory may reject further writes because it is char-bounded
- Brainstack does not yet guarantee that such rules become canonical policy authority

That means the system can "remember" a rule in multiple weak places without having one clearly authoritative Brainstack lane for it.

## current live facts

### 1. native memory is not the right authority surface

- native `MEMORY.md` and `USER.md` are intentionally small compatibility stores
- they can become full even while skills or Brainstack evidence continue to grow
- this makes them a poor primary authority for durable engineering doctrine

### 2. skills are real, but procedural

- a skill is a good place for reusable workflow know-how
- it is not the best sole authority for general user/project policy that later runtime logic may need to consume as canonical truth

### 3. Brainstack currently has the rule as evidence

- transcript and continuity evidence can carry the explicit "no bandaid/reactive patching" doctrine
- but evidence alone is weaker than a dedicated canonical policy lane

### 4. runtime work should not guess policy from residue

- later runtime-side inbox/intake/governance work should not infer rules from transcript text, skill prose, or native-memory spillover
- it should read a Brainstack-owned canonical policy snapshot

### 5. the new anti-rule must be explicit

- the user explicitly asked to freeze the principle that the memory kernel cannot be governor
- this is not implicit architecture taste anymore
- it is an explicit project rule

## generalized defect statement

Brainstack can already hold explicit rules as evidence or procedure, but it does not yet cleanly separate canonical durable policy authority from transcript residue, procedural skills, and native compatibility memory. That blurs what later runtime phases should trust.

## practical model to prefer

This phase should not overdesign five equal authority planes.

The preferred practical model is:

- `canonical policy`
- `operating/live state`
- `evidence`

And two supporting stores that are not equal-authority policy planes:

- `skill` as procedural support memory
- `native memory` as compatibility/support context

## why this is Brainstack-owned

- the missing authority lane is inside Brainstack
- the missing plane separation is inside Brainstack
- the missing promotion/demotion discipline is inside Brainstack
- runtime work should consume the outcome, not define the memory architecture

## why Phase 64 should wait

Phase `64` is runtime-heavy. If it runs before this phase, it risks consuming:

- transcript residue
- skill prose
- native-memory spillover
- ad hoc conventions

instead of a canonical Brainstack policy snapshot.

That would make runtime behavior appear to work while the memory authority model remains blurred.

## what the next execution must decide

### 1. what the canonical durable policy lane is

- record type
- storage owner
- projection surface

### 2. what belongs in each memory plane

- procedural skill memory
- durable policy/preference memory
- operating/live-state memory
- transcript/continuity evidence
- native compatibility memory

### 3. what may be promoted to canonical authority

- explicit durable user/project rules
- explicit approval constraints
- explicit workflow doctrines
- explicit user-declared engineering doctrine

Promotion should default to:

- `no promotion unless explicit`

### 4. what must remain evidence only

- assistant narration
- recap text
- incidental transcript overlap
- speculative explanations
- assistant-inferred "this seems important" judgments

### 5. how runtime will read the result later

- through an explicit Brainstack-owned read snapshot
- not by scraping prose from other surfaces

## non-goals to reject

- runtime governance in Brainstack clothing
- "memory decides everything" architecture
- save-everything memory design
- skill-only doctrine storage
- transcript-only doctrine storage
- local one-user rescue logic
