# Phase 63.1 Plan

## purpose

Freeze the Brainstack-owned durable policy canon and memory-plane boundaries so explicit long-lived user/project rules become canonical state without turning the memory kernel into a runtime governor.

## core question

How should Brainstack canonically store and project explicit durable rules and memory-plane roles so the runtime can consume them later, while keeping policy authority inside Brainstack and governance/enforcement outside it?

## primary goals

- define a Brainstack-owned canonical durable policy/preference lane for explicit user and project rules
- freeze a practical authority model with three canonical categories:
  - `canonical policy`
  - `operating/live state`
  - `evidence`
- keep `skill` as procedural support memory and `native memory` as compatibility/support context rather than equal-authority policy planes
- define what may be promoted into canonical policy authority and what must remain only evidence
- freeze the anti-rule that the memory kernel is not governor
- define a read-only runtime-facing policy snapshot that later runtime phases can consume

## explicit scope model

### Brainstack-owned scope

- canonical durable policy/preference storage
- policy projection into operating context or dedicated snapshots
- promotion/demotion discipline for durable authority
- memory-plane separation rules
- public Brainstack read seams for durable policy retrieval

### out-of-scope

- runtime session-start intake
- cron behavior
- approval enforcement
- task execution
- skill-manager rewrite
- generic Hermes prompt or style-governance cleanup

## anti-goals

- do not turn Brainstack into the scheduler, executor, or approval governor
- do not rely on transcript residue or native-memory overflow as the primary durable authority for important rules
- do not collapse skills, policy, operating truth, and transcript evidence into one memory bucket
- do not solve importance by heuristic keyword scoring or "save everything"
- do not bake one Discord thread or one exact rule wording into the architecture

## accepted evidence

- explicit user engineering rules like "no bandaid/reactive patching" are currently able to land in:
  - a skill
  - transcript/continuity evidence
- those rules are not yet guaranteed to exist in a canonical Brainstack policy lane
- native memory is char-bounded and therefore not a good primary authority surface for durable engineering doctrine
- later runtime work should consume canonical Brainstack policy state rather than infer rules from skills or transcript residue

## validated current reading

- the system already has multiple memory planes, but their roles are still too blurred for durable rule authority
- the right answer is not "save less by vibe" and not "save everything"
- the right answer is explicit promotion discipline plus explicit plane boundaries
- runtime-heavy work should wait until this Brainstack authority shape is frozen

## architecture direction to freeze before execution

### 1. durable-policy decision

- explicit long-lived user and project rules need a canonical Brainstack lane
- that lane must be separate from transcript residue and separate from procedural skills

### 2. authority-model decision

- the practical canonical model is:
  - `canonical policy` for explicit durable rules
  - `operating/live state` for current truth
  - `evidence` for transcript/continuity support
- `skill` remains procedural support memory
- `native memory` remains compatibility/support context
- these support stores are not equal-authority policy lanes

### 3. promotion decision

- only explicit durable rules may be promoted into canonical policy authority
- examples:
  - explicit user corrections
  - explicit workflow doctrines
  - explicit project rules
  - explicit approval/policy constraints
- default behavior is `no promotion unless explicit`
- assistant recap, speculative narrative, incidental transcript overlap, and inferred "importance" must not be promoted as canonical rules

### 4. non-governor decision

- Brainstack may project policy truth
- it may not enforce execution, schedule work, or decide approval as the acting governor
- later runtime code may consume Brainstack policy, but Brainstack itself does not become the acting control loop

### 5. runtime-handoff decision

- this phase must define a read-only policy snapshot contract that later runtime phases can consume
- the contract must not require transcript scraping, skill scraping, or natural-language policy inference

## workstreams

### 1. canonical policy lane design

- define the minimal record type, projection shape, and retrieval surface for durable policy/preference authority
- prefer extending an existing Brainstack-owned record family or snapshot if that keeps the topology simpler and clearer

### 2. memory-plane separation design

- freeze the exact role boundaries between:
  - canonical policy
  - operating/live state
  - evidence
  - procedural skill support
  - native compatibility support

### 3. promotion/demotion contract

- define what inputs may become canonical policy and what must remain only evidence

### 4. runtime-facing snapshot design

- define the minimal read-only snapshot that runtime may consume later without creating a governor loop inside Brainstack

### 5. proof design

- define how to prove that durable rules land in canonical policy state
- define how to prove that transcript residue is no longer the only authority
- define how to prove that Brainstack still does not become governor
