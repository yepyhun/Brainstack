# Phase 63.1 Architecture Decision

## decision

Brainstack will use a minimal canonical authority model:

- `canonical policy`
- `operating/live state`
- `evidence`

with:

- `skill` as procedural support memory
- `native memory` as compatibility/support context

This phase does not introduce a new governor loop. Brainstack stores and projects durable policy truth, but runtime ownership of scheduling, approval enforcement, and execution remains outside Brainstack.

## implementation choice

- reuse the existing `operating_records` substrate
- add a new explicit record type:
  - `canonical_policy`
- expose it through:
  - operating-context projection
  - a dedicated read-only provider snapshot

## promotion rule

Promotion is explicit-only.

Canonical policy authority may be written only through:

- explicit Brainstack API calls
- explicit typed envelopes that name `canonical_policy`

The system does not infer canonical policy from transcript importance, assistant recap, or heuristic scoring.

## stable-key rule

Canonical policy rows use explicit `rule_id` when available, so a rule can be updated without creating content-hash duplicates.

## runtime handoff rule

Later runtime phases may consume a read-only canonical policy snapshot. They must not scrape transcript residue, skill prose, or native-memory spillover to recover policy truth.

## rejected alternatives

- new standalone `policy_records` table
- transcript/continuity as sole durable rule authority
- skill-only doctrine storage
- any design where Brainstack decides execution or approval
