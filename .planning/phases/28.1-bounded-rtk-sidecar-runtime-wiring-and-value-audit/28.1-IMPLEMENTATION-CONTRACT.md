# Phase 28.1 Implementation Contract

## Objective

Settle the RTK-sidecar seam with the smallest correct shape:

- correct runtime wiring if needed
- optional tiny plain-text pre-normalization if and only if it proves value
- no broad filter-engine growth

## System doctrine this phase must preserve

- Brainstack ships the sidecar payload and installer path
- Hermes runtime remains the execution host, not a second source-of-truth
- `tool_result_storage` owns persistence/truncation/turn-budget behavior
- `rtk_sidecar` may shape config, telemetry, and at most one tiny preprocessing step
- explicit no-op is a valid success outcome

## Workstream A: Runtime wiring truth

- verify whether the current live/runtime path is missing the RTK sidecar import/init/calls that already exist in the source path
- if yes, define the narrowest parity correction
- if no, close the wiring question cleanly and move on

## Workstream B: Safe hook-point contract

- keep the sidecar hook immediately before `maybe_persist_tool_result(...)`
- keep per-turn aggregate pressure at `enforce_turn_budget(...)`
- do not create a second policy owner under `tool_result_storage`
- do not push sidecar logic down into tools themselves

## Workstream C: Bounded uplift audit

- evaluate only tiny candidates such as:
  - repeated-line collapse
  - blank-line compaction
  - obvious plain-text boilerplate trimming
- require each candidate to satisfy all of:
  - plain-text safe
  - generic
  - low-maintenance
  - measurable

## Workstream D: Measurement and verdict

- distinguish savings caused by:
  - normalization
  - persistence
  - truncation
  - turn-budget enforcement
- require a concrete verdict:
  - wiring-only
  - wiring plus tiny uplift
  - explicit no-op

## Protected boundaries

### Anti-overengineering boundary

- no command-specific filter registry
- no mini-RTK inside Hermes
- no heuristic pile that only solves hand-picked examples

### Structured-output safety boundary

- JSON and other structured outputs are presumed unsafe to rewrite
- the uplift must default to pass-through on structured outputs unless proven otherwise

### Ownership boundary

- `rtk_sidecar.py` owns:
  - config
  - mode
  - telemetry
  - optional tiny pre-normalization
- `tool_result_storage.py` owns:
  - path generation
  - persistence
  - persisted preview construction
  - turn-budget pruning behavior

### Runtime parity boundary

- if the sidecar ships via Brainstack installer, installer truth must match runtime truth
- `hermes-final` is a validation/use target, not a private fork with silent logic drift

## Minimum evidence required before calling Phase 28.1 done

- exact source/runtime wiring truth is written down
- exact safe hook points are written down
- the uplift decision is supported by real examples, not taste
- any landed change stays thin enough that a future strict audit would still call it justified
