# Phase 28.1 Plan

## Goal

Decide whether the local RTK sidecar should land as:

- wiring-only
- wiring plus a tiny plain-text normalization step
- or explicit no-op

without breaking runtime behavior or turning the sidecar into a second RTK.

## Scope

### In

- source-of-truth vs runtime wiring audit for RTK sidecar
- exact hook-point choice and ownership boundaries
- bounded candidate uplift for plain-text tool-result cleanup before persistence/truncation
- measurement plan for proving whether the uplift helps at all

### Out

- full donor port from upstream `rtk`
- command-specific filtering rules
- a new filter DSL
- broad host/runtime rewrites outside the sidecar seam

## Decision rule

Land only the smallest slice that proves real value.

If wiring truth is the only justified fix, stop at wiring.
If a tiny uplift cannot show measurable gain or threatens correctness, do not land it.

## Hypothesis -> Falsification -> Decision

### 1. Runtime wiring candidate

- hypothesis:
  - the main practical gap is that the local sidecar logic is not cleanly active on the real runtime path
- falsification:
  - runtime truth shows the deployed path is already correctly wired and parity-clean
- decision:
  - if false, do not patch runtime wiring

### 2. Tiny pre-normalization candidate

- hypothesis:
  - a tiny plain-text cleanup step before `maybe_persist_tool_result(...)` can reduce token surface better than raw truncation alone
- falsification:
  - sample outputs show the current persistence/truncation path already captures nearly all useful savings, or the cleanup step risks meaning distortion
- decision:
  - if false, keep the sidecar as budget/stats only

### 3. Structured-output safety candidate

- hypothesis:
  - the uplift can be bounded to plain-text/noisy outputs without touching structured outputs
- falsification:
  - the only meaningful gains require mutating JSON or other structured payloads
- decision:
  - if false, do not land the uplift

### 4. Telemetry-value candidate

- hypothesis:
  - slightly richer sidecar telemetry can prove whether savings came from:
    - normalization
    - persistence
    - truncation
    - aggregate turn-budget pressure
- falsification:
  - the extra telemetry adds complexity without changing any real decision
- decision:
  - if false, keep the current simpler stats

## File-level execution order

1. `run_agent.py`
   - verify exact RTK import/init/callsite truth in source and runtime
   - settle the safe hook locations
2. `agent/rtk_sidecar.py`
   - define whether the sidecar stays config/stats only or gains one tiny pre-normalization helper
3. `tools/tool_result_storage.py`
   - verify ownership stays here for persistence/truncation/turn-budget logic
   - avoid moving RTK logic into storage internals
4. `scripts/install_into_hermes.py`
   - confirm shipped payload/install path if runtime parity needs correction
5. tests / doctor
   - only after the boundary is settled

## Safe integration points

### Bind here

- `run_agent.py` constructor:
  - create sidecar config/stats once after config load
- per-tool result path:
  - immediately after the raw tool result returns
  - immediately before `maybe_persist_tool_result(...)`
- per-turn budget path:
  - keep aggregate `enforce_turn_budget(...)` where it already sits, after the tool messages for that turn are assembled

### Do not bind here

- inside individual tool implementations
- deep inside `tool_result_storage.py` as a second owner of policy
- after persisted-output sentinel generation
- on structured/JSON payloads unless they are explicitly proven safe

## Stop conditions before any patch

- if the runtime path is already parity-clean, stop calling wiring a problem
- if the only visible win needs broad heuristics, stop
- if structured-output safety cannot be preserved cleanly, stop
- if the phase starts inventing command-specific rules, stop

## Regression net

- no broken tool-result semantics
- no degraded JSON/tool-call continuity
- no double-ownership between sidecar and `tool_result_storage`
- no installer drift between Brainstack source and real runtime target

## Verification ladder

1. truth audit:
   - source vs runtime wiring map
2. bounded sample proof:
   - noisy plain-text outputs
   - structured outputs
   - already-small outputs
3. safety proof:
   - no semantic loss on structured outputs
4. value proof:
   - measured token/char reduction or clearer preview quality
5. final gate:
   - wiring only
   - tiny uplift
   - or no-op

## Required working style

- no blind donor copying
- no phase closure on “seems good”
- no RTK mythology:
  - only measured local value counts
- if the best answer is “keep it thin and do not add more logic”, record that explicitly
