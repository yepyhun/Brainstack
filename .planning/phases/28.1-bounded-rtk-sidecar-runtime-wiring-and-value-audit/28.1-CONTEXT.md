# Phase 28.1 Context

## Why this phase exists

The RTK sidecar question is now real for two different reasons:

- the project already has a local `rtk_sidecar` layer
- the live runtime wiring is not cleanly aligned across the planning/source path and the actual runtime target

At the same time, the latest upstream `rtk` project is **not** a drop-in code donor for that local layer.
It is a separate command-output filtering tool that lives earlier in the pipeline.

That creates a bounded product question:

- should the local sidecar remain almost purely a budget/telemetry wrapper
- should it gain a very small pre-normalization step
- or should it stay as-is because the extra logic would create more maintenance and risk than value

This phase exists to answer that question honestly before more code lands.

## Settled truths this phase must preserve

- Brainstack remains the source-of-truth for shipped sidecar payload files and installer behavior
- `hermes-final` is the runtime/use target, not an experimental fork
- the local sidecar must remain thin
- `tools/tool_result_storage.py` remains the owner of:
  - oversized-result persistence
  - truncation fallback
  - aggregate turn-budget enforcement
- the sidecar must not become a command-specific filter engine
- the phase must prefer explicit no-op over clever but fragile behavior
- structured tool outputs must not be “cleaned up” in ways that can distort meaning

## Latest donor reading at planning time

### RTK

Current reading:

- latest upstream `rtk` is a command-output proxy/filtering system
- its strongest ideas are:
  - smart filtering
  - grouping
  - truncation
  - deduplication
  - token-savings tracking
- those ideas are relevant in spirit, but not as a direct transplant into Hermes

Current local truth:

- the local `rtk_sidecar` already owns:
  - mode/config resolution
  - budget shaping
  - simple savings telemetry
- live Hermes wiring already exists in `hermes-agent-latest`
- the actual `hermes-final` runtime path is the place where wiring truth must be checked before any uplift is attempted

Current hypothesis:

- there is likely value in:
  - correct runtime wiring
  - a tiny plain-text pre-normalization step before persistence/truncation
- there is **not** likely value in:
  - command-specific filtering
  - a new DSL
  - a second RTK hidden inside Hermes

## Core question

What is the thinnest RTK-sidecar shape that improves real token/context discipline without damaging tool-result correctness or creating a second maintenance-heavy filter subsystem?

## Acceptable answers

- wiring-only correction
- wiring plus tiny measurable uplift
- explicit no-op / defer

## Why this phase is worth doing

- there is a real runtime/source drift question, not just an abstract design argument
- the present sidecar is thin enough that a small 80/20 improvement may exist
- the wrong move here would be cheap to start but expensive to maintain later, so the stop conditions matter as much as the plan
