# Phase 25 Implementation Contract

## Objective

Refresh the broader deployed-live quality baseline on top of the settled Phase 24 truth,
without turning the rerun into stealth corrective implementation.

## System doctrine this phase must preserve

- Brainstack remains the owner of durable personal-memory truth.
- Phase 22 boundary decisions remain the default baseline.
- Phase 24 correctness fixes remain the default reading unless broader live evidence
  directly falsifies them.
- This phase is primarily about refreshed product truth, not new capability growth.
- The project guardrails remain active and must be used explicitly:
  - donor-first
  - modularity / upstream updateability
  - truth-first
  - fail-closed on the owned axis
  - no benchmaxing
  - no overengineering

## Workstream A: Refreshed broader scenario matrix

- Define a post-Phase-24 scenario set across:
  - ordinary conversation quality
  - stateful continuity after reset
  - proactive continuity carry-through
  - long-range relation recall
  - larger knowledge-body use
  - packet/token usefulness where it meaningfully informs the read

Required artifact:

- one refreshed scenario matrix with explicit mapping to the former residual families and the broader product targets

## Workstream B: Real deployed-live execution

- Execute the scenario matrix on the real deployed-live path.
- Record outcomes by quality class, not only binary pass/fail.

Protected rule:

- do not let narrow proof confidence replace a real broader rerun

## Workstream C: Comparison and inspection

- Compare the refreshed read against the Phase 23 baseline.
- For ambiguous or weak outcomes, inspect:
  - packet composition
  - persisted state
  - provider / harness health

Protected rule:

- do not reopen Phase 24 seams without evidence that broader live behavior actually falsifies them

## Workstream D: Refreshed residual map

- Convert the rerun into a clean new baseline reading:
  - improved
  - unchanged but acceptable
  - reopened residual
  - provider / harness issue
  - future capability phase

Protected rule:

- no vague “maybe still weak” category
- every important miss must be classified concretely

## Workstream E: Bounded action if forced

- Only perform a change if:
  - the evidence points to a measurement seam
  - the change is thin
  - the change is needed for truthful interpretation of the refreshed baseline

Protected rule:

- do not hide another corrective phase inside the rerun

## Protected boundaries

### Anti-overengineering boundary

- no new large subsystem
- no broad host rewrite
- no opportunistic architecture expansion because the rerun found a weaker conversational patch

### Boundary-stability / donor-risk boundary

- Phase 22’s ownership/coexistence decisions remain in force unless the rerun directly falsifies them
- Phase 24’s seam fixes remain in force unless the rerun directly falsifies them

### Truth boundary

- read product behavior from live evidence first
- separate memory-supply failure from answer-synthesis weakness
- separate provider variance from product regression

## Minimum evidence required before calling Phase 25 done

- one refreshed broader deployed-live scenario matrix
- one refreshed broader live result set
- explicit comparison against the Phase 23 baseline
- supporting packet/state inspection for important ambiguous or failing results
- one honest new baseline verdict:
  - materially improved
  - roughly unchanged but acceptable
  - or still carrying a justified residual
