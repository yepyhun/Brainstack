# Phase 25 Plan

## Goal

Refresh the broader deployed-live quality baseline on top of the settled Phase 24 truth, so
the project has an honest post-fix product reading instead of relying on the older 9/10 Phase 23 baseline.

## Scope

### In

- rerunning a broader deployed-live scenario matrix on the real path after Phase 24
- comparing the new result against the prior Phase 23 baseline
- explicit re-check of the former residual families:
  - cross-principal profile bleed
  - proactive continuity carry-through
- packet/state inspection where the refreshed read is ambiguous or contradictory
- thin harness correction only if the measurement path itself is proven misleading
- explicit verdict on whether another corrective follow-up is justified

### Out

- new memory capability work
- SHIBA-style Tier-2 uplift
- reopening Phase 22 coexistence by default
- reopening Phase 24 seam fixes by default
- broad host cleanup
- benchmark-shaped expansion

## Validation method

This phase should behave like a truthful rerun, not a repair sprint.

The order is:

1. refresh the scenario matrix for post-Phase-24 reading
2. run the broader deployed-live matrix on the real path
3. compare against the Phase 23 baseline
4. inspect packet/state truth where the refreshed read is unclear
5. classify the new baseline honestly
6. only then decide whether any further follow-up is justified

## Decision rules

1. If the refreshed read materially improves and the prior residuals stay closed, do not invent new work just to keep momentum.
2. If one of the old residual families reappears, verify whether the failure is causal evidence against the Phase 24 seam reading before reopening corrective work.
3. If the matrix degrades for provider or harness reasons, classify that cleanly instead of treating it as automatic Brainstack regression.
4. If a new blocker would require broad implementation to fix, stop at classification rather than sneaking that build work into Phase 25.

## Hypothesis -> Falsification -> Repair decision

1. Post-Phase-24 baseline-improvement hypothesis.
   - Hypothesis:
     - the broader deployed-live baseline after Phase 24 is stronger than the Phase 23 `9 / 10` read, or at minimum cleaner in the former residual classes.
   - Falsification:
     - show that the broader rerun does not improve those classes, or that one of them reopens in normal live use.
   - Repair decision:
     - if falsified, classify whether this is a true product blocker, a narrow residual, or a measurement/provider issue

2. Correctness-holds-at-scale hypothesis.
   - Hypothesis:
     - the Phase 24 seam fixes hold outside their narrow proofs and remain stable under broader live conversation shapes.
   - Falsification:
     - show bleed or carry-through failures recurring in the broader matrix.
   - Repair decision:
     - if falsified, reopen only the proven seam, not the whole architecture

3. Measurement-still-truthful hypothesis.
   - Hypothesis:
     - the broader rerun path still reads the deployed product honestly enough to guide the next move.
   - Falsification:
     - show that provider, harness, or environment seams are distorting the read.
   - Repair decision:
     - if falsified, repair the measurement seam first before drawing product conclusions

## Stop conditions before any patch

No product code change is allowed until all of these are true:

- the broader post-Phase-24 matrix has run at least once on the real deployed path
- the former residual families have explicit rerun evidence
- any ambiguous result has supporting packet/state inspection where needed
- any proposed change is clearly classified as:
  - harness fix
  - bounded product fix
  - or future phase candidate

## Validation ladder

1. Post-Phase-24 scenario matrix refresh
2. Real deployed-live execution
3. Baseline comparison against Phase 23
4. Packet/state inspection for ambiguous or failing outcomes
5. Residual classification
6. Thin correction only if measurement truth demands it
7. Final refreshed baseline verdict

## Compatibility / donor-risk checkpoints

1. Boundary stability
   - do not casually reopen the Phase 22 coexistence decision during the rerun

2. Correctness stability
   - do not assume the Phase 24 seams failed just because one conversational answer is weak

3. Validation realism
   - prefer ordinary user-like conversations and resets over benchmark-shaped prompts

4. Maintenance safety
   - do not add new systems or persistent heuristics inside a rerun phase

## Execution steps

1. Workstream A: refreshed scenario matrix.
   - Reuse the broader product targets from Phase 23.
   - Make sure the two former residual families are represented explicitly.
   - Keep the matrix broad enough to be meaningful, but still small enough to rerun honestly.

2. Workstream B: real-path execution.
   - Run the refreshed matrix on the actual deployed-live path.
   - Record outcome classes, not just pass/fail:
     - strong pass
     - acceptable
     - ambiguous
     - clear fail

3. Workstream C: targeted inspection.
   - For weak or contradictory outcomes, inspect:
     - packet composition
     - persisted state
     - provider / harness health
   - Separate memory-supply failure from answer-synthesis weakness.

4. Workstream D: baseline comparison.
   - Compare the new matrix against Phase 23:
     - what improved
     - what stayed roughly the same
     - what reopened
     - what is still acceptable residual

5. Workstream E: bounded action only if forced.
   - If a thin measurement fix is clearly needed, do it.
   - Otherwise close the phase with an honest refreshed baseline reading.

## Regression net

1. Ownership regression net.
   - no personal-memory shadow owner reopens during the broader rerun

2. Boundary regression net.
   - `session_search` stays bounded as transcript forensics, not second durable memory

3. Correctness regression net.
   - principal bleed canary stays clean
   - carry-through negative control stays clean

4. Scope regression net.
   - the phase stays a baseline refresh phase unless evidence clearly forces a thin measurement fix

## Required working style

- Prefer honest comparison over optimism.
- Prefer broader truth over narrow green slices.
- Prefer bounded inspection over reactive code changes.
- If the system is now stronger, say so cleanly.
- If a residual still exists, classify it exactly instead of hiding it in vague language.
