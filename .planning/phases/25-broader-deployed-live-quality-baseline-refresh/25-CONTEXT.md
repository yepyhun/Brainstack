# Phase 25 Context

## Why this phase exists

Phase 23 established the broader deployed-live quality baseline, but it also surfaced
two concrete residuals:

- principal-scoped durable profile bleed
- proactive continuity carry-through weakness

Phase 24 then closed both at their proven seams:

- durable storage / scoped lookup for principal-safe profile truth
- continuation-salience / synthesis guidance for selective carry-through

That means the next correct move is not another corrective micro-phase.

The next correct move is to refresh the broader deployed-live baseline on top of the
new Phase 24 truth.

## What is already known

### Settled prior truths

- Phase 22 boundary still stands:
  - Brainstack owns durable personal memory
  - `session_search` may coexist as transcript forensics / browsing
  - `cronjob` remains native automation outside the owned axis
- Phase 24 closed the two named post-Phase-23 residuals at deeper seams, not with
  surface patching

### What is still unknown

- whether the wider live quality picture has materially improved after the Phase 24 fixes
- whether the former residual classes stay closed outside their focused proof shapes
- whether any new residual appears when the broader scenario family is rerun on the real
  deployed path

## Core doctrine for this phase

This is a baseline-refresh validation phase.

It is not a stealth feature phase.

It should answer:

- what is the new honest broader deployed-live baseline after Phase 24?
- did the narrower correctness fixes improve the broader product picture?
- is there any remaining blocker strong enough to justify another focused follow-up?

## Project guardrails with refreshed emphasis

- donor-first
  - do not react to every weak answer by inventing a new Brainstack subsystem
- modularity / upstream updateability
  - prefer thin truth-reading and bounded harness hygiene over reactive expansion
- truth-first
  - do not call the system healthier than the broader live read actually shows
- fail-closed on the owned axis
  - do not reopen shadow personal-memory paths during validation
- no benchmaxing
  - this is a live product baseline refresh, not a benchmark campaign
- no overengineering
  - do not hide another build phase inside a rerun unless the evidence is unusually clear

## Product targets this phase must read against

- coherent continuous conversation
- stateful continuity after reset
- proactive stateful continuity
- long-range accurate recall and relation-tracking
- larger knowledge-body use
- packet / token usefulness where it materially informs the reading

## Ideal outcome

The ideal outcome is:

- a refreshed broader live baseline
- explicit comparison against the old Phase 23 reading
- clear confirmation that the Phase 24 fixes stayed narrow and improved the product picture
- or a sharply classified remaining residual, if one still exists

## What this phase is not

- not a reopen of Phase 22 boundary work
- not a reopen of Phase 24 correctness work by default
- not a SHIBA-style uplift phase
- not a broad host cleanup
- not a benchmark campaign

## End-state invariants

1. Boundary invariant.
   - Phase 22’s coexistence decision remains the default baseline.

2. Correctness invariant.
   - Phase 24’s two seams remain treated as closed unless the broader live rerun directly
     falsifies that reading.

3. Product-truth invariant.
   - the new baseline must be read from real deployed-live behavior, not from narrow proof
     slices alone.

4. No-feature-creep invariant.
   - no new capability work is silently pulled into this phase unless the measurement path
     itself is broken.

5. Honest comparison invariant.
   - the Phase 25 result must state clearly whether broader live quality is:
     - materially improved
     - roughly unchanged
     - or still carrying one or more justified residuals
