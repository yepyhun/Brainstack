# Phase 1 Context

## Name
Brainstack Composite Provider Foundation

## Objective
Design the first valid Hermes-compatible architecture for a modular second-brain stack based on:

- Hindsight
- Graphiti
- MemPalace

without turning the system into another tangled multi-kernel experiment.

## Constraints
- Hermes latest only allows one external provider
- built-in prompt memory can be disabled, but the built-in memory tool may need a small compatibility patch
- user wants modularity and one-click updates for internal layers
- user wants token savings and actual agent smartening, not just memory storage
- user treats `agent smartening`, `token savings`, and `graph / temporal truth` as equally top priority
- user treats large corpus handling as near-top priority, not as a later nice-to-have
- user wants full replacement of built-in Hermes memory and built-in user-profile behavior
- user does not want planning questions framed in internal implementation jargon
- user wants a separate stable shelf for personal identity, preferences, and shared work continuity
- user wants a broad first-version scope rather than a deliberately narrowed pilot
- user wants a max-savings default control philosophy
- user wants RTK and My-Brain-Is-Full-Crew represented as real early architecture concerns
- user wants conflicts surfaced and asked about rather than silently auto-resolved
- user wants provenance visible by default, but not spammy; stronger provenance should appear when confidence is lower or stakes are higher
- user wants the system to preserve old states and temporal change rather than destructively replacing prior knowledge
- user wants deep corpus integration ambition rather than a shallow search-only corpus layer
- user wants preferences learned strongly and applied automatically
- user wants a highly automatic, low-devops operating model with contradiction surfacing
- user wants the agent's character to feel like a real agent/personality first, with study-companion value as a secondary mode

## Phase Output
This phase must produce:

1. a stable provider boundary
2. ownership boundaries for each layer
3. extension policy
4. concrete rules for early sidecars and shell integrations
5. a refined implementation direction based on user answers
6. a user-facing explanation of the remaining architecture tradeoffs in plain language
