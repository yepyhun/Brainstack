# Phase 20.3 Implementation Contract

## Purpose

Turn the Phase `20.2` forensic conclusions into one clean structural design contract for the next retrieval architecture step.

This phase is the handoff from:

- local overlay / forensic reasoning

to:

- structural retrieval routing design

## Required Workstreams

### Workstream A. Executive routing contract

Define the `L1` routing contract:

- what inputs the router sees
- which retrieval modes it may invoke
- which fallback guarantees it must preserve
- what metadata later proof/eval must surface

Hard requirement:

- the contract must be fail-open by design

Allowed routing mechanisms must also be explicit.

Allowed:

- one bounded single-LLM routing hint as the preferred language-agnostic mechanism
- language-agnostic structural query signals such as:
  - comparison structure
  - arithmetic / quantity structure
  - explicit list / multi-item structure
  - date-like or ordering-shape structure

Disallowed:

- multilingual keyword / alias tables as the primary routing mechanism
- classify-first then second LLM call just to route again
- lexical routing logic that quietly recreates the old decomposition gate under a new name

### Workstream B. Fact retrieval mode

Define the baseline relevance-first path for ordinary fact lookup.

Hard requirements:

- this remains the default/simple path
- it is decomposition-free by default
- it does not inherit the old decomposition gate

### Workstream C. Temporal retrieval mode

Define the dedicated path for:

- ordering
- before/after
- elapsed-time questions

Hard requirements:

- timestamp-aware retrieval must be first-class
- timestamp rendering must be part of the mode, not a prompt afterthought
- this path may bias toward temporal evidence, but generic fallback must remain

### Workstream D. Aggregate retrieval mode

Define the dedicated path for:

- totals
- sums
- multi-event rollups
- broader bounded recall over multiple relevant events

Hard requirements:

- this path may widen recall beyond top-K
- widening must remain bounded
- it must not degrade into “dump more rows and hope”

The bound form must be explicit during execution.

Acceptable bound dimensions include:

- maximum session count
- maximum row count beyond standard top-K
- bounded time-window
- bounded token budget for aggregate evidence

The chosen bound must be justified outside the benchmark itself.

### Workstream E. Decomposition boundary reset

Explicitly retire the old decomposition gate/path from inheritance.

Hard requirements:

- Phase `20.3` must not silently carry forward the old decomposition triggers
- if decomposition returns later, it must return as a new, narrow helper design
- decomposition may not be the universal prefetch path again

### Workstream F. Proof design for later execution

Define the evaluation shape that later execution must satisfy.

Minimum proof categories:

- fact retrieval improvements
- temporal retrieval improvements
- aggregate retrieval improvements
- fail-open safety when routing bias is wrong or weak

### Workstream G. Route-specific aggregate rendering

Close the presentation gap for aggregate retrieval.

Hard requirements:

- aggregate route output must not reuse the generic fact/snippet presentation unchanged
- aggregate evidence must be rendered as explicitly separate recalled items
- the rendering must help multi-item rollup questions without turning into unbounded dump-more-rows behavior
- fact and temporal rendering must remain intact unless an explicit route-specific reason requires change

### Workstream H. Retrieval-vs-answer proof split

Close the proof-method gap for structural routing.

Hard requirements:

- bounded proof output must record whether the correct evidence entered the prompt
- bounded proof output must record whether the LLM answered correctly from that evidence
- the proof report must classify misses so Brainstack is not blamed for pure answer-model reasoning failures
- this split must be added to the real-path proof tooling, not only to planning prose

## Protected Boundaries

### Architecture boundary

- `20.3` may not reopen the donor-first architecture
- `20.3` may not demote `Kuzu` or `Chroma`
- `20.3` may not turn the shell back into an all-local memory brain

### Heuristic boundary

- no multilingual regex routing tables as the main intent mechanism
- no alias-sprawl to simulate structural routing
- no revival of the old decomposition gate through renamed heuristics

### Donor boundary

- no new shell-side graph reconciliation logic as part of routing
- no permanent normalization of the current shell-side supersession heuristic as “the right design”
- donor-owned logic stays donor-owned

### Runtime boundary

- routing must be fail-open
- specialized mode selection may bias retrieval, not eliminate generic fallback
- proof must include a wrong-intent / weak-intent safety story

## Minimum Proof Required Before Calling Phase 20.3 Done

- the new structural next phase is defined as one coherent architecture phase, not a cluster of leftover tuning shards
- fact / temporal / aggregate retrieval modes are explicit and distinguishable
- the current decomposition gate is explicitly retired from inheritance
- fail-open routing is explicit and testable
- aggregate rendering is route-specific and explicitly separate from generic fact presentation
- later proof reports can distinguish retrieval correctness from answer correctness
- the resulting design can be executed later without reopening the `20.2` forensic debate
