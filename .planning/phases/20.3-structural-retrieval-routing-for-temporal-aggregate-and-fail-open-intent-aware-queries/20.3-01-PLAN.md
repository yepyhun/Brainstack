# Phase 20.3 Plan

## Goal

Define the structural retrieval-routing design that should replace further local overlay tuning after Phase `20.2`, with explicit fail-open paths for:

- straightforward fact lookup
- temporal / time-difference / ordering queries
- aggregate / exhaustive-recall queries

## Scope

### In

- structural routing design for `L1` executive retrieval
- explicit separation of retrieval modes by query class
- fail-open routing behavior and fallback rules
- route-specific rendering/packing where retrieval mode would otherwise be lost in presentation
- proof reporting that separates retrieval correctness from LLM answer correctness
- clean handling of decomposition as an optional helper, not a universal path
- carry-forward rules for donor-first boundaries
- proof shape for validating the new routing before implementation is called done

### Out

- fresh benchmark-only tuning
- new quote / `digit` / wording calibration
- reopening `L1/L2/L3` ownership
- replacing `Kuzu` or `Chroma`
- another shell-side donor-logic growth wave

## Planning Assumption

The main planning assumption is now:

The residual `10 / 15` plateau is not best attacked by more local ranking or wording tweaks.
It should be attacked by routing the query into the right retrieval mode while preserving a generic fallback.

That assumption is justified by:

- the closed `20.2` forensic matrix
- the measured dataset pressure toward temporal / multi-session questions
- the clean-slate rule against inheriting the old decomposition gate

## Execution Steps

1. Define the routing contract at the executive-retrieval boundary.
   The contract must state what the router is allowed to decide and what it must leave intact.

2. Define the minimal query classes for the first structural routing iteration.
   Minimum set:
   - `fact`
   - `temporal`
   - `aggregate`
   Optional later helper:
   - `multi_entity`

3. Define each retrieval mode precisely.
   - `fact`: relevance-first top-K / fused retrieval
   - `temporal`: timestamp-ordered retrieval and rendering
   - `aggregate`: widened / exhaustive bounded recall across relevant events

4. Define fail-open behavior.
   The specialized mode must bias retrieval, not hard-exclude the generic relevance path.

5. Define decomposition’s new status.
   - not inherited from the old gate
   - not the default path
   - only a later helper for truly multi-entity cases

6. Define the boundaries with donor-owned logic.
   - no shell-side graph reconciliation expansion
   - no new multilingual regex routing heuristics
   - no benchmark-shaped ad hoc ranking overlays as the main mechanism

7. Define the proof ladder for later execution.
   It must be attributable by retrieval class:
   - exact fact
   - temporal
   - aggregate
   - fail-open fallback safety

8. Complete the rendering half of structural routing.
   - aggregate routing must not stop at wider recall
   - aggregate evidence must be presented in an enumerated, aggregation-friendly form

9. Complete the proof-method split.
   - the bounded real-path proof must distinguish retrieval correctness from answer correctness
   - Brainstack must not be over-penalized for pure model reasoning failures

## Implementation Rules

- do not continue Phase `20.2` logic by accretion
- do not let “intent-aware” collapse into a hard classifier
- do not let “aggregate” mean “just fetch more noisy rows”
- do not let “temporal” mean “same rows, louder prompt”
- do not inherit the current decomposition gate or its trigger logic
- treat the benchmark as diagnostic evidence, not as the sole design source

## Output Shape Required From Phase 20.3

This phase must produce:

1. a structural retrieval context document
2. an implementation contract with explicit workstreams
3. a clear statement of:
   - routing classes
   - fail-open rules
   - decomposition boundaries
   - proof requirements for the later execution phase

## Exit Gate

Phase `20.3` is planned correctly only if:

- the next work is framed as one structural retrieval phase, not more local tuning shards
- fact / temporal / aggregate retrieval are explicitly separated
- fail-open routing is explicit, not implied
- aggregate routing has route-specific rendering instead of generic snippet reuse
- later proof can distinguish retrieval correctness from answer correctness
- the old decomposition gate is explicitly excluded from inheritance
- donor-first boundaries remain intact
- the later execution phase would be able to prove gains by retrieval class rather than only by one blended score
