## Result

Phase `20.3` is execution-complete in the source repo.

Quality standard for interpreting this phase:

- no component in Brainstack should be described as merely \"not buggy\" or \"good enough\"
- each component must be classified as one of:
  - correct for the current phase contract
  - explicitly pending, with the gap named and owned
  - wrong and fixed immediately
- the current routing gate is therefore recorded as a **calibrated iteration decision** for `20.3`, not as a vague acceptable compromise
- its known bound is explicit:
  - it suppresses trivial question-mark spam
  - it still admits longer temporal / aggregate questions through language-agnostic complexity thresholds
  - any later live false-positive / false-negative pattern must be treated as a named follow-up fix, not as tolerated fuzziness

What is now proven locally:

- structural routing can live in the executive layer without pushing new donor logic into the shell
- the old decomposition gate is no longer the default path for simple fact retrieval
- temporal and aggregate modes are now real code paths, not louder prompt variants
- specialized routes fail open instead of hard-zeroing memory recall when the route guess is thin
- the route gate is no longer broad enough to call the routing LLM for ordinary question-mark-only prompts
- temporal ordering is rendered with explicit date labels in the prompt; that part was already correct and remained intact through the hardening pass

What this phase does **not** yet claim:

- that the new routing has already moved the real-path LongMemEval score materially
- that routing intent quality is solved for all live multilingual cases
- that decomposition has a justified return path yet

## Explicit Pending Gaps

These are not \"nice to have\" notes. They are named pending gaps after `20.3` execute:

- route-specific aggregate rendering is still missing
  - retrieval now distinguishes `aggregate`, but rendering still mostly uses the generic snippet/bullet treatment
  - this is the strongest remaining implementation gap before the next benchmark proof because aggregate retrieval gains can still be lost in presentation to the LLM
- retrieval correctness and LLM answer correctness are not yet measured separately in the proof method
  - the next bounded proof needs to distinguish:
    - whether the correct evidence was present in the prompt
    - whether the LLM answered correctly from that evidence
  - without that split, Brainstack can be over-punished for model reasoning errors
- fail-open widening, if added later, must remain explicitly bounded
  - acceptable direction: a slightly wider bounded fallback for structurally-shaped misses
  - unacceptable direction: dumping extra rows and hoping the model figures it out
- exact-value supersession remains a separate pending debt
  - it stays recorded as temporary tech debt on the current shell/state path
  - it is not the next `20.3` cut even though it remains important
