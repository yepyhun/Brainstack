# Phase 20.3: Structural Retrieval Routing For Temporal, Aggregate, And Fail-Open Intent-Aware Queries - Context

**Gathered:** 2026-04-12
**Status:** Ready for planning

## Why this phase exists

Phase `20.2` closed its full low-cost forensic loop.

That loop proved:

- the best current exact-fact baseline is `10 / 15`
- disabling the first `20.2` decomposition path was the primary regression recovery
- quote, `digit`, and stronger grounding wording are all load-bearing in the current local stack
- the local overlay stack is therefore compensatory rather than fundamentally strong
- the net gain over the post-`20.1` rerun is still only `+1`

This means the next work is no longer another `20.2`-style tuning shard.
The remaining gap is now structural retrieval work.

## What this phase is

This phase is the first clean structural retrieval-routing phase after the `20.2` forensic closeout.

It must introduce separate retrieval routes for at least:

1. straightforward fact lookup
2. temporal / time-difference / ordering questions
3. aggregate / exhaustive-recall questions

And it must do that inside a fail-open executive retrieval design, not as a hard exclusive classifier.

## What this phase is not

This phase is **not**:

- another `20.2` overlay-tuning pass
- another bonus-calibration phase
- another prompt-wording phase
- a donor-first rollback
- a rebranding of the old decomposition gate
- a multilingual regex / alias-table expansion phase

## Architectural constraints carried forward

- Brainstack remains a thin shell around donor-aligned retrieval and storage layers
- `Kuzu` remains the active graph backend target
- `Chroma` remains the active semantic corpus backend target
- the shell does not become a new local memory brain
- the current shell-side exact-value auto-supersession remains temporary technical debt, not a model for more shell-side donor logic
- the benchmark remains diagnostic, not the product goal; the target is a better everyday kernel

## Clean-slate routing rule

The next retrieval design must not inherit the current decomposition gate/path as-is.

The current best baseline explicitly keeps decomposition disabled, so Phase `20.3` must start from a clean routing design:

- simple fact lookup -> relevance retrieval path, decomposition-free by default
- temporal queries -> timestamp-ordered retrieval path
- aggregate queries -> widened / exhaustive recall path
- decomposition may return only as a narrowly justified helper for true multi-entity queries, under a new design

## Fail-open routing rule

Intent-aware routing is required, but hard exclusive classification is not acceptable.

The routing shape must therefore be:

- intent-aware
- biased toward the right path for the query class
- but still fail-open, with generic relevance fallback preserved

This is important because a wrong intent guess must not zero out recall.

## Measured pressure carried forward

The current LongMemEval mix is now measured rather than guessed:

- `multi-session`: `26.6%`
- `temporal-reasoning`: `26.6%`
- `knowledge-update`: `15.6%`

This is strong evidence that one universal ranking path is too coarse for the real workload.

## Success condition

Phase `20.3` succeeds if it produces a structural retrieval design and execution contract that:

- stops treating the remaining gap as local overlay tuning debt
- cleanly separates fact / temporal / aggregate retrieval modes
- preserves donor-first shell boundaries
- keeps routing fail-open
- and gives Phase `20.4` a coherent execution target instead of another forensic loop
