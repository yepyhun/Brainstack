# Phase 20.19 Plan

## Goal

Enable the already-built donor-aligned graph/corpus capability in deployed Bestie, then produce a fresh isolated live re-baseline on the real intended runtime stack before any further retrieval or answer-quality tuning.

## Scope

### In

- deployed Bestie runtime dependency/config truth for graph/corpus enablement
- bounded runtime enablement for the existing Brainstack graph/corpus path
- fail-closed verification that `semantic` and `graph` are truly active after enablement
- one post-enablement isolated live re-baseline
- one explicit before/after reading against the degraded-stack `3 / 5` baseline

### Out

- broader aggregate trigger or planner expansion
- new scorer/reranker work
- new temporal micro-tuning on the degraded stack
- broad packaging cleanup unrelated to runtime enablement
- benchmark-only proof loops

## Execution Steps

1. Workstream A: runtime truth audit.
   - Prove exactly why deployed Bestie is still `semantic: degraded` and `graph: degraded`.
   - Localize the live blocker precisely:
     - missing runtime dependency
     - disabled config
     - bad import path
     - bad startup wiring
     - or another bounded enablement seam
   - Record the current fail-closed truth before changing anything.

2. Workstream B: bounded runtime enablement.
   - Enable the existing donor-aligned graph/corpus backends in deployed Bestie.
   - Reuse the already-built Brainstack capability; do not invent a new runtime path.
   - Keep this bounded:
     - only the minimal dependency/config/wiring needed for truthful live activation
     - no hidden new capability work

3. Workstream C: activation proof.
   - Produce one explicit runtime proof that the channels are no longer degraded:
     - `semantic` active
     - `graph` active
   - If one backend can be enabled and the other cannot, say so explicitly rather than pretending partial activation is full success.

4. Workstream D: post-enablement isolated live re-baseline.
   - Re-run the isolated deployed-live reset eval after enablement.
   - Compare against the latest degraded-stack baseline:
     - `3 / 5`
   - Read specifically:
     - whether `aggregate_total` now bridges to the native capability
     - whether `temporal_order` still fails once the intended runtime stack is live

5. Workstream E: honest next-phase decision.
   - If live enablement materially improves the remaining misses, say so directly.
   - If one residual class survives even on the enabled stack, identify it cleanly.
   - If full enablement cannot be completed honestly in this phase, stop with the blocker documented instead of forcing a fake green result.

## Implementation Rules

- do not fake “active” channel status with stubs or forced status strings
- do not widen keyword retrieval further just to compensate for missing backend enablement
- do not reopen planner/scorer debates before the enabled-stack re-baseline exists
- keep donor-first boundaries explicit:
  - runtime enablement should consume the existing Brainstack graph/corpus path
  - not replace it with a parallel ad hoc implementation

## Output Shape Required From Phase 20.19

- one explicit live runtime enablement diagnosis
- one bounded live enablement change set or one honest blocker report
- one activation proof for graph/corpus channel status
- one post-enablement isolated live eval artifact
- one explicit next-thread recommendation

## Exit Gate

- deployed Bestie runtime truth is no longer ambiguous about graph/corpus status
- at least one honest post-enablement isolated live re-baseline exists
- the next-thread recommendation is based on enabled-stack product truth rather than degraded-stack momentum
