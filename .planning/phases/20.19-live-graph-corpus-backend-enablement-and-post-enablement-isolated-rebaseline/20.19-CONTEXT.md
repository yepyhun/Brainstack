# Phase 20.19 Context

## Why this phase exists

Phase `20.18` established the current ceiling of the deployed Bestie runtime honestly:

- isolated live eval improved to `3 / 5`
- `coffee_order` and `gift_context` now pass on the active degraded stack
- `temporal_order` still fails as a low-overlap same-principal ordering miss
- `aggregate_total` still fails even though `20.15` already proved a bounded native aggregate path in oracle conditions

The key runtime truth is now explicit:

- deployed Bestie still reports `semantic: degraded`
- deployed Bestie still reports `graph: degraded`
- therefore the current live product truth is still dominated by the `keyword + temporal` stack

That means further keyword-stack micro-tuning is now lower-value than enabling the already-built graph/corpus capability in the live runtime.

## Inputs from prior phases

- `20.11–20.15` built the structural graph/corpus capability:
  - temporal restoration
  - typed extraction
  - bounded native Kuzu aggregate path
- `20.17` fixed principal-scoped live recall isolation and noun-`order` route disambiguation
- `20.18` proved that bounded low-overlap live surfacing can still improve the degraded runtime, but also showed diminishing returns

## Current product truth

- Bestie live runtime is not yet exercising the graph/corpus capability built in source
- the remaining live misses are now entangled with runtime enablement:
  - `aggregate_total` is the clearest example
  - `temporal_order` may or may not remain once semantic/graph channels are truly active

## What this phase is not

- not broader aggregate planner expansion
- not more degraded-stack keyword/temporal micro-tuning
- not a benchmark-shaped recovery phase
- not a hidden principal-scope redesign
- not a broad packaging/refactor campaign beyond what is needed to enable the runtime stack honestly

## Honest target

Enable donor-aligned graph/corpus backends in deployed Bestie, verify that the runtime actually reports those channels as active, then re-run the isolated live baseline so the next phase decision is based on the real intended stack rather than on the degraded fallback stack.
