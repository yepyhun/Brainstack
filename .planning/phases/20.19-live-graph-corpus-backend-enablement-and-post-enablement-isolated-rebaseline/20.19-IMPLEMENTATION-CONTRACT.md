# Phase 20.19 Implementation Contract

## Objective

Move deployed Bestie from the current degraded `keyword + temporal` runtime into the already-built donor-aligned graph/corpus runtime, then measure live truth again on that actual stack.

## Workstream A: Runtime Enablement Diagnosis

- Start from the live truth already recorded after `20.18`:
  - deployed Bestie reports `semantic: degraded`
  - deployed Bestie reports `graph: degraded`
- The phase must identify the concrete enablement blocker, not just restate the degraded status.
- Acceptable blocker classes:
  - missing dependency in the deployed `.venv`
  - disabled or absent runtime config
  - import/wiring failure
  - startup path not initializing the backend
  - another bounded integration seam

## Workstream B: Bounded Enablement

- Enable the existing graph/corpus capability in deployed Bestie.
- Preferred bounded directions:
  - install or restore the required runtime dependency
  - wire the existing config path correctly
  - remove only the minimal fallback condition that is keeping the backend disabled
- Do not broaden phase scope into unrelated packaging work.

## Workstream C: Fail-Closed Activation Proof

- After the enablement work, prove channel truth explicitly.
- The phase must not call itself done if the runtime still silently degrades.
- If only one backend activates:
  - record that partial truth explicitly
  - do not round it up to “enabled”

## Workstream D: Post-Enablement Live Re-Baseline

- Re-run the isolated deployed-live reset eval.
- Compare against the degraded-stack baseline:
  - `3 / 5`
- Read the result through product questions, not benchmark vanity:
  - does `aggregate_total` now bridge to native graph truth?
  - does `temporal_order` still fail when semantic/graph are actually available?

## Workstream E: Honest Stop Boundary

- If runtime enablement proves larger or riskier than expected, the phase may close with:
  - diagnosed blocker
  - bounded partial progress
  - explicit deferral to `20.20`
- Do not fake a clean enablement story if the live runtime is still degraded.

## Protected Boundaries

### Product Boundary

- coherent conversation and correct live recall outrank adding more fallback heuristics
- live deployed truth outranks oracle-only capability proofs

### Architectural Boundary

- donor-first runtime enablement only
- no parallel replacement stack
- no hidden new retrieval layer
- preserve modularity and upstream updateability

### Anti-Benchmaxing Boundary

- no tuning against named benchmark IDs
- no “proof-only” local hacks that do not survive in deployed Bestie
- no claiming success from degraded fallback answers when the intended runtime stack is still off

## Minimum Evidence Required Before Calling Phase 20.19 Done

- one explicit diagnosis of why graph/corpus were degraded in deployed Bestie
- one bounded enablement patch set or one explicit blocker report
- one runtime proof of actual post-change channel status
- one isolated post-enablement live eval artifact
- one explicit next-phase recommendation
