# Phase 63 Plan

## purpose

Replace mandatory remote `structured_understanding` in the ordinary Brainstack task/operating hot path with a local deterministic typed-understanding architecture that stays no-heuristic, multimodal-capable, and Brainstack-owned.

## core question

How should Brainstack determine ordinary task, operating, and current-work read/write behavior from local typed substrates instead of a mandatory remote semantic-classification seam, without rolling back to cue lists or phrase farms?

## primary goals

- define the local typed-understanding architecture for ordinary Brainstack read and capture behavior
- remove mandatory remote `structured_understanding` authority from the ordinary task/operating hot path
- preserve the no-heuristic direction from `v1.0.17` and the availability containment from Phase `61.1`
- preserve the current-state authority gains from Phase `62` without folding live-state truth back into transcript residue
- keep the design multimodal-capable so the next kernel shape is not text-only by construction

## explicit scope model

### Brainstack-owned scope

- ordinary-turn task/operating route determination inside Brainstack
- ordinary-turn task/operating capture eligibility inside Brainstack
- the local typed substrate that these decisions consume
- demotion of remote `structured_understanding` from mandatory hot-path authority to bounded auxiliary/off-path use, if any such use remains justified
- proof that the ordinary kernel can still behave safely when the remote seam is unavailable

### host-boundary scope

- generic Hermes runtime, cron, gateway, provider, or tool UX cleanup is not this phase
- no new host seam is justified unless execution proves Brainstack cannot obtain the needed typed substrate from existing owned inputs
- remote LLM/provider tuning is not the architecture answer

### multimodal rule

- the new understanding layer must not assume text-only inputs as its core contract
- the design must be able to consume normalized turn artifacts, typed durable records, tool outcomes, and future non-text event payloads without devolving into cue-list routing

## anti-goals

- do not roll back to `v1.0.16`
- do not restore locale-specific cue tables, phrase farms, or disguised keyword routing
- do not solve this by making `structured_understanding` faster and leaving it mandatory
- do not move the same semantic classification dependency into a hidden host helper
- do not treat `session_search`, transcript dumping, or tool fallback as a memory-kernel fix
- do not hardcode one user, one Discord thread, one phase name, or one Hungarian wording family

## accepted evidence

- Phase `61.1` proved the remote seam needed containment, but also left an explicit residual:
  - `control_plane` still performs one structured query-understanding call for ordinary packet analysis
- Phase `62` fixed current live-system authority for scheduler state, but did not remove remote-understanding authority from the task/operating hot path
- live evidence before and during Phase `61.1` showed that remote structured understanding failures were real runtime facts, not theoretical edge cases
- the immutable project rule remains:
  - no code-level heuristic routing as the recovery path

## validated current reading

- the kernel is now safer when remote understanding fails
- the kernel is not yet architecturally correct, because ordinary route/capture authority still depends too much on a remote semantic seam
- a full rollback would violate the no-heuristic rule
- another round of timeout/fallback/provider patching would only deepen the wrong dependency

## architecture direction to freeze before execution

### 1. no-rollback decision

- the recovery target is not `v1.0.16`
- Phase `63` must not reintroduce cue-driven routing as the price of availability

### 2. hot-path authority decision

- ordinary task/operating read and capture behavior must become locally decidable from typed Brainstack-owned substrates
- remote `structured_understanding` must no longer be the mandatory authority for those ordinary decisions

### 3. typed-substrate decision

- execution must define the exact normalized substrates that local understanding may use
- preferred inputs are:
  - typed durable records
  - normalized turn structure
  - explicit record classes
  - provenance
  - timestamps
  - current operating-context state
- execution must justify any additional substrate it introduces

### 4. ambiguity decision

- unresolved ambiguity must degrade to bounded safe behavior:
  - fact-only read path
  - no-capture write path
- execution must not replace remote ambiguity with local guessing

### 5. multimodal decision

- the design must remain valid for non-text-bearing turns and mixed-modality turn envelopes
- if a proposed local parser only works because the input is natural-language text, the phase has not solved the right problem

### 6. LLM-role decision

- remote or local model-based understanding may remain only as bounded enrichment, consolidation, or backfill if explicitly justified
- the phase must state which LLM uses remain and why they are no longer hot-path authority

## workstreams

### 1. hot-path evidence freeze

- inventory every remaining mandatory `structured_understanding` call site in the ordinary kernel
- separate read-path authority, write-path authority, and optional/off-path enrichment

### 2. typed-understanding substrate design

- define the exact local inputs available for deterministic typed understanding
- define which of those inputs are already authoritative versus merely supportive

### 3. read-path replacement design

- define how ordinary task/operating/current-work lookup is decided locally
- keep current-state authority from Phase `62` separate from recap and task semantics

### 4. write/capture replacement design

- define how ordinary task/operating capture eligibility is decided locally
- preserve no-capture behavior when proof is insufficient

### 5. migration and proof design

- define how the old remote seam is demoted without creating a half-wired hybrid
- define the proof matrix for availability, correctness, and no-heuristic compliance

## deliverables

- a frozen architecture decision for local typed understanding in the Brainstack hot path
- a call-site cutover map showing which ordinary decisions stop depending on remote understanding
- a bounded migration plan for any remaining off-path LLM usage
- an execution proof plan covering ordinary reads, ordinary capture, degraded mode, and multimodal discipline

## acceptance criteria

- the execution plan makes it possible to remove mandatory remote `structured_understanding` authority from ordinary task/operating routing and capture
- the design remains compliant with the no-heuristic rule
- the design remains multimodal-capable by contract, not just by aspiration
- the design does not create a second memory owner or a shadow host classifier
- the phase states clearly what still remains off-path and why
