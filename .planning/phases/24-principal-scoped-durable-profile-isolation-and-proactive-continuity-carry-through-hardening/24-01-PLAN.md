# Phase 24 Plan

## Goal

Restore principal-scoped durable profile isolation and harden proactive continuity carry-through on the live path, using shared-seam diagnosis first so the fix does not become a ragtapasz patch on top of a deeper boundary problem.

## Scope

### In

- reproducing and localizing the cross-principal durable profile bleed
- reproducing and localizing the proactive continuity dietary carry-through miss
- packet/state inspection for both residuals
- bounded repair of the deepest proven shared seam
- thin follow-up repair for carry-through if it remains after scope isolation is fixed
- explicit decision checkpoints during execute if a real architecture fork appears

### Out

- broad SHIBA-style Tier-2 uplift
- broad scorer/reranker redesign
- reopening the Phase `22` Brainstack/native boundary by default
- broad host cleanup
- benchmark-shaped work

## Validation method

This phase should behave like a correctness-first seam diagnosis followed by bounded repair.

The order is:

1. reproduce the principal-scope bleed with persisted-state evidence
2. reproduce the carry-through miss with packet/state evidence
3. localize each failure to the deepest credible seam
4. check for a shared underlying seam
5. only then patch
6. rerun live proof on both residual families

Cheap validation gates should be used to keep the phase headache-free:

- a minimal principal-bleed canary before broader reruns
- a carry-through negative control so a fix does not simply start hallucinating constraints everywhere

## Decision rules

1. If principal isolation is broken, fix that before making extraction or carry-through logic more aggressive.
2. If carry-through failure turns out to be retrieval/packet/synthesis rather than extraction/persistence, do not “fix” Tier-2 first.
3. If both residuals share one lower seam, prefer the shared repair over two symptom patches.
4. If the evidence points to a real architecture fork during execute, bring it back to the user in simple language before locking the repair path.
5. If SHIBA-style uplift looks tempting but is not the narrowest credible fix, defer it.

## Hypothesis -> Falsification -> Repair decision

1. Principal-isolation hypothesis.
   - Hypothesis:
     - the durable profile bleed originates in a shared scope/keying/filtering seam, not in the broader Brainstack/native boundary decision
   - Falsification:
     - show that the bleed only exists because the current boundary or coexistence policy is itself wrong
   - Repair decision:
     - if not falsified, repair the deepest scope seam and keep the Phase `22` boundary intact

2. Carry-through localization hypothesis.
   - Hypothesis:
     - the dietary carry-through miss can be localized to one of:
       - extraction
       - persistence
       - retrieval
       - packet composition
       - answer synthesis
   - Falsification:
     - show that the miss is too noisy or provider-random to localize meaningfully
   - Repair decision:
     - if localized, patch the narrowest responsible seam instead of broad “continuity improvement”

3. Shared-seam hypothesis.
   - Hypothesis:
     - there may be one lower seam contributing to both the scope bleed and the carry-through miss
   - Falsification:
     - show that the two residuals are causally separate
   - Repair decision:
     - if shared, repair once at the shared seam
     - if separate, still repair in the order:
       - isolation first
       - carry-through second

4. SHIBA-deferral hypothesis.
   - Hypothesis:
     - these residuals can be fixed without broad SHIBA-style capability uplift
   - Falsification:
     - show that the narrow fix path is not credible and a deeper Tier-2 capability gap is the real blocker
   - Repair decision:
     - if falsified, stop and reframe honestly instead of sneaking SHIBA uplift into this phase

## Stop conditions before any patch

No product code change is allowed until all of these are true:

- the principal-scope bleed has been reproduced or explicitly falsified with persisted-state evidence
- the carry-through miss has been reproduced or explicitly falsified with packet/state evidence
- the likely failing layer for each residual is written down
- any proposed repair is classified as:
  - shared correctness seam
  - narrow carry-through seam
  - or future capability thread

1. Scope-bleed reproduction

7. Residual classification update

## Compatibility / donor-risk checkpoints

1. Boundary stability.
   - do not casually reopen `session_search` / `cronjob` / Brainstack-only ownership decisions during this phase

2. Scope safety.
   - do not widen durable profile extraction before principal isolation is repaired

3. Capability discipline.
   - do not turn SHIBA-style missing pieces into emergency scope unless the narrower path truly fails

4. Communication discipline.
   - during execute, only escalate to the user when there is a real fork with a meaningful tradeoff

## Execution steps

1. Workstream A: principal-scope truth pass.
   - reproduce the bleed
   - start with a minimal canary:
     - principal A gets a few strong profile/style markers
     - principal B gets none
     - verify whether B receives any A-owned durable profile truth after reset
   - inspect persisted profile rows, stable keys, scope keys, and injection path
   - determine whether the fault is in:
     - ingest / principal tagging
     - durable write keying
     - retrieval filtering
     - contract assembly
     - finalize lifecycle

2. Workstream B: proactive carry-through truth pass.
   - rerun the dietary continuity scenario
   - include one negative control:
     - a nearby scenario with no dietary constraint
     - verify the system does not start inventing or over-carrying constraints after the fix
   - inspect extraction/persistence/retrieval/packet/synthesis layers
   - classify the first real drop point

3. Workstream C: shared-seam diagnosis.
   - ask whether the two residuals connect at a lower seam
   - if yes, repair there
   - if no, keep the isolation-first order

4. Workstream D: bounded repair.
   - fix the deepest proven seam
   - keep the repair thin and donor-friendly

5. Workstream E: rerun and residual map refresh.
   - rerun the exact residual families
   - rerun the cheap canary and negative control first
   - update the reading:
     - fixed
     - improved but residual remains
     - misdiagnosed, needs reframing

## Regression net

1. Ownership regression net.
   - no reopening of personal-memory shadow owners

2. Boundary regression net.
   - no accidental rollback of the settled Phase `22` coexistence model

3. Scope regression net.
   - no broader SHIBA-style capability expansion hidden inside a correctness phase

4. Truth regression net.
   - no fix is accepted without persisted-state or packet evidence at the claimed seam
   - no carry-through fix is accepted if the negative control starts inventing missing constraints

## Required working style

- correctness first, then quality
- shared seam first, then local symptom
- simple-language decision checkpoints if a real fork appears
- no speculative “smarter memory” leap before scope hygiene is clean
