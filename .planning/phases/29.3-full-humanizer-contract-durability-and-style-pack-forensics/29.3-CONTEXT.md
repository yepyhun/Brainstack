# phase 29.3 context

## problem statement

The first 29.3 implementation attempt was rejected and reverted.

It violated a core rule:

- it introduced style-query regex routing
- it reconstructed the long-form Humanizer pack from transcript text with section parsing

That is not acceptable as the final Brainstack architecture for a kernel that is meant to stay multilingual and general.

The real problem remains:

- the compact communication contract is durable and useful
- the full Humanizer rule pack still needs a durable, queryable, user-owned representation
- the solution must not depend on language-bound heuristics

## what is already correct

- the compact active communication contract should remain compact
- ordinary turns should not carry the full style pack
- this memory is user-specific durable style truth, not generic public knowledge
- this should stay on the personal-memory axis, not escape into persona files, skill files, or transcript luck

## what was learned from the rejected attempt

### good insight that remains valid

- the system needs a two-layer model:
  - compact operational contract for ordinary turns
  - detailed contract recall for explicit questions

### wrong implementation choices that are now rejected

- regex-based style-query detection
- transcript section parsing and reconstruction as a standing runtime path
- corpus-backed long-form carrier as the default final lane for this use case

## current architectural reading

The strongest clean model now is:

1. compact lane
   - existing canonical communication slots remain the ordinary-turn carrier

2. detailed lane
   - one canonical principal-scoped `preference:style_contract` durable row on the personal-memory axis

3. explicit recall path
   - explicit Humanizer/rules questions must target that durable slot through a model-backed routing seam, not regexes

4. compatibility bridge
   - if a previously written corpus-style Humanizer artifact exists from the rejected 29.3 attempt, migrate it once into the canonical profile slot and retire it

## why this is stronger

- it keeps style truth on the same personal-memory axis as other user preferences
- it avoids creating a second quasi-document system for personal style memory
- it removes language-specific parsing from the durability path
- it lets ordinary prompt projection stay compact
- it gives explicit recall a first-class, exact durable owner

## evidence that shaped this reading

- compact contract projection already works
- generic profile search alone is not enough for explicit long-form style recall
- a long-form style row cannot simply be thrown into generic profile retrieval, or it leaks into ordinary prompts/retrieval
- the existing route-hint seam is the right place to solve explicit recall intent
- the existing Tier-2 extraction seam is the right place to solve durable style-pack ingest

## open design questions this phase must settle

1. exact canonical slot name and content rules for the detailed style contract
2. how the routing seam asks for `style_contract` mode without regexes
3. how generic profile retrieval excludes the long-form style slot on ordinary turns
4. how the one-shot compatibility migration retires the rejected corpus-backed artifact if it exists

## model level

Recommended model level: `xhigh`
