# phase 29.3 implementation contract

## phase type

Architecture correction phase.

The previous 29.3 implementation was rejected and reverted.

This phase is complete only if the replacement path is both correct and clean.

## required final architecture

The final implementation must satisfy all of these:

1. one compact ordinary-turn lane
   - existing canonical communication slots remain the source for the active prompt

2. one detailed durable lane
   - one canonical principal-scoped `preference:style_contract`

3. one explicit recall route
   - explicit style/rules questions can target the detailed slot without regexes

4. one owner only
   - transcript, corpus, persona, and skill files are not authoritative owners once the final slot exists

## minimum evidence required before calling the phase done

- structured Tier-2 output can emit the detailed style contract directly
- the canonical `preference:style_contract` row can be upserted and retrieved
- generic ordinary-turn prompt projection excludes that row
- explicit style/rules recall can target that row without regexes
- the old corpus-backed style-contract path, if present, is migrated or retired

## explicitly disallowed outcomes

- any regex-based style-query routing
- any transcript section parser in the active runtime path
- any corpus-backed authoritative detailed-style lane in the final design
- any dual-read or dual-write path for the detailed style contract
- any dead code from the reverted attempt left active or importable in the runtime path

## cleanup requirement

The phase is not done when the new path works.

It is done only when:

- the rejected heuristic path is removed
- the runtime no longer references it
- tests validate only the surviving path
- source-of-truth can be read without understanding the abandoned approach

## compatibility requirement

If the rejected 29.3 attempt already wrote durable data, the replacement phase must not simply abandon the user’s stored content.

Allowed compatibility bridge:

- a one-shot, deterministic migration from the old corpus-backed style artifact into canonical `preference:style_contract`

Disallowed compatibility bridge:

- keeping both corpus and profile as active authoritative detailed-style owners
- transcript-based re-synthesis of the style pack as a standing fallback

## ordinary-turn protection

The fix must preserve all of these:

- compact active contract
- no detailed style-pack prompt bloat
- no ordinary-query leakage of the long-form style contract
- no regression in Phase 29.1 and 29.2 behavior

## code hygiene requirement

At close, the final diff must not contain:

- abandoned helper modules
- unused imports from the rejected path
- unreachable migrations or one-off debug hooks
- secondary retrieval branches that are no longer selected

## recommended model level

`xhigh`
