# Phase 20.7 Implementation Contract

## Purpose

Fix the live retrieval-quality path after `20.6` by closing the execution-path anomaly first, then restoring fact-path and route-quality behavior on the now-activated live route system.

## Required Workstreams

### Workstream A. Execution-Path Anomaly Hardening

This workstream exists because `6a1eabeb` and `gpt4_7f6b06db` showed blank route metadata and missing `<memory-context>` blocks.

It must:

- reproduce the anomaly if possible
- identify which live path skips normal memory-packet or prompt injection
- fix that correctness bug before downstream retrieval-quality claims rely on those rows
- if the anomaly does not reproduce cleanly, capture the exact trigger surface and residual uncertainty explicitly

It must not:

- flatten the anomaly into generic retrieval failure
- speculate about the cause without a concrete traced path

### Workstream B. Fact-Path Parity Restoration

This workstream exists because live fact behavior is still far below the earlier stronger decomp-off baseline.

It must:

- document the exact older stronger baseline path again before comparing
- inspect fact-path candidate ordering, packet shape, rendering, and budgets
- identify what drifted inside the fact path itself
- recover fact-path behavior as far as the evidence supports

It must not:

- let fact-path parity fall behind route-quality work again
- explain away the gap as “structural routing is still immature”

### Workstream C. Route-Quality Recovery

This workstream exists because route activation now happens, but the activated routes still perform badly.

It must explicitly separate three sublayers:

1. route selection
   - is the wrong mode being chosen?
2. candidate quality
   - is the chosen route retrieving irrelevant evidence?
3. evidence packing
   - are the right candidates found but packed badly?

It must:

- name which of these sublayers is dominant for each reproduced route-quality failure class
- keep route widening bounded and language-agnostic
- preserve donor-first boundaries while improving route behavior

It must not:

- collapse all route failures into one undifferentiated “route quality” label
- reintroduce the old decomposition gate or multilingual keyword routing

### Workstream D. Profile / Identity Retrieval Miss Recovery

This workstream exists because `5d3d2817` exposed a distinct fact-query miss class.

It must:

- audit identity-like / profile-like fact questions as their own retrieval class
- inspect semantic retrieval, ranking, and packing on those questions
- state explicitly whether the class reproduced in `20.7`

It must not:

- bury the class inside generic fact-path failure language
- speculate if the class does not reproduce

### Workstream E. Bounded Grounding Leak Carry-Through

This workstream exists because `d682f1a2` showed a clean retrieval-correct / answer-wrong case.

It must:

- stay bounded to proven retrieval-correct / answer-wrong rows
- verify whether the leak still reproduces after the retrieval fixes
- report explicitly whether the class remains actionable or evaporates once upstream retrieval quality improves

It must not:

- turn into a broad prompt / wording retuning phase
- be used to excuse unresolved retrieval failures

### Workstream F. Comparable Live Reruns

This workstream exists to prove whether `20.7` improved the live stack.

It must run:

- answer-only
- split-proof

It must compare against `20.6`, not against older pre-repair runs only.

## Protected Boundaries

### Proof Boundary

`20.7` must not reopen proof-calibration as its main mission.

Proof output is used here as a measurement surface, not as the product of the phase.

### Restoration Boundary

This phase is allowed to repair live execution and retrieval quality.

It is not allowed to drift into abstract new architecture design.

### Heuristic Boundary

This phase must not fall back to broad overlay tuning, new decomposition heuristics, or multilingual keyword tables.

Any bounded retrieval tweak must be justified by the named live failure class it fixes.

### Donor Boundary

This phase must stay donor-first.

It may repair selection, ordering, packing, and execution-path correctness in the Brainstack shell.

It must not replace donor retrieval with new shell-owned pseudo-retrievers.

## Minimum Proof Required Before Calling Phase 20.7 Done

Before Phase `20.7` can be called done, the summary must show:

- whether the execution-path anomaly was fixed or remains unresolved
- whether fact-path parity materially improved or remains sharply explained
- whether route-quality failures were primarily:
  - selection
  - candidate quality
  - packing
- whether the profile / identity miss class improved
- whether the grounding leak class still reproduces
- comparable `20.7` answer-only and split reruns

If live restoration still fails, the summary must say so directly.
