# Phase 20.7 Plan

## Goal

Restore live retrieval quality after `20.6` by fixing the execution-path correctness bug first, then recovering fact-path parity and route-quality on the now-real live route path.

## Scope

### In

- execution-path anomaly hardening for missing memory-injection cases
- fact-path candidate quality / parity restoration
- route-quality recovery split into:
  - route selection
  - candidate quality
  - evidence packing
- profile / identity retrieval miss recovery
- bounded grounding leak carry-through audit
- comparable answer-only and split reruns at the end

### Out

- new proof-calibration theory
- new decomposition experiments
- broad bonus retuning
- broad evidence-priority wording loops
- donor replacement or donor bypass logic

## Planning Assumption

`20.6` already proved that the live path is wired enough to expose the real quality problem.

`20.7` should therefore spend its effort on execution correctness and retrieval quality, not on reopening measurement theory.

## Execution Steps

1. Harden the execution-path anomaly first.
   - reproduce the blank-route / missing-memory-block cases such as `6a1eabeb` and `gpt4_7f6b06db`
   - identify where the normal memory packet / prompt injection path is being skipped
   - fix that correctness bug before treating those rows as normal retrieval failures

2. Restore fact-path parity next.
   - compare the current live fact path against the older stronger decomp-off baseline
   - inspect candidate ordering, packet shape, rendering, and budgets on the fact path itself
   - identify what changed inside the fact path that kept parity far below the old baseline

3. Recover route quality with explicit layer separation.
   - route selection:
     - is the mode choice wrong?
   - candidate quality:
     - is the selected route retrieving the wrong evidence?
   - evidence packing:
     - are the right candidates found but packed badly?
   - the phase must name which of those layers is dominant for the remaining route failures

4. Recover the profile / identity retrieval miss class explicitly.
   - treat `5d3d2817`-type misses as their own bounded class
   - inspect semantic retrieval, ranking, and packing for identity-like fact questions
   - if the class does not reproduce, say so explicitly instead of speculating

5. Re-check bounded grounding leak carry-through.
   - inspect only proven retrieval-correct / answer-wrong rows such as `d682f1a2`
   - verify whether the leak still reproduces after the upstream retrieval fixes
   - if it no longer reproduces, say so explicitly

6. Run comparable reruns.
   - answer-only
   - split-proof
   - the phase output must distinguish real live restoration from merely cleaner diagnosis

## Implementation Rules

- Do not let the execution-path anomaly remain implicit under generic retrieval-failure language.
- Do not let fact-path parity slide behind route-quality work again.
- Do not let “route quality” remain a single mushy category; split it into selection, candidate quality, and packing.
- Do not reopen proof-calibration as the main work of this phase.
- Do not let profile / identity misses disappear into general fact-retrieval noise.
- Do not let the bounded grounding leak audit grow into a broad prompt-tuning phase.
- Do not declare improvement unless the final reruns actually improve live behavior.

## Output Shape Required From Phase 20.7

The final summary must explicitly say:

- whether the missing-memory-block execution path was fixed
- whether fact-path parity materially improved
- whether route-quality failure was mainly:
  - selection
  - candidate quality
  - packing
- whether profile / identity retrieval improved
- whether the known grounding leak still reproduces
- whether live restoration improved or remained insufficient

## Exit Gate

Phase `20.7` is done only when:

- the execution-path anomaly has been explicitly reproduced and either fixed or sharply explained
- fact-path parity has been explicitly audited and either improved or sharply explained
- route-quality has been split into selection / candidate / packing findings
- profile / identity retrieval has been explicitly audited as a named class
- bounded grounding leak has been re-checked after the retrieval work
- final answer-only and split reruns are comparable to `20.6`
- if restoration remains weak, the summary says so directly instead of describing the phase as a partial win
