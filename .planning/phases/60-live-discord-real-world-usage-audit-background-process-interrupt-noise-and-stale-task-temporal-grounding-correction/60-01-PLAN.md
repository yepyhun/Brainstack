# Phase 60 Plan

## purpose

Use the user's current live Discord usage only as a case study to isolate and correct Brainstack-owned universal defects.

This phase is not a Hermes cleanup phase. It is not a "make this one messy thread look nicer" phase. It is a bounded audit-and-correction phase for Brainstack behavior that could also fail for another user on a different but structurally similar path.

## core question

Which parts of the current live-thread failures reveal a Brainstack-universal defect in extraction, retrieval, grounding, authority, or durable-write behavior, and which parts are merely host/runtime noise that must only be documented as boundary conditions?

## primary goals

1. freeze the live case-study evidence before it drifts out of view
2. separate Brainstack-owned universal defects from host-native Hermes behavior with an explicit ownership model
3. correct only the Brainstack-owned universal defects that the case study exposed
4. leave a defensible proof story that would still make sense if the next reproducer came from another user, another reminder, another tool flow, or another language

## explicit scope model

### case-study input

The user's current live Discord sessions are evidence, not the design target.

Accepted use of the case study:
- reveal what kinds of content Brainstack is over-trusting
- reveal how stale task/reminder text becomes eligible ordinary-turn evidence
- reveal when assistant-authored narration is promoted into durable state
- reveal whether reflection-driven writes are contaminating durable memory

Rejected use of the case study:
- coding for the exact `11:15` reminder sentence
- coding for one specific Discord thread shape
- coding for one specific Hungarian phrasing
- papering over one ugly response without proving the universal defect is removed

### Brainstack-owned scope

In scope only if the evidence shows Brainstack ownership:
- stale temporal resurfacing of expired reminder/task text through transcript, continuity, profile, decision, or other Brainstack-managed evidence lanes
- assistant-authored claim extraction contamination
- speculative implementation-claim promotion into durable Brainstack state
- reflection-triggered durable memory/skill mutations insofar as Brainstack extraction trusts or persists them too aggressively
- Brainstack-side evidence ranking, gating, provenance, trust, temporal validity, and durable-write criteria
- Brainstack-owned installer or wizard changes only where they are required to preserve Brainstack's universal contract across installs

### host-boundary scope

Allowed only as boundary analysis or as the narrowest necessary seam correction:
- gateway interruption text
- generic execute_code/background-process churn
- provider fallback noise
- generic Hermes cron scheduler design
- Discord adapter surface behavior

These may be documented, and their effect on Brainstack may be proven, but they are not Phase 60 design targets unless a narrow seam change is strictly required to preserve Brainstack correctness.

## anti-goals

- no generic Hermes bugfix pass hidden inside a Brainstack phase
- no broad cron rewrite
- no "fix the current symptom at all costs" approach
- no user-specific rescue logic
- no locale-specific heuristics
- no prompt-only sebtapasz that leaves bad evidence eligible underneath
- no broad host surgery just because the thread looked chaotic
- no deleting historical state wholesale to hide the problem
- no claiming Brainstack ownership for issues that are only host/runtime ugliness

## phase hypotheses to test

1. Brainstack is currently allowing expired reminder/task text to survive as ordinary-turn evidence without enough temporal re-validation.
2. Brainstack is currently over-trusting assistant-authored self-explanation, self-diagnosis, and speculative implementation narrative when forming durable continuity/decision state.
3. Reflection-driven write paths are currently treated too much like ordinary user-authored truth and can contaminate durable state.
4. Some live cron/pulse drift may matter to Phase 60 only when it is re-imported or reinforced by Brainstack durable extraction, not merely because the scheduler itself changed.

## workstreams

### 1. evidence freeze

- preserve the exact live evidence chain
- annotate each preserved artifact with:
  - what happened
  - what is proven
  - what is only reported
  - what owner it belongs to

### 2. ownership audit

- classify each finding as:
  - Brainstack-owned
  - host/runtime-owned
  - wizard seam
  - mixed
- require a written reason for each classification

### 3. universal-defect extraction

- turn the live case study into generalized Brainstack defect statements
- reject any statement that only describes the user's exact thread instead of the universal failure mode

### 4. Brainstack correction design

- design minimal changes only for Brainstack-owned universal defects
- prefer durable-write, evidence-selection, provenance, or temporal-grounding fixes over surface wording patches
- before code changes, freeze one compact architecture decision note for:
  - temporal-grounding mechanism
  - provenance/trust mechanism
  - reflection-path handling
- that note must explicitly use the current Brainstack architecture rather than inventing a blank-slate model
- it must account for the fact that:
  - `task_memory` and `operating_truth` already exist as real Brainstack lanes
  - session-end message ingest already rejects non-user role content
  - Tier-2 batch extraction still reads merged `User: ... / Assistant: ...` turn transcript rows

### 5. boundary validation

- document host/runtime behaviors that remain out of scope
- only propose a host/wizard seam correction if Brainstack cannot remain correct across installs without it

### 6. proof design

- define proof that demonstrates universal correction, not just one cleaned-up replay
- include a "different user / different reminder / different wording" reasoning test in the acceptance story

## accepted findings to carry forward

- stale `11:15` reminder resurfacing was not backed by active scheduler truth and therefore exposed a real temporal-grounding leak
- assistant-authored self-diagnosis and speculative operational narrative were promoted into durable Brainstack continuity/decision state
- reflection prompts are causing real writes and therefore cannot be treated as inert background behavior
- some live cron/pulse state drift is now real scheduler state and must only remain in scope where Brainstack extraction/authority makes it durable, trusted, or wrongly resurfaced

## acceptance criteria

- the phase text makes it impossible to confuse "user case study" with "design target"
- Brainstack-owned universal issues are explicitly named
- out-of-scope Hermes issues are explicitly fenced off
- anticélok forbid sebtapasz and host-wide cleanup drift
- the later execute step will only be allowed to touch host seams when a Brainstack-universal correctness requirement proves it necessary

## canonical principle reference

This phase must remain consistent with:
- donor-first
- fail-closed upstream compatibility
- truth-first / no "good enough"
- zero heuristic sprawl
- modularity / upstream updateability
