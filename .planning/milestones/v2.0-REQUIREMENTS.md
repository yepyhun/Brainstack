# Requirements

## Product Requirements

### R1. Real Second Brain
The system must support large knowledge corpora including:

- books
- textbooks
- notes
- occasional legal or reference documents

### R2. Cross-Session Continuity
The agent must remember:

- who the user is
- how the user prefers to work
- what the agent and user were working on together
- recent relevant project continuity

### R3. Temporal Truth
The system must represent:

- current vs previous facts
- changed-over-time facts
- superseded preferences or states
- durable relationships between memories

### R4. Agent Smartening
The memory system must make Hermes feel noticeably smarter through:

- continuity
- relevant recall
- reduced re-explanation by the user
- stronger project and personal context retention

### R5. Token Efficiency
The system must aggressively minimize token usage through:

- just-in-time recall
- selective context assembly
- collapse/expand document views
- avoiding unnecessary tool calls where memory suffices

### R6. Update Safety
The three main layers must be independently replaceable or updatable without breaking the Hermes-facing provider contract.

### R7. Hermes Compatibility
The stack must fit Hermes latest without a deep core fork.

Acceptable:

- one external composite provider
- small compatibility patches
- host-level config changes

Not acceptable:

- large long-lived divergence from Hermes core
- multiple competing external provider hacks

## Architecture Requirements

### A1. One Host-Facing Provider
Hermes must interact with only one external provider:

- `brainstack`

### A2. Clear Ownership
Each internal layer must have a clear source of truth:

- `Hindsight` owns recency and continuity
- `Graphiti` owns graph and temporal truth
- `MemPalace` owns corpus/document memory
- `brainstack orchestrator` owns packing and retrieval decisions

### A3. No Dual-Write Ambiguity
No fact class may have two competing canonical stores without an explicit synchronization rule.

### A4. Extensible Slots
Future additions must plug in through explicit slots:

- ingest hooks
- retrieval rerank hooks
- packing hooks
- consolidation hooks
- audit/debug hooks

## Sidecar Policy

### S1. RTK
Use as token-efficiency infra and command/output filtering sidecar, not as memory.

### S2. NeuronFS
Use as donor for selected ideas only, not as a live memory core.

### S3. My-Brain-Is-Full-Crew
Treat as an upper UX / orchestration shell if useful, not as the memory substrate.

### S4. Mira
Treat as control-plane inspiration for working-memory orchestration and context packing, not as a drop-in memory core.
